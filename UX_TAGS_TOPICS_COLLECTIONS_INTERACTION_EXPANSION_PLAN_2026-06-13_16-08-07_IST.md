# UX Tags, Topics, and Collections Interaction Expansion Plan

Created: 2026-06-13 16:08:07 IST
Project: AI Brain Phase 2 redesign
Scope: Web and Android high-fidelity UX interaction design for Tags, Included topics, and Collections on item detail and discovery surfaces.
Status: Plan ready for review.

## 1. Executive Decision

Tags, Included topics, and Collections should not behave like three visual variations of the same pill. They should become three distinct rediscovery paths:

| Surface | Meaning | Generated by | User can edit? | Primary click/tap result |
|---|---|---|---|---|
| Tags | Organization labels for retrieval and personal taxonomy | AI plus user | Yes | Open all items with that tag |
| Included topics | AI-detected concepts covered inside the item | AI only | Not directly | Open a topic exploration view |
| Collections | Saved groupings or projects | User, later AI-assisted | Yes | Open the collection detail page |

The item detail page remains the origin point, but the deeper interaction happens in focused destination views:

```text
Item detail -> Tag -> Tagged items
Item detail -> Included topic -> Topic exploration
Item detail -> Collection -> Collection detail
```

## 2. Current UX Gap

The current item-detail right rail shows:

- Tags as static-looking pills plus `Add tag`.
- Included topics as AI-detected pills that look clickable but do not clearly explain what clicking does.
- Collections as an add/manage affordance without showing the working model of an existing collection destination.

The result is ambiguous:

- Users cannot tell whether a pill is a filter, an editor, a link, or metadata.
- Topics and tags visually compete despite having different ownership rules.
- Collections feel like a form field, not a real library structure.

## 3. Product Model

### 3.1 Tags

Tags are lightweight labels attached to an item. They are good for personal language, quick filtering, and user-controlled organization.

Rules:

- AI can create initial tags during enrichment.
- Users can add manual tags.
- Users can remove a tag from an item.
- Users can promote, rename, or merge tags from the tag management surface.
- Auto-generated and manually added tags can coexist.
- If the same tag was generated by AI and later added by the user, show it as one tag with `AI + You` provenance in the management detail, not as duplicates.

Recommended visual treatment:

- Manual tag: normal tag pill.
- AI-generated tag: tag pill with small `AI` dot or subtle sparkle marker.
- User-confirmed AI tag: same pill, but provenance reads `AI suggested, kept by you`.

### 3.2 Included Topics

Included topics are concepts the AI believes are covered by the item. They are for semantic exploration, not manual filing.

Rules:

- Topics are generated from readable item content, transcript, PDF text, or notes.
- Topics do not have an `Add topic` action on item detail.
- Topics should be clickable.
- Users can convert a topic into a tag from the topic view if they want to use it as personal organization.
- Low-confidence topics should not be shown as normal pills. If needed, show them in a `Possible topics` collapsed section later, not in this first pass.

Recommended visual treatment:

- Keep the `AI DETECTED` label.
- Topic pills should include a subtle semantic/explore affordance, such as hover label `Explore topic`.
- Topics should look related to tags, but not identical: slightly lighter border, AI label nearby, no add button.

### 3.3 Collections

Collections are saved groupings with their own destination and management actions.

Rules:

- Collections can be manually created.
- A saved item can belong to zero, one, or many collections.
- A collection pill is clickable and opens the collection detail page.
- `Add to collection` opens a picker, not a free text-only input.
- The picker supports creating a new collection inline.
- Later, AI-generated auto-collections can be introduced, but this plan focuses on manual collection interaction while leaving room for `Auto collection` badges.

Recommended visual treatment:

- Existing collection pills should show a folder icon plus collection name.
- Empty state: `Not in a collection yet`.
- Add action: `Add to collection`.
- Collection pill hover/menu: `Open collection`, `Remove from this item`.

## 4. Primary Interaction Rules

### 4.1 Clicking A Tag

Web:

- Single click on a tag pill opens `Library` filtered to that tag.
- URL pattern: `/library?tag=design`.
- The Library header changes to a tag-filter context:
  - Title: `Tag: design`
  - Count: `12 items`
  - Active filter chip: `design`
  - Actions: `Ask this tag`, `Manage tag`, `Clear filter`

Android:

- Tapping a tag pill opens the Library filtered to that tag.
- Header uses compact context:
  - `Tag`
  - `design`
  - item count
- Back returns to the previous item detail.
- Tag management is not hidden behind long-press; use a visible `More` menu in the tag-filter screen.

Secondary actions:

- `Ask this tag`: starts Ask scoped to items with the tag.
- `Manage tag`: opens tag editor or settings tag detail.
- `Remove from item`: available from the item-detail tag edit mode, not from the normal click target.

### 4.2 Tapping Or Clicking An Included Topic

Web:

- Single click opens a topic exploration view.
- URL pattern: `/topics/attention-mechanisms` or, if a dedicated topic route is not ready, `/library?topic=attention-mechanisms`.
- Preferred first pass: a dedicated Topic view because topics are not the same as filters.

Topic view contains:

- Topic name.
- AI explanation: `Why Brain detected this topic`.
- Items covering the topic.
- Evidence snippets from the current item and related items.
- Related topics.
- Actions:
  - `Ask this topic`
  - `Create tag from topic`
  - `Add matching items to collection`

Android:

- Tapping a topic opens a Topic detail screen.
- Top app bar title: topic name.
- Sections:
  - `Covered in this item`
  - `More from your library`
  - `Ask this topic`
  - `Related topics`
- Back returns to the item detail.

Secondary actions:

- `Create tag from topic`: creates a user-managed tag with the same normalized name.
- `Add to collection`: opens collection picker using all selected topic-matched items.

### 4.3 Clicking A Collection

Web:

- Single click on a collection pill opens the Collection detail page.
- URL pattern: `/collections/product-research`.
- Collection detail contains:
  - Collection title and description.
  - Item count.
  - Sort controls.
  - Search within collection.
  - Items in the collection.
  - Actions: `Ask collection`, `Add items`, `Rename`, `Remove current item`.

Android:

- Tapping a collection pill opens Collection detail.
- Bottom navigation remains visible if this is a normal library destination.
- Back returns to the item detail.
- `Add item` and `Remove from collection` live in the collection overflow menu or item row menu.

Empty collection state:

```text
No items in this collection yet.
Add from Library or use Add to collection on an item.
```

## 5. Item Detail UX Changes

### 5.1 Tags Card

Current pattern:

```text
Tags
[design] [research] [+ Add tag]
```

Proposed pattern:

```text
Tags
[AI design] [research] [+ Add tag]
```

Required behavior:

- Every tag pill is clickable.
- Hover/focus copy: `View items tagged design`.
- `Add tag` opens an inline editor on web and a bottom sheet on Android.
- Edit mode is explicit. Do not put remove actions on the primary pill click.

Web edit mode:

- `Add tag` opens a compact popover or right-rail inline editor.
- Search existing tags first.
- Suggested tags appear under `AI suggestions`.
- User can type a new tag and press Enter.
- Existing attached tags show a remove `x` only inside edit mode.

Android edit mode:

- `Add tag` opens a bottom sheet.
- Search input at top.
- Existing attached tags with remove buttons.
- AI suggestions below.
- Primary action: `Done`.

### 5.2 Included Topics Card

Current pattern:

```text
Included topics        AI DETECTED
Auto-detected from this item
[Attention mechanisms] [Positional encoding] ...
```

Proposed behavior:

- Keep no add action.
- Every topic pill opens Topic exploration.
- Add hover/focus hint:
  - Web: `Explore topic`
  - Android: visible sheet/screen title after tap
- Add optional microcopy:
  - `Tap a topic to see related items`

Avoid:

- Do not let users manually add topics here.
- Do not show remove controls by default.
- Do not merge topics into tags visually.

### 5.3 Collections Card

Current pattern should become:

```text
Collections
[Folder Product research] [Folder Reading list]
+ Add to collection
```

If no collections:

```text
Collections
Not in a collection yet.
+ Add to collection
```

Required behavior:

- Existing collection pill opens Collection detail.
- `Add to collection` opens a picker.
- Picker allows:
  - search existing collections,
  - select one or many collections,
  - create new collection inline,
  - apply/cancel.
- After apply, newly attached collections appear as pills.

Web picker:

- Modal or side drawer.
- Show current item title.
- Show existing collections with checkbox state.
- Show `Create new collection` input.

Android picker:

- Bottom sheet.
- Search field.
- Collection list with checkboxes.
- `Create new collection` row.
- Sticky `Apply` action.

## 6. Destination Views

### 6.1 Tag Filter View

Purpose: show every item with a tag and make tag-level work possible.

Web layout:

- Library list with contextual header.
- Header:
  - `Tag: design`
  - item count,
  - tag provenance summary: `AI suggested on 8 items · manually added on 4`,
  - actions: `Ask this tag`, `Manage tag`, `Clear`.
- Items list uses the existing Library item card pattern.

Android layout:

- Library screen with contextual top bar.
- Filter chip pinned under top bar.
- Items list below.
- Overflow menu: `Ask this tag`, `Manage tag`, `Clear filter`.

### 6.2 Topic Exploration View

Purpose: help the user understand what the AI found across their saved material.

Web layout:

- Topic header with `AI detected topic`.
- Short explanation:
  - `Brain found this topic in 6 saved items.`
- Current item evidence card:
  - title,
  - snippet,
  - `Open item`.
- Related items list.
- Related topics pills.
- Actions:
  - `Ask this topic`
  - `Create tag from topic`
  - `Add items to collection`

Android layout:

- Topic detail screen.
- Sections stacked vertically:
  - `Why this topic`
  - `In this item`
  - `More from your library`
  - `Related topics`
- Sticky bottom action can be `Ask this topic` when there are enough items.

### 6.3 Collection Detail View

Purpose: make Collections feel like real saved spaces, not just labels.

Web layout:

- Header:
  - Collection name,
  - description,
  - item count,
  - type badge: `Manual` or later `AI collection`,
  - actions: `Ask collection`, `Add items`, `Rename`.
- Main:
  - item list,
  - sort by recently added / saved date / title,
  - search within collection.

Android layout:

- Collection detail screen.
- Top app bar with title and overflow menu.
- Item list.
- Floating or sticky `Add items` action if needed.
- Overflow includes `Rename`, `Remove collection`, `Ask collection`.

## 7. Ask Integration

These interactions should all have Ask scope support:

| Source | Ask scope label | Behavior |
|---|---|---|
| Tag | `Tag: design` | Retrieve only items with that tag |
| Topic | `Topic: attention mechanisms` | Retrieve items where AI detected that topic |
| Collection | `Collection: Product research` | Retrieve only items in the collection |

Ask entry points:

- Tag filter header: `Ask this tag`.
- Topic view header: `Ask this topic`.
- Collection detail header: `Ask collection`.

Ask should show the scope near the composer so users understand the boundary.

## 8. Search And Library Integration

Library filters should support:

- `tag`
- `topic`
- `collection`
- `source type`
- `quality`
- date

Filter chip behavior:

- Multiple filters can stack.
- Each filter chip can be removed independently.
- `Clear all` appears when more than one filter is active.
- Deep links preserve filters.

Suggested URL patterns:

```text
/library?tag=design
/library?topic=attention-mechanisms
/collections/product-research
/ask?scope=tag&tag=design
/ask?scope=topic&topic=attention-mechanisms
/ask?scope=collection&collection=product-research
```

## 9. AI Generation Rules

### 9.1 Auto Tags

AI-generated tags should be:

- 3 to 8 per readable item.
- short,
- reusable across items,
- normalized to lowercase slug form internally,
- user-facing label can be title case or natural case depending on existing style.

Auto tags should favor organization:

- `retrieval-quality`
- `android-offline`
- `local-first`
- `capture-repair`

### 9.2 Included Topics

Included topics should be:

- more descriptive than tags,
- content-specific,
- generated from actual readable text,
- supported by evidence snippets,
- shown only when confidence is high enough.

Topics can be longer:

- `Attention mechanisms`
- `Source quality evaluation`
- `Android offline reading`
- `Capture repair flow`

### 9.3 Collections

Collections remain user-controlled in this pass.

Future-compatible rule:

- AI-suggested auto-collections may appear later with an `AI collection` badge.
- Auto-collections should not silently mix with user-created collections without a visible badge.

## 10. Accessibility Requirements

Shared:

- Every pill must be keyboard/focus reachable.
- Clickable pills need accessible labels:
  - `View items tagged design`
  - `Explore topic attention mechanisms`
  - `Open collection Product research`
- Do not rely on color alone to distinguish AI-generated tags, topics, and collections.
- Focus ring must be visible on all pills and add actions.
- Remove controls must have explicit labels:
  - `Remove design from this item`
  - `Remove item from Product research`

Web:

- Tag/topic/collection pills should be real buttons or links.
- `Enter` opens the primary destination.
- `Esc` closes tag editor, collection picker, and popovers.
- Modal focus is trapped while open.

Android:

- All taps must have visible feedback.
- Bottom sheets must have a clear close affordance.
- No required action depends only on long-press.

## 11. Edge Cases

| Case | UX behavior |
|---|---|
| Item has no tags | Show `No tags yet` plus `Add tag` |
| Item has only AI tags | Show them with AI provenance marker; allow user to keep/remove |
| Item has no included topics | Show `Topics appear after Brain reads the content` |
| Metadata-only item | Topics card says topics need readable text; tags can still be manually added |
| No collections | Show empty state and `Add to collection` |
| Offline on Android | Viewing cached tag/topic/collection lists is allowed if data exists; editing shows `Reconnect to update organization` |
| Tag renamed | Deep links resolve old slug or show redirected label |
| Topic merged by AI later | Keep old topic link stable and show merged topic name |
| Collection deleted | Item detail removes the collection pill and shows toast |

## 14. Visual Design Guidance

Use the current design system, but make the role of each object visible:

- Tags: neutral/tag styling, optional small AI marker.
- Included topics: lighter semantic pills with `AI DETECTED` context.
- Collections: folder icon and magenta/accent treatment.

Avoid:

- Making all pills identical.
- Hidden-only hover controls for core actions.
- Long-press as the only management path.
- Putting Add controls inside Included topics.
- Mixing Tags, Topics, and Collections into one combined card.

## 15. Acceptance Criteria

Web:

- Clicking a tag from item detail opens a tag-filtered Library view.
- `Add tag` opens a working editor state.
- Clicking an included topic opens a topic exploration view.
- Topic view has `Ask this topic` and `Create tag from topic`.
- Existing collection pills are visible and clickable.
- `Add to collection` opens a collection picker.
- Collection detail shows item list and `Ask collection`.

Android:

- Tapping a tag opens a tag-filtered Library view.
- `Add tag` opens a bottom sheet.
- Tapping a topic opens Topic detail.
- Tapping a collection opens Collection detail.
- `Add to collection` opens a bottom sheet.
- All exits/back paths are visible and predictable.

Shared:

- Tags, topics, and collections have distinct meanings and visual treatments.
- AI-generated versus user-managed organization is understandable.
- Ask scope is clear when asking across a tag, topic, or collection.
- Empty, offline, and metadata-only states are represented.

## 16. No-Go Gates

Do not execute the Magic Patterns update if:

- Tags, topics, and collections are collapsed into one generic metadata pattern.
- Topic pills are clickable but have no destination.
- Collection pills do not open a collection detail page.
- Tags cannot be manually added.
- Auto-generated tags are indistinguishable from manual tags.
- Android relies on long-press for core actions.
- The implementation changes Focus mode details by adding Tags, Topics, or Collections into Focus mode. These organization interactions belong on normal item detail and discovery views.

## 17. Recommended First Pass

The first high-fidelity pass should implement the visible design review version of:

1. Clickable tag pills.
2. Add tag editor.
3. Clickable topic pills with Topic detail.
4. Existing collection pills.
5. Add to collection picker.
6. Collection detail.
7. Ask scope from tag, topic, and collection.

This is enough to prove the mental model without overbuilding full settings-level taxonomy management.
