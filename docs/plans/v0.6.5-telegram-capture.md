## Architecture decision (locked from self-critique on 2026-06-02)

**B1 — Webhook handler as a new Next.js API route inside `brain.service`.** Not long polling. Not a separate systemd unit. Not a Cloudflare Worker.

Rationale:
- `brain.arunp.in` already has a public TLS endpoint via the Cloudflare named tunnel — webhook works for free.
- Webhook needs no daemon to manage, monitor, or restart.
- Telegram's built-in retry protocol handles brain.service downtime (up to 24 h).
- Auth is solved by Telegram's `secret_token` header + sender user-ID allowlist.
- Same process as the rest of brain — no new deploy unit, no new dependency injection seam.

## Tasks (atomic, ordered by dependency)

### T-G — Pre-flight: bot setup with @BotFather

Manual one-shot, before any code. Cannot be automated.

1. In Telegram, message `@BotFather` → `/newbot` → name `Arun Brain` → username `arun_brain_bot` (or whatever's free).
2. Copy the bot token from BotFather's reply. **Never** commit, log, or paste into chat.
3. `/setdescription`: "Personal capture bot for AI Brain. Forward links, send notes or PDFs."
4. `/setuserpic`: optional.
5. `/setprivacy` → `Disable` (so the bot sees all messages in DMs, not just commands). Default is already DM-only by Telegram; the privacy toggle matters in groups, which we don't use.

Provisioning on Hetzner (one-shot, when T-2 is ready to deploy):
1. SSH to Hetzner. `sudo nano /etc/brain/.env`.
2. Append three vars (values from BotFather + `openssl rand -hex 32`):
   ```
   TELEGRAM_BOT_TOKEN=<bot-token-from-botfather>
   TELEGRAM_WEBHOOK_SECRET=<32-byte-hex>
   TELEGRAM_OWNER_USER_ID=943125412
   ```
3. `sudo systemctl restart brain` (after T-2 ships) so Next.js picks up the new env.
4. Register the webhook with Telegram (one-shot curl):
   ```
   curl -X POST "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook" \
     -d "url=https://brain.arunp.in/api/telegram/webhook" \
     -d "secret_token=$TELEGRAM_WEBHOOK_SECRET" \
     -d 'allowed_updates=["message"]'
   ```
   Expect: `{"ok":true,"result":true,"description":"Webhook was set"}`.
5. Verify: `curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/getWebhookInfo"` shows the URL + a recent `last_synchronization_error_date` of null/zero.

**Acceptance:** Webhook registered. Sending a test message gets a 200 from brain (logs visible via `journalctl -u brain`).

### T-1 — Telegram client wrapper

**File:** `src/lib/telegram/client.ts` *(new)*

A thin wrapper around the Telegram Bot API. Two methods used in v0.6.5:

```ts
sendMessage(chatId: number, text: string, opts?: { reply_to_message_id?: number; parse_mode?: 'Markdown' | 'HTML' }): Promise<TelegramMessage>
editMessageText(chatId: number, messageId: number, text: string, opts?: ...): Promise<void>
```

Internally uses `fetch` against `https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}`. No new npm dep. Token read from env at module load; throws if missing.

**File:** `src/lib/telegram/types.ts` *(new)* — Telegram API type definitions for `Update`, `Message`, `User`, `Document`. Hand-rolled, narrow scope, no `@types/telegraf` or similar.

**File:** `src/lib/telegram/client.test.ts` *(new)* — 4 tests stubbing `fetch`: sendMessage success, sendMessage 4xx surfaces error, editMessageText, getFile. No real network in CI.

### T-2 — Webhook route handler

**File:** `src/app/api/telegram/webhook/route.ts` *(new)*

```ts
export const runtime = "nodejs";
export const dynamic = "force-dynamic";

export async function POST(req: NextRequest) {
  // 1. Verify Telegram secret-token header
  const headerSecret = req.headers.get("x-telegram-bot-api-secret-token");
  if (headerSecret !== process.env.TELEGRAM_WEBHOOK_SECRET) {
    return new NextResponse("unauthorized", { status: 401 });
  }

  // 2. Parse Update payload (zod-validated; existing pattern)
  const update = TelegramUpdateSchema.safeParse(await req.json());
  if (!update.success) return NextResponse.json({ ok: true });  // accept-and-ignore malformed

  const msg = update.data.message;
  if (!msg) return NextResponse.json({ ok: true });

  // 3. Sender allowlist
  const ownerId = Number(process.env.TELEGRAM_OWNER_USER_ID);
  if (msg.from?.id !== ownerId) return NextResponse.json({ ok: true });

  // 4. Dispatch to capture handler (T-3..T-6)
  try {
    await handleCaptureMessage(msg);
  } catch (err) {
    logError("telegram.capture.failed", err);
    await sendMessage(msg.chat.id, `⚠️ Capture failed: ${(err as Error).message}`).catch(() => {});
  }

  // ALWAYS return 200 to Telegram so it doesn't retry indefinitely
  return NextResponse.json({ ok: true });
}
```

**File:** `src/lib/telegram/dispatch.ts` *(new)* — `handleCaptureMessage(msg)` inspects the message and routes:
- Has `entities` of type `url`? → URL capture (T-3).
- Has `document` with PDF mime? → PDF capture (T-5).
- Plain text (no entities) → Note capture (T-4).
- Has `entities` but no URL (e.g., bot-command, mention) → reply with help text.

**File:** `src/app/api/telegram/webhook/route.test.ts` *(new)* — 6 tests: secret-token mismatch → 401, missing secret → 401, valid secret + non-owner → 200 silent, valid secret + owner + URL → calls capture, valid secret + owner + text → calls note capture, malformed body → 200 silent.

**Acceptance:** Tests green. Manual smoke: POST a synthetic update to the local dev server with the right secret + owner ID + URL → see capture log; POST with wrong secret → 401.

### T-3 — URL capture path

**File:** `src/lib/telegram/capture-url.ts` *(new)*

Extracts URL from `msg.text` or `msg.entities` (Telegram annotates URLs in the message). Calls the existing `extractArticleFromUrl` helper directly (the same function `/api/capture/url` uses) — no internal HTTP round-trip. Returns the new item ID.

YouTube and Substack PDFs use the existing detection paths. Dedup hits `findItemByUrl`. The bot ack message includes:
- New: `✅ Captured: '<title>'  →  https://brain.arunp.in/items/<id>`
- Duplicate: `↩️ Already captured: '<title>'  →  https://brain.arunp.in/items/<id>`

**File:** `src/lib/telegram/capture-url.test.ts` *(new)* — 4 tests: plain URL, YouTube URL, duplicate URL, capture-error.

**Acceptance:** Tests green. Manual smoke: send a Substack URL via Telegram → item appears in `/library`; send same URL again → bot replies "Already captured".

### T-4 — Note capture path

**File:** `src/lib/telegram/capture-note.ts` *(new)*

When `msg.text` has no URL entities, treat it as a note. First-line-as-title (max 80 chars), full text as body. Calls existing `insertCaptured` with `{ source: 'note', title, body }`.

Bot ack: `✅ Saved as note  →  https://brain.arunp.in/items/<id>`.

Edge case: text under 3 chars → bot replies "🤔 That's awfully short. Did you mean to send a URL?" and creates nothing. Prevents typos becoming clutter.

**File:** `src/lib/telegram/capture-note.test.ts` — 3 tests: normal note, very-short text rejected, multi-line note title extraction.

**Acceptance:** Tests green. Manual smoke: send a note via Telegram → item appears in `/library`.

### T-5 — PDF capture path

**File:** `src/lib/telegram/capture-pdf.ts` *(new)*

When `msg.document.mime_type === 'application/pdf'`:
1. Send "📄 Capturing PDF…" ack immediately (return the message_id for later edit).
2. `getFile(document.file_id)` → returns Telegram CDN path.
3. Download bytes via `fetch('https://api.telegram.org/file/bot<token>/<file_path>')`.
4. Hand bytes to existing `extractPdfText` helper.
5. `insertCaptured` with `{ source: 'pdf', title, body, originalFilename }`.
6. `editMessageText` to update the ack: `✅ Captured: '<title>'  →  https://brain.arunp.in/items/<id>`.

Cap PDF size at 20 MB at the bot layer (Telegram's max document size for bots is 20 MB; users hit a separate limit at 50 MB but bots can only fetch 20 MB).

**File:** `src/lib/telegram/capture-pdf.test.ts` — 3 tests: small PDF success, oversize rejection, paywall-error path (mocks the PDF extractor to throw).

**Acceptance:** Tests green. Manual smoke: forward a PDF document via Telegram → "📄 Capturing PDF…" appears, then edits to "✅ Captured" within 10s.

### T-7 — Smoke test

**File:** `scripts/smoke-v0.6.5.mjs` *(new)*

Driven by env vars `BRAIN_HOST` + `TELEGRAM_WEBHOOK_SECRET`. Builds synthetic `Update` payloads and POSTs them to `/api/telegram/webhook` with the right secret-token header. Asserts:
1. URL update → item exists with that URL.
2. Note update → item exists with that title.
3. PDF update → item exists (PDF download path stubbed for the smoke; requires real network test for full proof).
4. Non-owner update → no item created.
5. Wrong secret → 401.

Runs against either local `npm run dev` or live Hetzner.

**Acceptance:** All 5 pass against Hetzner after deploy.
