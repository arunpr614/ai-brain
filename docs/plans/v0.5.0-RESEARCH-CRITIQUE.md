## 3. BLOCKERS (3)

### B-1: §4.4 proxy sketch has a token-comparison control-flow bug

**Location:** `v0.5.0-RESEARCH.md` §4.4 lines 173–215.

**Finding:**
The code sketch does:

```typescript
if (pathname.startsWith('/api/') && BEARER_ROUTES.some(p => pathname.startsWith(p))) {
  const auth = req.headers.get('authorization');
  if (auth?.startsWith('Bearer ')) {
    const provided = auth.slice(7);
    const expected = process.env.BRAIN_LAN_TOKEN ?? '';
    if (provided.length === expected.length && provided === expected) {
      return NextResponse.next();
    }
  }
}

// 4. Reject — unchanged redirect/401 logic
if (pathname.startsWith('/api/')) {
  return NextResponse.json({ error: 'unauthenticated' }, { status: 401 });
}
```

This is **correct** — the fallthrough hits the 401. I initially misread this as falling through to `.next()`. **Revising: this specific line of the sketch is OK.**

**However, a related real issue remains:** The sketch silently accepts `BRAIN_LAN_TOKEN = ''` (the `?? ''` default) as a valid token when the env var is unset. If `BRAIN_LAN_TOKEN` is missing in `.env`, then any request with `Authorization: Bearer ` (empty bearer) **and** `provided.length === expected.length` (both zero) **and** `provided === expected` (both empty string) passes the check. The plan must add an explicit guard that rejects when `BRAIN_LAN_TOKEN` is empty or < 32 chars, independently of the comparison.

**Corrected sketch:**

```typescript
const expected = process.env.BRAIN_LAN_TOKEN ?? '';
if (expected.length < 32) {
  // Refuse to accept bearer tokens until the env is configured.
  return rejectWith401(req);
}
if (provided.length === expected.length && provided === expected) {
  return NextResponse.next();
}
```

**Disposition:** The plan's T-2 (`BRAIN_LAN_TOKEN` auto-gen on first start) and T-3 (bearer-token check) must be sequenced so that the token is guaranteed to exist before the check runs. Current DAG already has this (T-2 → T-3), but the guard is implicit. Make it explicit in T-3's acceptance criterion: "reject bearer-authenticated requests when `BRAIN_LAN_TOKEN` is unset or shorter than 32 characters, with a distinct error log line".

**Severity reconsidered:** BLOCKER → **HIGH**. Downgrading because the control flow is correct; only the empty-token edge case is a real bug. Fold into plan T-3 acceptance, not a pre-plan blocker.

### B-2: §5.1 `bonjour.publish({host: 'brain.local'})` may not produce the expected hostname

**Location:** `v0.5.0-RESEARCH.md` §5.1 lines 280–296.

**Finding:**
The sketch uses `bonjour.publish({name: 'Brain', type: 'http', port: 3000, host: 'brain.local'})`. The `host` field in the `bonjour-service` API corresponds to the **SRV target** for the service advertisement — i.e., the hostname that clients should connect to for the service `_http._tcp`. It does *not* set the `.local` name that a browser resolves when the user types `brain.local` in the address bar.

For Chrome on Android to resolve `brain.local` to the Mac's LAN IP, the Mac must be advertising a **hostname record (A record over mDNS)** for `brain.local` itself — not just a service record (`Brain._http._tcp.local` → SRV → brain.local → A).

macOS's `mDNSResponder` publishes `<mac-hostname>.local` (usually the Mac's `scutil --get LocalHostName`) automatically. If the Mac's LocalHostName happens to be `brain`, then `brain.local` already resolves without any help from `bonjour-service`. If it's something else (like `Arun-MacBook-Pro`), then we need to either:

1. Change the Mac's LocalHostName to `brain` via `sudo scutil --set LocalHostName brain` (one-time manual step).
2. Use a library that can publish arbitrary hostname records, not just services. `bonjour-service` may or may not support this; the `multicast-dns` package under it definitely does.

The research doc **assumes** that advertising a service named "Brain" on port 3000 will cause `brain.local` to resolve. This is not how mDNS works.

**Empirical check needed:**
```bash
# On Mac after bonjour-service is running:
dns-sd -G v4 brain.local
# Expect: timestamp  flags  interface  name  address  ttl

# If no result: the Mac is not publishing brain.local as a hostname.
```

**Disposition:** Must be resolved before T-5 lands.

**Recommended fix:** Add a pre-plan decision in the plan §2 "Decisions" block:

> **D-v0.5.0-1:** Publish `brain.local` via either (a) `sudo scutil --set LocalHostName brain` as a one-time Mac setup step, OR (b) direct `multicast-dns` hostname publish in `instrumentation.ts` via `new Dnsd().respond({answers: [{name: 'brain.local', type: 'A', data: <lan-ip>}]})`.
>
> **Recommendation:** Option (a) — cleaner, uses macOS's native mDNS stack, no Node.js process has to keep answering queries. The one-time setup cost is acceptable for a local-first tool.

**Severity:** **BLOCKER** — must be resolved in plan drafting; otherwise T-5 will produce an advertisement that doesn't actually make `brain.local` resolvable on the Pixel.

### H-2: §4.5 CSRF reasoning omits the "browser same-origin GET with cookies" case

**Location:** §4.5 CSRF posture table.

**Finding:**
The table covers:
- Browser PIN-unlocked (same-origin cookie)
- APK WebView (same-origin cookie)
- Extension (bearer token)

It does not cover: **a malicious webpage on a *different* origin making a `<form action="http://brain.local:3000/api/capture/url" method="POST">` submission.** With `SameSite=Strict`, the browser won't send the cookie, so the request arrives unauthenticated → 401. That's correct behaviour — but the doc doesn't walk through it and therefore doesn't prove the posture is complete.

Additionally: **`SameSite=Strict` has a known gap** on top-level navigations. If a malicious page opens `window.location = "http://brain.local:3000/some-destructive-GET"`, the cookie *is* sent (same-site for top-level nav). This is why destructive actions must be POST, not GET — which is already the project convention (server actions), but not documented in the research.

**Action:** In the plan's F-036 task, add an acceptance criterion: "No destructive endpoint accepts GET; all mutations go through POST + server action or API route with method check". This is implicit today; make it explicit.

**Severity:** **HIGH**. Omission, not an error. But a planner + code-reviewer would benefit from the explicit check.

---

### H-3: §4.6 rate limiter does not handle X-Forwarded-For / missing IP

**Location:** §4.6 rate limiter sketch.

**Finding:**
`isRateLimited(ip)` takes an `ip` parameter but the sketch doesn't show how `ip` is derived from `NextRequest`. In Next.js 16:

- `req.ip` was removed (it was a Vercel-specific helper).
- `req.headers.get('x-forwarded-for')` is available but trivially spoofable by any client on the LAN.
- Without a real proxy in front, the "client IP" is effectively whatever `req.headers.get('x-forwarded-for')` says — or empty.

If the limiter keys on a spoofable header, the rate limit is trivially bypassed (send a random value each request).

**Action:** The plan's T-3 rate-limiter acceptance should say: "Key the rate limiter on a server-assigned identifier (e.g., bearer-token fingerprint: `sha256(token).slice(0, 16)`), NOT on client IP. There is one user; IP-based keying adds spoofability without adding security."

**Severity:** **HIGH**. The sketch's `ip` parameter is a security anti-pattern for this threat model.

### H-4: §5.1 `bonjour.destroy()` comment says "do not call" but doesn't wire shutdown

**Location:** §5.1 line 295 — `// Do not call bonjour.destroy() — keep advertising for server lifetime`.

**Finding:**
"Never call `destroy()`" during server lifetime is fine, but the doc doesn't say **what happens on graceful shutdown** (SIGTERM, `npm run start:lan` → Ctrl+C, macOS sleep, etc.). If the process exits without calling `destroy()`, the advertisement persists in the mDNS cache on neighbour devices for up to 2 hours (default mDNS TTL). On restart, we get a "zombie" cached advertisement pointing at a stale IP if DHCP moved the Mac.

**Action:** The plan's T-5 should add a `SIGTERM` handler in `instrumentation.ts` that calls `bonjour.destroy()` to send a "goodbye" packet. One `process.on('SIGTERM', () => bonjour.destroy())` line.

**Severity:** **HIGH**. Small code, real symptom. Skipping this guarantees a "works on first boot, broken after Mac reboot" class of bug.

### H-6: §7 RISK-9 is mis-disposed — dynamic LAN IPs are the default, not the exception

**Location:** §7 RISK-9 lines 494–497.

**Finding:**
The doc says `network_security_config.xml` with `brain.local` only is fine because IPs "vary by network". But the *whole point* of the QR-code fallback (§5.3 Layer 1) is that the user may end up using a raw IP when mDNS fails. So "IPs vary by network" isn't a reason to accept the risk — it's a reason to **design for it**.

The recommendation "use `cleartext: true` for v0.5.0 simplicity" at the end of RISK-9 effectively unwinds §3's "use `network_security_config.xml` for production correctness" decision. These two sections **contradict each other**.

**Action:** Pick one. The plan must land on a single approach. Recommended: `network_security_config.xml` with cleartext allowed for `brain.local`, `localhost`, `10.0.2.2` (AVD), and the `192.168.*`/`10.*`/`172.16.*` RFC1918 ranges. This is 10 lines of XML and covers all realistic home-network topologies. The "simplicity" of `cleartext: true` isn't worth the security debt.

**Severity:** **HIGH**. Internal contradiction between §3 and §7 RISK-9. Planner needs a decision.

### H-7: §7 RISK-8 (SameSite=Strict first-navigation) conflates two different flows

**Location:** §7 RISK-8 lines 489–492.

**Finding:**
RISK-8 says: "When the APK WebView first navigates to `http://brain.local:3000`, there is no `brain-session` cookie (it's first launch). The user must PIN-unlock via the `/unlock` page."

But the *very same APK* uses a `BRAIN_LAN_TOKEN` bearer token for API calls on `/api/capture/*`. So there are two authentication layers mixed in one APK:

1. **WebView navigation** → PIN via `/unlock` → session cookie → same-origin cookie works.
2. **Programmatic API call from Capacitor native code** (e.g., `CapacitorHttp` in the share handler) → bearer token in `Authorization` header.

The research doc doesn't distinguish these. Planner will be confused about which auth layer applies to which code path. Specifically: **does the share handler's POST to `/api/capture/url` use (1) the cookie set by a prior `/unlock` PIN unlock, or (2) the bearer token baked into `BuildConfig`?**

**Action:** The plan must declare: "All programmatic writes from the APK (share handler, settings writes, etc.) use bearer token. The cookie-based session exists only for WebView browser interaction (PIN unlock, navigation, form posts to /api/capture/\* when triggered from the WebView UI)." Then the share-intent path uses bearer — the user never needs to PIN-unlock to receive a share.

**Severity:** **HIGH**. This is the "what's the actual UX on first launch" question. Gets it wrong and the APK either forces a PIN on every share (annoying) or silently discards shares before PIN (confusing).

### M-3: §3 `capacitor.config.ts` example has `androidScheme: 'http'` — confirm this isn't overriding the default

Capacitor 8's default `androidScheme` is `https` to force-bump cleartext warnings. Setting it to `http` explicitly opts into the cleartext path. This is intentional (we're pointing at `http://brain.local:3000`), but the research doesn't explain the side effect: the WebView's **own** origin becomes `http://localhost` (not `https://localhost`), which can affect `secure` cookies, crypto.subtle availability on some APIs, and `service workers` inside the Next.js app itself.

**Action:** Plan T-7 acceptance: "Smoke-check that the WebView's `window.location.protocol` is `http:` and that no Next.js features depend on `https:` origin (secure cookies, webcrypto, push notifications)." None of our current features do, but worth a ticked box.

**Severity:** MEDIUM. No blockers in the current codebase, but is a latent scheme-origin gotcha.

`bonjour-service` has historically accepted both `'http'` (canonicalised internally to `_http._tcp`) and `_http._tcp` directly. The research uses the short form. Verify in library docs at T-5 start; update sketch if needed.

### M-6: §12 Assumptions log omits 3 load-bearing assumptions

## 7. Gaps — research items that weren't covered but should have been

### G-1: No coverage of **how the APK reaches the Mac from the Pixel over Wi-Fi when the Mac is asleep**

macOS aggressively sleeps idle laptops. If the Mac is closed and in a bag, the Pixel APK can't reach the server. Is this an accepted user-experience trade-off ("Mac must be awake to save")? Is there a wake-on-LAN path? Or does the APK queue shares locally and sync on next reachable?

This is arguably a UX-design question, not pure research. But it materially shapes what the APK does when the server is unreachable (F-020's "Mac-unreachable offline screen"). Without a clear stance, the offline screen becomes handwavy.

**Recommendation:** Plan should include an open decision: "When Mac is unreachable, APK **(a)** shows an error and discards the share, **(b)** queues locally in IndexedDB/SharedPreferences and retries on next reachable tick, **(c)** wakes Mac via WoL packet." Default (a) for v0.5.0 with local queue deferred to v0.5.1.

**Severity: HIGH (from plan perspective; MEDIUM from research perspective).**

---

### G-2: No coverage of **PDF share handling specifics** beyond "intent filters for application/pdf"

Android's share-sheet for PDFs delivers a `content://` URI, not a `file://` path. The APK must call `ContentResolver.openInputStream(uri)` to get the bytes, then stream-upload via `CapacitorHttp`. The research mentions `CapacitorHttp` as the wrapper (§4, §8 T-12) but doesn't walk through the content-URI → bytes → upload chain.

This is the part of F-039 ("native file-stream upload path, avoid WebView heap") that is most likely to have a cross-platform quirk. The Capgo plugin delivers the URI in `files[]`; does it also resolve the ContentResolver for us, or do we have to write Kotlin for that?

**Recommendation:** Plan T-12 acceptance must include: "share a PDF from Google Drive → APK → Mac. Verify the POST body bytes match the original PDF by SHA256 comparison. No WebView blob conversion; stream straight from ContentResolver to HTTP."

**Severity: HIGH.** F-039's whole value prop is streaming without WebView heap. Missing this means a 50 MB PDF OOMs the WebView — the exact failure mode F-039 was supposed to prevent.

---

### G-3: No mention of **keystore rotation, APK signing identity, or "how do I update the APK after v0.5.0 ships"**

The debug keystore is fine for v0.5.0 sideload. But Android requires all updates to be signed with the same key, else it refuses to install over the existing APK. If the debug keystore is ever lost (e.g., fresh Mac, accidental `rm`), all future updates require a **full uninstall + reinstall** (losing APK-side SharedPreferences).

**Recommendation:** Plan T-18 acceptance: "Back up `android/app/debug.keystore` to `data/backups/debug.keystore.backup` and add to the 6-hour backup rotation. Document recovery procedure." Tiny task; prevents a self-inflicted lockout.

**Severity: MEDIUM.** Only bites after first install; easy to add.

---

### G-4: No mention of **how the extension's Bearer token is set on first install**

§6.5 says "Token retrieval: chrome.storage.local.get" but not how it got there. Does the user manually copy-paste? Does the extension read it from a clipboard on first install? Does it require opening the Mac's Settings UI and scanning a QR?

The APK has a well-designed first-run-QR flow (§5.3). The extension does not. This is a UX dead spot.

**Recommendation:** Plan T-25 (extension settings page) must include: "Settings page has a 'Paste token from Mac Settings' field and a 'Test connection' button. The Mac's `/settings/lan-info` page must show the token as copyable text next to the QR code, so desktop Chrome on the Mac can just paste it."

**Severity: MEDIUM.** Planner will eventually figure this out, but calling it out now avoids a late "oh wait, how do we bootstrap the extension" scramble.

---

### G-5: No error-log namespace for v0.5.0

v0.3.1 added `data/errors.jsonl` with namespaces like `enrich.worker`, `ask.orphan-citation`, etc. v0.5.0 will add new error surfaces (mDNS publish failure, bearer-token reject, rate-limit trigger, share-intent double-fire, offline detection). Without a convention defined in the plan, these will get scattered namespaces.

**Recommendation:** Plan §3 "Conventions" section should declare: all v0.5.0 error logs use `lan.*` prefix (`lan.mdns.publish-failed`, `lan.bearer.reject`, `lan.ratelimit.triggered`) and `share.*` prefix (`share.intent.duplicate`, `share.pdf.upload-failed`).

**Severity: LOW.** Cosmetic but hygiene-matters given the v0.3.1 F-050 pattern.
