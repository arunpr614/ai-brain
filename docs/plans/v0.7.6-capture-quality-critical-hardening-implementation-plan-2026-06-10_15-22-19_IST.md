# v0.7.6 Capture Quality Critical Hardening Implementation Plan

**Created:** 2026-06-10 15:22:19 IST  
**Author:** Codex  
**Source report:** `ReviewReport/AI_BRAIN_CAPTURE_QUALITY_SELF_CRITIQUE_2026-06-10_15-15-31_IST.md`  
**Purpose:** turn the adversarial self-critique findings into an executable implementation plan before any production deployment of the v0.7.5 capture-quality foundation.  
**Scope:** all P0 and P1 findings from the self-critique, plus the P2 guardrails needed to prove the fixes are real.

---

## 1. Executive Summary

The v0.7.5 implementation created a good foundation for platform-aware captures, but the self-critique identified several critical risks:

1. YouTube smoke tests now allow metadata-only captures to pass, which can hide a real transcript-quality regression.
2. Optional YouTube Data API enrichment has no timeout or cache, which can hang capture flows and burn quota.
3. Artifact storage is too fragile for production: synchronous writes, absolute paths, no truncation metadata, no cleanup, and collision risk.
4. LinkedIn metadata-only capture still stores raw HTML snapshots, creating avoidable privacy/compliance risk.
5. Substack paywall detection can incorrectly label long previews as full text.
6. Pasting full LinkedIn text after a weak metadata-only capture creates a duplicate item instead of upgrading the existing weak item.
7. Builds still mutate local runtime data and can recreate `.next/standalone/data`.

This plan should be executed as a hardening release before deployment. The intended outcome is not more features; it is safer behavior, sharper quality gates, and fewer misleading captures.

---

## 2. Release Goal

Ship a v0.7.6 hardening pass that makes capture quality observable, bounded, and honest.

Success means:

- A live YouTube save can still succeed with metadata-only fallback, but quality regression is visible and cannot quietly pass as "healthy".
- YouTube Data API enrichment cannot hang a capture and does not repeatedly spend quota for the same video.
- Capture artifacts are stored predictably and can be cleaned up.
- LinkedIn metadata capture stores only normalized metadata unless the user explicitly provides text.
- Substack paid/preview pages are not over-labeled as `full_text`.
- User-provided text can upgrade a weak existing capture instead of creating confusing duplicates.
- `npm run build` no longer leaves runtime DB state inside `.next/standalone/data`, and the deploy path fails safely if that directory appears.

---

## 3. Non-Goals

- Do not build a full background job system.
- Do not implement browser-authenticated capture for LinkedIn, Substack, or YouTube.
- Do not scrape private LinkedIn content.
- Do not auto-delete real user data as part of this hardening pass.
- Do not redesign the Library UI beyond the labels/messages needed to make upgraded or weak captures understandable.
- Do not deploy until the verification checklist in this plan passes.

---

## 4. Implementation Principles

- Prefer honest weak captures over false confidence.
- Separate "saved successfully" from "captured high-quality content".
- Keep network calls bounded by explicit timeouts.
- Treat raw third-party HTML as sensitive retention unless there is a clear product reason.
- Make artifact writes recoverable and inspectable.
- Add tests around the exact failure modes from the self-critique, not just happy paths.

## 5. Workstream A — YouTube Quality Gates

### Critical Finding Covered

**P0 — Smoke Tests Now Allow A Major YouTube Quality Regression**

The current live smoke can pass even when YouTube and YouTube Shorts fall back to metadata-only content.

### Design

Split YouTube verification into two concepts:

1. **Save reliability:** the product should save a usable item even when transcripts are blocked.
2. **Transcript quality:** known public videos should still produce transcripts often enough that a drop is visible and actionable.

Do not remove the metadata-only product fallback. The fix is to make the regression observable and to fail the right quality gate.

### Implementation Tasks

1. Update `scripts/smoke-youtube.mjs`.
   - Keep the current reliability behavior: metadata-only fallback is acceptable when timed text is blocked.
   - Add explicit output fields:
     - `save_ok`
     - `quality_ok`
     - `capture_quality`
     - `transcript_present`
     - `transcript_expected`
     - `failure_mode`
   - When a known transcript fixture returns metadata-only, print a loud warning with the video ID and reason.

2. Add a separate quality-focused script or mode.
   - Preferred option: add `scripts/smoke-youtube-quality.mjs`.
   - Alternative option: add `--quality-gate` to `scripts/smoke-youtube.mjs`.
   - Quality mode should fail when all known transcript fixtures return metadata-only.
   - If live YouTube is flaky, allow `YOUTUBE_QUALITY_GATE=warn` for local development and `YOUTUBE_QUALITY_GATE=strict` for release validation.

3. Update `package.json`.
   - Keep `smoke:youtube` as save reliability.
   - Add:
     - `smoke:youtube:quality`
     - optionally `smoke:capture-quality:strict`

4. Update `scripts/spikes/capture-quality-eval.mjs`.
   - Add per-platform minimum quality expectations.
   - YouTube/Shorts should not silently pass with average score `2.00`.
   - Output a `quality_regression` boolean in the summary JSON.
   - Include `quality_gate_status: pass | warn | fail`.

5. Add or update docs.
   - In the smoke script comments or README, explain that save reliability and quality are different checks.

### Files Likely Touched

- `scripts/smoke-youtube.mjs`
- `scripts/smoke-youtube-quality.mjs`
- `scripts/spikes/capture-quality-eval.mjs`
- `package.json`
- Possibly `docs/testing` or the relevant plan/runbook file if present.

### Tests

- Unit-test the smoke summary scoring if the script has testable helpers.
- Add fixture-driven tests for quality scoring if practical.
- Run:
  - `npm run smoke:youtube`
  - `npm run smoke:youtube:quality`
  - `npm run smoke:capture-quality`

### Acceptance Criteria

- A YouTube metadata-only save passes the reliability smoke.
- A known transcript fixture returning metadata-only is shown as a quality warning or strict failure.
- The capture-quality summary cannot show YouTube average score `2.00` without a visible `quality_regression` signal.
- Release validation has a command that can fail on YouTube transcript-quality collapse.

## 6. Workstream B — YouTube Data API Timeout And Cache

### Critical Finding Covered

**P0 — Optional YouTube Data API Fetch Has No Timeout Or Cache**

`fetchYoutubeDataApiMetadata` currently calls `fetch(url)` directly. This can hang captures and repeatedly spend quota.

### Design

Make optional YouTube Data API enrichment bounded and cacheable.

Recommended design:

- Add a small DB-backed metadata cache keyed by `platform + cache_key`.
- Use `youtube:video:<videoId>` as the YouTube Data API cache key.
- Cache successful responses for 7 to 30 days.
- Cache "not found/private/quota/auth failure" briefly only if useful, but avoid long negative cache for transient errors.
- Add a timeout using `AbortController`.
- Treat API timeout/cache failure as non-fatal; capture should continue with oEmbed/page metadata and transcript fallback.

### Implementation Tasks

1. Add migration `015_capture_metadata_cache.sql`.
   - Proposed schema:

   ```sql
   CREATE TABLE IF NOT EXISTS capture_metadata_cache (
     id TEXT PRIMARY KEY,
     platform TEXT NOT NULL,
     cache_key TEXT NOT NULL,
     payload_json TEXT NOT NULL,
     status TEXT NOT NULL DEFAULT 'ok',
     expires_at TEXT NOT NULL,
     created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
     updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
     UNIQUE(platform, cache_key)
   );

   CREATE INDEX IF NOT EXISTS idx_capture_metadata_cache_lookup
     ON capture_metadata_cache(platform, cache_key, expires_at);
   ```

2. Add DB helper.
   - New file: `src/db/metadata-cache.ts`.
   - Functions:
     - `getCaptureMetadataCache(platform, cacheKey, now?)`
     - `upsertCaptureMetadataCache(input)`
     - `deleteExpiredCaptureMetadataCache(now?)`

3. Update `src/lib/capture/youtube-metadata.ts`.
   - Add timeout helper.
   - Default timeout: 2,500ms or 3,000ms.
   - Optional env override: `YOUTUBE_DATA_API_TIMEOUT_MS`.
   - Check cache before fetch.
   - Write cache after successful parse.
   - On timeout, quota, auth, malformed JSON, or non-OK response, return `null` and let capture continue.

4. Add instrumentation.
   - Return or log non-sensitive failure reason:
     - `youtube_data_api_timeout`
     - `youtube_data_api_quota_or_auth`
     - `youtube_data_api_not_found`
     - `youtube_data_api_parse_error`
     - `youtube_data_api_cache_hit`
   - Do not expose API keys in logs.

5. Add tests.
   - Cache miss then fetch success.
   - Cache hit avoids fetch.
   - Timeout returns `null`.
   - Quota/auth error returns `null`.
   - Malformed JSON returns `null`.
   - Expired cache triggers fetch.

### Files Likely Touched

- `src/db/migrations/015_capture_metadata_cache.sql`
- `src/db/client.ts`
- `src/db/metadata-cache.ts`
- `src/lib/capture/youtube-metadata.ts`
- `src/lib/capture/youtube.test.ts` or new `youtube-metadata.test.ts`
- `.env.example`

### Acceptance Criteria

- YouTube Data API fetch never blocks capture beyond configured timeout.
- Repeated captures of the same video do not call the Data API when a valid cache entry exists.
- All API failures degrade gracefully.
- Tests prove timeout, cache hit, cache miss, and malformed response behavior.

## 7. Workstream C — Artifact Storage Hardening

### Critical Finding Covered

**P1 — Artifact Storage Is Too Naive For Production**

Artifact storage currently writes sync files in the request path, stores absolute paths, lacks truncation metadata, and has no file cleanup.

### Design

Keep artifact storage simple, but make it deterministic and recoverable.

Recommended design:

- Store artifacts under a configured artifact root.
- Store relative paths in SQLite.
- Add `truncated` metadata.
- Generate collision-resistant filenames.
- Record partial failures without failing the entire capture.
- Add cleanup helpers for item deletion and orphan artifacts.

### Implementation Tasks

1. Add migration `016_capture_artifacts_hardening.sql`.
   - Add columns:
     - `relative_path TEXT`
     - `truncated INTEGER NOT NULL DEFAULT 0`
     - `size_bytes INTEGER`
     - `write_status TEXT NOT NULL DEFAULT 'ok'`
     - `error_message TEXT`
   - Keep old `file_path` temporarily for backward compatibility.
   - Backfill `relative_path` where possible if path is under artifact root.

2. Update artifact row type in `src/db/client.ts`.
   - Include the new fields.
   - Prefer `relative_path` for new code.

3. Update artifact DB helper in `src/db/capture-artifacts.ts`.
   - Add:
     - `createCaptureArtifact`
     - `listCaptureArtifactsForItem`
     - `deleteCaptureArtifactRowsForItem`
     - possibly `markCaptureArtifactFailed`
   - If item deletion currently relies only on DB cascade, add explicit file cleanup flow at the service layer.

4. Update `src/lib/capture/artifacts.ts`.
   - Replace synchronous write with async `fs.promises.writeFile`.
   - Calculate byte length before and after truncation.
   - Set `truncated = 1` when content exceeds limit.
   - Use stable path shape:
     - `data/capture-artifacts/<itemId>/<artifactId>.<safe-kind>.<ext>`
   - Return per-artifact result:
     - `saved`
     - `skipped`
     - `failed`
   - Log failures but do not fail the captured item unless the artifact is mandatory.

5. Add cleanup helper.
   - New helper: `deleteArtifactsForItem(itemId)`.
   - Ensure it removes files and DB rows.
   - If there is no centralized item delete path yet, document that cleanup helper must be wired when item deletion is implemented.

6. Add retention guard.
   - Add default max artifact size.
   - Keep existing cap, but record truncation.
   - Consider lower default for HTML snapshots.

### Files Likely Touched

- `src/db/migrations/016_capture_artifacts_hardening.sql`
- `src/db/client.ts`
- `src/db/capture-artifacts.ts`
- `src/lib/capture/artifacts.ts`
- Capture call sites:
  - `src/app/api/capture/url/route.ts`
  - `src/lib/telegram/dispatch.ts`
  - `src/app/capture-actions.ts`
- Tests for artifact write, failure, truncation, and cleanup.

### Acceptance Criteria

- New artifact records store relative paths.
- Artifact filenames cannot collide for the same item.
- Truncated artifacts are marked as truncated.
- Artifact write failure does not crash a successful capture unless explicitly configured.
- There is a tested helper to remove artifact files for an item.
- Existing absolute-path artifact rows remain readable or are safely ignored.

## 8. Workstream D — LinkedIn Retention And Pasted-Text Safety

### Critical Findings Covered

- **P1 — LinkedIn Still Stores Raw HTML Snapshots**
- **P1 — Existing Metadata-Only Items Are Duplicated Instead Of Upgraded**
- Pull-forward guardrail: **P2 — User-Pasted LinkedIn Text Loses Formatting And URLs**

### Design

LinkedIn capture should have two distinct modes:

1. **Link-only:** metadata-only, no raw HTML retention.
2. **Link + user-pasted text:** user-provided full text, preserve meaningful formatting, upgrade weak existing captures when safe.

### Implementation Tasks

1. Stop storing LinkedIn raw HTML snapshots.
   - Update `src/lib/capture/linkedin.ts`.
   - Remove `html_snapshot` artifact for LinkedIn.
   - Keep normalized metadata artifact only, such as:
     - title
     - description
     - image
     - canonical URL
     - fetch status
   - If HTML retention is needed for debugging, gate it behind an env var like `CAPTURE_RETAIN_LINKEDIN_HTML=1`, default off.

2. Preserve user-pasted LinkedIn formatting.
   - Update `meaningfulUserText` in `src/lib/capture/capture-url.ts`.
   - Remove only the captured source URL, not every URL.
   - Preserve paragraph breaks.
   - Preserve list-like lines.
   - Normalize excessive blank lines.
   - Keep secondary links inside the pasted post.

3. Add upgrade behavior.
   - In `src/app/api/capture/url/route.ts`, when:
     - existing item has same `source_url`
     - existing item `capture_quality = metadata_only`
     - incoming capture quality is `user_provided_full_text`
   - Then update the existing item instead of creating a duplicate.
   - Update fields:
     - `title`
     - `body`
     - `source_platform`
     - `capture_quality`
     - `extraction_method`
     - `extraction_version`
     - `description`
     - `updated_at`
   - Consider whether to preserve the old metadata body as an artifact before replacement.

4. Add DB update helper if missing.
   - New or updated helper in `src/db/items.ts`:
     - `updateItemCaptureContent(id, fields)`
   - Keep the update narrow. Do not overwrite unrelated user edits if the app already supports item edits.

5. Update API response.
   - Return:
     - `ok: true`
     - `item`
     - `action: "created" | "duplicate" | "upgraded"`
   - The UI can later use this to tell the user that a weak capture was improved.

6. Apply the same logic to Telegram if feasible.
   - If Telegram user sends a LinkedIn URL plus pasted text in the same message, upgrade weak existing capture.
   - If Telegram currently creates through a different path, keep the same DB helper and response concept internally.

### Files Likely Touched

- `src/lib/capture/linkedin.ts`
- `src/lib/capture/capture-url.ts`
- `src/app/api/capture/url/route.ts`
- `src/lib/telegram/dispatch.ts`
- `src/db/items.ts`
- `src/lib/capture/linkedin.test.ts`
- `src/lib/telegram/dispatch.test.ts`
- Route-level capture tests, if present.

### Acceptance Criteria

- Link-only LinkedIn capture no longer stores raw HTML by default.
- Pasted LinkedIn text keeps paragraphs, bullets, and secondary links.
- Capturing a LinkedIn URL first as metadata-only, then again with pasted text, upgrades the original item or returns a clear upgraded result.
- No duplicate item is created in the upgrade case.
- Tests cover web API and Telegram paths.

## 9. Workstream E — Substack Paywall Truthfulness

### Critical Finding Covered

**P1 — Substack Paywall Detection Can Mislabel Previews As Full Text**

The current classifier can mark a long paywalled preview as `full_text`.

### Design

Strong paywall signals should win over body length unless the body source is known to be complete.

Introduce a body-source concept so the app can distinguish:

- `readability`
- `jsonld`
- `rss`
- `user_paste`
- `email`
- `unknown`

Do not claim `full_text` for a paywalled page unless the full content came from an explicit full source.

### Implementation Tasks

1. Update Substack extraction to track body source.
   - In `src/lib/capture/substack.ts`, annotate the selected body with source.
   - Add body source to extraction metadata or artifacts.

2. Tighten quality classification.
   - If strong paywall signal exists:
     - `user_paste` or `email` body may be `user_provided_full_text` or `full_text`.
     - `readability`, `rss`, `jsonld`, or `unknown` should be `paywall_preview` unless there is a reliable complete-content marker.
   - Remove the current "paywall only if body length < 2,000" rule.

3. Improve paywall signals.
   - Use text and HTML markers such as:
     - "subscribe to keep reading"
     - "this post is for paid subscribers"
     - subscription call-to-action containers
     - locked content markers
   - Keep these signals conservative to avoid false positives.

4. Add tests and fixtures.
   - Long paid preview with more than 2,000 characters should be `paywall_preview`.
   - Free full post should be `full_text`.
   - Short preview should be `paywall_preview`.
   - User-pasted full post should not be downgraded due to source URL paywall markers.

### Files Likely Touched

- `src/lib/capture/substack.ts`
- `src/lib/capture/types.ts`
- `src/lib/capture/substack.test.ts`
- Test fixtures under the existing test fixture location, if present.

### Acceptance Criteria

- A long paid preview is no longer labeled as `full_text`.
- A normal free Substack post can still be labeled `full_text`.
- The selected body source is visible in metadata or artifacts for future debugging.
- Tests cover long-preview, short-preview, and free-post cases.

## 10. Workstream F — Build Data Safety

### Critical Finding Covered

**P1 — Build Still Mutates Local Runtime Data**

`npm run build` can apply migrations to a runtime DB and recreate `.next/standalone/data`.

### Design

Builds should be deterministic and should not carry runtime data into deployable artifacts.

There are two acceptable mitigation paths:

1. Prevent app routes/pages from opening the runtime DB during build.
2. Force build-time DB access to use a disposable DB path.

The second path is likely faster for v0.7.6.

### Implementation Tasks

1. Inspect current Next build behavior.
   - Identify which route/page opens the DB during build.
   - Check if dynamic rendering flags can prevent static generation from reading the DB.

2. Add disposable build DB configuration.
   - Update `scripts/deploy.sh` and/or `package.json` build command.
   - Use a temporary path:
     - `BRAIN_DB_PATH="$(mktemp -d)/build.sqlite"`
   - Ensure migrations during build apply only to the disposable DB.

3. Add standalone data guard.
   - New script: `scripts/check-build-artifacts.mjs`.
   - Fail if `.next/standalone/data` exists after build.
   - Optionally remove it only in a cleanup command, but release validation should fail first so the issue is visible.

4. Wire guard into verification.
   - Add package script:
     - `check:build-artifacts`
   - Add it after build in release/deploy script.

5. Re-run build.
   - Confirm:
     - No real runtime DB is mutated.
     - `.next/standalone/data` is absent or the guard catches it.

### Files Likely Touched

- `package.json`
- `scripts/deploy.sh`
- `scripts/check-build-artifacts.mjs`
- Possibly pages/routes that are statically generated during build.

### Acceptance Criteria

- `npm run build` no longer mutates the real local runtime DB.
- `.next/standalone/data` does not remain after build, or the guard fails loudly before deployment.
- Release/deploy validation includes the build artifact guard.

## 13. Migration Strategy

Use additive migrations only.

Recommended migrations:

- `015_capture_metadata_cache.sql`
- `016_capture_artifacts_hardening.sql`

Avoid rewriting historical item rows in this pass unless there is a clear, reversible migration. This hardening pass should not attempt to "fix" legacy capture quality labels broadly.

Before applying migrations to a real user DB:

1. Confirm backup exists.
2. Run migrations locally.
3. Inspect row counts:
   - items by `source_platform`
   - items by `capture_quality`
   - artifact count
   - artifact paths with absolute vs relative path
4. Only then consider cleanup or backfill.

## 15. Risks And Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---:|---:|---|
| Live YouTube transcript endpoints remain flaky | High | Medium | Separate save reliability from quality gate; strict mode only for release readiness |
| Metadata cache adds migration complexity | Medium | Medium | Keep schema generic and additive; all API failures degrade to `null` |
| Artifact migration leaves old absolute paths | Medium | Low | Keep old `file_path`; use new `relative_path` only for new artifacts |
| Upgrade flow overwrites user-edited content | Medium | High | Only upgrade when existing item is metadata-only and body is weak/system-generated |
| Substack paywall classifier false positives | Medium | Medium | Conservative strong signals plus body source tracking |
| Build DB mitigation hides a deeper static-generation issue | Medium | Medium | Also inspect which route opens DB during build; add guard |

## 16. Definition Of Done

The hardening release is done when:

- All P0/P1 findings from the self-critique have concrete code fixes.
- All acceptance criteria in Workstreams A through G pass.
- No new critical self-critique findings remain after a fresh adversarial review.
- A completion report is written with:
  - files changed
  - migrations added
  - tests run
  - smoke summaries
  - any known residual risk

If committing later, use small commits:

2. `fix: bound and cache youtube metadata enrichment`
3. `fix: harden capture artifact storage`
4. `fix: make linkedin capture retention and upgrades safer`
5. `fix: tighten substack paywall quality classification`
