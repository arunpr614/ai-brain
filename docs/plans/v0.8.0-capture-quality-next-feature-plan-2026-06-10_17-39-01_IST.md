# v0.8.0 Capture Quality Next Feature Plan

**Created:** 2026-06-10 17:39:01 IST
**Author:** Codex
**Status:** planning starter after production stabilization
**Source evidence:** production Telegram capture smoke after v0.7.7 deployment hygiene merge
**Goal:** improve the quality of saved captures without weakening the reliability gains from v0.7.6.

---

## 1. Current Production Baseline

Production is healthy and mainline now matches the deployed release stack.

The latest real Telegram-path smoke used four representative captures:

| Case | Result | Quality | Notes |
| --- | --- | --- | --- |
| YouTube video | Saved | `metadata_only` | Title/metadata saved; transcript blocked by YouTube anti-bot behavior |
| YouTube Short | Saved | `metadata_only` | Short normalized to canonical watch URL; transcript blocked |
| LinkedIn with pasted text | Saved | `user_provided_full_text` | Best current LinkedIn path works when the user pastes text with the link |
| Substack public post | Saved | `paywall_preview` | Long preview/body captured, but classification stayed conservative |

The important stabilization goal is met: Telegram captures no longer fail just because a platform blocks rich extraction. The next feature should now focus on making weak captures more useful and easier to upgrade.

---

## 2. Recommended Feature Theme

Build **Capture Quality Upgrade Flows** rather than another broad extraction rewrite.

This means:

1. Keep reliable save-first behavior.
2. Make weak captures obvious.
3. Give the user a simple way to improve weak captures.
4. Add better platform-specific extraction only where it is legally and operationally safe.

---

## 3. Product Priorities

## Priority 1 - YouTube Transcript Upgrade Path

### Problem

YouTube videos and Shorts now save reliably, but both commonly land as `metadata_only` because transcript extraction is blocked.

### User Outcome

When Brain cannot fetch a transcript, the saved item should still become useful with minimal effort.

### Recommended Design

Add a user-provided transcript/text upgrade flow:

- Telegram reply after metadata-only YouTube capture:
  - "Saved metadata. If you want the transcript/notes attached, paste it here with the same link."
- Web item page action:
  - "Add transcript or notes"
- API behavior:
  - if the same YouTube URL already exists as `metadata_only`, a later paste upgrades the existing item rather than creating a duplicate.

### Acceptance Criteria

- A metadata-only YouTube item can be upgraded to `user_provided_full_text`.
- The upgraded item keeps the original source URL, title, platform, thumbnail, and duration metadata where available.
- No duplicate item is created for the same YouTube URL during upgrade.
- The item page clearly shows the upgraded quality.

---

## Priority 2 - LinkedIn Capture UX Polish

### Problem

The technical LinkedIn pasted-text path works, but the product flow is still invisible. A user who sends only a LinkedIn URL may not know how to get a higher-quality capture.

### User Outcome

LinkedIn capture should feel intentional:

- link only: save metadata and explain the limitation
- link plus pasted text: save high-quality user-provided content
- later pasted text: upgrade the weak item

### Recommended Design

Improve Telegram acknowledgements and item-page hints:

- For `linkedin + metadata_only`:
  - "LinkedIn blocks most anonymous reads. Saved metadata. Paste the post text with this link to upgrade it."
- For `linkedin + user_provided_full_text`:
  - "Saved LinkedIn text."
- For web app item page:
  - show a small improvement hint only for weak LinkedIn captures

### Acceptance Criteria

- A LinkedIn URL-only capture does not store raw HTML snapshots.
- A LinkedIn URL-only capture shows an upgrade hint.
- A LinkedIn URL plus pasted text saves as `user_provided_full_text`.
- A later paste for the same LinkedIn URL upgrades the existing weak capture.

---

## Priority 3 - Substack Classification Cleanup

### Problem

The latest Substack smoke captured a useful 6,903-character body but still classified it as `paywall_preview`. That may be honest, but it needs sharper rules so long public posts are not under-labeled.

### User Outcome

Substack captures should distinguish:

- public full article
- long public preview
- paid/paywalled preview
- metadata-only fallback

### Recommended Design

Add a Substack classifier refinement:

- Use JSON-LD article body, RSS entry content, and readability body length together.
- Treat explicit paywall markers as stronger than body length alone.
- Treat long body with no paywall marker as `full_text` or `long_preview`, not always `paywall_preview`.
- Keep conservative behavior when paywall markers are present.

### Acceptance Criteria

- Known paid fixture remains `paywall_preview`.
- Known public fixture becomes `full_text` or an explicit `long_preview` quality.
- Export and item detail pages show the classification consistently.
- Tests cover public, paid, and ambiguous Substack examples.

---

## Priority 4 - Capture Quality Review Surface

### Problem

The system now records `capture_quality`, but the user still has to infer what to do next.

### User Outcome

The Library and item detail pages should help the user quickly spot which captures are useful and which need attention.

### Recommended Design

Add simple review affordances:

- Library filter or facet:
  - All
  - Full text
  - Metadata only
  - Needs upgrade
- Item detail hints:
  - YouTube metadata-only: add transcript/notes
  - LinkedIn metadata-only: paste post text
  - Substack preview: open source / confirm whether preview is enough

### Acceptance Criteria

- Weak captures are easy to find.
- Hints are platform-specific and do not read like developer diagnostics.
- No giant redesign; keep the current calm Library UI.

---

## 4. Suggested Execution Order

1. **YouTube/LinkedIn upgrade flow**
   - Highest user value.
   - Uses existing duplicate/upgrade patterns.
   - Does not depend on brittle scraping.

2. **Substack classification cleanup**
   - Improves trust in the quality labels.
   - Should be fixture-driven.

3. **Library review filter**
   - Helps after quality data is trustworthy.
   - Keep this lightweight.

4. **Advanced YouTube transcript research**
   - Only after the upgrade flow exists.
   - Evaluate browser-assisted capture, official API/caption paths, or third-party transcript providers with privacy/legal review.

---

## 5. Non-Goals For v0.8.0

- Do not bypass LinkedIn authentication or scrape private LinkedIn content.
- Do not make YouTube transcript extraction mandatory for saving.
- Do not add a paid third-party transcript service without a separate review.
- Do not build a large capture dashboard before the basic upgrade flow works.
- Do not sacrifice the current save reliability for richer extraction.

---

### Validation

---

## 7. Recommendation

Start v0.8.0 with the upgrade flow, not with deeper scraping.

Reason: the production evidence says reliability is fixed, but YouTube and LinkedIn quality depends on content that platforms often block. The fastest honest improvement is to let the user upgrade weak captures intentionally, then use deeper extraction research as a second layer.
