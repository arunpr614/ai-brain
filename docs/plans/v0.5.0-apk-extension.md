## 1. Goal

Ship a sideloadable Android APK and a Chrome MV3 extension that let the user capture URLs and PDFs into the Brain library from their phone and desktop browser without touching the Mac UI. The APK is a thin WebView shell pointed at the live Next.js server running on the Mac; it receives Android share-sheet intents for text/URL and `application/pdf`. The extension saves the active tab URL via a popup or right-click context menu. Both authenticate with a single LAN bearer token; `SameSite=Strict` session cookies continue to gate the web UI.

Primary user-visible success: pressing "Share → Brain" in Chrome on a Pixel puts the URL into the Mac's library within 2 s. Pressing the Brain icon in desktop Chrome does the same for the current tab.

**Execution rule:** auth foundation (T-3 bearer-token + rate limiter) lands before anything that issues a bearer-authenticated request. mDNS advertisement (T-6) lands before any Android device test. Debug keystore is backed up (T-20) before the first APK is signed — otherwise a lost key locks the user out of updates.

## 4. Decisions (locked before drafting; do not re-open without user)

### D-v0.5.0-1 — `brain.local` resolvability (resolves critique B-2)

**Problem:** `bonjour-service.publish({type: 'http', host: 'brain.local'})` advertises a service record (`Brain._http._tcp.local` → SRV → `brain.local` A-record), but does **not** guarantee that `brain.local` itself is resolvable by Chrome on Android. Chrome's address-bar resolution queries the system resolver for a hostname A-record, not a service record.

**Chosen mechanism (both layers must be in place):**

1. **One-time Mac setup (T-7):** `sudo scutil --set LocalHostName brain`. macOS's `mDNSResponder` then publishes `brain.local` as a direct A-record for the Mac's current LAN IP, tracked by the OS across DHCP changes. This is the **primary** resolution path.

2. **Node.js advertisement layer (T-6):** `bonjour-service` still publishes the service `Brain._http._tcp.local` on port 3000. Used for mDNS-aware service discovery tools (future) and as a signal that the server is live. Does NOT own `brain.local` resolution.

**Rationale:** macOS's native mDNS stack is more robust, survives process restarts, and is the expected way for `.local` names to work on macOS. The Node.js layer is supplementary. If the user rejects running `scutil` (a `sudo` command), fallback is the QR-IP path (D-v0.5.0-3 below), which is mandatory regardless.

**Verification (T-7 exit):** `dns-sd -G v4 brain.local` from the Mac itself returns the Mac's LAN IP with a timestamp within 2 s. Pre-Pixel-test gate.

### D-v0.5.0-2 — Mac-asleep UX (resolves gap G-1)

**Problem:** When the Mac is asleep, the APK cannot reach the server. Options considered: (a) show error + discard; (b) queue share locally in APK, retry on reachable; (c) wake-on-LAN packet from APK to Mac.

**Chosen:** **(a) for v0.5.0 — show error + one-tap retry.** The WebView displays the offline screen (F-020) with a "Retry" button; the share intent's payload is held in memory until the user retries or dismisses. No IndexedDB persistence, no WoL magic packet.

**Rationale:** (b) adds a sync-reconciliation problem (what if the same URL is shared twice while offline, then both reach the server on reconnect? Need F-041-style dedup at the server). (c) requires LAN broadcast permission on Android 14 which is restricted. Neither is worth the complexity for v0.5.0; revisit for v0.5.1 if the user reports frequent lose-a-share incidents.

**User-visible promise:** "If your Mac is asleep, your share won't save. Open the Brain app once to retry." This is documented in the APK's offline screen (F-020) and in the README.

### D-v0.5.0-3 — QR-IP fallback is mandatory, not optional (research §5.3)

**Problem:** Even with D-v0.5.0-1 in place, `brain.local` may not resolve on a specific Pixel due to carrier WebView builds, VPN apps intercepting DNS, or other unforeseen factors. Research §5.2 flags this as MEDIUM confidence.

**Chosen:** First-run APK setup screen ALWAYS shows a QR scanner, regardless of whether `brain.local` might work. The QR encodes `{ip: "192.168.x.x", token: "<32-byte-hex>"}`. APK stores both in `SharedPreferences`. Runtime decision tree:

```
On APK boot:
  1. Try `http://brain.local:3000/api/health` (timeout 2s)
     → success: mark 'mdns-green', use brain.local
     → fail: go to (2)
  2. Try `http://<stored-ip>:3000/api/health` (timeout 2s)
     → success: mark 'ip-fallback', use stored IP
     → fail: show offline screen with "Re-scan QR" option
```

**Verification (T-8 exit):** T-8 tests force-fail (1) and verify (2) kicks in.

### D-v0.5.0-4 — Network security config with base-level cleartext (resolves critique H-6 + REVIEW P0-1)

**Problem:** Research §3 recommends `network_security_config.xml` for production correctness; research §7 RISK-9 suggests falling back to `cleartext: true` for simplicity. These contradict.

**Original v1.0 attempt** (CORRECTED in v1.1 — see REVIEW P0-1): used `<domain-cidr>` child elements for RFC1918 ranges. `<domain-cidr>` is not a valid element in Android's Network Security Configuration schema; the valid children of `<domain-config>` are `<domain>`, `<trust-anchors>`, `<pin-set>`, and nested `<domain-config>`. Silently ignored or build-time-fails. Since the QR-IP fallback (D-v0.5.0-3) needs cleartext to arbitrary LAN IPs, this would have broken SC-3 and SC-4.

**Chosen (v1.1):** Keep `network_security_config.xml` to document which domains are cleartext-permitted, but use `<base-config cleartextTrafficPermitted="true" />` as the catch-all. The file documents intent ("cleartext is for LAN only"); the base-config makes the QR-IP fallback actually work on any LAN IP. `android:usesCleartextTraffic` is NOT set at the `<application>` level — the config file owns the policy.

**Rationale for broad cleartext:** v0.5.0 is a sideloaded personal tool on a single user's home Wi-Fi. The threat model (§5) accepts LAN-adjacent eavesdropping; bearer tokens + 256-bit entropy are the primary defence, not transport security. Domain-locked cleartext with CIDR support is not available in this XML schema, and a manual per-IP allow-list is incompatible with DHCP-assigned fallback IPs. Global cleartext is the only option that satisfies SC-3/SC-4; documenting the intent in the XML file is the hygiene we can still preserve.

**Verification (T-10 exit):** APK builds with the network_security_config; runtime smoke: APK successfully POSTs to `http://brain.local:3000` AND to `http://<arbitrary-rfc1918-ip>:3000` (simulate via `adb forward` or a second device); no cleartext errors in logcat.

**Future-tightening path:** when LAN HTTPS lands (post-v1.0.0), swap `<base-config cleartextTrafficPermitted="true" />` to `false` and add `<domain-config>` entries for only the specific hosts still needing cleartext (none if the Mac serves HTTPS).

### D-v0.5.0-5 — Auth boundary: WebView-JS uses Preferences bearer, browser-nav uses cookie (resolves critique H-7 + REVIEW P0-2)

**Problem:** Research conflates session-cookie auth (browser WebView) and bearer-token auth (native APK code). v1.0 plan said "baked into `BuildConfig` at APK build time" — but `share-handler.tsx` runs in the WebView JavaScript context, which cannot read `BuildConfig` (a native Android compile-time constant) without a custom Capacitor plugin bridge. REVIEW P0-2 flagged this as a broken auth boundary.

**Chosen (v1.1):** declared convention, documented in §6 Conventions:

- **Web UI browser-style flows (WebView navigating to `/`, `/items/*`, `/unlock`, form posts, server actions):** uses `SameSite=Strict` `brain-session` cookie set by `/unlock` PIN flow. First APK launch: user must unlock once with PIN (cookie persists across relaunches until expiry).
- **WebView-JS bearer flows (share-intent handler, `CapacitorHttp` uploads, any `fetch()` to `/api/capture/*` or `/api/errors/client` triggered by native intent):** reads bearer token at runtime from `@capacitor/preferences` under key `brain_token`. The key is populated by T-16's QR scanner. On first share before QR scan: handler detects missing token, shows toast "Open Brain settings to scan the setup QR" and opens the setup route. `BuildConfig` is NOT used — the token never crosses the native/JS boundary as a compile-time constant.
- **Chrome extension (popup fetch, service worker fetch, context-menu handler):** uses bearer token from `chrome.storage.local`, set by user on first install (see T-27 for the bootstrap UX).
- **`SameSite=Strict`** on all cookies, unchanged from v0.3.1 F-043.

Both auth paths coexist on the server: the proxy checks cookie first (browser flows in WebView), then bearer if cookie is absent and the route is in `BEARER_ROUTES` (WebView-JS programmatic POSTs + extension + CLI).

**Token storage map:**

| Context | Token store | Writer | Reader |
|---|---|---|---|
| Mac server | `.env` `BRAIN_LAN_TOKEN` | T-3 auto-gen / T-17 rotate script | `src/lib/auth/bearer.ts::verifyBearerToken` |
| APK (WebView JS) | `@capacitor/preferences` key `brain_token` | T-16 QR scanner | `share-handler.tsx`, any native-initiated fetch |
| Chrome extension | `chrome.storage.local` key `brain_token` | T-27 options page | `popup.ts`, `background.ts` |

**Verification:** SC-7 (bearer rejects) + SC-1/SC-5 (cookie flow for WebView navigation) + new T-12 acceptance (Preferences read + missing-token toast) all pass.

### D-v0.5.0-6 — Rate limiter keyed on token fingerprint (resolves critique H-3 + REVIEW P2-1 framing)

**Problem:** Client IP is trivially spoofable via `X-Forwarded-For` on a LAN without a trusted proxy. Token-fingerprint keying is spoof-proof.

**What it actually protects against (v1.1 framing per REVIEW P2-1):** brute-force of the bearer token is already defeated by 256-bit entropy — at 10 req/min it would take ~10^71 years. The rate limiter's real job is to **cap damage from runaway client bugs** (a loop in the extension, a misconfigured share target firing repeatedly, a broken auto-retry loop). Not a primary security control; a stability/cost control.

**Single-user caveat:** with one token, the entire rate-limit bucket is shared across APK + extension + any future clients. A legitimate burst (PDF upload with progress polling, share-burst after a browsing session) can reasonably approach 30 req/min, which is why the v1.1 limit was raised from v1.0's 10.

**Chosen:** rate-limit key is `sha256(token).slice(0, 16)`. In-process `Map<keyFingerprint, number[]>`, 60 s sliding window, **30 requests per window** (raised from v1.0's 10 after REVIEW P2-1 analysis; still catches runaway loops while leaving headroom for normal bursts). Limit exposed as env var `BRAIN_LAN_RATE_LIMIT` (default 30); operator can tune without a code change. Session-cookie-authenticated requests (browser) are exempt — the user has already PIN-unlocked.

### D-v0.5.0-7 — No destructive GETs (resolves critique H-2)

**Problem:** `SameSite=Strict` cookies ARE sent on top-level navigations (including `window.location = "..."`), so a malicious page doing `location = "http://brain.local:3000/some-destructive-GET"` would bypass CSRF. Requires all mutations to be non-GET.

**Chosen:** explicit plan-wide audit at T-5: grep all route handlers for `export async function GET` that performs side effects; add regression test. All mutations in v0.5.0 new code use POST. Continue to avoid server-action-via-GET patterns.

| Cold-start share intent double-fire | 2 s Map-based dedup at APK (F-041) AND server-side dedup on `(url, captured_at_trunc_1min)` as defense-in-depth. | T-12 |

| Debug keystore loss → locked out of APK updates | Keystore copied to `data/backups/debug.keystore.backup` on every build; never pruned by F-009 (SQLite-only). Operator additionally copies to external path (e.g., iCloud) to survive `git clean`. | T-20 |

## 6. Architecture

### 6.1 Conventions (v0.5.0-wide)

**Error-log namespaces (resolves gap G-5):**

| Prefix | Source | Example events |
|---|---|---|
| `lan.mdns.*` | mDNS advertise / teardown | `lan.mdns.publish-failed`, `lan.mdns.destroyed-on-sigterm` |
| `lan.bearer.*` | Bearer-token proxy decisions | `lan.bearer.reject-empty`, `lan.bearer.reject-short`, `lan.bearer.reject-mismatch`, `lan.bearer.accept` |
| `lan.ratelimit.*` | Rate-limit triggers | `lan.ratelimit.triggered` |
| `share.intent.*` | APK share handler | `share.intent.received`, `share.intent.duplicate`, `share.intent.pdf-upload-failed` |
| `share.http.*` | APK HTTP errors | `share.http.unreachable`, `share.http.401`, `share.http.429` |
| `ext.*` | Extension errors | `ext.fetch-failed`, `ext.token-missing` |

All events append to `data/errors.jsonl` using the existing F-050 `logError()` helper. Client-side (APK WebView / extension) events POST to `/api/errors/client` (new endpoint in T-5).

**Auth boundary convention (resolves critique H-7):** see Decision D-v0.5.0-5. Any code path touching `Authorization` headers lives in `src/lib/auth/bearer.ts`. Any code path touching `brain-session` cookies lives in `src/lib/auth/session.ts`. The proxy (`src/proxy.ts`) chooses which helper to call.

**Keystore protection (resolves gap G-3 + REVIEW P1-2):** `scripts/build-apk.sh` ALWAYS copies the active keystore to `data/backups/debug.keystore.backup` before invoking Gradle. NOTE: F-009's `pruneOldBackups()` only prunes files matching `*.sqlite` (verified in `src/lib/backup.ts`), so the keystore backup is a persistent fixed file that survives all SQLite rotations — good for durability, but it is NOT protected from `git clean` or manual folder deletion. T-20 acceptance additionally requires the operator to copy the backup to an external path outside the repo (e.g., iCloud-synced `~/Documents/Brain-keystore-backup`).

### 6.2 Schema additions

**No new migrations.** v0.5.0 is app-shell + auth + transport; all data still lands in existing tables (`items`, `chunks`, `chat_*`, `settings`). The `BRAIN_LAN_TOKEN` lives in `.env` (not the DB); the in-process rate-limiter map is ephemeral.

One `settings` table row is read (not a schema change): `lan_info_shown_at` (timestamp of when the user first viewed the QR). Used to decide whether to show the QR setup screen on load (T-16).

### 6.3 Proxy middleware (src/proxy.ts)

Layered checks, in order. First-match-wins. All other branches fall through to 401.

```
1. Path is public (/unlock, /setup, /api/auth, static)  → next()
2. brain-session cookie present + HMAC-valid           → next()
3. Path matches BEARER_ROUTES                          → bearer check:
   a. Authorization: Bearer header present
   b. process.env.BRAIN_LAN_TOKEN.length >= 32
   c. provided === expected (length-first short-circuit)
   d. rate-limit check on sha256(token).slice(0,16)
   → on pass: next()
   → on fail: 401 with structured log
4. Path is HTML (non-/api/)                            → redirect to /unlock
5. Path is /api/*                                      → 401 JSON
```

**`BEARER_ROUTES`** (T-4 exports from `src/lib/auth/bearer.ts`):

```
/api/capture/url
/api/capture/pdf
/api/capture/note  (used by extension, not APK)
/api/items         (GET only — used by extension to check for duplicates before capture)
/api/health        (GET only — used by APK reachability probe)
/api/errors/client (POST only — used by APK + extension to log errors)
```

```xml
    hosts, not enforcement. Bearer token + 256-bit entropy are the primary
    security layer on LAN (§5 threat model).
  -->
  <base-config cleartextTrafficPermitted="true" />

  <domain-config cleartextTrafficPermitted="true">
    <domain includeSubdomains="false">brain.local</domain>
    <domain includeSubdomains="false">localhost</domain>
    <domain includeSubdomains="false">10.0.2.2</domain>
  </domain-config>
</network-security-config>
```

**Future-tightening hook:** when LAN HTTPS ships post-v1.0.0, change `<base-config>` to `cleartextTrafficPermitted="false"`; the documented domain-config above continues to work for the subset still needing cleartext.

### 6.5 Share-intent handler (APK)

File: `src/components/share-handler.tsx` mounted in `src/app/layout.tsx`. Registers `SharePlugin.addListener('shareReceived', handler)` on mount (Capgo plugin API per research §10 corrections — NOT `appShareReceived`).

Handler flow:

```
shareReceived({title, texts[], files[]}) →
  1. Dedup check: hash = sha256(title + texts.join() + files.map(f=>f.path).join())
     if Map has hash with ts < 2000ms ago → drop (log share.intent.duplicate)
  2. Reachability probe: GET /api/health with bearer token, timeout 2s
     fail → render offline screen with retry (D-v0.5.0-2)
     pass → continue
  3. Branch by content:
     - texts[0] looks like URL → POST /api/capture/url {url, title, note}
     - files[0].type === 'application/pdf' → POST /api/capture/pdf (multipart stream, T-13)
     - otherwise → POST /api/capture/note {title, body: texts.join('\n')}
  4. On success: navigate to /items/<id>  (WebView handles this)
  5. On failure: show toast, leave user on current screen
```

### 6.6 PDF stream upload path (F-039, resolves gap G-2)

On Android, shared PDFs arrive as `content://` URIs, not file paths. The flow must:

1. Plugin delivers `{files: [{uri: "content://...", type: "application/pdf", name: "doc.pdf"}]}`.
2. APK calls `CapacitorHttp.post({url: '/api/capture/pdf', data: <stream from content-URI>, ...})`.
3. CapacitorHttp reads the content-URI via Android's `ContentResolver.openInputStream()` internally (confirmed in Capacitor 8 docs).
4. Multipart upload stream — does NOT load full PDF into WebView heap.
5. Server (existing `/api/capture/pdf` from v0.2.0) receives the multipart stream.

**T-13 acceptance includes SHA256 round-trip:** upload a 5 MB PDF, compute SHA256 on Android before upload (via `CapacitorHttp`'s `progress` hook reading a fresh stream) and on server after receive; must match. Prevents WebView blob-ification from corrupting binary data.

### 6.8 Rate limiter

```typescript
// src/lib/auth/bearer.ts
const WINDOW_MS = 60_000;
const LIMIT = Number(process.env.BRAIN_LAN_RATE_LIMIT ?? 30);   // REVIEW P2-1
const rateMap = new Map<string, number[]>();

export function checkBearerRateLimit(token: string): 'ok' | 'limited' {
  const key = sha256(token).slice(0, 16);
  const now = Date.now();
  const times = (rateMap.get(key) ?? []).filter(t => now - t < WINDOW_MS);
  if (times.length >= LIMIT) return 'limited';
  rateMap.set(key, [...times, now]);
  return 'ok';
}
```

In-process Map resets on server restart; acceptable for single-user tool.

## 10. Rollback

All v0.5.0 work is additive — no destructive DDL, no data migration. Rollback path:

1. **Code:** `git revert` the merge or individual task commits.
2. **Database:** no schema changes except optional T-30 WebAuthn migration — if reverting with WebAuthn in flight, drop the `webauthn_credentials` table manually (docs in rollback runbook).
3. **APK:** uninstall from device (`adb uninstall com.arunprakash.brain`) OR downgrade install.
4. **Extension:** disable in `chrome://extensions` or uninstall.
5. **Mac setup:** `sudo scutil --set LocalHostName <original-name>` to undo D-v0.5.0-1. README documents this.
6. **Hard rollback:** `scripts/restore-from-backup.sh` (F-034) → restore pre-v0.5.0 snapshot.

Release guard T-34 requires a pre-v0.5.0 backup within the last 6 h.
