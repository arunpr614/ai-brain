# v0.6.x — Library Offline From DB

> **DEFERRED 2026-05-15** — user chose track (a) v0.6.0 cloud migration (Hetzner + Anthropic batch) over track (b) library-offline. This plan is preserved in `BACKLOG.md` v7.2 and may be picked up after v0.6.0 ships in Phase E. No work has been done against this plan.

> **Read this top-to-bottom before touching code.** Skim §1–3 to load context, then walk §6 task-by-task. The plan is intentionally task-decomposed so a fresh agent can execute one task per work session.

---

## §0. TL;DR

**Problem.** Today, tapping a Library item with no network falls through to `/offline.html`. v0.5.6 cached the four shell tabs (Library, Inbox, Ask, Settings) into the service worker but cannot cache item-detail pages because Next.js client-side navigation fetches RSC payloads, not standalone HTML — and item pages are server components that read SQLite on the laptop. There is no cacheable HTML to serve.

**Solution.** Stop fetching items over HTTP from the APK. Replicate the user's library into an **on-device store** (IndexedDB inside the Capacitor WebView) and render the item-detail page **client-side** from that store. The laptop's SQLite remains the system of record; the device store is a read-replica that stays in sync via a delta-pull endpoint.

**User-visible outcome.** With airplane mode on, tap any item in the Library → see its full title, body, summary, quotes, tags, and collections. Search the library offline. The only feature that requires network is enrichment of newly-captured items (already handled by the v0.5.5 outbox).

**Out of scope (deferred).**
- Vector search / semantic similarity offline (uses sqlite-vec — too heavy for the WebView; defer to a later plan).
- Ask / RAG offline (depends on Ollama, which is laptop-only).
- Edit-on-device (writes to title / tags / collections offline). v0.5.5 outbox handles new captures; in-place edits to existing items remain online-only for this plan.
- Desktop-web parity. Plan targets the APK first; desktop-web continues to render server-side and is unaffected.

**Size.** ~5–8 working days for one agent. 12 tasks (LIBOFF-1..12). MVP gate is LIBOFF-9 (item-detail offline on APK with 8-item test library).

## §2. Architecture decision — why IndexedDB, not bundled SQLite

Three options were considered. Decision: **IndexedDB inside the WebView, mirrored from server SQLite via a delta-pull JSON API.**

| Option | Description | Verdict |
|---|---|---|
| **A. Ship full sqlite-vec WebAssembly to the device** | Run SQLite + vector index inside the WebView. Single source of truth on device. | **Rejected.** Bundle weight (~2 MB sqlite-wasm + sqlite-vec WASM + indexing pipeline). Capacitor 8.3.3 WebView quota constraints (per `docs/plans/v0.6.x-offline-mode-apk.md` §0 evidence — quota measured per origin, not unlimited). Re-implements indexing on a constrained runtime. |
| **B. Add `/api/items` GET endpoints + cache them via SW** | Convert server-component data fetches into JSON routes; SW caches per-item JSON. | **Rejected.** SW cache lifecycle is hard to reason about (we already paid this cost for `_rsc` mismatch in v0.5.6 — three cache-version bumps in one day). Cache invalidation becomes the user's problem. No semantic for "give me only items I haven't seen." |
| **C. IndexedDB device store + delta-pull JSON API** ← **CHOSEN** | Server exposes `/api/sync/items?since=<ts>`. Device pulls deltas, stores rows in IndexedDB. Item page reads from IndexedDB client-side. | **Chosen.** Uses the IndexedDB infrastructure already proven by v0.5.5 outbox. Sync semantics are explicit (timestamp-based). Bundle weight is minimal (no WASM). Failure modes are debuggable in DevTools. Can grow to bidirectional sync later without rewrite. |

**Trade-off accepted.** Item pages render via client component fetch from IndexedDB → first paint shows a skeleton, hydrates in 50–150 ms. Slightly worse perceived perf online than today's server-rendered page. Massively better offline (today: dead end → /offline.html; this plan: full content). The user has explicitly said offline is the priority (HANDOVER §10).

**Database choice: IndexedDB direct, not Dexie / Lovefield / RxDB.** v0.5.5 outbox uses raw IndexedDB with a small wrapper at `src/lib/outbox/storage.ts`. We extend the same pattern. Adding a new dependency would violate the user's zero-new-dep norm without an exceptional reason.

### LIBOFF-1 — Server-side: define the sync endpoint contract (DOC ONLY)

**Output:** `docs/plans/v0.6.x-library-offline-from-db-API.md` — JSON schema doc.

**Contract:**

```http
GET /api/sync/items?since=<unix_ms>&limit=<n>&cursor=<id>
Authorization: bearer <BRAIN_LAN_TOKEN>   (or brain-session cookie)
```

Response:

```json
{
  "items": [
    {
      "id": "abcd1234",
      "source_type": "url",
      "source_url": "https://…",
      "title": "…",
      "author": null,
      "body": "…",
      "summary": "…",
      "quotes": "[\"q1\",\"q2\"]",
      "category": null,
      "captured_at": 1715000000000,
      "enriched_at": 1715000300000,
      "enrichment_state": "done",
      "extraction_warning": null,
      "total_pages": null,
      "total_chars": 5421,
      "duration_seconds": null,
      "tags": ["productivity", "ai"],
      "collections": [{"id": "col_ab", "name": "Reading list"}],
      "deleted": false
    }
  ],
  "next_cursor": "abcd1234",
  "server_time": 1715000301234,
  "more": false
}
```

**Semantics:**
- `since` defaults to 0 (full pull) if absent.
- Items where `captured_at > since` OR `enriched_at > since` OR `updated_at > since` (we'll add `updated_at` to schema in LIBOFF-2) are included.
- Soft-deleted items appear with `deleted: true` so device can purge.
- Page size: 50 items default, 200 max.
- `next_cursor` = last item's `id` for stable pagination across ties.
- `server_time` returned so device's `last_pulled_at` is server-authoritative (avoids clock-drift).

**Why a single endpoint, not per-table.** Tags / collections are small enough to embed in each item row. This avoids client-side JOIN logic. If tags grow to 1000s per item, revisit.

**Rationale notes** (write into the doc): why this shape, why timestamp-not-CRDT, why server_time return.

**No code changes in LIBOFF-1.** Doc only. Commit message: `docs(plan): v0.6.x library-offline sync API contract (LIBOFF-1)`.

### LIBOFF-2 — Server-side: schema migration for `updated_at` + `deleted_at`

**Output:** `src/db/migrations/008_sync_columns.sql`.

```sql
-- Add columns needed for delta-pull semantics. Existing rows get
-- updated_at = COALESCE(enriched_at, captured_at) so first pull is correct.
ALTER TABLE items ADD COLUMN updated_at INTEGER;
ALTER TABLE items ADD COLUMN deleted_at INTEGER;

UPDATE items SET updated_at = COALESCE(enriched_at, captured_at) WHERE updated_at IS NULL;

CREATE INDEX IF NOT EXISTS idx_items_updated_at ON items(updated_at);
CREATE INDEX IF NOT EXISTS idx_items_deleted_at ON items(deleted_at) WHERE deleted_at IS NOT NULL;
```

**Update existing write paths** (`src/db/items.ts`):
- `insertCaptured` sets `updated_at = captured_at`.
- Any UPDATE that touches `summary`, `quotes`, `category`, `enrichment_state`, `enriched_at`, `title`, `author`, `body` also sets `updated_at = Date.now()`.
- Soft-delete (used by `deleteItemAction`) sets `deleted_at = Date.now()` and `updated_at = Date.now()`. **Confirm with the user whether to convert hard-delete to soft-delete** — this is a behavior change. Alternative: keep hard-delete but emit a "deleted_ids" array on the next sync response. **Default: ASK before implementing.**

**Tests:** extend `src/db/items.test.ts` to assert `updated_at` is set on insert and on any update path.

Commit message: `feat(db): updated_at + deleted_at for delta sync (LIBOFF-2)`.

### LIBOFF-3 — Server-side: `/api/sync/items` route handler

**Output:** `src/app/api/sync/items/route.ts`.

Implementation:

```ts
import { NextResponse, type NextRequest } from "next/server";
import { listItemsForSync } from "@/db/items"; // new export

export async function GET(req: NextRequest) {
  const url = new URL(req.url);
  const since = Number.parseInt(url.searchParams.get("since") ?? "0", 10);
  const limit = Math.min(Number.parseInt(url.searchParams.get("limit") ?? "50", 10), 200);
  const cursor = url.searchParams.get("cursor") ?? null;

  if (!Number.isFinite(since) || since < 0) {
    return NextResponse.json({ error: "bad_since" }, { status: 400 });
  }

  const { items, more, next_cursor } = listItemsForSync({ since, limit, cursor });

  return NextResponse.json(
    { items, more, next_cursor, server_time: Date.now() },
    { headers: { "cache-control": "no-store" } },
  );
}
```

`listItemsForSync` (new in `src/db/items.ts`) — joins `items` + `tags` + `collections` server-side; emits the contract from LIBOFF-1; orders by `(updated_at ASC, id ASC)` for deterministic pagination.

**Bearer route allow-list.** Add `/api/sync/items` to `BEARER_ROUTES` in `src/lib/auth/bearer.ts` so the APK can authenticate via bearer token (no cookie possible from a fresh install before pairing).

**Tests** (`src/app/api/sync/items/route.test.ts`):
- empty store → empty items array.
- 3 items, all newer than `since` → all returned.
- soft-deleted item → present with `deleted: true`.
- `since` = max(updated_at) → empty items array.
- pagination: 100 items + limit=50 → first call returns 50 + cursor; second call returns remaining 50.
- bad `since` → 400.

Commit message: `feat(api): /api/sync/items delta-pull endpoint (LIBOFF-3)`.

### LIBOFF-6 — Client-side: sync engine (delta-pull → IndexedDB)

**Output:** `src/lib/library-store/sync.ts`.

Algorithm:

```
async function pullDelta() {
  const meta = await getMeta();
  const since = meta?.last_pulled_at ?? 0;
  let cursor: string | null = null;
  let pulledCount = 0;

  while (true) {
    const url = new URL("/api/sync/items", origin);
    url.searchParams.set("since", String(since));
    url.searchParams.set("limit", "100");
    if (cursor) url.searchParams.set("cursor", cursor);

    const res = await fetch(url, { headers: authHeaders() });
    if (!res.ok) throw new SyncError(`http_${res.status}`);
    const { items, more, next_cursor, server_time } = await res.json();

    await applyBatch(items);
    pulledCount += items.length;

    if (!more) {
      await setMeta({ last_pulled_at: server_time, last_full_pull_completed_at: meta?.last_full_pull_completed_at ?? server_time });
      return { pulledCount };
    }
    cursor = next_cursor;
  }
}

async function applyBatch(items: SyncItem[]) {
  // Single transaction across items + tags_by_item + collections_by_item.
  // For each item: if deleted=true, delete by id from all three stores.
  // Else: upsert into items, replace tags_by_item rows for this item,
  // replace collections_by_item rows for this item.
}
```

**Triggers** (LIBOFF-7 wires these in):
- App focus (visibilitychange + focus).
- 30s interval when foregrounded.
- After successful outbox sync (emit a `library-store:sync-requested` event from outbox; sync engine listens).
- Manual: a "Sync now" button in Settings (LIBOFF-11).

**Concurrency.** Use a module-level `isSyncing` flag + a "pending request" boolean. If sync requested while one is in flight, queue exactly one re-sync after current finishes. No N-deep queue.

**Error handling.** Network failure → swallow, retry on next trigger. 401 → emit `library-store:auth-failed` (rendered as banner in shell). 5xx → exponential backoff (1s, 4s, 16s, then idle until next trigger).

**Tests** (`sync.test.ts`):
- empty device + non-empty server → all items pulled in pages of 100.
- subsequent call with no server changes → 0 items pulled, last_pulled_at advanced to server_time.
- soft-deleted item → removed from device.
- network error mid-pagination → partial state preserved, next trigger resumes from same cursor.

Commit message: `feat(library-store): delta-pull sync engine (LIBOFF-6)`.

### LIBOFF-8 — Render path: convert `/items/[id]/page.tsx` to a hybrid component

**The biggest change.** Today's page is a server component. We need it to:
- Render server-side when online from desktop-web (unchanged).
- Render client-side from IndexedDB when on APK.

**Approach.** Split into two:

1. `src/app/items/[id]/page.tsx` — keeps server-side `getItem(id)` for the desktop-web path. Detects `User-Agent` containing the Capacitor user-agent string OR a custom `x-brain-client: capacitor` header injected by `MainActivity` on every request. If APK detected, returns a thin shell that loads `<DeviceItemDetail id={id} />` client-side. Otherwise returns today's full server-rendered page.

2. `src/components/device-item-detail.tsx` — **NEW**. Client component. Reads from IndexedDB via `getItemFromDevice(id)`. Renders the same UI as today's server component but from device data. Skeleton state while loading. Empty state if not found in device store ("Item not yet synced — connect to wifi and reopen").

**Why split, not unify on client.** Desktop-web has no IndexedDB by default in this app, and adding a client-side path for desktop introduces a new failure mode (fresh browser → empty device store → nothing renders). Keep desktop-web on the server-rendered path; APK switches to client. Both paths share the same JSX components for body, summary, quotes, tags, collections — refactor those into `src/components/item-detail-view.tsx` consumed by both server and client wrappers.

**Test plan.**
- Desktop-web: existing item page tests pass unchanged.
- APK: new component test in `device-item-detail.test.tsx` using fake-indexeddb pre-seeded with one item.
- Integration: `/items/[id]` rendered with the Capacitor UA returns the client-shell HTML, not the server-rendered body.

Commit message: `feat(items): hybrid render path — server on web, client-from-IndexedDB on APK (LIBOFF-8)`.

## §8. Watch-outs (read or you will burn time)

These come from the prior session's session-self-critique and architectural risks specific to this plan.

1. **Bundle size.** Adding the IndexedDB read path will ship to every desktop-web page too unless the import is lazy. Use a dynamic `import()` for `library-store` so it only loads when the hybrid component decides to use it (i.e., on APK).
2. **Don't unify desktop and APK on the client path.** Desktop-web users with no prior sync should not be told "item not yet synced." Keep server-render the default; APK opts in.
3. **`updated_at` must be set on EVERY mutation path.** If a code path forgets to bump it, that change is invisible to the device. Add a Vitest that walks every UPDATE statement in `src/db/items.ts` (regex `db.prepare(.*UPDATE`) and asserts each touches `updated_at`. This is paranoid but cheap.
4. **Soft-delete needs user approval.** Hard-delete today is permanent. Soft-delete leaves rows in SQLite forever unless we add a vacuum job. **ASK the user** before implementing soft-delete in LIBOFF-2. Alternative: emit a separate `deleted_ids: string[]` array on the sync response and the device deletes those rows from IndexedDB — preserves hard-delete on the server.
5. **First-sync UX.** A fresh install with 1000+ items will spend 30–60 seconds pulling. Show a progress toast: "Syncing your library: 230 of ~1000". Use the `more` flag + a count estimate from `/api/sync/items?count_only=1` (add this query param to LIBOFF-3).
6. **Capacitor user-agent detection.** Check `window.Capacitor?.isNativePlatform()` from a client component as the most robust signal. Server-side UA sniff is brittle. Prefer server reads `x-brain-client: capacitor` header injected by `android/app/src/main/java/.../MainActivity.java` (or its TS equivalent if generated). **Confirm whether Capacitor 8.3.3 supports a custom request header on every WebView request — if not, fall back to UA sniff and document the brittleness.**
7. **Don't break the v0.5.6 service worker.** The SW intercepts all `/_next/...` and shell paths. Make sure `/api/sync/items` is in the SW's `NETWORK_ONLY_PATHS` (or implicit network-only via the `/api/` prefix already in `NETWORK_ONLY_PATHS`). It is — but verify `public/sw.js` `NETWORK_ONLY_PATHS = ["/api/", ...]` is unchanged.
8. **No new npm deps.** If you find yourself reaching for `dexie` / `idb-keyval` / `rxdb` / `pouchdb`: stop. Reuse `src/lib/outbox/storage.ts` patterns. If you genuinely need a dep, ASK first.
9. **Test on localhost first.** Use `npm run dev:lan` against a local IP. Do not push half-working changes to `brain.arunp.in`.
10. **Empirical evidence first.** If the device store appears empty: open `chrome://inspect` → `brain-library` → confirm row count. Don't guess from server logs.

## §9. Risk register

| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| WebView IndexedDB quota too small for full library | High (drop items) | Low | Measure once during LIBOFF-4 dev. v0.5.5 outbox proved IndexedDB works. Library data is text-only — 8 items × 50 KB avg = 400 KB. 1000 items projected = 50 MB. Quota should be ≥ 200 MB. If quota fails, fall back to "last 200 items" cap. |
| `Capacitor.isNativePlatform()` returns false during the JS-execution race window before plugins load | Medium (wrong render path) | Low | Use `window.location.protocol === "https:" && /Capacitor/.test(navigator.userAgent)` as a synchronous check; treat plugin async path as a confirmation, not the gate. |
| Server adds tag while device offline; device pulls; UI shows new tag but user expected to see their OWN edit | Medium (confusing UX) | Medium | Out of scope: edit-on-device. Tags edits remain online-only with a toast. Document explicitly in CHANGELOG. |
| Sync engine retries on a flaky Cloudflare tunnel and blocks UI | Medium | Medium | Sync is fire-and-forget from the UI thread; results are observed by `library-store:sync-complete` event. UI never `await`s a sync. |
| Bundle size grows by > 30 KB | Low | Low | Lazy-import the entire `library-store` module from the hybrid component. |
| v0.5.6 SW caches the old item-detail HTML and serves it instead of the new client component | High (silent regression) | Medium | Bump SW cache version `brain-*-v3` → `brain-*-v4` as part of LIBOFF-8. The activate handler purges old caches automatically. |
