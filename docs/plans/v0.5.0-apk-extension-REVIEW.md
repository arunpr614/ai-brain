## P0 — Blocker (must fix before execution starts)

### P0-1: `<domain-cidr>` is not a valid Android `network_security_config.xml` element

**Location:** §6.4 XML block, lines 272–289. Implements D-v0.5.0-4.

**Finding:** The plan's `network_security_config.xml` uses `<domain-cidr>` elements for the RFC1918 ranges:

```xml
<domain-cidr>10.0.0.0/8</domain-cidr>
<domain-cidr>172.16.0.0/12</domain-cidr>
<domain-cidr>192.168.0.0/16</domain-cidr>
```

`<domain-cidr>` is not in the Android Network Security Configuration schema. The valid child elements of `<domain-config>` are `<domain>` (hostname or IP literal), `<trust-anchors>`, `<pin-set>`, and nested `<domain-config>`. There is no CIDR notation support. This XML will either silently ignore the unknown elements (meaning any stored fallback IP gets blocked) or fail to parse during the APK build.

D-v0.5.0-4's entire rationale ("10 lines of XML covers all RFC1918") is built on a non-existent Android API. The QR-fallback path (D-v0.5.0-3) — which the plan calls mandatory — depends on cleartext to an arbitrary LAN IP working. If `<domain-cidr>` is silently dropped, the APK will only permit cleartext to `brain.local`, `localhost`, and `10.0.2.2`; any stored IP like `192.168.1.17` will be blocked.

**Consequence:** SC-3 (QR-IP fallback) and SC-4 (DHCP reassignment) both fail on real device. The plan's mandatory fallback path is broken before a line of code is written.

**Patch:** D-v0.5.0-4 must be revised. The real options are:

- (a) Revert to `android:usesCleartextTraffic="true"` at the application level (the option the plan rejected but which actually works for LAN IPs). Since this is a sideloaded personal tool, the security trade-off is acceptable.
- (b) Keep `network_security_config.xml` for `brain.local` and `localhost`, add a `<domain>` entry per known static IP (untenable for DHCP).
- (c) Use `<base-config cleartextTrafficPermitted="true" />` globally while keeping the `network_security_config.xml` file (removes per-domain granularity but is syntactically valid; fallback to arbitrary LAN IPs works).

**Recommended:** option (c) — valid XML that achieves broad cleartext + file can be tightened later. Rewrite the §6.4 XML block accordingly and update D-v0.5.0-4 rationale.

### P0-2: Bearer token is not accessible to WebView JavaScript — the auth boundary is architecturally incomplete

**Location:** D-v0.5.0-5 (§4), §6.1 conventions, §6.5 share handler flow, T-12/T-13.

**Finding:** D-v0.5.0-5 declares: "APK native code (share-intent handler, settings page writes, `CapacitorHttp` uploads): uses `Authorization: Bearer <BRAIN_LAN_TOKEN>`. No PIN prompt required — the token is baked into `BuildConfig` at APK build time."

However, `share-handler.tsx` is React code running inside the **WebView JavaScript context**, not native Kotlin/Java. `BuildConfig.BRAIN_LAN_TOKEN` is a native Android constant — it is not accessible from WebView JavaScript unless exposed via a custom Capacitor plugin bridge. No such bridge is mentioned in any task.

`CapacitorHttp.post(...)` is called from JavaScript, so the `Authorization` header must be set in the JavaScript call. The token must be readable in JS. The plan has two competing token sources:

- D-v0.5.0-5 says "baked into BuildConfig at APK build time"
- D-v0.5.0-3 + T-16 says "QR scanner writes token to Preferences plugin at runtime"

These can in principle coexist (build-time default, runtime override), but the plan never states: (a) which source the share-handler reads at runtime, (b) how either source gets into JS scope, or (c) what happens if neither is set (first share before QR scan).

The Preferences plugin path (`@capacitor/preferences`) can work for JS — `Preferences.get({key: 'brain_token'})` returns a Promise. But `@capacitor/preferences` is not in the version table (§2), not in the scope table (§3), and not in any task's install step. T-16 says "new Capacitor Preferences plugin install" in the task description but it is the only mention and does not version-pin it.

**Consequence:** T-12 (share handler) will fail at implementation — the developer will reach for `BuildConfig.BRAIN_LAN_TOKEN` in TypeScript and find it does not exist. The auth boundary convention in §6.1 is stated but the mechanism to honour it in WebView code is undefined.

**Patch:**

- Revise D-v0.5.0-5: "The bearer token is stored in `@capacitor/preferences` under key `brain_token`, populated by T-16's QR scan. `share-handler.tsx` reads it via `Preferences.get({key: 'brain_token'})` before each bearer request. `BuildConfig` is NOT used by WebView JavaScript."
- Add `@capacitor/preferences@6.x` (current stable) to the version table (§2) with npm-verified version.
- Add `@capacitor/camera@6.x` to the version table (also missing).
- T-12 acceptance: add "reads token from Preferences plugin, fails gracefully if unset with toast 'Open Brain settings to re-scan QR'."

**Finding:** SC-4 says: "QR-stored IPs are updated via the 'Test connection' button in APK settings." There is no APK settings page task. T-16 creates the first-run QR scanner. The second option is simpler and matches what the plan actually builds.

### P1-2: F-009 `pruneOldBackups()` only prunes `*.sqlite` — keystore backup claim is misleading

**Location:** §6.1 "Keystore protection", T-20, §5 threat model row "Debug keystore loss", §9 risk table.

**Finding:** The plan states: "the 6h `data/backups/` rotation already exists (F-009); this file participates automatically." Confirmed from `src/lib/backup.ts`:

```typescript
.filter((f) => f.endsWith(".sqlite"))
```

`pruneOldBackups()` only considers `.sqlite` files. `debug.keystore.backup` has a `.backup` extension and is never pruned, not counted in the retention window, and not subject to rotation. The keystore file is written once per build and accumulates without limit.

This is good news for durability (never deleted by F-009), but the claim that it "participates in the 6h rotation" is factually wrong and could mislead a future operator who interprets "rotation" as "it gets rolled across backups." It does not — it is a persistent fixed file.

**Real risk this surfaces:** the `data/backups/` tree is NOT copied to iCloud or any external sink. The keystore is safer than the plan implies (never pruned) but NOT protected from `git clean` or manual folder deletion.

**Patch:**

- §6.1: Replace "The 6 h `data/backups/` rotation already exists (F-009); this file participates automatically" with "F-009 prunes only `*.sqlite` files; `debug.keystore.backup` is a persistent fixed file that survives all SQLite rotations. However, it is NOT protected from `git clean` or manual folder deletion. T-20 acceptance: operator copies the backup to a second location (e.g., iCloud-synced `~/Documents/Brain-keystore-backup`)."
- §5 threat-model row: change mitigation to "Keystore copied to `data/backups/debug.keystore.backup` (never pruned by F-009); operator should additionally copy to an iCloud-accessible path after first build."
- T-20 acceptance: add "operator manually confirms backup exists outside the repo directory."

## P2 — Should patch before release

### P2-1: Rate limiter is security theatre for the stated threat, but serves a different real threat

**Location:** D-v0.5.0-6, §5 threat model row "Attacker on LAN brute-forces bearer token", §6.8, SC-8.

**Finding:** The plan claims the rate limiter "defeats IP spoofing" and keying on `sha256(token).slice(0,16)` is the fix. For a single-user deployment with one token, `sha256(token)` always produces the same fingerprint — there is exactly one key in the rate-limit map. Every bearer request, regardless of sender (APK, extension, attacker), shares the same bucket. If the attacker sends 10 requests, the APK's next legitimate share gets a 429. This is a self-DoS scenario, not a brute-force defence.

Against actual brute-force (attacker trying random tokens): the 256-bit entropy means there are 2^256 possible tokens. At 10 req/min = 600 req/h, cracking would take ~10^71 years. The rate limiter contributes zero meaningful additional protection against brute-force at these entropy levels. **The brute-force is already defeated by token entropy alone.**

What the rate limiter actually protects against: accidental or buggy clients (e.g., a loop bug in the extension, a misconfigured share target repeatedly firing) that hammer the server. That is a legitimate use case but it should be stated correctly.

**Self-DoS scenario (minor):** attacker burns the single bucket. With one token and one user, 10 req/min is also probably too low — a PDF upload with progress polling could hit that limit.

**Patch:** §5 threat model row: restate the rate limiter's purpose as "prevents runaway client bugs from saturating the capture pipeline; not a primary brute-force defence (entropy handles that)." Bump the limit to 30 req/min or make it configurable. Note the self-DoS scenario in the risk table as Low/Low.

### P2-2: `scutil --set LocalHostName brain` verification gap — T-7 only confirms Mac side

**Location:** D-v0.5.0-1, T-7.

**Finding:** The plan says `scutil --set LocalHostName brain` causes macOS `mDNSResponder` to publish `brain.local` as an A-record, "tracked by the OS across DHCP changes." This is correct for Apple devices querying via mDNS — macOS's `mDNSResponder` does publish the hostname as an A-record over mDNS automatically.

However, the query path on the Android Pixel is: Chrome WebView → Android system resolver → Android mDNS stack (NsdManager or system-level mdnsresponder). The Android mDNS stack must receive the multicast DNS response from macOS `mDNSResponder`. This chain is correct in theory but the research itself rates it as MEDIUM confidence and explicitly states "never tested on a real Pixel."

The plan's T-7 verification (`dns-sd -G v4 brain.local` from the Mac itself) only confirms the Mac publishes the record — it does not confirm an Android device can receive and resolve it. `dns-sd` queries the local mDNSResponder, which obviously knows about itself.

**Patch:** T-7 exit criterion: add "From Android device (not from Mac): `adb shell ping -c 1 brain.local` (requires `ping` on device) OR open `http://brain.local:3000/api/health` in Chrome on device returns 200; OR document as pending-Pixel-test at T-22." Makes the Android resolution a hard gate before T-8 rather than a hope.

### P2-3: `/api/errors/client` endpoint contract undefined

**Location:** §6.1, T-5, §14 C-13.

**Finding:** C-13 resolution points to §6.1 conventions for `lan.*` / `share.*` / `ext.*` namespaces. §6.1 says "Client-side (APK WebView / extension) events POST to `/api/errors/client` (new endpoint in T-5)." However, T-5's task description focuses on F-036 (Origin validation + no-destructive-GETs audit) and mentions `/api/errors/client` in the Files column as a side note. The endpoint's acceptance criteria are not stated.

If T-5 ships without defining the error schema (JSON body? fields? does it append to `errors.jsonl` using the existing F-050 `logError()` helper?), APK and extension will be writing to an endpoint with no defined contract.

**Patch:** T-5 acceptance: add "the `/api/errors/client` endpoint accepts `POST {namespace: string, message: string, context?: object}`, validates namespace is in `lan.*|share.*|ext.*` allow-list, and appends to `data/errors.jsonl` using the existing `logError()` helper. Rejects empty namespace with 400. Unit test included."

## Missing risk

**Risk not in §9:** Token present in QR URI scheme (`brain://setup?ip=...&token=...`) persists in Android recents/activity history and in the QR PNG stored server-side in `data/`. If user screenshots the QR or the `qr_png_data_uri` is served without auth, token leaks. The plan serves the QR from `/settings/lan-info` which is behind cookie auth, so in-flight risk is low — but the QR PNG is the 32-byte hex token in scan-able format.

**Patch:** §9 risk table row: "`/settings/lan-info` is cookie-gated; never serve the QR PNG over an unauthenticated endpoint; do not cache in browser history (add `Cache-Control: no-store` header)."

## Summary

**2 P0 blockers, 3 P1 must-patches, 3 P2 should-patches, 3 P3 nits. Verdict: BLOCK.**

The two blockers are independent but both hit before T-9/T-10 execution:

1. `<domain-cidr>` does not exist in Android Network Security Configuration, silently breaking the mandatory QR-IP fallback on non-`brain.local` IPs.
2. `share-handler.tsx` runs in WebView JS and cannot read `BuildConfig` — the Preferences plugin path is the right fix but unversioned, unscoped, and not specified in any task.

Neither P0 requires plan restructuring — both are targeted rewrites to one decision entry and one XML block. Fix both, patch P1 items (SC-4 orphan, keystore rotation claim, serial dep reintroduction), and the plan is executable.
