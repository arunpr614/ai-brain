# v0.8.4 YouTube Transcript Provider Resilience Implementation Plan

**Created:** 2026-06-11 15:09:51 IST
**Starts after:** PR #3 merged as recovery/Review improvement
**Merged PR:** https://github.com/arunpr614/ai-brain/pull/3
**Merge commit:** `89219dc Merge PR #3: capture review and YouTube recovery`
**Primary objective:** improve automatic YouTube transcript recovery reliability without weakening the successful save/review/manual-resolution fallback shipped in PR #3.

## 1. Release Context

PR #3 intentionally shipped as a recovery and Review improvement:

- YouTube links are saved even when transcripts are unavailable.
- Weak YouTube captures are queued for transcript recovery.
- Duplicate YouTube resends requeue recovery instead of dead-ending.
- Review links focus the exact item with `/review?focus=<itemId>`.
- Manual pasted transcript/notes upgrades are auditable.
- One bad transcript job cannot stop the worker loop.

The known accepted limitation is that live YouTube timed-text requests returned HTTP `429` from the local validation environment on 2026-06-11. That means automatic transcript extraction was not reliable enough to block the PR #3 release, but it is important enough to handle as its own follow-up.

## 2. Product Goal

When Arun saves a YouTube link, Brain should prefer this path:

1. Save the link immediately.
2. Try transcript recovery safely.
3. If YouTube throttles or blocks transcript fetching, avoid noisy repeated failures.
4. Retry later with a sensible schedule.
5. Make the item easy to resolve in Review.
6. Eventually recover transcripts in batches when the provider is healthy again.

The user experience should stay calm:

- No dead-end messages.
- No false claim that transcript recovery succeeded.
- No repeated retry storm against YouTube.
- One clear Review page for weak captures.

## 3. Non-Goals

- Do not remove the metadata-only fallback.
- Do not make YouTube transcript fetching synchronous again.
- Do not block capture saving on automatic transcript success.
- Do not scrape with user credentials unless there is an explicit user-authorized design.
- Do not run an unbounded backfill.
- Do not add ASR/download-heavy fallback before simpler provider and retry controls are proven.

## 4. Workstream A - Provider Observability

### Goal

Make transcript provider behavior visible enough to debug without reading raw logs.

### Implementation Shape

Add structured events for transcript recovery attempts:

- provider name
- video id
- item id
- job id
- attempt number
- HTTP status when available
- error code
- retryable vs terminal
- duration
- chosen caption track language/kind if available
- next retry time

Likely files:

- `src/lib/capture/youtube-transcript/recovery.ts`
- `src/lib/queue/transcript-worker.ts`
- `src/db/transcript-jobs.ts`
- existing error/event sink modules under `src/lib/errors/`

### Acceptance Criteria

- A timed-text HTTP `429` produces a structured event with `status_code=429`.
- A no-captions terminal outcome is distinguishable from a throttled provider outcome.
- Review/debug output can show why an item is waiting.

## 5. Workstream B - Global Provider Cooldown For 429

### Problem

If YouTube is returning HTTP `429`, immediately retrying many transcript jobs will only create noise and may extend throttling.

### Goal

When timed-text returns HTTP `429`, Brain should pause YouTube timed-text attempts for a cooldown window instead of burning retries item-by-item.

### Implementation Shape

Add a small provider-health/cooldown layer:

- provider key: `youtube_timedtext`
- last failure code/status
- cooldown until timestamp
- failure count
- last success timestamp

Possible storage options:

1. Use the existing `settings` table for a compact JSON blob.
2. Add a tiny `provider_cooldowns` table if the data becomes relational.

Recommended first implementation:

- Use the existing `settings` table for minimal schema churn.
- Promote to a table only if dashboards/queries need it.

Worker behavior:

1. Before claiming or processing YouTube transcript jobs, check provider cooldown.
2. If cooldown is active, leave jobs pending/retryable and avoid calling timed-text.
3. On `429`, set a cooldown such as 30-60 minutes with jitter.
4. On success, clear or reduce cooldown state.

### Acceptance Criteria

- One `429` does not cause every queued YouTube item to immediately hit timed-text.
- Jobs remain visible and retryable.
- The worker logs a clear "provider cooldown active" reason.
- Tests prove cooldown prevents provider calls while still keeping jobs recoverable.

## 6. Workstream C - Retry Policy Tuning

### Goal

Make retry behavior honest and cost-aware.

### Required Changes

- Separate retry classes:
  - provider throttled: retry later, do not mark manual-needed too quickly
  - no captions: terminal/manual-needed depending on product decision
  - private/deleted/unavailable: terminal
  - parsing drift: retryable with alert
  - network failure: retryable with backoff
- Use longer delays for `429` than generic network failures.
- Avoid increasing user-facing urgency when the problem is provider health.

### Acceptance Criteria

- `429` schedules a later retry and records the reason.
- Private/deleted/unavailable videos do not burn generic retries.
- A repeated `429` sequence does not mark the item as permanently failed too quickly.

## 7. Workstream D - Provider Strategy Spike

### Goal

Evaluate safer ways to improve transcript success beyond the current InnerTube + timed-text path.

### Options To Research And Test

1. **Current InnerTube timed-text path with better cooldown**
   - Lowest complexity.
   - Best first fix.

2. **Alternate YouTube client contexts**
   - Test whether a WEB or TV client context returns different caption track URLs.
   - Must be implemented conservatively and with tests.

3. **Caption track fallback expansion**
   - Try multiple ranked tracks, languages, and ASR/manual captions.
   - Already partly implemented; verify behavior under live drift.

4. **Browser-authorized capture**
   - Chrome extension or local browser session captures transcript text the user can access.
   - More privacy-sensitive, but user-authorized.
   - Good candidate if server IP throttling persists.

5. **`yt-dlp` operator fallback**
   - Useful for local/manual backfill or server-side recovery.
   - Adds binary dependency and operational complexity.
   - Should be optional, not required for normal capture.

6. **ASR fallback**
   - Download audio and transcribe locally or via model.
   - Heavier, slower, and more operationally expensive.
   - Defer until the above options are insufficient.

### Acceptance Criteria

- Produce a short decision note with measured results.
- Pick one implementation path for v0.8.4.
- Do not mix multiple heavy fallback systems into one release.

## 8. Workstream E - Backfill After Provider Controls

### Goal

Recover old YouTube metadata-only items safely once provider cooldown/retry behavior is stable.

### Requirements

- Dry-run mode.
- Limit per run.
- Idempotent enqueue.
- Skip terminal failures.
- Respect provider cooldown.
- Summarize scanned/eligible/enqueued/skipped counts.

Likely file:

- `src/lib/capture/youtube-transcript/backfill.ts`

### Acceptance Criteria

- Running backfill twice does not duplicate active jobs.
- Backfill does not hammer YouTube during a `429` cooldown.
- Operator can run a tiny batch first.

## 9. Testing Plan

Automated:

```bash
npm run typecheck
npm test
npm run lint
npm run build
git diff --check
```

Targeted tests:

- provider cooldown stores and expires correctly
- worker skips timed-text call during cooldown
- `429` creates retryable attempt with long delay
- terminal failures do not consume generic retries
- backfill respects active jobs and cooldown

Live smoke:

```bash
npm run smoke:youtube
YOUTUBE_QUALITY_GATE=warn npm run smoke:youtube:quality
npm run smoke:youtube:quality
```

Interpretation:

- Save reliability must pass.
- Warning-mode quality should document provider state.
- Strict quality passing is desirable, but if it fails, the provider cooldown behavior must prevent noisy retries.

## 10. Rollout Plan

1. Implement observability and cooldown first.
2. Run local tests.
3. Run warning-mode YouTube quality smoke.
4. Run strict quality smoke when provider appears healthy.
5. Deploy only after SQLite backup.
6. Observe transcript job volume and `429` rate.
7. Start backfill in dry-run mode.
8. Run a tiny bounded backfill.
9. Increase batch limits only after stable provider behavior.

## 11. Rollback Plan

- Disable transcript worker.
- Clear provider cooldown state if it becomes stuck.
- Stop scheduled backfill.
- Leave saved metadata-only items and manual resolutions intact.
- Restore SQLite only if a migration or write bug corrupts data.

## 12. Definition Of Done

This follow-up is complete when:

- YouTube timed-text `429` produces calm, observable, bounded retry behavior.
- The worker avoids retry storms during provider cooldown.
- Review explains the item's recovery state clearly.
- Backfill can safely queue old weak YouTube captures without hammering the provider.
- Automatic transcript extraction is more reliable when the provider is healthy, but save reliability remains independent of transcript success.
