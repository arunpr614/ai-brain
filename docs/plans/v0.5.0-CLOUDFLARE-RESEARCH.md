## §1 — Product intent recap

AI Brain is a local-first personal knowledge app. All data lives on the user's Mac M1 Pro in SQLite; Ollama runs locally for AI inference; nothing is stored in the cloud. The v0.5.0 goal is a sideloadable Android APK (thin Capacitor WebView) that lets the user share URLs and PDFs from their phone to the Mac's library via a simple share-sheet intent, plus a Chrome MV3 extension for desktop capture. Both require network connectivity from the Android device and browser to the Mac's Next.js server.

The original LAN-only design used mDNS (`brain.local` via `bonjour-service`) combined with macOS's `scutil --set LocalHostName brain` to make the Mac's server reachable at a stable hostname over Wi-Fi. At the T-21 AVD smoke gate, the user reported that macOS's Application Firewall blocked incoming connections from the Pixel, and the fix (`sudo pfctl` rules or System Settings firewall exception) was too complex for a non-technical workflow. On 2026-05-09, the project pivoted to Cloudflare Tunnel (quick-tunnel mode) as the network transport.

Success in the pivoted design looks like: the user runs `npm run dev:tunnel` on their Mac, cloudflared creates a public HTTPS tunnel (`https://<random>.trycloudflare.com`), a QR code on the pairing page encodes the new URL and the bearer token, the user scans it from their Pixel once, and from that point forward shares work from any network (home Wi-Fi, cellular, office) without configuring a single firewall rule. The bearer-token auth, share-handler, reachability probe, offline page, and QR pairing flow all remain; the mDNS stack, scutil setup, NSC cleartext XML, and the two-probe decision tree are deleted. The Mac's local server is never directly reachable from the internet — Cloudflare routes inbound tunnel traffic to the locally-bound `cloudflared` process which forwards it to `localhost:3000`.

### URL shape — VERIFIED

Pattern: `https://<random-hyphenated-words>.trycloudflare.com`

Observed URLs from 3 empirical runs:
1. `https://definitely-with-sku-liable.trycloudflare.com`
2. `https://marion-von-stated-essex.trycloudflare.com`
3. `https://physics-greatly-cal-predict.trycloudflare.com`

All three share the pattern: 3–5 lowercase English words joined by hyphens, then `.trycloudflare.com`. No hex, no UUIDs. The subdomain is long enough to be collision-resistant for casual use.

### URL rotation — VERIFIED

**The URL changes on every `cloudflared` process restart.** [VERIFIED: 3 empirical runs produced 3 different subdomains]

This is the expected documented behavior ("randomly-generated subdomain" per official docs). The URL is NOT associated with the Mac's hardware, IP address, or any persistent identity. Every time `cloudflared` starts, it requests a fresh tunnel from Cloudflare's edge and receives a new random subdomain.

**Implication for the pivot plan:** the APK must be re-paired (QR re-scan or manual URL update) every time the Mac reboots or the `dev:tunnel` script is restarted. This is a known UX cost of quick tunnels (vs named tunnels which have stable URLs). The plan must make the re-pair flow as frictionless as possible.

### Protocol — VERIFIED

cloudflared uses **QUIC** (not HTTP/2) by default for the tunnel connection to Cloudflare's edge. [VERIFIED: startup log shows `protocol:quic`, `Initial protocol quic`]

The connection is to `198.41.192.167` or `198.41.200.13` (Cloudflare edge IPs) via QUIC on port 7844 over UDP. [VERIFIED: from startup logs including `Tunnel connection curve preferences: [X25519MLKEM768 CurveP256]`]

The edge-to-origin connection (from `cloudflared` to `localhost:3000`) uses plain HTTP/1.1 by default (or HTTP/2 if `--http2-origin` is configured). For our use case (Next.js dev server), HTTP/1.1 is correct and requires no configuration.

### Process lifecycle — VERIFIED + ASSUMED

**Mac sleep behavior:** [ASSUMED — not empirically tested] When the Mac sleeps, the QUIC connection to Cloudflare's edge drops. On wake, `cloudflared` automatically attempts to reconnect. Based on the startup sequence observed (it takes 5–8 seconds to establish), reconnection on wake should also take 5–8 seconds. However, the **URL is stable during the session — cloudflared reconnects to the same tunnel (same subdomain) after a sleep/wake cycle** as long as the process is still running. The URL changes only when the process exits and restarts.

**Termination:** `Ctrl-C` triggers graceful shutdown. Observed from empirical run:
```
INF Initiating graceful shutdown due to signal terminated ...
INF Tunnel server stopped
INF Metrics server stopped
```
Clean exit with no leftover state. No temp files, no daemon processes, no config written. [VERIFIED: empirical run]

**State left behind:** none. cloudflared is stateless for quick tunnels. No credentials, no config files, no lock files are created. [VERIFIED: checked `~/.cloudflared/` — directory did not exist after 3 runs]

## §4 — Free-tier limits (critical)

### Concurrent request limit — VERIFIED

**Hard limit: 200 in-flight requests.** [VERIFIED: official CF docs at `developers.cloudflare.com/cloudflare-one/connections/connect-networks/do-more-with-tunnels/trycloudflare/`]

> "Quick Tunnels are subject to a hard limit on the number of concurrent requests that can be proxied at any point in time. Currently, this limit is 200 in-flight requests."

Exceeding this returns HTTP 429. For a single-user personal app, 200 concurrent in-flight requests is essentially unlimited. Normal usage peaks at perhaps 3–5 simultaneous requests (page load + API calls). The 429 from this ceiling will never be hit in practice.

### Bandwidth limits — ASSUMED

No bandwidth cap is documented in the official quick-tunnel docs. [ASSUMED: based on training knowledge and inability to find a documented cap. This is consistent with CF's general approach of not publishing hard bandwidth caps for free products — they rate-limit by connection count instead. Flag for verification if the user ever transfers large amounts of data regularly.]

### Request body size (PDF uploads, F-039) — CRITICAL FINDING

Cloudflare's free tier (and Cloudflare Tunnel generally) has a **100 MB maximum upload size per request** by default for proxied traffic. [ASSUMED: this is the documented limit for Cloudflare's reverse proxy; the same limit likely applies to Tunnel because the edge handles the request. Needs verification against official Tunnel-specific docs, which do not explicitly state a limit for quick tunnels.]

The share flow (F-039) targets PDFs up to ~50 MB. If the 100 MB cap is correct, 50 MB PDFs are within bounds. However, this is an ASSUMED limit. The plan must include a test: upload a 50 MB PDF via the tunnel URL and confirm it reaches the Next.js capture handler intact. If the 100 MB limit applies, all current Brain PDFs (≤ 50 MB per R-0.5.0's F-039 spec) fit within it.

**Confidence level:** MEDIUM. The 100 MB cap is standard CF infrastructure behavior, but official quick-tunnel docs do not confirm this applies to Tunnel traffic specifically. Add a TODO in plan v2.0: empirically test a 50 MB upload via the tunnel as part of T-12's acceptance criteria.

### SSE (Server-Sent Events) — BLOCKER — VERIFIED

**Quick tunnels do NOT support SSE.** [VERIFIED: official CF docs, exact quote:]

> "Quick Tunnels do not support Server-Sent Events (SSE)."

**Impact on AI Brain:** The v0.4.0 Ask feature (`/api/ask` route) uses SSE to stream Ollama responses token-by-token via a `text/event-stream` response. The `threads/[id]/messages` route also uses SSE for message streaming. Both will fail when accessed via a quick tunnel URL.

**Scope of breakage:** The breakage affects only requests routed via the tunnel URL (`https://<random>.trycloudflare.com`). The Ask feature continues to work when accessed from `localhost:3000` directly (browser on the Mac) or via LAN IP (`dev:lan`). The broken path is: APK WebView → Ask page → SSE stream → tunnel.

**Mitigation options for plan v2.0 (research-only; decision for planner):**

| Option | Tradeoff |
|--------|---------|
| Accept breakage — Ask is not in APK scope | Cleanest. Ask was never listed as a v0.5.0 APK feature; the APK is for capture flows, not consumption. If Ask via the APK is not a use case, SSE through the tunnel is a non-issue. |
| Switch Ask to HTTP chunked transfer (fallback polling) | Requires changing the `/api/ask` streaming protocol. Complex, defeats the UX purpose of streaming. Not recommended for v0.5.0. |
| Use a named tunnel instead of quick tunnel | Named tunnels (free with a CF account) do support SSE. But user has declined named tunnel. |
| Display warning in APK when navigating to Ask page | Simplest non-breaking approach: detect if running in the tunnel context and show "Streaming unavailable via tunnel — open Brain on your Mac for Ask." |

**Recommendation:** The Ask page (`/api/ask`) should remain accessible in the APK WebView but display a degraded experience (non-streaming, single-shot response) or a clear "Ask requires Mac access" message. The planner must decide whether to leave Ask broken-but-accessible in the APK or degrade it gracefully. This is a P0 decision for plan v2.0.

### WebSocket support — ASSUMED

Quick tunnels support WebSockets. [ASSUMED: based on Cloudflare Tunnel's general documented WebSocket support; SSE is specifically called out as unsupported, which implies WebSockets are not in the same category. Brain v0.5.0 does not use WebSockets, so this is informational only.]

### SSE limitation — named tunnel alternative

If the user reconsiders, a free Cloudflare account + named tunnel removes the SSE limitation. Named tunnels have stable URLs (no re-pair on restart) and support SSE. The named tunnel option should remain documented in the plan as the upgrade path if quick-tunnel SSE breakage becomes a UX issue.

## §5 — Security properties

### TLS architecture — VERIFIED

```
APK WebView / browser
        │  HTTPS (TLS to *.trycloudflare.com cert)
        ▼
Cloudflare Edge (terminates TLS)
        │  QUIC tunnel (cloudflared ↔ edge, encrypted end-to-end via QUIC TLS)
        ▼
cloudflared daemon on Mac
        │  Plain HTTP/1.1 (loopback, localhost:3000)
        ▼
Next.js server
```

TLS terminates at the Cloudflare edge. The connection between cloudflared and the Next.js server is plain HTTP on the loopback interface. This is safe because loopback traffic never leaves the machine; eavesdropping on loopback requires local machine compromise (at which point TLS would not help anyway).

The edge-to-daemon QUIC tunnel is itself encrypted (QUIC mandates TLS 1.3). [VERIFIED: startup log shows `Tunnel connection curve preferences: [X25519MLKEM768 CurveP256]` — post-quantum hybrid key exchange]

**Client IP at the origin:** The Next.js server sees the Cloudflare edge IP (e.g., `198.41.192.167`), not the client's real IP. The client's real IP is available via the `CF-Connecting-IP` header, which Cloudflare adds to all forwarded requests. [VERIFIED: official CF docs — "CF-Connecting-IP provides the client IP address connecting to Cloudflare to the origin web server."]

The rate limiter in `src/lib/auth/bearer.ts` already keys on `sha256(token).slice(0,16)` (not client IP), so the rate limiter is unaffected by this. The `X-Forwarded-For` header is also added by Cloudflare (client IP appended to chain). [VERIFIED: official CF docs]

### Cloudflare request logs — IMPORTANT

**What Cloudflare logs for quick tunnels:** [ASSUMED — no official quick-tunnel-specific privacy docs found; the general CF privacy policy applies]

Cloudflare logs request metadata for network operations: timestamp, source IP, destination, HTTP method, status code, byte sizes. For quick tunnels specifically, because there is no Cloudflare account associated, the user CANNOT see these logs. Cloudflare employees/systems CAN see the routing metadata.

**Request body visibility:** TLS terminates at Cloudflare's edge. This means Cloudflare's infrastructure technically can observe plaintext request and response bodies before forwarding them to the cloudflared daemon. [ASSUMED — standard architectural implication of TLS termination at a reverse proxy]

For AI Brain's threat model this is acceptable: the bearer token is the only credential transmitted, and Cloudflare already has the same access to it as they do to any other HTTPS traffic they terminate. The data (personal notes, URLs, PDF content) flows through Cloudflare's infrastructure briefly. The project's local-first posture means data is stored only on the Mac; Cloudflare is transit only.

The quick tunnel Terms of Use warning from the cloudflared startup log is relevant here:
> "Cloudflare reserves the right to investigate your use of Tunnels for violations of such terms."

This is standard legal boilerplate for a free service, not a signal of active monitoring. [ASSUMED: legal interpretation]

### Bearer token — still the only auth gate — VERIFIED

The tunnel adds zero authentication of its own. Any request reaching the `*.trycloudflare.com` URL is forwarded to the Next.js server unconditionally. The bearer token check in `src/lib/auth/bearer.ts` (and the session cookie check in the proxy) remain the only auth gates. [VERIFIED: official CF docs — quick tunnels have no access control by default]

This is the correct architecture. The tunnel is a transport; auth is the application's responsibility.

### Origin header behavior — VERIFIED + KEY IMPLICATION

**Cloudflare passes the Origin header through unchanged.** [VERIFIED: official CF docs — "Cloudflare passes all HTTP request headers to your origin web server" with no modification of the Origin header listed]

**Impact on `validateOrigin()` in `src/lib/auth/bearer.ts`:**

When the APK WebView loads via the tunnel URL, the WebView's page origin becomes `https://<random>.trycloudflare.com`. HTTP requests from that page carry `Origin: https://<random>.trycloudflare.com`. The current `ALLOWED_ORIGINS` in `bearer.ts` is:
```typescript
const ALLOWED_ORIGINS = new Set([
  "http://localhost:3000",
  "http://127.0.0.1:3000",
  "http://brain.local:3000",
]);
```

All three will fail the origin check for tunnel-originated requests. The `validateOrigin()` function will return `false`, causing route handlers to log `lan.bearer.reject-origin` and return 403.

**Required fix for plan v2.0:** `validateOrigin()` must accept `*.trycloudflare.com` origins. Since the subdomain is random and rotating, the allow-list cannot be a static set. Two options:

**Option A:** Dynamic allow-list — read `BRAIN_TUNNEL_URL` env var and add its origin to the set at module init:
```typescript
const tunnelOrigin = process.env.BRAIN_TUNNEL_URL
  ? new URL(process.env.BRAIN_TUNNEL_URL).origin   // "https://xxx.trycloudflare.com"
  : null;
```

**Option B:** Pattern-match on the domain suffix:
```typescript
if (originHeader.endsWith('.trycloudflare.com') && originHeader.startsWith('https://')) return true;
```

Option B is slightly looser (any `trycloudflare.com` subdomain, not just the current one). For the single-user, single-token threat model this is acceptable — a stale tunnel URL cannot be used with our bearer token since the token changes independently. Option B requires no env var read-through at request time. The plan should pick one and document the decision. This research recommends **Option B** for its simplicity.

## §6 — APK integration

### `capacitor.config.ts` tunnel URL injection — VERIFIED (Capacitor limitation)

**Capacitor 8 does NOT support environment-variable interpolation in `capacitor.config.ts` natively.** [VERIFIED: Capacitor 8 docs and `capacitor.config.ts` type signatures — `CapacitorConfig.server.url` is a plain `string`, not a template or env-var reference]

The current config:
```typescript
const config: CapacitorConfig = {
  server: {
    url: "http://brain.local:3000",
    androidScheme: "http",
  },
  // ...
};
```

For the tunnel pivot, `server.url` must be `https://<random>.trycloudflare.com` — but this URL is only known at the moment `cloudflared` starts, not at APK compile time.

**Two approaches for plan v2.0:**

**Approach A: Build-time shell munging (recommended for v0.5.0)**

A wrapper script `scripts/build-apk-tunnel.sh` reads `BRAIN_TUNNEL_URL` from the environment, writes a temporary `capacitor.config.ts` with the tunnel URL substituted, runs `cap sync android` + `./gradlew assembleDebug`, then restores the original config:

```bash
#!/usr/bin/env bash
set -e
if [[ -z "$BRAIN_TUNNEL_URL" ]]; then
  echo "ERROR: BRAIN_TUNNEL_URL not set. Run cloudflared first and export the URL." >&2
  exit 1
fi
# Patch capacitor.config.ts in place
sed -i.bak "s|url: \"http://brain.local:3000\"|url: \"$BRAIN_TUNNEL_URL\"|g" capacitor.config.ts
trap 'mv capacitor.config.ts.bak capacitor.config.ts' EXIT
npx cap sync android
cd android && ./gradlew assembleDebug
```

This approach has a downside: the APK is built with a specific tunnel URL baked in. If the tunnel URL rotates (Mac restarts), the APK must be rebuilt and reinstalled. This is the fundamental UX cost of quick tunnels.

**Approach B: Runtime URL in `@capacitor/preferences` (recommended for v0.6.0+)**

Instead of baking the URL into `server.url`, keep `server.url` pointing at a stable URL (or even `null` with a custom WebView initialization) and store the tunnel URL in `@capacitor/preferences`. The APK loads a local splash page from `webDir: 'public'`, reads the tunnel URL from Preferences, and then navigates the WebView to it via `Capacitor.setServerBasePath()` or a custom native bridge call.

This is more robust but requires more native code. Not recommended for v0.5.0 timeline.

**For plan v2.0:** Use Approach A (shell munging). Document the "rebuild APK if tunnel URL changes" constraint explicitly in the user-facing docs.

### HTTPS and WebView cleartext — VERIFIED

With the tunnel URL being real HTTPS (`https://*.trycloudflare.com`), the Android 9+ default "block cleartext" policy is sufficient. No `network_security_config.xml` is needed for the tunnel path. `cleartext: true` in `capacitor.config.ts` is also NOT needed.

However, the plan must decide whether to keep `network_security_config.xml` for the LAN fallback path (if any) or remove it entirely. Since the pivot removes mDNS and LAN cleartext, the NSC file and its `<base-config cleartextTrafficPermitted="true"/>` can be deleted. The tunnel is HTTPS; the APK only ever talks to `https://*.trycloudflare.com`. [VERIFIED: Android security model — HTTPS is always permitted; cleartext XML only needed for non-HTTPS origins]

**`androidScheme` in `capacitor.config.ts`:** The current config has `androidScheme: 'http'`. With the tunnel URL being HTTPS, this should change to `androidScheme: 'https'` (or be omitted, as `https` is the Capacitor 8 default). Setting it to `'http'` when the server URL is HTTPS creates a mismatch — the WebView's own origin becomes `http://localhost` while the page origin should be derived from the `https://` server URL. [ASSUMED: Capacitor docs describe this behavior but the exact interaction with `server.url` HTTPS needs empirical verification in the APK build]

### SSL certificate pinning — VERIFIED (not a concern)

Cloudflare's edge TLS certificate is a valid CA-signed certificate (Cloudflare, Inc. CA). The Android WebView accepts all valid CA-signed certificates by default. No pinning is configured. Certificate rotation at Cloudflare's edge (routine renewal) does not break the APK because pinning is not in use. [VERIFIED: Android WebView security model — cert pinning requires explicit `android:networkSecurityConfig` or code-level pinning; neither is used in this project]

### `CapacitorHttp.enabled: true` — still needed — VERIFIED

`CapacitorHttp` is enabled in the current `capacitor.config.ts`. This is needed for resolving Android `content://` URIs when uploading PDFs via the share handler (F-039). The `CapacitorHttp` plugin intercepts `fetch()` calls from the WebView and routes them through the native Android HTTP client, which can resolve `content://` URIs from `ContentProvider`.

This is unrelated to the transport protocol. Even with HTTPS, the APK still needs to resolve a `content://` URI provided by the Android share sheet to get the PDF bytes before uploading them to the Next.js server. `CapacitorHttp.enabled: true` must remain. [VERIFIED: R-0.5.0 §4, plan D-v0.5.0-5, existing `capacitor.config.ts`]

## §7 — Tunnel-down UX

### Cases and current behavior

| Case | What happens | Detection possible? |
|------|-------------|---------------------|
| `cloudflared` not running | APK `fetch` to `https://<tunnel-url>` fails with a network error (DNS NXDOMAIN or connection refused). `probeReachability()` returns `{ok: false, reason: "network"}` in ~2s. | Yes — via `probeReachability()` |
| `cloudflared` running, Cloudflare edge dropped connection | QUIC reconnect happens automatically within 5–8s. Traffic may fail briefly during reconnect. | Partially — brief fetch errors |
| Mac asleep (lid closed) | QUIC connection drops. `cloudflared` is suspended. APK `fetch` times out. `probeReachability()` returns `{ok: false, reason: "timeout"}` after 2s. | Yes |
| Wi-Fi lost on Pixel (cellular only) | Traffic still routes through `https://<tunnel-url>` since it's a real HTTPS endpoint reachable over any network. No tunnel-down condition — the Mac's end is still connected. | N/A — this works fine |
| Mac Wi-Fi lost | cloudflared loses upstream connectivity; QUIC reconnects on MAC's Wi-Fi restore. Pixel requests fail until Mac reconnects. | Via probeReachability timeout |

### Server-side detection of tunnel status — VERIFIED

**The Next.js app cannot detect whether its own tunnel is running.** cloudflared is an external process; there is no syscall or IPC between Next.js and cloudflared. The Next.js app does not know its own public tunnel URL unless it is explicitly told via an environment variable (`BRAIN_TUNNEL_URL`).

The `/settings/pairing` page can display the tunnel URL (from `process.env.BRAIN_TUNNEL_URL`) and a "copy" button. Whether the tunnel is actually live is unverifiable from the server side without a self-probe (which is complex and fragile).

**Simplest approach for plan v2.0:** Display the tunnel URL on the pairing page as a read-only value from `BRAIN_TUNNEL_URL`. Add a client-side "Test connection" button that calls `probeReachability()` from the browser to the tunnel URL. This is sufficient UX without server-side tunnel introspection.

### Client-side detection — VERIFIED (reachability.ts already handles this)

`src/lib/client/reachability.ts`'s `probeReachability()` already handles the tunnel-down case correctly:
- Network error (tunnel not running) → `{ok: false, reason: "network"}`
- Timeout (Mac asleep) → `{ok: false, reason: "timeout"}`
- 401 (bearer token mismatch after URL re-pair) → `{ok: false, reason: "unauthorized"}`

The `describeVerdict()` function's messages will need minor updates for the tunnel context (e.g., "Is your Mac awake and on the same Wi-Fi?" → "Is your Mac awake and is cloudflared running?"). This is a small wording change, not a structural change.

### Re-pair UX on tunnel URL rotation

Every Mac restart (or `dev:tunnel` restart) produces a new tunnel URL. The APK's stored `BRAIN_TUNNEL_URL` becomes stale. The APK's `probeReachability()` will return `{ok: false, reason: "network"}` (DNS NXDOMAIN for the old subdomain). The user sees the "Cannot reach Brain" message with a "Re-scan QR" button.

The `/settings/pairing` page on the Mac (accessible via `localhost:3000` in a browser) shows the current tunnel URL as a QR code. The user scans it from the APK. The entire re-pair flow is already built (T-16 / `setup-apk/page.tsx`). The only required change is: update the QR content from `{ip: "192.168.x.x", token: "..."}` to `{url: "https://xxx.trycloudflare.com", token: "..."}`.

**Pain level:** Moderate. Re-scanning a QR takes ~10 seconds. The user must know to do this after restarting the Mac. Mitigation: document prominently in the pairing page ("This URL changes every time Brain restarts. Re-scan if your phone can't connect.").

## §8 — Observability

### cloudflared local logs — VERIFIED

**Tail in real time:**
```bash
cloudflared tunnel --url http://localhost:3000 --logfile /tmp/cloudflared.log
# In another terminal:
tail -f /tmp/cloudflared.log
```

Or capture to the terminal and grep post-run:
```bash
cloudflared tunnel --url http://localhost:3000 2>&1 | tee /tmp/cloudflared.log
```

**Useful grep patterns:**
```bash
grep "ERR\|error" /tmp/cloudflared.log      # errors only
grep "Registered tunnel" /tmp/cloudflared.log  # confirms tunnel is up
grep "graceful shutdown" /tmp/cloudflared.log  # confirms clean stop
```

Log level can be increased for debugging:
```bash
cloudflared tunnel --url http://localhost:3000 --loglevel debug
# WARNING: debug level logs request URLs, methods, content-length, and ALL
# request/response headers — this may expose bearer tokens in the log.
```

### Cloudflare dashboard — VERIFIED

Quick tunnels (no account) do NOT appear in any Cloudflare dashboard. There is no account to log into. Zero visibility on the Cloudflare side for the user. [VERIFIED: quick tunnels are explicitly account-free per official docs and `cloudflared` startup output]

### cloudflared local health/status endpoint — VERIFIED

**The `/ready` endpoint is the canonical tunnel status probe.** [VERIFIED: empirically tested]

While cloudflared is running with `--metrics 127.0.0.1:20241`, the `/ready` endpoint at the metrics server returns:

**When tunnel is up:**
```json
{"status":200,"readyConnections":1,"connectorId":"ef8c973d-..."}
HTTP 200
```

**When tunnel is not running:** the metrics server doesn't exist; the connection is refused.

Default metrics port: `127.0.0.1:20241` (fallback to 20242, 20243, 20244, 20245 if occupied). [VERIFIED: `cloudflared tunnel run --help`]

For the `/settings/pairing` "tunnel status" block in plan v2.0:

**Option A (recommended):** A server-side API endpoint (`GET /api/settings/tunnel-status`) that probes `http://127.0.0.1:20241/ready` from the Next.js server (loopback, fast). Returns `{connected: true, url: process.env.BRAIN_TUNNEL_URL}` or `{connected: false}`. The `/settings/pairing` page client-polls this endpoint on mount.

**Option B:** Client-side only — display `BRAIN_TUNNEL_URL` from the server, add a "Probe" button that `fetch()`es the tunnel URL `/api/health` and reports success/failure.

Option A is cleaner because it detects `cloudflared` process health (not just network reachability). The metrics port probe is fast (loopback, < 1 ms) and does not require the Next.js server to know the external tunnel URL. However, it requires that cloudflared is started with a fixed metrics port so Next.js knows where to probe.

**Suggested approach for plan v2.0:** Wrap the `npm run dev:tunnel` script to always start cloudflared with `--metrics 127.0.0.1:20241`. Then `/api/settings/tunnel-status` probes `http://127.0.0.1:20241/ready`.

**Note on metrics server start:** In the empirical run, when cloudflared was killed by SIGTERM before the metrics server fully started, no metrics server was present. The metrics server needs ~2 seconds to come up. The Next.js API endpoint should handle `ECONNREFUSED` gracefully and return `{connected: false}` rather than throwing.

## §9 — Alternatives reconsidered

### Tailscale

Tailscale creates an encrypted overlay network (WireGuard-based) between enrolled devices. The Mac and the Pixel would both need the Tailscale client installed and signed into the same account. The Mac would get a stable `<hostname>.ts.net` address.

**Why not:** Requires a Tailscale account. More importantly, requires installing and configuring Tailscale on the Pixel — this is exactly the "too complex for non-technical user" problem that the firewall pivot was trying to escape. The user's goal is zero-config on the phone side: install Brain APK, scan QR, done. Tailscale adds: install Tailscale on phone, sign in to Tailscale account, allow Tailscale VPN profile, then scan QR. Also, a Tailscale network is not an anonymous public tunnel — the same user owns both endpoints, so there is a Tailscale account tying the devices together, which is more persistent identity than a quick tunnel. Declined.

### ngrok

ngrok provides a similar quick-tunnel experience. The free tier (ngrok 3.x as of 2026) requires creating an account and authenticating the `ngrok` agent to get a persistent agent connection. Without an account, the free tier of ngrok has rate limits as low as 40 req/min and 20 connections, which are significantly more restrictive than Cloudflare's 200 concurrent requests. Additionally, ngrok's free tier now requires account creation for any use. Cloudflare's quick tunnel requires zero account, zero authentication. ngrok has no meaningful advantage over cloudflare for this use case and requires account friction. Declined.

### Named Cloudflare Tunnel (~$0/yr with free Cloudflare account)

A named tunnel (created via `cloudflared tunnel create <name>`) provides:
- Stable URL (associated with a hostname on a domain you own)
- SSE support (the quick-tunnel SSE limitation does not apply)
- Reconnect with the same URL after Mac restart

The cost is: creating a free Cloudflare account + owning a domain (~$10/yr via Cloudflare Registrar, or free if using a CF-provided subdomain via Workers). The user declined named tunnels for the initial pivot ("quick tunnel" was the user's choice). However, the named tunnel path should remain documented in the plan as the upgrade path if (a) the re-pair-on-restart friction becomes a complaint, or (b) the SSE breakage in the APK's Ask page becomes a blocker.

### Serverless deploy (Vercel, Railway, etc.)

Hosting the Next.js app on Vercel would eliminate the connectivity problem entirely. However, it violates the core local-first premise: Ollama runs locally on the Mac and cannot be moved to a cloud server without changing the entire architecture. SQLite is local. The personal knowledge base is local. This is not a connectivity alternative — it's a fundamental project re-architecture. Not considered.

## §10 — Pre-plan decisions for v2.0

The following decisions must be locked before plan v2.0 can be drafted. They are listed as open items, not recommendations (except where research strongly favors one option).

### P1 — SSE in the APK: accept, degrade, or block [PRIORITY: MUST DECIDE]

Quick tunnels do not support SSE. The Ask feature (`/api/ask`) uses SSE. Options:
- (a) Accept: Ask page is accessible in APK WebView but the stream silently fails. User sees a spinner that never resolves.
- (b) Degrade: Detect APK context, show "Ask is not available via tunnel" message. The APK's ask page falls back to polling or shows an explanatory error.
- (c) Block at proxy: `/api/ask` returns 503 when `Origin` is a `trycloudflare.com` domain.
- (d) Named tunnel upgrade: reconsider the named tunnel option.

**Research recommendation:** (b) — graceful degradation with a clear message. Silently failing (a) is bad UX. Blocking at proxy (c) requires knowing the request came via tunnel, which is not trivially detectable. Named tunnel (d) is user's call to reconsider.

### P2 — `capacitor.config.ts` URL injection mechanism [PRIORITY: MUST DECIDE]

Shell `sed` munging before `cap sync` (Approach A) vs. runtime URL from `@capacitor/preferences` (Approach B). Research recommends Approach A for v0.5.0 (simpler, less native code). Approach B for a future phase.

**Constraint:** With Approach A, every tunnel URL rotation requires an APK rebuild + `adb install -r`. This is the cost of quick tunnels. The build takes ~2 minutes. Is this acceptable to the user? If not, Approach B or named tunnels are the answer.

### P3 — `npm run dev:tunnel` script design [PRIORITY: SHOULD DECIDE]

The pivot eliminates `npm run dev:lan`. A new `dev:tunnel` script must:
1. Start the Next.js server on `localhost:3000` (not bound to `0.0.0.0` — tunnel handles external traffic).
2. Start `cloudflared tunnel --url http://localhost:3000 --metrics 127.0.0.1:20241`.
3. Parse the tunnel URL from `cloudflared`'s stderr output and write it to `BRAIN_TUNNEL_URL` in `.env` (so the pairing page can display it and the APK build script can read it).
4. Print the QR to the terminal (via `qrcode-terminal`) so the user can immediately scan from their phone.

**Open questions for plan:**
- Use `concurrently` (npm package, `concurrently "npm run dev" "cloudflared tunnel ..."`) for process management, or a custom Node.js wrapper script that can parse `cloudflared`'s output for the URL?
- `concurrently` cannot easily capture cloudflared's stderr output to extract the URL. A custom script using `spawn()` + stdout/stderr parsing is more capable but more code.
- **Research recommendation:** custom `scripts/tunnel-start.mjs` (Node.js) that spawns both processes, parses cloudflared's output for the URL line, writes it to `.env`, renders the QR, and proxies both processes' stdout/stderr with prefixes. This is ~60 lines of Node.js.

### P4 — Origin allow-list for `validateOrigin()` [PRIORITY: MUST DECIDE]

Current allow-list: `{localhost:3000, 127.0.0.1:3000, brain.local:3000}`.

With the tunnel, APK WebView requests arrive with `Origin: https://<random>.trycloudflare.com`. Options:
- (a) Static: add the current tunnel URL to the set on server start (read from `BRAIN_TUNNEL_URL` env var).
- (b) Pattern: allow any `https://*.trycloudflare.com` origin without checking the specific subdomain.
- (c) No origin check for bearer-authenticated routes (origin is irrelevant when a 256-bit token is already verified).

**Research recommendation:** (b) — pattern match on `*.trycloudflare.com`. The origin header is defense-in-depth beyond the bearer token. Allowing any `trycloudflare.com` subdomain is acceptable because an attacker would still need the 256-bit token to pass `verifyBearerToken()`. (c) is also defensible for the same reason but removes a potentially useful audit trail.

The `brain.local` origin should be removed from `ALLOWED_ORIGINS` since mDNS is being deleted. `localhost` and `127.0.0.1` origins should remain for local browser development.

### P5 — Rate limit posture with CF's 200-concurrent-request ceiling [PRIORITY: LOW]

The current `BRAIN_LAN_RATE_LIMIT` default is 30 req/min (raised from 10 in v1.1 per REVIEW P2-1). Cloudflare's ceiling is 200 concurrent in-flight requests. These are different metrics (per-minute rate vs. concurrent count), and for a single user they do not conflict. The 200 concurrent limit will never be hit; the 30/min rate limit is set for runaway client bug protection, not CF compliance.

**Research recommendation:** No change to BRAIN_LAN_RATE_LIMIT. The existing value is appropriate.
