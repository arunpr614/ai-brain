## 1. Goal

Ship the first end-to-end retrieval-augmented chat over the personal library. The user types a question; the app retrieves the top-k relevant chunks from across all captured items, sends them + the question to the local Ollama LLM, streams a cited answer, and persists the conversation. No cloud calls. Single user. Latency target: first token < 2 s, full answer < 8 s for a typical 3-paragraph reply on a 10k-chunk library.

**Execution rule:** schema + embeddings pipeline land before any Ask UI work. Retrieval correctness (citations point to real chunks) gates streaming.

## 5. Architecture

### 5.1 Schema additions

New migration `005_vector_index.sql`:

```sql
-- vec0 virtual table for chunk embeddings (768-dim float32).
-- Rowid is joined to chunks.id via a separate TEXT→INTEGER bridge because vec0
-- requires integer rowids. See chunks_rowid below.
CREATE TABLE IF NOT EXISTS chunks_rowid (
  chunk_id TEXT PRIMARY KEY REFERENCES chunks(id) ON DELETE CASCADE,
  rowid    INTEGER NOT NULL UNIQUE
);

CREATE VIRTUAL TABLE IF NOT EXISTS chunks_vec USING vec0(
  embedding float[768]
);
```

Rowid assignment: `chunks_rowid.rowid` is a monotonically increasing BigInt, assigned at insert time by a small helper that SELECTs `MAX(rowid) + 1` inside the same transaction that inserts the chunk. Documented gotcha: vec0 rowids must be `BigInt`, not `Number` (learned in R-VEC S-1).

### 5.2 Embedding pipeline

- New file: `src/lib/embed/client.ts` — thin wrapper on Ollama `/api/embed` endpoint, model `nomic-embed-text` (~274 MB, single pull).
- New file: `src/lib/embed/pipeline.ts` — `embedChunks(chunks: ChunkRow[])` batched (16 per call; `/api/embed` supports array input), writes to `chunks_vec` + `chunks_rowid`.
- Integration point: `src/lib/queue/enrichment-worker.ts` — after `enrichItem` succeeds, enqueue a follow-up `embed_item` job. Re-use the existing `enrichment_jobs` table by adding a `stage` column (migration 006) or add a sibling `embedding_jobs` table.
  - **Decision:** sibling table. Cleaner separation of retry counters; avoids schema churn on `enrichment_jobs`. (Plan note: revisit in v0.4.x if the two queues duplicate logic.)

### 5.3 Retriever

New file: `src/lib/retrieve/index.ts`. Public interface:

```ts
type RetrievedChunk = {
  chunk_id: string;
  item_id: string;
  item_title: string;
  body: string;
  similarity: number;   // 1 - cosine distance
};

async function retrieve(
  query: string,
  opts: { topK?: number; itemId?: string; minSimilarity?: number }
): Promise<RetrievedChunk[]>;
```

SQL shape:

```sql
SELECT
  c.id, c.item_id, i.title, c.body,
  1 - v.distance AS similarity
FROM chunks_vec v
JOIN chunks_rowid r ON r.rowid = v.rowid
JOIN chunks c ON c.id = r.chunk_id
JOIN items  i ON i.id = c.item_id
WHERE v.embedding MATCH ? AND [i.id = ? ONLY WHEN scope=item]
ORDER BY v.distance LIMIT ?
```

### 5.4 Generator

System prompt template (inline, not a file — iterate):

```
You are AI Brain, a personal knowledge assistant. You answer ONLY from the provided library chunks below. If nothing in the chunks answers the question, say "I don't have anything on this in your library."

Cite every claim with [CITE:chunk_id]. Never output a chunk_id that isn't in the list below.

=== Library chunks ===
[id=<chunk_id>] from "<item_title>"
<chunk body>
---
[id=<chunk_id>] from "<item_title>"
<chunk body>
---

=== User question ===
<question>
```

Streaming via Ollama `/api/generate` with `stream: true` — already supported in `src/lib/llm/ollama.ts::generate`. The route reads the response body as an async iterator and re-emits SSE frames.

## 9. Rollback

Plan uses one additive migration (`005_vector_index.sql`) and one additive sibling (`006_embedding_jobs.sql`). No destructive DDL, no data migration.

Rollback path: `git revert` the merge or the individual task commits. `data/brain.sqlite` retains orphan `chunks_vec` / `chunks_rowid` rows; they're inert without the code paths.

Hard rollback (if embedding data is corrupt): `scripts/restore-from-backup.sh` (F-034) → restore the last pre-v0.4.0 6-hour snapshot. Release guard T-19 must include "a pre-v0.4.0 backup exists within the last 6 h" as a checklist item.
