**Release objective:** YouTube links saved as metadata-only must enter a durable transcript recovery workflow, including existing items, duplicate-link resends, Review status, retries, and safe manual fallback.

---

## 1. Executive Summary

The research recommendation was right but not implementation-safe. This revised plan fixes the gaps found in the adversarial review:

- Existing metadata-only YouTube items must be backfilled, including the item that triggered this work.
- Duplicate YouTube link saves must not dead-end at "Already captured".
- Transcript recovery needs a durable attempt history, not a single `last_error`.
- `/review` should be extended from the v0.8.2 Review Inbox implementation, not rebuilt in the stale phase2 root.
- `yt-dlp` must be gated behind a subprocess safety contract and is not part of the first shippable slice.
- ASR/audio transcription remains explicitly out of scope until separate cost, retention, and policy gates exist.

This release should ship the queue and user experience first. Provider expansion comes after the recovery loop is visible, observable, and safe.

## 3. Non-Negotiable Requirements

1. Existing YouTube metadata-only items are backfilled into transcript recovery.
2. Duplicate YouTube resends create, reset, or surface the recovery job instead of only saying "Already captured".
3. Transcript recovery state is visible in Review and on the item page.
4. Every provider attempt is recorded durably with retryability and error code.
5. Successful transcript recovery updates the existing item, does not create a duplicate, and resets derived enrichment/search state.
6. `yt-dlp` is disabled by default and cannot ship without subprocess safety tests.
7. ASR/audio transcription is out of scope for v0.8.3.
8. Official YouTube `captions.download` is not used for arbitrary public videos.
9. Failures are honest: "queued", "retrying", "manual help needed", or "captions unavailable" must mean what they say.

### `transcript_jobs`

```sql
CREATE TABLE IF NOT EXISTS transcript_jobs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id TEXT NOT NULL REFERENCES items(id) ON DELETE CASCADE,
  source_platform TEXT NOT NULL,
  video_id TEXT,
  state TEXT NOT NULL DEFAULT 'pending'
    CHECK (state IN (
      'pending',
      'running',
      'retryable_error',
      'manual_needed',
      'ignored',
      'done'
    )),
  priority INTEGER NOT NULL DEFAULT 0,
  attempts INTEGER NOT NULL DEFAULT 0,
  max_attempts INTEGER NOT NULL DEFAULT 5,
  next_run_at INTEGER,
  claimed_at INTEGER,
  completed_at INTEGER,
  last_attempt_id INTEGER,
  last_provider TEXT,
  last_error_code TEXT,
  last_error_message TEXT,
  created_at INTEGER NOT NULL DEFAULT (unixepoch() * 1000),
  updated_at INTEGER NOT NULL DEFAULT (unixepoch() * 1000),
  UNIQUE (item_id)
);

CREATE INDEX IF NOT EXISTS idx_transcript_jobs_state_next_run
  ON transcript_jobs(state, next_run_at, priority, created_at);

CREATE INDEX IF NOT EXISTS idx_transcript_jobs_item
  ON transcript_jobs(item_id);
```

### `transcript_attempts`

```sql
CREATE TABLE IF NOT EXISTS transcript_attempts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id INTEGER NOT NULL REFERENCES transcript_jobs(id) ON DELETE CASCADE,
  item_id TEXT NOT NULL REFERENCES items(id) ON DELETE CASCADE,
  attempt_number INTEGER NOT NULL,
  provider TEXT NOT NULL,
  state TEXT NOT NULL
    CHECK (state IN ('success', 'retryable_error', 'terminal_error', 'skipped')),
  retryable INTEGER NOT NULL DEFAULT 0,
  error_code TEXT,
  error_message TEXT,
  status_code INTEGER,
  started_at INTEGER NOT NULL,
  finished_at INTEGER,
  duration_ms INTEGER,
  transcript_language TEXT,
  transcript_is_generated INTEGER,
  transcript_is_translated INTEGER,
  transcript_chars INTEGER,
  artifact_ids_json TEXT,
  created_at INTEGER NOT NULL DEFAULT (unixepoch() * 1000),
  UNIQUE (job_id, attempt_number, provider)
);

CREATE INDEX IF NOT EXISTS idx_transcript_attempts_job_created
  ON transcript_attempts(job_id, created_at);

CREATE INDEX IF NOT EXISTS idx_transcript_attempts_item_created
  ON transcript_attempts(item_id, created_at);
```

### Backfill Existing Items

The migration must enqueue already-broken YouTube items:

```sql
INSERT OR IGNORE INTO transcript_jobs (
  item_id,
  source_platform,
  video_id,
  state,
  priority,
  next_run_at
)
SELECT
  id,
  COALESCE(source_platform, source_type),
  NULL,
  'pending',
  10,
  unixepoch() * 1000
FROM items
WHERE
  (
    source_platform IN ('youtube', 'youtube_short')
    OR source_type = 'youtube'
  )
  AND (
    capture_quality = 'metadata_only'
    OR extraction_warning IN (
      'no_transcript',
      'youtube_transcript_fetch_metadata_only',
      'youtube_antibot_metadata_only'
    )
  );
```

Implementation should fill `video_id` opportunistically in application code by parsing `source_url`; the SQL migration should not depend on URL parsing.

```

Required functions:

- `enqueueTranscriptJobForItem(item, options)`
- `enqueueTranscriptJobForExistingYoutubeItem(itemId, reason)`
- `backfillTranscriptJobsForExistingYoutubeItems()`
- `getTranscriptJobForItem(itemId)`
- `listTranscriptJobsForReview(options)`
- `claimNextTranscriptJob(now)`
- `recordTranscriptAttempt(input)`
- `markTranscriptJobDone(jobId, attemptId)`
- `markTranscriptJobRetryable(jobId, attemptId, nextRunAt, error)`
- `markTranscriptJobManualNeeded(jobId, attemptId, error)`
- `retryTranscriptJobNow(itemId)`
- `ignoreTranscriptJob(itemId)`

Rules:

- Enqueue is idempotent by `UNIQUE(item_id)`.
- Retry-now clears `claimed_at`, sets `state = 'pending'`, sets `next_run_at = now`, and does not erase attempt history.
- Ignore sets `state = 'ignored'` and removes it from active Review lanes.
- Manual upgrade or successful provider recovery marks any active job as `done`.

```

Provider result shape:

```ts
type TranscriptProviderResult =
  | {
      ok: true;
      provider: string;
      transcript: string;
      language: string | null;
      isGenerated: boolean | null;
      isTranslated: boolean;
      artifacts?: CaptureArtifactCandidate[];
      metadata?: Record<string, unknown>;
    }
  | {
      ok: false;
      provider: string;
      retryable: boolean;
      errorCode:
        | "no_captions"
        | "timedtext_429"
        | "timedtext_http_error"
        | "timedtext_empty"
        | "antibot"
        | "video_unavailable"
        | "provider_timeout"
        | "provider_unavailable"
        | "provider_parse_error";
      errorMessage: string;
      statusCode?: number | null;
      artifacts?: CaptureArtifactCandidate[];
      metadata?: Record<string, unknown>;
    };
```

### v0.8.3 Providers

Only these are enabled in v0.8.3:

1. `innertube-ranked-timedtext`
2. `innertube-alt-client-timedtext` if the first provider gets retryable 429, timeout, empty timedtext, or client-specific block

### Track Ranking

Rank caption tracks in this order:

1. User configured language if available.
2. English manual captions.
3. English generated captions.
4. Hindi manual captions.
5. Hindi generated captions.
6. Any manual captions.
7. Any generated captions.
8. Translatable caption translated to English, only if provider supports it and provenance is stored.

Default language preference for v0.8.3:

```text
["en", "en-US", "en-IN", "hi"]
```

This can become configurable later.

### Provider Attempt Rules

- Try multiple tracks before declaring `no_captions` or `timedtext_empty`.
- Treat HTTP 429 as retryable.
- Treat live stream without captions as retryable until max attempts, then `manual_needed`.
- Treat private/deleted/unavailable as terminal.
- Treat anti-bot as retryable for early attempts, then `manual_needed`.
- Do not overwrite item body on provider failure.

## 8. Worker And Scheduling

Add:

```text
src/lib/queue/transcript-worker.ts
src/lib/queue/transcript-cron.ts
```

Register from `src/instrumentation.ts` only when:

```text
YOUTUBE_TRANSCRIPT_RECOVERY_ENABLED !== "false"
```

Recommended default:

```text
YOUTUBE_TRANSCRIPT_RECOVERY_ENABLED=true
YOUTUBE_TRANSCRIPT_WORKER_ENABLED=true
YOUTUBE_TRANSCRIPT_YTDLP_ENABLED=false
```

### Worker Pattern

Mirror `src/lib/queue/enrichment-worker.ts`:

- globalThis guard for exactly one worker per process;
- polling loop with idle backoff;
- stale-claim sweep;
- atomic claim;
- max attempts;
- JSONL error logging;
- no crash on one failed job.

### Timing

Backoff:

```text
attempt 1: now
attempt 2: +15 minutes
attempt 3: +2 hours
attempt 4: next nightly recovery window
attempt 5: next nightly recovery window
then: manual_needed
```

Stale claim:

```text
5 minutes
```

Concurrency:

```text
1 transcript job per process for v0.8.3
```

### Success Path

On successful transcript recovery:

1. Build updated YouTube content using the existing YouTube body format.
2. Use `upgradeItemCaptureContent` from v0.8.2, or an equivalent shared helper, to update the existing item.
3. Reset derived state:
   - summary;
   - quotes;
   - category;
   - enriched_at;
   - enrichment_state;
   - chunks;
   - embedding jobs;
   - auto-tags;
   - batch_id.
4. Save transcript artifacts.
5. Mark transcript job `done`.
6. Revalidate `/review`, `/items/[id]`, and `/`.

### Enrichment Timing Policy

For v0.8.3, accept initial metadata-only enrichment if it already happened, but make recovered transcript upgrades reset derived state exactly once.

Do not delay all enrichment in this release. That would add scheduler coupling. Instead:

- Review and item page must clearly show "Transcript recovery pending" even if enrichment says done.
- Recovered transcripts must requeue enrichment/search.

### Telegram New YouTube Capture

When `extractUrlCapture()` returns YouTube metadata-only:

1. Insert item as today.
2. Save capture artifacts as today.
3. Enqueue transcript job.
4. Send acknowledgement:

```text

Review or add notes:
https://brain.arunp.in/review?focus={itemId}
```

If transcript succeeds inline:

```text
Saved YouTube video with transcript: {title}
{itemUrl}
```

### Telegram Duplicate YouTube Capture

Current behavior returns early for duplicates. Change it:

If existing item is YouTube/Shorts and `capture_quality = metadata_only`:

1. Enqueue or retry the transcript job.
2. Return:

```text
Already captured, and transcript recovery is queued:
{reviewFocusUrl}
```

If existing item is full text/transcript:

```text
Already captured with transcript:
{itemUrl}
```

### API Capture Duplicate Path

Apply the same behavior in `/api/capture/url`:

- duplicate metadata-only YouTube returns existing item plus transcript job state;
- duplicate full-text YouTube remains duplicate/no-op.

### Manual Text Upgrade

If user pastes transcript or notes:

- keep v0.8.2 manual upgrade behavior;
- mark transcript job `done` or `ignored` with reason `manual_upgrade`;
- record a transcript attempt with provider `manual_user_text` and state `success`.

## 11. Artifact, Log, And Privacy Rules

### Attempt History

Attempt history lives in `transcript_attempts`.

Artifacts are optional supporting evidence, not the primary attempt log.

### Artifact Kinds

Allowed v0.8.3 transcript artifact kinds:

- `youtube_transcript_attempt_json`
- `youtube_timedtext_xml`
- `youtube_caption_track_json`

Future `yt-dlp` artifact kinds:

- `youtube_ytdlp_info_json`
- `youtube_ytdlp_stdout_txt`
- `youtube_ytdlp_stderr_txt`
- `youtube_ytdlp_subtitle_vtt`
- `youtube_ytdlp_subtitle_json3`

### Caps

Add or enforce caps:

```text
youtube_transcript_attempt_json: 128 KB
youtube_caption_track_json: 128 KB
youtube_timedtext_xml: 2 MB
youtube_ytdlp_stdout_txt: 32 KB
youtube_ytdlp_stderr_txt: 32 KB
youtube_ytdlp_subtitle_vtt: 2 MB
youtube_ytdlp_subtitle_json3: 2 MB
```

### Redaction

Add redaction for all text artifacts, not only HTML:

- access tokens;
- bearer tokens;
- cookies;
- signed URLs;
- query params named `token`, `signature`, `key`, `api_key`, `auth`, `expire`;
- local temp paths if they include sensitive user directories.

### Retention

For v0.8.3:

- keep at most 5 attempts per job by default;
- keep all `success` attempt artifacts;
- failed attempt artifacts can be capped/truncated but not silently dropped if they are needed to diagnose 429/anti-bot/provider errors;
- expose artifact write failures through attempt history.

## 12. `yt-dlp` Safety Contract For Future Phase

`yt-dlp` is not enabled by default in v0.8.3.

Before enabling:

1. Production host dependency decision is documented.
2. Binary/install source is pinned.
3. Provider is behind:

```text
YOUTUBE_TRANSCRIPT_YTDLP_ENABLED=true
```

4. Provider only accepts canonical YouTube watch/shorts URLs from the platform detector.
5. It rejects playlists, channels, non-YouTube hosts, and live streams.
6. It uses `child_process.spawn(file, args, { shell: false })`.
7. It never shells out through a command string.
8. It creates one isolated temp directory per attempt.
9. It validates every output path remains inside that temp directory.
10. It enforces timeout.
11. It caps stdout/stderr bytes.
12. It runs with concurrency 1 by default.
13. It uses subtitle-only options.
14. It does not pass cookies.
15. It does not download video or audio.
16. It cleans temp files in success, failure, and timeout paths.

Required tests before enabling:

- non-YouTube URL rejected;
- playlist URL rejected;
- shell metacharacters remain inert;
- timeout kills process;
- output outside temp directory rejected;
- stdout/stderr cap enforced;
- temp cleanup happens after success and failure.

## 16. Acceptance Criteria

### Product

- Telegram no longer tells the user to paste transcript as the first step for metadata-only YouTube.
- Metadata-only YouTube items show recovery status in Review and item detail.
- Existing metadata-only YouTube items are included after migration/backfill.
- Resending an already-captured metadata-only YouTube link queues or surfaces recovery.
- Manual paste remains available and marks recovery resolved.

### Data Integrity

- No duplicate item is created during transcript recovery.
- Successful recovery updates the existing item.
- Summary/search/enrichment state is reset on successful transcript upgrade.
- Attempt history survives process restart.
- Item delete removes jobs and attempts.

### Reliability

- 429 is retryable.
- Stale running jobs are resurrected.
- Terminal failures become `manual_needed`.
- Worker can be disabled by env flag.

### Safety

- `yt-dlp` is disabled unless explicitly enabled and tested.
- ASR is absent from v0.8.3.
- Transcript artifacts are capped and redacted.

### Observability

- Every enqueue, attempt, retry, manual-needed transition, and success emits a structured log.
- Review displays last provider, last error, attempt count, and next retry.

## 17. No-Go Gates

Do not ship v0.8.3 if any are true:

- Existing metadata-only YouTube rows are not backfilled.
- Duplicate metadata-only YouTube resends still return only "Already captured".
- Review cannot show transcript recovery state.
- Successful recovery does not reset stale summaries/chunks/embeddings.
- Attempt history is not durable.
- Worker has no stale-claim recovery.
- `yt-dlp` is enabled without subprocess safety tests.
- ASR code path is added without explicit separate approval.
- Implementation is done in the wrong baseline and loses v0.8.2 Review/upgrade behavior.

## 18. Rollout And Rollback

### Before Deploy

- Take production DB backup.
- Verify migrations locally against a copied DB when possible.
- Run tests and smokes.
- Confirm env flags:

```text
YOUTUBE_TRANSCRIPT_RECOVERY_ENABLED=true
YOUTUBE_TRANSCRIPT_WORKER_ENABLED=true
YOUTUBE_TRANSCRIPT_YTDLP_ENABLED=false
```

### Deploy

- Deploy with worker enabled only after migration succeeds.
- Watch logs for:
  - backfill count;
  - enqueue count;
  - attempt failures by error code;
  - manual-needed count.

### Rollback

Fast rollback:

```text
YOUTUBE_TRANSCRIPT_RECOVERY_ENABLED=false
YOUTUBE_TRANSCRIPT_WORKER_ENABLED=false
```

This stops new processing without deleting tables. Existing items remain safe because recovery updates are ordinary item upgrades.

DB rollback should not drop transcript tables unless restoring from backup. Keeping unused transcript tables is safer than destructive rollback.
