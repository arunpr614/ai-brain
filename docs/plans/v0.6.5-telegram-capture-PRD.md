## 1. Problem statement

Today AI Brain has three capture surfaces: the web app at `brain.arunp.in`, the Android APK (share-target), and the Chrome extension. All three require the user to actively switch context to "save this" — open the app, tap share, click the extension button.

Telegram is the user's most-used "throw it somewhere I'll deal with later" surface. Mobile chat is faster than mobile share-target on iOS (which the user does not have an APK for) and faster than browser tab-jockeying on the desktop. A Telegram bot would let the user forward a message, paste a link, or type a quick note and have it land in the library with no app switching.

## 2. Goal

Add a fourth capture surface: a Telegram bot at `@<bot-username>` (TBD-during-implementation) that captures any link, text note, or PDF the owner sends to it, and replies with an acknowledgement that confirms what was saved (or surfaces the error if it failed).

## 3. Success criteria

| # | Criterion | How verified |
|---|---|---|
| **SC-1** | Owner sends a URL to the bot. Item appears in `/library` within 5 seconds. Bot replies with the item's title + library link. | End-to-end smoke from owner's Telegram → web library |
| **SC-2** | Owner sends a forwarded Substack post (link + preview). Bot extracts the underlying URL and captures it. | Smoke with a real Substack forward |
| **SC-3** | Owner sends a plain-text note (no URL). Bot creates a Note item with that text as the body. | Smoke with text-only message |
| **SC-4** | Owner sends a PDF document. Bot uploads + extracts text via the existing PDF pipeline. | Smoke with a 1–10 MB PDF |
| **SC-5** | Owner sends a YouTube link. Bot captures via the existing YouTube transcript path. | Smoke with a known-good YouTube URL |
| **SC-6** | Any non-owner sends a message. Bot ignores it (no reply, no item created). | Probe from a second Telegram account |
| **SC-7** | Brain.service is down when a message arrives. Telegram retries the webhook automatically (built-in protocol behavior); when brain comes back, the message is delivered + captured. | Stop `brain.service`, send message, restart, observe capture. |
| **SC-8** | Webhook handler runs only after verifying the Telegram-issued secret-token header AND the sender's user ID. Any spoofed request is rejected with 401. | Curl probe with bad/missing token |

## 4. Non-goals

- **Photos / images.** Out of scope for v1. OCR is not in the system. Defer to a future patch when image-text extraction is on the roadmap.
- **Voice messages.** Out of scope. Whisper is not deployed yet (R-WHISPER spike not done; v0.10.0 lane).
- **Group / channel support.** Bot ignores messages from anything other than the owner's direct chat with it.
- **Multiple owners / multi-tenancy.** This is a single-user personal tool; the allowlist is hardcoded for one Telegram user ID.
- **Inline-mode `@bot query` queries.** Capture only. No retrieval / Ask path through Telegram in v1. (Could be a v0.6.6 follow-up.)
- **Two-way edits** ("/edit reply to update title"). Capture-only.
- **Tagging via bot commands** (`/tag startup foo`). Defer; existing library UI handles tagging.
- **Replies to bot ack messages should NOT mutate the item.** Bot only listens to top-level messages, not threaded replies — keeps the surface narrow.

## 5. User flow — happy path

1. Owner is on the phone, reads an interesting article in Telegram or a browser
2. Owner forwards / shares the link to `@<brain-bot>` chat
3. Within 1–2 seconds: bot replies "✅ Captured: '<article title>'  →  https://brain.arunp.in/items/<id>"
4. Owner taps the link if they want to read / tag now, or ignores and continues — item is in the library either way

For a plain-text note:
1. Owner types "remember to ask Jess about the offsite" into the bot
2. Bot replies "✅ Saved as note  →  https://brain.arunp.in/items/<id>"

For a PDF:
1. Owner taps the paperclip in the bot chat, picks a PDF
2. Bot replies "📄 Capturing PDF…" (because PDF extraction takes 3–10 s)
3. When extraction finishes: bot edits the message to "✅ Captured: '<extracted title>'  →  https://brain.arunp.in/items/<id>"

## 6. Failure modes (must handle)

| Mode | Behavior |
|---|---|
| Brain.service is down | Webhook returns 5xx. Telegram retries up to 24 hours (built-in). Owner sees no immediate reply, will see the ack once brain is back. |
| Brain is up but capture fails (bad URL, paywalled PDF, etc.) | Bot replies "⚠️ Couldn't capture: <reason>". Reason is the existing capture error message from the API. |
| Non-owner sends a message | Bot returns 200 silently. No reply, no log spam. Webhook handler tracks the count in a metric for ops visibility. |
| Spoofed webhook (no secret-token header) | Handler returns 401. No item created. Logged to `data/errors.jsonl`. |
| Telegram message > 4 KB (size limit on text) | Telegram itself truncates; not our problem. |
| PDF > 50 MB | Telegram itself rejects on send; not our problem. |
| YouTube link with no transcript | Bot replies "⚠️ Couldn't capture: this video has no transcript available". Existing YouTube error path. |
| Duplicate URL | Bot replies "↩️ Already captured  →  https://brain.arunp.in/items/<existing-id>". Same dedup as Chrome extension. |

## 7. Auth model — single-owner allowlist

**Hardcoded allowlist of one Telegram user.** Owner's identity:
- **Username:** `@Arun614`
- **Telegram user ID:** `943125412`
- **First name:** Arun
- **Last name:** Prakash

The handler verifies on every webhook call:
1. Telegram-issued `X-Telegram-Bot-Api-Secret-Token` header matches our pre-shared secret (defends against random POSTs to our webhook URL)
2. `message.from.id === 943125412` (defends against the case where someone discovers the bot username and sends it messages)

The user ID is the load-bearing check — usernames can be changed at any time by the user; numeric IDs are immutable per Telegram's design. Username is documented as a convenience for humans, not a security check.

If either check fails, the handler returns 200 (Telegram requires 2xx to stop retrying) but takes no other action. We don't return 401 for non-owner messages because that would cause Telegram to retry indefinitely; instead we silently accept-and-ignore.

The pre-shared secret-token is generated at first deploy and stored in `/etc/brain/.env` as `TELEGRAM_WEBHOOK_SECRET`.

## 8. Privacy

- **Bot token + webhook secret** live only in `/etc/brain/.env` on Hetzner. Same protection class as `BRAIN_API_TOKEN`.
- **Message content** transits Telegram → Cloudflare tunnel → brain.service → SQLite. Same trust boundary as the existing capture endpoints.
- **No third party** sees message content beyond Telegram itself (which already has it).
- **Capture history** is the same as today's library — visible only to authenticated brain sessions.

## 11. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Telegram bot token leaks via tracker docs / handover docs / git history | High | Token lives only in `/etc/brain/.env`. Public-repo audit (per HARDEN-HETZNER-SSH precedent) before merging any tracker entry that mentions the token. |
| Telegram changes webhook contract (rare but possible) | Low | Pin to a versioned API endpoint. Telegram has not made breaking changes in years. |
| Brain.service crashes specifically due to bad webhook payload | Medium | Wrap the handler in `try/catch` with `logError` + 200 return. Never let a malformed webhook take the service down. |
| Bot sends ack but item didn't actually save (write race) | Medium | Bot ack runs AFTER `INSERT INTO items` returns successfully. The dedup window (existing) handles double-sends. |
| Telegram rate-limits us | Low | Bot's reply rate is bounded by user's send rate; not a real concern for a personal-tool surface. |
| Webhook URL discovered by attacker | Low | Mitigated by the secret-token header check. URL alone is useless without the matching secret. |
