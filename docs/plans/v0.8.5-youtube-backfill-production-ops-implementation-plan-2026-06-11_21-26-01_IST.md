## 3. Guiding Rules

- Do not run real enqueue-mode backfill until the P0 runner is implemented, deployed, and dry-run verified.
- The default command must be dry-run.
- Real enqueue mode must require an explicit flag such as `--run`.
- Any real run must require a small explicit `--limit`.
- The runner must respect `provider_health.youtube_timedtext` unless the operator passes an explicit override.
- The runner must never duplicate active transcript jobs.
- The runner must skip terminal transcript jobs by default.
- The runner must write a clear machine-readable result summary.
- Clearing run summaries must never delete transcript jobs, transcript attempts, items, or capture artifacts.

## 4. P0 Workstream A - Production-Safe Backfill Runner

### Goal

Provide a command that can be run safely on Hetzner from `/opt/brain/scripts` without requiring the full TypeScript source tree or dev dependencies.

### Recommended Implementation

Create a production runner:

- `scripts/backfill-youtube-transcripts-prod.mjs`

Use this script for production instead of the existing TypeScript runner.

Why:

- Production uses the Next standalone artifact.
- Dev dependency `tsx` is not installed in production.
- The deployed artifact does not include the full source tree in a way that makes `scripts/backfill-youtube-transcripts.ts` reliable.
- A small production MJS script can use `better-sqlite3` directly, just like the existing production embedding backfill script pattern.

### Required CLI Shape

Default dry-run:

```bash
node /opt/brain/scripts/backfill-youtube-transcripts-prod.mjs --limit=25
```

Real bounded enqueue:

```bash
node /opt/brain/scripts/backfill-youtube-transcripts-prod.mjs --run --limit=5
```

Cooldown override, only for deliberate operator use:

```bash
node /opt/brain/scripts/backfill-youtube-transcripts-prod.mjs --run --limit=5 --ignore-cooldown
```

Inspect stored run summaries:

```bash
node /opt/brain/scripts/backfill-youtube-transcripts-prod.mjs --list-runs
```

Clear old stored run summaries:

```bash
node /opt/brain/scripts/backfill-youtube-transcripts-prod.mjs --clear-runs --older-than-days=30
```

### CLI Safety Requirements

The script must:

- Treat missing `--run` as dry-run.
- Reject `--run` if `--limit` is missing.
- Reject `--run` if `--limit` is greater than a small production cap, recommended `25`.
- Clamp dry-run limits to a safe cap, recommended `500`.
- Print exactly what DB path it is using before doing work.
- Print whether cooldown is active.
- Print whether it is a dry-run or real enqueue.
- Return non-zero on invalid flags.
- Never silently ignore malformed flags.

### Environment Requirements

The script should support:

- `BRAIN_DB_PATH`, if explicitly set.
- Otherwise default to `/opt/brain/data/brain.sqlite` when present.
- For local snapshot tests, allow:

```bash
BRAIN_DB_PATH=/tmp/brain-youtube-backfill-snapshot.sqlite node scripts/backfill-youtube-transcripts-prod.mjs --limit=5
```

### Backfill Selection Logic

The production script should mirror the app helper:

Scan items where:

- `source_platform IN ('youtube', 'youtube_short')`, or `source_type = 'youtube'`
- and either:
  - `capture_quality = 'metadata_only'`
  - or `extraction_warning IN ('no_transcript', 'youtube_transcript_fetch_metadata_only', 'youtube_antibot_metadata_only')`

Skip:

- `transcript_jobs.state IN ('pending', 'running', 'retryable_error')`
- `transcript_jobs.state IN ('manual_needed', 'ignored', 'done')`
- rows with no usable YouTube video id
- rows blocked by active provider cooldown, unless `--ignore-cooldown` is explicit

Enqueue only when:

- item is a recoverable YouTube weak capture
- no active job exists
- no terminal job exists
- cooldown is inactive or explicitly ignored
- `--run` is present

### Job Insert Behavior

Use the same semantic behavior as `enqueueTranscriptJobForItem`:

- `state = 'pending'`
- `priority = 10` by default
- `next_run_at = now`
- `attempts = 0`
- `claimed_at = NULL`
- `completed_at = NULL`
- `last_* = NULL`
- `ON CONFLICT(item_id)` must not duplicate jobs

For a first production version, prefer conservative behavior:

- Do not reset terminal jobs.
- Do not reset retryable jobs.
- Do not alter existing jobs unless the row is truly missing a job.

### Result Summary

The runner must output JSON containing at least:

```json
{
  "dryRun": true,
  "limit": 25,
  "dbPath": "/opt/brain/data/brain.sqlite",
  "scanned": 0,
  "eligible": 0,
  "enqueued": 0,
  "skippedExisting": 0,
  "skippedTerminal": 0,
  "skippedCooldown": 0,
  "skippedInvalidVideoId": 0,
  "cooldownActive": false,
  "cooldownUntil": null,
  "startedAt": "2026-06-11T15:56:01.000Z",
  "finishedAt": "2026-06-11T15:56:01.100Z"
}
```

### Run Summary Storage

Store operator run summaries under:

- `/opt/brain/data/operator-runs/youtube-transcript-backfill/`

Use timestamped JSON files:

- `2026-06-11T15-56-01-000Z-dry-run.json`
- `2026-06-11T15-58-22-000Z-run.json`

The summary file should contain:

- the JSON result
- the exact mode
- the limit
- whether cooldown was ignored
- host name if easy to include
- app version or git SHA if available

Do not store secrets, Telegram tokens, API tokens, pasted transcript text, or full item bodies.

### Clear Logs / Results

Add run-summary management to the same production script or a small sibling script.

Required behavior:

- `--list-runs` prints available stored run summaries.
- `--clear-runs --older-than-days=N` deletes only summary JSON files under the operator-run directory.
- `--clear-runs` without an age or exact run id should fail.
- The command must never delete:
  - `data/errors.jsonl`
  - `brain.sqlite`
  - `transcript_jobs`
  - `transcript_attempts`
  - capture artifacts

Acceptance wording for the operator:

> This clears saved backfill run summaries only. It does not clear transcript jobs, attempts, or item data.

- Preserve transcript attempts and logs for audit.

## 11. P1 Workstream H - Handle The 5 Existing Retryable Jobs

### Goal

Avoid guessing. Triage the existing retryable jobs and decide the right action for each.

### Triage Query

```bash
ssh brain "sqlite3 /opt/brain/data/brain.sqlite -json \"SELECT tj.id, tj.item_id, tj.video_id, tj.state, tj.attempts, tj.max_attempts, tj.next_run_at, tj.last_error_code, tj.last_error_message, i.title, i.source_url, i.capture_quality, i.extraction_warning FROM transcript_jobs tj JOIN items i ON i.id = tj.item_id ORDER BY tj.updated_at DESC;\""
```

### Decision Per Job

Classify each job as:

- Wait for cooldown/retry.
- Manual transcript or notes needed.
- Terminal because video is private/deleted/unavailable.
- Candidate for future provider fallback.
- Candidate for future ASR fallback.

### Acceptance Criteria

- Each of the 5 jobs has a clear next action.
- No job is blindly reset.
- No item is deleted as part of transcript triage.
- The result is captured in the running log or a small ops note.

## 12. P1 Workstream I - Daily / Batched Recovery Path

### Goal

Turn manual backfill into a quiet, auditable routine only after the production runner proves safe.

### Recommended Sequence

Phase 1: manual dry-run only.

- Operator runs dry-run on demand.
- No scheduled mutation.

Phase 2: scheduled dry-run.

- Add a daily dry-run schedule.
- Store run summaries.
- Alert or log if eligible count is greater than 0.

Phase 3: scheduled tiny real run.

- Only after dry-run has been stable.
- Use small cap, recommended `--run --limit=5`.
- Respect cooldown.
- Never use `--ignore-cooldown` in automation.

### Scheduling Options

Option A: cron, consistent with existing backup deploy pattern.

- Add `scripts/deploy/brain-youtube-transcript-backfill.cron`.
- Install manually or through a gated deploy step.
- Recommended initial schedule: daily dry-run.

Option B: systemd timer.

- Better status visibility.
- More setup.
- Use if the project starts accumulating more operator jobs.

Recommended first step:

- Use cron for consistency with the existing backup script.
- Keep the first scheduled mode dry-run.

### Daily Summary

Each scheduled run should write:

- run summary JSON
- one JSONL event in `data/errors.jsonl`
- clear console output for cron logs

### Acceptance Criteria

- Daily job respects provider cooldown.
- Daily job does not duplicate active jobs.
- Daily job has a small cap.
- Daily job has an audit trail.
- Operator can disable the job without changing app code.

## 18. Definition Of Done

This plan is done when:

- Production has a runnable YouTube backfill command under `/opt/brain/scripts`.
- The command defaults to dry-run.
- A real run requires `--run --limit=N`.
- Cooldown is respected by default.
- Duplicate and terminal transcript jobs are skipped.
- Run summaries are written and can be listed/cleared safely.
- Deploy wiring preserves the runner across releases.
- Local tests cover the safety boundaries.
- A production dry-run has been executed and recorded.
- A post-v0.8.4 Telegram send/resend smoke has passed.
- Provider cooldown/events have been inspected after at least one retry window.
- The 5 existing retryable jobs have been triaged.
