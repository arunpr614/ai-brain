**Created:** 2026-06-10 17:55:32 IST
**Author:** Codex
**Status:** revised implementation plan, replacing the earlier v0.8.0 planning starter

**Release objective:** upgrade weak captures safely, visibly, and measurably without duplicates or stale Ask/search state.

---

## 1. Executive Summary

The earlier v0.8.0 plan had the right product instinct but was not executable enough. This revised plan narrows v0.8.0 to one crisp release:

> If Brain saves only weak metadata or a preview, the user can upgrade that same item with pasted text, and Brain's Library, Ask, search, enrichment, artifacts, and production smoke checks all reflect the upgraded content.

This release should not chase deeper scraping. It should make the current save-first capture system trustworthy and upgradeable.

The highest-value missing behavior is **YouTube/Shorts pasted text upgrade**. LinkedIn pasted-text upgrade already mostly exists. The plan below keeps LinkedIn safe, finishes YouTube, adds downstream invalidation, adds a minimal Library review filter, tightens copy, and validates Substack classification without expanding scope into a new quality model.

---

## 2. Current Baseline

### Already Available

- Production Telegram captures save reliably instead of failing when YouTube blocks transcript extraction.
- YouTube and Shorts canonicalize to watch URLs and can save as `metadata_only`.
- LinkedIn link-only capture saves metadata without retaining raw LinkedIn HTML by default.
- LinkedIn link plus pasted text saves as `user_provided_full_text`.
- Existing `metadata_only` LinkedIn captures can be upgraded to pasted text via Telegram/API paths.
- Item detail pages show platform, quality, extraction metadata, and improvement hints.
- Library rows show platform and quality labels.
- Full-text search is automatically refreshed by SQLite triggers when `items.title` or `items.body` changes.
- Artifact storage is capped, uses relative paths, has write status/truncation metadata, and cleans up item artifacts on item delete.
- Build/deploy hygiene is currently healthy.

### Not Yet Available

- YouTube/Shorts pasted transcript or notes do not currently become captured content.
- A metadata-only YouTube item does not reliably upgrade when the same link is sent with pasted text.
- Upgrade behavior does not explicitly reset enrichment, chunks, embeddings, summaries, quotes, category, auto-tags, or job state.
- The web item page does not have a real "Add transcript or notes" action.
- The Library does not have a `Needs upgrade` filter.
- Upgrade decisions do not yet emit enough structured observability.
- Substack classification still needs fixture-backed validation before changing labels.

---

## 3. Non-Negotiable Release Principles

1. **Do not regress save reliability.** A blocked transcript still saves metadata.
2. **Do not create duplicate rows when the user is clearly upgrading a weak capture.**
3. **Do not let Ask/search use stale metadata-only chunks after an item is upgraded.**
4. **Do not imply pasted user text is an official transcript unless the user explicitly labels it that way.**
5. **Do not add LinkedIn scraping, LinkedIn-specific browser DOM extraction, or authenticated platform bypasses.**
6. **Do not add a new `long_preview` quality in v0.8.0 unless a separate quality-model migration is planned.**
7. **Do not ship a UI prompt for an action that does not actually exist.**

## 4. Scope Decision

### In Scope

- YouTube/Shorts pasted text upgrade.
- Preserve YouTube metadata during upgrade.
- Generic weak-capture upgrade decision matrix.
- Downstream derived-state reset after item body/title mutation.
- Telegram copy updates for weak captures and successful upgrades.
- Web item-page "Add transcript or notes" action for weak YouTube/Shorts and LinkedIn captures.
- Minimal Library filter:
  - `All`
  - `Needs upgrade`
  - `Full text`
  - `Metadata only`
- Structured logs for capture creation, duplicate prevention, upgrade rejection, and upgrade success.
- Production-safe synthetic smoke covering YouTube and LinkedIn upgrade paths.
- Fixture-backed Substack validation using current quality values only.

### Out Of Scope

- Browser extension selected-text capture.
- YouTube Data API expansion or new transcript providers.
- Jina/Firecrawl/Browserless integration.
- Substack email ingestion.
- Generic item revision history across all item types.
- Large capture dashboard.
- New `CaptureQuality` value such as `long_preview`.
- Server-side LinkedIn full-text scraping.

## 6. Product Behavior Requirements

### 6.1 YouTube/Shorts URL Only

When the user sends only a YouTube/Shorts URL:

- If transcript extraction succeeds, keep current behavior.
- If transcript extraction is blocked or unavailable, save the item as `metadata_only`.
- The Telegram acknowledgement must say exactly what happened and how to upgrade.

Telegram copy:

```text
Saved the YouTube link, but I could not read the transcript. To upgrade this item, send the same link again with the transcript or your notes pasted below it.
<item link>
```

### 6.2 YouTube/Shorts URL Plus Pasted Text

When the user sends a YouTube/Shorts URL plus meaningful pasted text:

- If a weak item with the same canonical URL exists, upgrade that item.
- If no item exists, create a new YouTube/Shorts item using the pasted text and available metadata.
- Do not run the normal YouTube transcript extractor as the only path when meaningful user text exists.
- Preserve paragraph breaks, bullet lines, and timestamp lines.
- Remove only the captured source URL and its canonical variant from the pasted text.
- Keep secondary URLs inside the pasted text.
- Store provenance as user-provided text, not official transcript.

Quality decision:

- Use `capture_quality = user_provided_full_text` for v0.8.0.
- Use `extraction_method = youtube_user_provided_text`.
- Keep platform as `youtube` or `youtube_short`.
- Keep source type as `youtube`.

Telegram upgrade copy:

```text
Updated the existing YouTube item with your pasted text.
<item link>
```

### 6.3 LinkedIn URL Only

When the user sends only a LinkedIn URL:

- Save metadata/preview only.
- Do not store raw HTML snapshots.
- Explain the safe upgrade path.

Telegram copy:

```text
Saved the LinkedIn link as a preview. To upgrade it, send the same link again with the post text pasted below it.
<item link>
```

### 6.4 LinkedIn URL Plus Pasted Text

Current behavior mostly works and must not regress:

- URL plus pasted text saves as `user_provided_full_text`.
- Existing weak LinkedIn captures upgrade instead of duplicating.
- Paragraphs, bullet lines, and secondary URLs remain intact.

### 6.5 Web Item Page Upgrade

Do not ship only a hint. Ship a real action.

For weak captures, show an "Add transcript or notes" control on the item detail page.

Eligible item states:

- `source_platform IN ('youtube', 'youtube_short', 'linkedin')`
- `capture_quality = 'metadata_only'`

Optional future eligible state:

- `source_platform = 'substack'`
- `capture_quality = 'paywall_preview'`

Web action requirements:

- Textarea label: `Transcript or notes`
- Helper copy: "Paste the content you want Brain to remember for this item."
- Minimum text rule: same as Telegram meaningful text rule, currently 8+ words after URL removal.
- Max text rule: 100,000 characters unless existing server limits require a lower value.
- Submit behavior: update existing item, reset downstream derived state, revalidate the item page and Library.
- Success state: show that the item was updated and enrichment/search are being refreshed.
- Error state: explain too-short text or ineligible item without technical terms.
- Mobile behavior: textarea and submit button must fit within the existing item detail layout.

## 7. Upgrade Decision Matrix

Create a single shared upgrade policy so API, Telegram, and web actions agree.

| Existing quality | Incoming content | Action |
|---|---|---|
| `metadata_only` | meaningful user text | Upgrade existing item |
| `paywall_preview` | meaningful user text or email body | Upgrade existing item only when explicitly supported |
| `transcript` | user notes | Do not overwrite in v0.8.0; return duplicate/existing item guidance |
| `metadata_plus_transcript` | user notes | Do not overwrite in v0.8.0; return duplicate/existing item guidance |
| `full_text` | user text | Do not overwrite; return existing item |
| `user_provided_full_text` | user text | Do not overwrite; return existing item |
| `failed` | meaningful user text | Create or upgrade only after explicit code path decision; not required for v0.8.0 |
| duplicate legacy rows | any | Choose newest canonical matching item and do not create another duplicate |

Shared helper recommendation:

```text
src/lib/capture/upgrade-policy.ts
```

Export:

- `classifyCaptureUpgrade(existing, incoming)`
- `isWeakCapture(item)`
- `isNeedsUpgrade(item)`

## 8. Content And Provenance Model

### 8.1 User-Provided Text Builder

Add a reusable builder for user-provided URL content.

Recommended file:

```text
src/lib/capture/user-provided.ts
```

Responsibilities:

- Normalize line endings.
- Preserve paragraph breaks.
- Preserve timestamp lines.
- Preserve bullets and secondary URLs.
- Remove only the raw source URL and canonical source URL.
- Enforce minimum meaningful text.
- Build a content body with a clear header:

```text
Title: <original title>
Source: <platform>
URL: <canonical URL>
Capture quality: user_provided_full_text
Provided by: user paste

Pasted text:
<text>
```

### 8.2 YouTube-Specific Builder

Recommended file:

```text
src/lib/capture/youtube-user-text.ts
```

Responsibilities:

- Accept canonical URL, detected platform, pasted text, and optional existing item.
- Preserve title, author/channel, duration, published date, thumbnail, and description from existing item where available.
- If no existing item exists, attempt cheap metadata extraction through existing YouTube metadata/oEmbed path, but do not fail the user-provided capture if metadata is unavailable.
- Return `CapturedContent` with:
  - `source_platform = 'youtube' | 'youtube_short'`
  - `capture_quality = 'user_provided_full_text'`
  - `extraction_method = 'youtube_user_provided_text'`
  - `extraction_warning = null`
  - artifact kind `user_provided_text`

### 8.3 Previous Content Preservation

Because upgrades mutate existing rows, preserve the previous weak item state before overwrite.

Use artifacts instead of adding a full revision table in v0.8.0.

Add artifact kind:

```text
pre_upgrade_item_json
```

Store:

- previous title
- previous body
- previous quality
- previous extraction method
- previous warning
- previous total chars
- previous timestamp of upgrade

This supports manual recovery without expanding v0.8.0 into generic revision history.

## 9. Derived-State Reset Requirement

This is a release blocker.

When an item body/title changes through an upgrade, Brain must not keep stale summaries, chunks, vectors, auto-tags, or job states.

Add a shared repository/helper:

```text
src/db/item-upgrades.ts
```

Recommended function:

```text
upgradeItemCaptureContent(input): ItemRow
```

It should perform the content update and derived-state reset in one transaction where practical.

Required behavior:

1. Save previous weak state as `pre_upgrade_item_json` artifact or prepare artifact candidate before/after transaction with failure logged.
2. Update item content and metadata.
3. Clear old enrichment output:
   - `summary = NULL`
   - `quotes = NULL`
   - `category = NULL`
   - `enriched_at = NULL`
   - `batch_id = NULL`
   - `enrichment_state = 'pending'`
4. Clear auto-generated tags for the item.
5. Re-arm enrichment job:
   - if row exists, set `state = 'pending'`, `attempts = 0`, `last_error = NULL`, `claimed_at = NULL`, `completed_at = NULL`
   - if missing, insert it
6. Clear old embedding/chunk state:
   - delete `chunks_vec` rows for the item's chunk rowids
   - delete `chunks` rows for the item
   - allow `chunks_rowid` to cascade, or delete explicitly if needed
   - delete existing `embedding_jobs` row so the enrichment-state transition to `done` can enqueue a fresh embedding job later
7. Rely on existing FTS trigger for `items.title/body` update.
8. Log one structured `capture.upgrade.completed` event.

Batch safety:

- Clearing `enrichment_state='batched'` and `batch_id` should make old batch poll results no-op because batch result writes require `enrichment_state = 'batched'`.
- Add a regression test proving a stale batch result cannot overwrite upgraded content.

## 12. Substack Classification Plan

Do not add `long_preview` in v0.8.0.

Reason:

- It would require quality type changes, UI labels, export changes, tests, and downstream behavior.
- The immediate release goal is safe upgrades, not quality taxonomy expansion.

Required Substack work for v0.8.0:

1. Add or update fixtures for:
   - known public full article
   - known paid/paywalled preview
   - ambiguous long public preview
2. Run current Substack classifier against fixtures.
3. Only change classifier logic if evidence shows current labels are wrong.
4. Keep values limited to existing qualities:
   - `full_text`
   - `paywall_preview`
   - `metadata_only`
5. Keep strong paywall markers conservative.
6. Document fixture result in the implementation report.

This makes Substack validation part of v0.8.0 without letting it dominate the release.

## 16. Acceptance Criteria

v0.8.0 is complete only when all criteria pass:

- Metadata-only YouTube can be upgraded from Telegram.
- Metadata-only YouTube can be upgraded from the web item page.
- Metadata-only LinkedIn continues to upgrade from Telegram/API.
- No duplicate item is created during the upgrade happy path.
- Strong captures are not overwritten by casual pasted text.
- Upgraded items preserve platform, source URL, title, thumbnail, duration, and description when available.
- User-provided text is clearly labeled as pasted/user-provided.
- Previous weak content is preserved as an artifact or equivalent recovery record.
- Enrichment summaries, quotes, category, auto-tags, chunks, vectors, and job state are reset correctly.
- Ask/search no longer rely on stale metadata-only chunks after upgrade.
- Library has a working `Needs upgrade` filter.
- Substack validation is documented and does not introduce an unplanned quality enum.
- Production smoke passes and cleans up after itself.

## 17. Rollback And Data Safety

### Before Production Deploy

- Create a fresh production DB backup.
- Confirm backup quick-check passes.
- Run full local verification.

### During Upgrade Mutations

- Only overwrite weak captures in v0.8.0:
  - `metadata_only`
  - `paywall_preview` only if explicitly supported
- Preserve previous content in `pre_upgrade_item_json`.
- Do not overwrite `full_text`, `transcript`, `metadata_plus_transcript`, or `user_provided_full_text`.

### If Production Smoke Fails

- Stop further testing.
- Use smoke IDs/source URLs to remove any partial smoke rows and artifacts.
- If user data was touched unexpectedly, restore from backup after confirming impact.
- Do not continue to new feature work until the upgrade path is repaired.

## 20. Final Recommendation

Execute this revised plan instead of the earlier v0.8.0 starter.

The first coding step should be the shared upgrade policy plus YouTube user-provided content builder. Do not touch UI first. The real foundation is making one canonical upgrade operation safe for API, Telegram, and web actions.
