## 1. Problem statement

Brain stores items with semantic embeddings (chunks_vec) and auto-generated tags. The only way the user currently sees connections between items is the per-item "Related" panel (`EXP-3`, shipped v0.4.0), which shows 5 nearest neighbors for the currently-open item. There's no way to see the library as a whole — no "show me the shape of my knowledge."

Recall.it ships a rich graph view (v2 audit catalogued 36 distinct capabilities). Obsidian has one. Brain doesn't.

## 2. Goal

One new page at `/graph`. Desktop only (≥1024px viewport, not inside APK). Renders every item as a node, every high-similarity pair as an edge, in a d3-force canvas simulation. Click a node → go to the item. Pan + zoom. Nothing more in MVP.

## 3. Scope — what's in the MVP

Per user decision 2026-05-12 (plan umbrella §D): **global graph only**. No local-neighborhood mode, no tag-cluster coloring, no filters.

## 4. Architecture

### 4.1 Data flow

```
Item captured → enriched → embedded (chunks_vec populated)
  ↓
embedItemWithRetry returns ok
  ↓
[NEW] updateEdgesForItem(item_id) called inline:
  ├─ findRelatedItems(item_id, { limit: 20, minSimilarity: 0.5 })  [existing]
  ├─ deleteEdgesForItem(item_id)
  └─ for each related: upsertEdge(sorted_pair, similarity)
  ↓
user navigates to /graph (desktop only)
  ↓
server component reads: listItems() + listAllEdges()
  ↓
client component receives props → d3-force simulation on canvas
  ↓
user clicks node → router.push(`/items/${id}`)
```

### 4.2 Schema (migration 009)

```sql
-- Undirected pairs: always store with source_item_id < target_item_id (lexicographic)
-- to guarantee uniqueness without double-rows.
CREATE TABLE edges (
  source_item_id TEXT NOT NULL,
  target_item_id TEXT NOT NULL,
  weight REAL NOT NULL CHECK (weight >= 0.0 AND weight <= 1.0),
  kind TEXT NOT NULL DEFAULT 'semantic',  -- future: 'manual', 'tag-shared', etc.
  created_at INTEGER NOT NULL,
  PRIMARY KEY (source_item_id, target_item_id),
  FOREIGN KEY (source_item_id) REFERENCES items(id) ON DELETE CASCADE,
  FOREIGN KEY (target_item_id) REFERENCES items(id) ON DELETE CASCADE,
  CHECK (source_item_id < target_item_id)  -- enforce sorted-pair invariant
);

CREATE INDEX idx_edges_source ON edges(source_item_id);
CREATE INDEX idx_edges_target ON edges(target_item_id);
CREATE INDEX idx_edges_weight ON edges(weight);
```

**Why sorted-pair invariant:** the similarity relation is undirected (item A related to B == B related to A). Storing both `(A, B)` and `(B, A)` doubles storage and complicates queries. Canonical ordering enforced by `CHECK (source_item_id < target_item_id)` — application layer sorts before insert.

**Why `FOREIGN KEY ... ON DELETE CASCADE`:** deleting an item must remove its edges. Matches existing cascade pattern in `item_tags` (migration 001).

### 4.3 Edge-maintenance hook

In `src/lib/embed/pipeline.ts`, after `embedItem()` returns `ok: true` inside `embedItemWithRetry`:

```ts
// After line ~145 (after successful embedItem return)
if (result.ok && result.chunk_count > 0) {
  try {
    await updateEdgesForItem(item_id);
  } catch (err) {
    // Non-fatal: graph will be stale for this item until next embed or backfill.
    console.warn(`[edges] updateEdgesForItem failed for ${item_id}:`, err);
  }
}
```

`updateEdgesForItem` signature:

```ts
export function updateEdgesForItem(item_id: string): void {
  const related = findRelatedItems(item_id, { limit: 20, poolSize: 60 });
  const db = getDb();
  const tx = db.transaction(() => {
    deleteEdgesForItem(item_id);
    for (const r of related) {
      if (r.similarity < 0.5) continue;
      const [a, b] = [item_id, r.item.id].sort();  // sorted-pair invariant
      upsertEdge(a, b, r.similarity, 'semantic');
    }
  });
  tx();
}
```

Performance: ~6ms per KNN call at 10k chunks (per R-VEC benchmark). Negligible added to the enrichment pipeline.

### 4.4 Backfill script

`scripts/backfill-edges.mjs`:

```ts
// Walk all items with enrichment_state = 'done' and chunks > 0
// For each: call updateEdgesForItem
// Progress log every 50 items
// BigInt-safe (per src/lib/related/index.ts:84 precedent)
```

Serial loop, no parallelism (better-sqlite3 is synchronous). At 1k items × ~6ms = ~6 seconds. At 10k items = ~60 seconds. Acceptable for a one-shot run.

### 4.5 Page + client component

`src/app/graph/page.tsx` (server):
- Reads `listItems({ limit: 10000 })` + `listAllEdges()`.
- Passes both as props to `<GraphClient nodes={items} edges={edges} />`.
- No client-detection in the server component (that's impossible for RSC).

`src/app/graph/graph-client.tsx` (client, `"use client"`):
- Detects APK: `window.Capacitor?.isNativePlatform()`. If true → renders `<GraphUnavailable reason="apk" />`.
- Detects viewport: CSS `@media (min-width: 1024px)` wrapper. Smaller screens → renders `<GraphUnavailable reason="viewport" />`.
- Otherwise: mounts canvas + d3-force simulation.

`<GraphUnavailable>` fallback copy (plain-language style from commit 59bba64):
- `apk` → "Open Brain on your Mac to see the graph."
- `viewport` → "The graph needs a bigger screen. Open Brain on a laptop or resize this window wider."

### 4.8 Rendering choice — canvas vs SVG

**Decision: canvas.** At 500+ nodes SVG redraws get slow; canvas stays ≥30fps through ~5k nodes with d3-force on M1 Pro.

Force simulation config:
- `forceManyBody().strength(-30)` — repulsion
- `forceLink(edges).distance(d => 100 * (1 - d.weight))` — stronger edges pull closer
- `forceCenter(width/2, height/2)` — anchor
- `forceCollide().radius(6)` — prevent node overlap
- Cooldown: simulation stops after 300 ticks or when alpha < 0.01

## 7. Risks + mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| d3-force adds ~57kB gzipped to bundle | Certain | Low | Lazy-load graph-client.tsx; bundle only on `/graph` route |
| Graph performance degrades at 5k+ nodes | Low (current library ~hundreds) | Medium | Canvas rendering; cooldown on alpha threshold; defer optimization until library grows |
| Edge backfill blocks for ~60s at 10k items | Low | Low | Serial script; progress logs; user runs once |
| Capacitor guard fails (new Capacitor version changes API) | Very low | High (graph leaks into APK) | Additional CSS viewport guard as second line of defense; APK WebView also enforces `min-width: 1024px` false since typical phone is ~400px |
| Migration 009 conflicts with Lane C migration 008 | Low | High | Lane C owns 008 per `DUAL-AGENT-HANDOFF-PLAN.md §1`; 009 is Lane L's first. Migration runner is additive; no collision at runtime |
| Edge maintenance adds ~6ms to every enrichment | Certain | Negligible | R-VEC benchmark shows <10ms headroom; not user-perceptible |

## 8. Non-goals (explicitly)
