## 1. Problem statement

Brain stores items with semantic embeddings (`chunks_vec`) and auto-generated tags. The only way the user currently sees connections between items is the per-item "Related" panel (`EXP-3`, shipped v0.4.0), which shows 5 nearest neighbors for the currently-open item. There is no way to see the library as a whole — no "show me the shape of my knowledge."

Recall.it ships a rich graph view (v2 audit catalogued 36 distinct capabilities). Obsidian has one. Logseq has one. Brain doesn't.

## 2. Goal

One new page at `/graph`. **Desktop only** (≥1024px viewport, not inside the APK WebView). Renders every item as a node, every top-K / high-similarity pair as an edge, in a **Sigma.js WebGL simulation**. Click a node → go to the item. Pan + zoom. Keyboard navigation + screen-reader text-list fallback. Nothing more in MVP.

## 3. Scope — what's in the MVP

Per user decision 2026-05-12 (plan umbrella §D): **global graph only**. No local-neighborhood mode, no tag-cluster coloring, no filters.

## 4. Architecture

### 4.1 Data flow

```
Item captured → enriched → embedded (chunks_vec populated)
  ↓
embedItemWithRetry returns ok
  ↓
[NEW] updateEdgesForItem(item_id) called inline:
  ├─ findRelatedItems(item_id, { limit: 20, poolSize: 60, minSimilarity: GRAPH_EDGE_MIN_SIM })
  ├─ take top-4 by similarity (v2.1)
  ├─ deleteEdgesForItem(item_id)
  └─ for each related: upsertEdge(sorted_pair, similarity)
  ↓
user navigates to /graph (desktop, ≥1024px, not APK)
  ↓
server component reads: listItems({ limit: 10000 }) + listAllEdges()
  ↓
client component:
  ├─ APK check: Capacitor.isNativePlatform() → <GraphUnavailable reason="apk" />
  ├─ Viewport check: CSS @media (min-width: 1024px) → <GraphUnavailable reason="viewport" />
  ├─ Accessibility toggle: pref=list → router.replace('/graph/list')
  └─ else: mount Sigma renderer with seeded forceatlas2
  ↓
user clicks node → router.push(`/items/${id}`)
user presses Tab → focus traversal via keyboard handler
```

### 4.2 Schema (migration 009)

```sql
-- 009_edges.sql — Undirected semantic edges between items
-- Sorted-pair invariant: source_item_id < target_item_id (CHECK enforced)

CREATE TABLE IF NOT EXISTS edges (
  source_item_id  TEXT NOT NULL,
  target_item_id  TEXT NOT NULL,
  weight          REAL NOT NULL CHECK (weight >= 0.0 AND weight <= 1.0),
  kind            TEXT NOT NULL DEFAULT 'semantic',
  created_at      INTEGER NOT NULL DEFAULT (unixepoch() * 1000),
  PRIMARY KEY (source_item_id, target_item_id),
  FOREIGN KEY (source_item_id) REFERENCES items(id) ON DELETE CASCADE,
  FOREIGN KEY (target_item_id) REFERENCES items(id) ON DELETE CASCADE,
  CHECK (source_item_id < target_item_id)
);

CREATE INDEX IF NOT EXISTS idx_edges_source ON edges(source_item_id);
CREATE INDEX IF NOT EXISTS idx_edges_target ON edges(target_item_id);
CREATE INDEX IF NOT EXISTS idx_edges_weight ON edges(weight DESC);
```

**Why sorted-pair invariant:** the similarity relation is undirected (A ↔ B == B ↔ A). Storing both rows doubles storage + complicates queries. Canonical ordering enforced by `CHECK`; application layer sorts before insert.

**Why `ON DELETE CASCADE`:** deleting an item must remove its edges. Matches existing cascade pattern in `item_tags` (migration 001).

**Why `weight DESC` index:** the most common read ("top-weighted edges for graph render") benefits from descending order.

### 4.3 Edge threshold — the basis

Two parameters drive which pairs become edges:

| Parameter | Default | Basis |
|---|---|---|
| `GRAPH_EDGE_MIN_SIM` | `0.75` (v2.1) | "Strongly related" threshold — sparser graph than 0.65, fewer false-positive connections. Below 0.5 is near-random; 0.5–0.65 weakly thematic; 0.65–0.75 topically coherent; ≥0.75 strongly related. User decision 2026-05-13 §0.1. |
| `GRAPH_EDGE_TOP_K` | `4` (v2.1) | Per-item neighbor cap. Slightly stricter than the Related panel's 5; prevents a single high-centrality item from saturating the graph with 50+ spoke edges. User decision §0.1. |

Both read from `process.env` in the backfill script and the pipeline hook. If unset, hardcoded defaults apply. Tuning is a user-visible knob for later iteration.

### 4.4 Edge-maintenance hook

In `src/lib/embed/pipeline.ts`, after `embedItem()` returns `ok: true` inside `embedItemWithRetry`:

```ts
// After line ~145 (after successful embedItem return)
if (result.ok && result.chunk_count > 0) {
  try {
    updateEdgesForItem(item_id);
  } catch (err) {
    // Non-fatal: graph will be stale for this item until next embed or backfill.
    console.warn(`[edges] updateEdgesForItem failed for ${item_id}:`, err);
  }
}
```

`updateEdgesForItem` signature (synchronous; better-sqlite3 is sync):

```ts
export function updateEdgesForItem(item_id: string): void {
  const minSim = Number(process.env.GRAPH_EDGE_MIN_SIM ?? 0.75);  // v2.1: 0.65 → 0.75
  const topK = Number(process.env.GRAPH_EDGE_TOP_K ?? 4);          // v2.1: 5 → 4

  const related = findRelatedItems(item_id, { limit: 20, poolSize: 60 })
    .filter(r => r.similarity >= minSim)
    .slice(0, topK);

  const db = getDb();
  const tx = db.transaction(() => {
    deleteEdgesForItem(item_id);
    for (const r of related) {
      const [a, b] = [item_id, r.item.id].sort();  // sorted-pair invariant
      upsertEdge(a, b, r.similarity, 'semantic');
    }
  });
  tx();
}
```

**Performance budget:** ~6ms per KNN call at 10k chunks (R-VEC benchmark). Inline cost on enrichment pipeline: ≤ 15ms including transaction overhead. Not user-perceptible.

### 4.6 Library stack — committed after GRAPH-4 benchmark

**Primary choice (subject to benchmark confirmation):** `sigma` + `graphology` + `@react-sigma/core` + `graphology-layout-forceatlas2`.

Evidence per `docs/research/graph-view-tooling.md`:
- `sigma@3.0.3`, MIT, 970 KB unpacked, last publish 2026-04-30. WebGL renderer. 50k+ node ceiling.
- `graphology@0.26.0`, MIT, 2.7 MB unpacked (tree-shakeable). Data structure + stdlib (includes layouts).
- `@react-sigma/core@5.0.6`, MIT, 147 KB unpacked. Thin React wrapper for Sigma v3.
- `graphology-layout-forceatlas2` — deterministic with seed, WebGL-friendly, converges faster than d3-force on large graphs.

**Why not d3-force (v1 pick):** last publish 2022-06-14, unmaintained, 1–2k node Canvas ceiling, no built-in hit-testing or pan/zoom (would need `d3-zoom` + `d3-drag`, both also unmaintained).

**Why not reagraph (rank 2 in research doc):** less proven at scale; secondary choice if Sigma fails the benchmark.

**Why not cosmograph (rank 4 in research doc):** CC-BY-NC-4.0 license blocks non-personal use. Disqualified.

**Benchmark gate (GRAPH-4):** 3-way prototype at `tmp/graph-spike/{d3,sigma,reagraph}/` with **500 + 1,000 node fixtures (v2.1: optimized for ≤500 items, 1,000 is the 2× growth stress test)**. Metrics: cold render < 0.8s @ 500, ≥ 45 FPS pan/zoom @ 500, hit-test < 50ms, bundle cost. Results land in `docs/research/graph-view-benchmark-<date>.md`.

**Failure path (v2.1 user decision §0.1 — Option A):** if Sigma misses any §6 perf gate at 500 nodes, **block GRAPH-7 execution entirely**. Halt; re-evaluate library choice; possibly revert to reagraph or write a new research doc. **No "ship anyway with footnote" path.** Plan v2 does not assume the benchmark's outcome.

### 4.7 Page + client components

`src/app/graph/page.tsx` (server):
- Reads `listItems({ limit: 10000 })` + `listAllEdges()`.
- Passes both as typed props: `<GraphClient nodes={items} edges={edges} />`.
- No client-detection (impossible from RSC).

`src/app/graph/graph-client.tsx` (client, `"use client"`):
- APK detect: `window.Capacitor?.isNativePlatform()` → render `<GraphUnavailable reason="apk" />`.
- Viewport detect: CSS `@media (min-width: 1024px)` wrapper → smaller screens render `<GraphUnavailable reason="viewport" />`.

`src/app/graph/graph-canvas.tsx` (client, lazy):
- Builds `graphology` graph from props.
- Applies seeded `forceatlas2` layout (seed derived from hashing all item IDs in lexicographic order — deterministic per library state).
- Reads cached positions from `localStorage.graph-layout-v1` on mount; writes back on `stable` event.
- Mounts `<SigmaContainer>` from `@react-sigma/core`.
- **Tag-cluster coloring (GRAPH-9R):** for each node, look up its dominant `item_tags` row (highest-weighted manual tag, falling back to first auto tag). Map each distinct tag to one of 10 palette hues (deterministic hash → palette index); 11+ tags collapse into "other" (neutral grey). Color applied via Sigma's per-node `color` attribute.
- **Hover preview cards (GRAPH-10R):** on `enterNode` event, after a 250 ms hover-delay timer, render a floating card with title + summary (first ~120 chars) + thumbnail (favicon for URLs, generic icon for note/PDF). Card lives in a separate React portal so Sigma's WebGL canvas doesn't fight CSS for layering.
- **Wires click handler** → `router.push(/items/${nodeId})`.

`src/app/graph/graph-search.tsx` (GRAPH-11R, client):
- Floating search input top-right of the canvas.
- Fuzzy match (substring + case-insensitive) on `title` of every node currently in the graph.
- On match: dim non-matches (set `alpha=0.15` via Sigma's reducer API), highlight matches; pan camera to first hit; Enter cycles to next.
- Esc clears search; Sigma reducer reset.

### 4.8 Layout stability (GRAPH-10)

Force-directed layouts drift between reloads unless seeded. Two-layer strategy:

1. **Seed the forceatlas2 layout** with a deterministic seed = hash of sorted item IDs. Same library state → same layout.
2. **Cache computed positions** in `localStorage.graph-layout-v1` keyed by library-state hash. On reload, if hash matches, skip simulation and render directly from cache. Saves ~1.5s cold-load on stable libraries.

Invalidation: when library hash changes (new item added or deleted), cache is discarded; layout re-runs with new seed.
