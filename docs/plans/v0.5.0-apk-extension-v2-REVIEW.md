## Overall Verdict

**PROCEED-WITH-CHANGES.** The plan is structurally sound and well-researched: locked decisions are correctly derived from the critique and spikes, the dependency wave ordering is valid for the major flows, and the threat model adequately covers the pivoted attack surface for a personal-use app. However, five HIGH findings must be resolved before execution begins.

The single most critical: T-CF-9 acceptance criterion 4 (line 252) commands `curl ... Authorization: Bearer <token>` against `/api/settings/lan-info`, but that route is cookie-gated only and is not in `BEARER_ROUTES` — the curl test will return 401 and the criterion will always fail. Additionally, no T-CF-* task owns creation of `scripts/smoke-sse.sh` (referenced by T-CF-11, §12, and §7 but absent from every task's Files list), and the `parseSetupUri` return-type change in T-CF-4 will silently break `setup-apk/page.tsx` unless T-CF-6 also updates the `parsed.ok` check. These are execution-blocking.

**Findings: 5 HIGH, 5 MEDIUM, 5 LOW.**

## 2. Task Acceptance Criteria Realism — **HIGH** (3 findings)

### AC-1 (HIGH): T-CF-9 acceptance criterion 4 is unfalsifiable

**Plan line 252:**
```
4. curl https://brain.arunp.in/api/settings/lan-info -H "Authorization: Bearer <token>"
   returns {"url":"https://brain.arunp.in","token":"<token>"} (no ip field).
```

`/api/settings/lan-info` is NOT in `BEARER_ROUTES` (verified: `src/lib/auth/bearer.ts` lines 67–75 list only `/api/capture/url`, `/api/capture/pdf`, `/api/capture/note`, `/api/items`, `/api/health`, `/api/errors/client`). The route is cookie-gated (`src/app/api/settings/lan-info/route.ts` line 38: `if (!req.cookies.get(SESSION_COOKIE)?.value) return unauth()`). A bearer-token curl returns 401, not 200. This acceptance criterion will always fail.

**Same error propagates to T-CF-23 assertion 5** (line 513): `GET /api/settings/lan-info with bearer → 200` will always return 401 from the automated smoke script.

**Fix:** Change criterion 4 to use a cookie-based curl (e.g., `curl -b "brain-session=<value>" ...`) or add a note that manual browser verification is the gate. Alternatively, if the plan intends to add `lan-info` to `BEARER_ROUTES`, that must be an explicit task edit to `bearer.ts`.

### AC-2 (HIGH): T-CF-4 return-type change will break `setup-apk/page.tsx`

**Plan lines 179–181** specify `parseSetupUri` returning `{kind: 'ok', ...}` / `{kind: 'invalid', ...}`. The existing `SetupUriVerdict` type is `{ok: true, ip, token} | {ok: false, reason}` (verified in `src/lib/lan/setup-uri.ts` line 28). The caller in `setup-apk/page.tsx` line 49 checks `if (!parsed.ok)`. If T-CF-4 changes the discriminant from `ok: boolean` to `kind: string`, `setup-apk/page.tsx` fails `tsc --noEmit` immediately.

T-CF-4's Files list (line 174) includes only `setup-uri.ts` and its test — not `setup-apk/page.tsx`. T-CF-6 owns `setup-apk/page.tsx` changes but its scope is limited to the reachability probe wiring.

**Fix:** Either (a) keep the discriminant as `ok: boolean` in T-CF-4 (change `ip` → `url` field, keep `ok` semantics) and update acceptance-criteria notation to match, or (b) explicitly add `setup-apk/page.tsx` to T-CF-4's Files list with the instruction to update `if (!parsed.ok)` → `if (parsed.kind !== 'ok')`. **Option (a) is lower risk.**

### AC-3 (HIGH): `scripts/smoke-sse.sh` has no owning T-CF-* task

`scripts/smoke-sse.sh` appears in:
- §7 with full content (lines 575–621)
- T-CF-11 step 10: "Run SSE smoke (see §7)" (line 294) — used but not created
- §12 Definition of Done (line 801) — required gate
- T-CF-23 assertion 9 (line 517) — referenced as "passed"
- `package.json` `smoke:sse` wiring (line 621)

No T-CF-* task lists `scripts/smoke-sse.sh` in its **Files** list as a file to create. T-CF-23's Files list (line 520) names only `scripts/smoke-v0.5.0.mjs` and `package.json`.

**Fix:** Add `scripts/smoke-sse.sh` (NEW) and `package.json` (add `smoke:sse` script) to T-CF-11's Files list. T-CF-11 is the first task to run the smoke, so it's the logical owner.

### AC-4 (LOW): `smoke-sse.sh` ignores its own `BRAIN_TOKEN` env var

**Plan lines 594–595:** Script declares `: "${BRAIN_TOKEN:?required ...}"` — aborts if unset. But script body (line 605) uses `Cookie: brain-session=<obtain-cookie-from-browser-session>` — a hardcoded placeholder. `$BRAIN_TOKEN` is never used. The script is internally inconsistent.

**Fix:** Remove the guard (since `/api/ask` is cookie-only per CF-D-08) or replace with an explanatory comment.

## 5. Threat Model Coverage — **MEDIUM**

The STRIDE table at §10 is adequate for a personal-use app. Three gaps:

### TM-1 (MEDIUM): Cloudflare edge caching of token-bearing QR response

The `/api/settings/lan-info` route returns a QR PNG as a data URI containing the bearer token. While the route sets `Cache-Control: no-store, no-cache, must-revalidate`, Cloudflare's edge behavior for `no-store` responses is not verified. If cached, a token rotation followed by a QR scan could deliver a stale QR. Threat register §10 has nothing about this.

**Fix:** Add STRIDE row T-CF-I-04: "Info Disclosure: Cloudflare edge caches token-containing QR response" → mitigation: `cache-control: no-store` + `Pragma: no-cache` already in place; `force-dynamic` on the route ensures no static generation.

### TM-2 (LOW): DDoS via unauthenticated path scanning

Public IPs can now probe `https://brain.arunp.in`. Paths like `/unlock`, `/setup`, `/offline.html` are intentionally open. Threat register §10 has T-CF-D-01 (rate limit) but only for authenticated routes. A PIN brute-force against `/unlock` is not rate-limited. Not blocking for v0.5.0 (personal use) but should be acknowledged.

### TM-3 (LOW): Cloudflare free-tier WAF characterization

Line 748: "free tier has no WAF rules" understates actual defense posture. Free tier includes DDoS protection and baseline bot detection. Minor wording issue.
