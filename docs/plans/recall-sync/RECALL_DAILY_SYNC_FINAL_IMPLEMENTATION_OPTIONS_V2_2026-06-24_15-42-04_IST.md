## Option A - REST API Daily Pull

### Product Shape

A daily server-side job lists new Recall cards by date window, fetches card detail chunks, imports eligible content into AI Brain with Recall provenance, and lets the existing enrichment and embedding queues process the item.

### Production Flow

```text
systemd timer
  -> scripts/recall-scheduled-apply.sh
  -> scripts/sync-recall-prod.mjs
  -> Recall REST /cards date-window enumeration
  -> Recall REST /cards/{card_id}?max_chunks=50
  -> Recall mapper and fidelity classifier
  -> Recall importer
  -> items, recall_sync_items, recall_sync_runs, recall_sync_state
  -> enrichment_jobs and embedding_jobs
  -> Library, Ask, Search
```

### Why This Is Still The Primary Recommendation

- It is the only option that fits an unattended daily job cleanly.
- It uses server-side API-key auth rather than browser state.
- It has first-class local idempotency by Recall card ID.
- It can run dry-run-first with strict import caps.
- It refuses to proceed when Recall `total_count` indicates the returned card list is capped or inconsistent.
- It has proof gates before live API, production dry-run, production apply, deploy, and scheduler enablement.
- It maps naturally into AI Brain's existing capture/enrichment model.

### Remaining Risks

- Live `/cards` enumeration may miss cards or expose pagination/cap behavior that the docs do not describe.
- Card detail chunks may be summary-only, metadata-only, or truncated for long PDFs/videos.
- Recall is read-only, so V1 cannot propagate AI Brain edits back to Recall.
- No webhook is documented, so "whenever I add content" means "the next scheduled pull imports it," not instant import.

### Current Verdict

Proceed to approved live validation. Do not deploy or enable scheduling yet.

## Option B - REST Daily Pull With Weak-Item Upgrade

### Product Shape

Option B extends Option A. If a Recall card has an exact source URL match to an existing weak AI Brain item, the job can repair the weak item instead of creating a duplicate Recall item.

### Current Implementation State

The exact-URL weak-upgrade path is implemented and proven in synthetic tests, but it remains disabled by default.

Default policy:

```text
--allow-weak-upgrade-by-url omitted
BRAIN_RECALL_ALLOW_WEAK_UPGRADE_BY_URL=0
```

Opt-in policy after approval:

```text
--allow-weak-upgrade-by-url
BRAIN_RECALL_ALLOW_WEAK_UPGRADE_BY_URL=1
```

### Why It Should Stay Out Of V1 Default Apply

- Exact URL matching is safe but conservative.
- Non-exact canonical URL matching needs a separate PRD.
- Weak upgrades depend more heavily on live chunk fidelity because the system is replacing weak local content.
- The UI does not yet have a dedicated "Upgraded by Recall" badge or review queue.

### Current Verdict

Keep as a V1.1/V2 opt-in. Use live dry-run output to inspect upgrade candidates before enabling.

## Option C - MCP-Assisted Pull

### Product Shape

Use Recall MCP as a human-operated or agent-assisted fallback to inspect, reconcile, or retrieve content from Recall.

### Why It Is Not V1 Scheduler Path

- MCP auth is browser/OAuth based.
- It is suitable for assistants and interactive clients, not obviously for a headless Hetzner timer.
- Its outputs are useful for research and reconciliation, but less attractive as a stable production import contract than REST.

### Current Verdict

Keep as fallback/research path if REST live enumeration or fidelity fails.

## Option D - Markdown Export Reconciliation

### Product Shape

Use Recall's Markdown export for one-time bootstrap, reconciliation, or repair of cards where REST chunks are insufficient.

### Why It Is Not V1 Scheduler Path

- Export is a web-app flow, not a documented headless daily API.
- Full knowledge-base export can create high-risk local private files.
- Stable mapping back to Recall card IDs is not guaranteed from the current docs alone.

### Current Verdict

Keep as bootstrap/reconciliation fallback, not the daily automation path.

## Option E - Browser Automation

### Product Shape

Use Chrome or Playwright against the Recall web app to discover/export new cards.

### Why It Is Last Resort

- It is fragile against UI changes.
- It requires session/profile handling.
- It has the highest privacy and operational risk.
- It is harder to deploy safely on the production host.

### Current Verdict

Only consider if REST, MCP, and export paths fail to satisfy the live account requirements.

10. Only after repeated clean manual runs, deploy and explicitly enable the disabled scheduler.

## Final Decision For The Product

The feature should ship as a cautious, one-way Recall snapshot importer:

- Daily scheduled pull, not instant webhook.
- Snapshot import, not two-way sync.
- Recall card ID is the idempotency anchor.
- Content fidelity is visible and policy-relevant.
- Long or uncertain content is blocked or reviewed rather than silently treated as complete.
- First production write requires a reviewed dry-run and backup proof.
- Scheduler remains disabled until live evidence and manual apply have both succeeded.

The final product option is therefore:

```text
Option A for V1, with Option B as a disabled opt-in after live fidelity proof.
```

The goal is not complete yet because the live Recall account, production dry-run/apply, production deploy, and scheduler verification remain unexecuted by design.
