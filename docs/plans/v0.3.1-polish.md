### T-3. F-302 — Inline tag editor on item detail (S)

- **Files touched:**
  - `src/app/items/[id]/page.tsx` — replace the read-only tag pill row with a new client component.
  - `src/components/tag-editor.tsx` **(new)** — mirrors `CollectionEditor`'s shape. Form with `<input name="tag_name">` + submit; existing tags render as pills with × form buttons.
- **APIs reused:**
  - `listTagsForItem(itemId)` → `src/db/tags.ts`
  - `addTagToItemAction` / `removeTagFromItemAction` → `src/app/taxonomy-actions.ts` (already revalidate `/items/[id]`)
- **UI contract:**
  - Keep the existing tag pill style for consistency (`rounded-sm border …`).
  - Auto-tags (kind = `auto`) should keep their current distinct subtle style; manual tags get a stronger border. Include this styling cue in the component so we can see at a glance which tags the LLM proposed.
  - Input is a single-line text field with `maxlength=60` (matches the zod `max(60)` in `taxonomy-actions.ts`).
  - Enter submits; blank submissions are rejected client-side before the form action runs.
- **Edge cases:**
  - Submitting a tag that already exists on the item → `attachTagToItem` is idempotent via the unique constraint; rely on the DB, don't double-check client-side.
  - Submitting a tag name with only whitespace → client-side `trim()` and reject if empty. Server-side zod already enforces `min(1)`.
- **Test plan:** Add a tag → verify pill renders → remove it → add a duplicate of an existing auto-tag (should promote/merge safely).
- **Commit:** `feat(v0.3.1, F-302): inline tag editor on item detail page`

### T-4. B-301 — Title de-hyphenation post-processor (S)

**Why:** the fix lives in one function; small unit-testable surface.

- **Files touched:**
  - `src/lib/enrich/pipeline.ts` — add `postProcessTitle(raw: string): string` helper. Call it on `output.title` before the DB `UPDATE items` write.
  - `src/lib/enrich/pipeline.test.ts` **(new — first test file in the project)** — Node's built-in `node:test` for parity with the existing zero-dep posture. If that feels heavier than the project wants, inline an assertion block in a `scripts/test-post-process-title.ts` one-shot; `docs/plans/v0.3.1-polish.md` records either choice.
- **Heuristic (proposed):**
  - If the title contains **more hyphens than spaces** AND every hyphen is surrounded by non-space on both sides, treat it as a slug.
  - Replace hyphens with spaces, then Title Case each word (capitalize first letter of each word except common small words: `a, an, and, or, the, of, in, on, for, to, with, at, by, but, vs`).
  - Preserve original if no hyphens at all.
- **Test cases:**
  | Input | Expected |
  |---|---|
  | `Growth-Loops-Messy-Draft` | `Growth Loops Messy Draft` |
  | `Why-the-Best-PMs-Say-No` | `Why the Best PMs Say No` |
  | `Long-term thinking` | `Long-term thinking` (only one hyphen, more spaces) |
  | `State-of-the-Art 2026` | `State-of-the-Art 2026` (hyphens ≤ spaces after counting the space) — verify exact rule; may need tweak |
  | `Already Clean Title` | `Already Clean Title` |
  | `HYPHENATED-ALL-CAPS` | `Hyphenated All Caps` |
- **Edge case:** if the rewrite produces a title identical to the item's original filename-derived title, that's fine — idempotent.
- **Commit:** `fix(v0.3.1, B-301): de-hyphenate slug-shaped enrichment titles`

### T-5. F-207 — Library bulk-select UI + batch ops (M)

**Why last:** biggest change; everything else stabilizes the ergonomics it depends on (tag + collection editing).

#### T-5a. UI affordance
- **Files touched:**
  - `src/app/page.tsx` — convert the `<li>` rows into a client-component wrapper that tracks a `Set<string>` of selected ids in local state.
  - `src/components/library-bulk-bar.tsx` **(new)** — floating action bar that appears when `selectedIds.size > 0`, with three buttons: **Tag**, **Add to collection**, **Delete**, plus a **Clear** button. Count indicator shows `{n} selected`.
- **Selection UX:**
  - Checkbox at the leading edge of each row (hidden until hover OR until at least one item is selected — "quiet" by default).
  - Click on the card still navigates to the item; clicking the checkbox does not.
  - Shift-click should extend a contiguous range (nice-to-have; skip if it adds more than ~20 lines).
  - `Esc` clears selection.

#### T-5b. Batch server actions (`src/app/actions.ts`)
- `bulkTagItemsAction(formData)` — accepts `item_ids: string[]`, `tag_name: string`; reuses `upsertTag("manual")` + `attachTagToItem` in a transaction.
- `bulkAttachCollectionAction(formData)` — accepts `item_ids: string[]`, `collection_id: string`; reuses `attachItemToCollection` in a transaction.
- `bulkDeleteItemsAction(formData)` — accepts `item_ids: string[]`; reuses `deleteItem` in a transaction.
- All three `revalidatePath("/")` on success.
- Use a shared zod schema: `z.object({ item_ids: z.array(z.string().min(1)).min(1).max(500) })`. The 500 cap guards against malformed requests. Larger batches are a later concern.

#### T-5c. Confirm flows
- Delete must show a browser `confirm("Delete N items? This cannot be undone.")` client-side before submit. Matches the single-item delete pattern.
- Tag + collection batch ops should show a transient toast-style confirmation on success ("Tagged 3 items with `product`"). If there's no toast infrastructure yet, a simple `<div role="status">` that auto-dismisses after 2s is fine — don't introduce a toast library just for this.

#### T-5d. Test plan
- Select 3 items → click Tag → enter a new name → all 3 acquire the tag → verify on their detail pages.
- Select 3 items → click Add to collection → pick one → all 3 appear under `/collections/[id]`.
- Select 1 item → click Delete → cancel the confirm → item still present.
- Select 2 items → click Delete → confirm → both gone from library and from collections/tags.

- **Commit sequence:**
  1. `feat(v0.3.1, F-207a): selection state + bulk bar on library page`
  2. `feat(v0.3.1, F-207b): bulk tag/collection/delete server actions`
  3. `feat(v0.3.1, F-207c): wire bulk actions to bulk bar + confirm dialogs`

Three commits let a bisect land on the smallest failing surface if anything regresses.

## 5. Risk + mitigation

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Bulk-select client state breaks library server rendering | M | M | Keep the `<li>` server-rendered; extract only the checkbox + bulk-bar into a thin client boundary. |
| Title de-hyphenation fires on genuinely hyphenated titles (`State-of-the-Art`) | M | L | Heuristic requires **more hyphens than spaces**. Add test cases for false-positives; refine rule if one slips through. |
| Bulk-delete foot-gun wipes a batch by accident | L | H | Hard-require a `confirm()` with the count; no auto-submit on checkbox keyboard events. |
| Inline tag editor duplicates tags already proposed by the LLM | M | L | `upsertTag` + unique constraint silently dedupes; acceptable. |
| Turbopack cache drift during dev | L | L | Known from handover §M9. Kill `.next` + restart if class names mutate stale. |
