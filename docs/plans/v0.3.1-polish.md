### T-B-4. B-301 — Title de-hyphenation post-processor (S)

**Heuristic tightened per critique P-1** — the original "more hyphens than spaces" rule misfired on compound adjectives like `State-of-the-Art 2026`. New rule:

- **Fire only if** the title contains **zero spaces** AND at least one hyphen (pure slug, e.g. `Growth-Loops-Messy-Draft`), **OR** the title has `hyphens > 2 × spaces` (strong dominance).
- **Do not fire** on `State-of-the-Art 2026` (3 hyphens, 1 space → ratio is 3:1 which is ≤ 2× threshold? **No — 3 > 2×1**. So we need a stricter rule.)

**Resolved rule (v2):** fire only if **zero spaces and ≥ 2 hyphens**. Drop the ratio heuristic entirely — it introduces too many edge cases. Compound-adjective titles retain their hyphens because they always contain a space somewhere.

- **Files touched:**
  - `src/lib/enrich/pipeline.ts` — add `postProcessTitle(raw: string): string`; call on `output.title` before the DB write.
  - `src/lib/enrich/pipeline.test.ts` **(new)** — uses the `node:test` runner established by T-A-7.
- **Test cases (must pass):**
  | Input | Expected | Rule path |
  |---|---|---|
  | `Growth-Loops-Messy-Draft` | `Growth Loops Messy Draft` | fire (0 spaces, 3 hyphens) |
  | `Why-the-Best-PMs-Say-No` | `Why the Best PMs Say No` | fire (0 spaces, 5 hyphens) |
  | `State-of-the-Art 2026` | `State-of-the-Art 2026` | do not fire (contains space) |
  | `Long-term thinking` | `Long-term thinking` | do not fire (contains space) |
  | `Already Clean Title` | `Already Clean Title` | do not fire (0 hyphens) |
  | `HYPHENATED-ALL-CAPS` | `Hyphenated All Caps` | fire (0 spaces, 2 hyphens) |
  | `single-word` | `single-word` | do not fire (only 1 hyphen) |
- **Title-casing:** preserve small-word lowercasing for `a, an, and, or, the, of, in, on, for, to, with, at, by, but, vs` unless they're the first word.
- **Commit:** `fix(v0.3.1, B-301): de-hyphenate slug-shaped enrichment titles`

### T-B-5. F-207 — Library bulk-select UI + batch ops (M)

**Why last:** biggest change; everything else stabilizes the ergonomics it depends on (tag + collection editing).

#### T-B-5a. UI affordance
- **Files touched:**
  - `src/app/page.tsx` — convert the `<li>` rows into a client-component wrapper that tracks a `Set<string>` of selected ids in local state.
  - `src/components/library-bulk-bar.tsx` **(new)** — floating action bar that appears when `selectedIds.size > 0`, with three buttons: **Tag**, **Add to collection**, **Delete**, plus a **Clear** button. Count indicator shows `{n} selected`.
- **Selection UX:**
  - Checkbox at the leading edge of each row (hidden until hover OR until at least one item is selected — "quiet" by default).
  - Click on the card still navigates to the item; clicking the checkbox does not.
  - Shift-click should extend a contiguous range (nice-to-have; skip if it adds more than ~20 lines).
  - `Esc` clears selection.

#### T-B-5b. Batch server actions (`src/app/actions.ts`) — implements F-053
- `bulkTagItemsAction(formData)` — accepts `item_ids: string[]`, `tag_name: string`; reuses `upsertTag("manual")` + `attachTagToItem` in a transaction.
- `bulkAttachCollectionAction(formData)` — accepts `item_ids: string[]`, `collection_id: string`; reuses `attachItemToCollection` in a transaction.
- `bulkDeleteItemsAction(formData)` — accepts `item_ids: string[]`; reuses `deleteItem` in a transaction.
- **All three revalidate:** `revalidatePath("/")`, `revalidatePath("/collections/[id]", "layout")`, `revalidatePath("/settings/tags")` (per critique P-6 / F-053). `bulkDeleteItemsAction` additionally revalidates `/items/[id]` for any just-deleted item visible in another tab.
- Use a shared zod schema: `z.object({ item_ids: z.array(z.string().min(1)).min(1).max(500) })`. The 500 cap guards against malformed requests.
- **Back-pressure (per critique P-3):** the UI wraps each submit in `useTransition` and disables the bulk bar buttons while `isPending`. Add an explicit note in T-B-5c that the bar uses `useTransition`; not a guess for the implementer.

#### T-B-5c. Confirm flows
- Delete must show a browser `confirm("Delete N items? This cannot be undone.")` client-side before submit. Matches the single-item delete pattern.
- Tag + collection batch ops should show a transient toast-style confirmation on success ("Tagged 3 items with `product`"). If there's no toast infrastructure yet, a simple `<div role="status">` that auto-dismisses after 2s is fine — don't introduce a toast library just for this.

#### T-B-5d. Test plan
- Select 3 items → click Tag → enter a new name → all 3 acquire the tag → verify on their detail pages.
- Select 3 items → click Add to collection → pick one → all 3 appear under `/collections/[id]`.
- Select 1 item → click Delete → cancel the confirm → item still present.
- Select 2 items → click Delete → confirm → both gone from library and from collections/tags.

- **Commit sequence:**
  1. `feat(v0.3.1, F-207a): selection state + bulk bar on library page`
  2. `feat(v0.3.1, F-207b, F-053): bulk tag/collection/delete server actions with full revalidate`
  3. `feat(v0.3.1, F-207c): wire bulk actions to bulk bar + confirm dialogs + useTransition back-pressure`
