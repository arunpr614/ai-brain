## §2 — Blockers

### B-1: Named-tunnel SSE support is asserted, not verified

**What the research claims:** §4 and §9 both state that named tunnels "do support SSE" and that switching to a named tunnel removes the SSE limitation. This is the entire rationale for the user's upgrade decision.

**Why it's wrong or unverifiable:**

Fetching the current Cloudflare docs (`developers.cloudflare.com/cloudflare-one/connections/connect-networks/do-more-with-tunnels/trycloudflare/`) confirms the exact quote: "Quick Tunnels do not support Server-Sent Events (SSE)." and "These limitations only apply to Quick Tunnels." This is a negative scope claim — limitations *of quick tunnels* — not a positive assertion that named tunnels support SSE.

No page in the official Cloudflare Tunnel docs explicitly states "named tunnels support SSE." The named-tunnel origin configuration docs (`origin-configuration/`) list `disableChunkedEncoding` as an origin parameter and note that "long-lived HTTP requests (for example, WebSocket) and TCP connections will be dropped" during config updates — but contain zero SSE-specific documentation. The `tunnel-run-parameters` page has no SSE-related flags.

SSE is a server-push streaming protocol (chunked `text/event-stream` responses). It is architecturally different from WebSockets. Cloudflare's network-level proxying can interfere with SSE via response buffering, chunked-encoding rewriting, or keepalive timeouts — none of which are addressed in the research doc.

**The `keepAliveTimeout` default of 90 seconds (1m30s) is an undisclosed risk.** The `originRequest.keepAliveTimeout` defaults to 1 minute 30 seconds. An Ollama inference that takes longer than 90 seconds of idle time on the SSE stream could trigger a keepalive timeout and drop the connection mid-stream, even on a named tunnel. The research doc doesn't mention this parameter at all.

**Risk:** Plan v2.0 is built on the premise that named tunnels fix Ask streaming. If SSE is in practice broken or flaky on named tunnels (e.g., via buffering at the edge, 90s keepalive drop, or undocumented per-proxy buffering), the user discovers this at T-CF-3 smoke testing, not at plan-drafting time.

**Fix:** Empirically verify SSE over a named tunnel before plan v2.0 is drafted. Run `cloudflared tunnel run brain` against `http://localhost:3000`, curl a streaming endpoint (`curl -N https://brain.arunp.in/api/ask -H "Authorization: Bearer ..."`) and confirm tokens stream through in real time, not buffered. Also verify the `/api/threads/[id]/messages` SSE route. If buffering occurs, set `disableChunkedEncoding: false` (the default) in `config.yml` and test again.

**Which task owns the fix:** This must be a pre-planning verification step, not a task in the plan itself. The plan cannot assume SSE works on a named tunnel without an empirical result.

### B-2: DNS nameserver prerequisite is not flagged as a real-time blocker

**What the research claims:** The research describes `cloudflared tunnel route dns brain brain.arunp.in` as a straightforward step (§10, P3 discussion). It notes the CNAME is created, and that DNS propagation takes "standard" time.

**Why it's wrong or unverifiable:**

The official Cloudflare Tunnel `create-local-tunnel` docs state two explicit prerequisites:
1. "Add a website to Cloudflare"
2. "Change your domain nameservers to Cloudflare"

These are prerequisites that must be satisfied before `cloudflared tunnel route dns` can work. The command creates a CNAME record in Cloudflare's DNS — which requires Cloudflare to be the authoritative DNS for the zone. If `arunp.in` has not yet completed its nameserver change to Cloudflare, the `cloudflared tunnel route dns` command will fail with an error like `failed to add route: Failed to update DNS: no such zone`.

The user's situation at time of writing: **nameserver change in-flight, not yet confirmed complete.** The research document acknowledges "DNS moving to Cloudflare, nameserver change in-flight" in the project context but does not:
1. Flag this as a potential blocker to T-CF-1 (the first tunnel task).
2. Distinguish between "Cloudflare account created" and "domain active on Cloudflare."
3. State how to verify the NS change is complete (`dig NS arunp.in @8.8.8.8` should return Cloudflare nameservers).

**Second DNS propagation race:** After `cloudflared tunnel route dns` succeeds, the CNAME `brain.arunp.in → <UUID>.cfargotunnel.com` needs to propagate before the APK can resolve it. Cloudflare's own DNS is fast (typically <1 minute for records within their system), but external resolvers may cache the old NS records for up to 24–48 hours during propagation. The APK will get NXDOMAIN on `brain.arunp.in` until this resolves.

**Risk:** The user starts T-CF-1, the `cloudflared tunnel route dns` command fails with a zone-not-found error, and there's no guidance on what to do next. The plan appears blocked for a non-technical user with no context.

**Fix:** Plan v2.0 must have a preflight step before T-CF-1: verify the NS change is complete. Add a concrete check command and expected output. Add a "if this fails, here's why and how long to wait" note.

**Which task owns the fix:** The plan's prerequisites section (before T-CF-1) or a dedicated T-CF-0 preflight task.

### B-3: `sed`-based `capacitor.config.ts` munging will not work as described — `cap sync` reads from compiled TypeScript, not raw `.ts`

**What the research claims (§6):** Approach A proposes `sed -i.bak "s|url: \"http://brain.local:3000\"|url: \"$BRAIN_TUNNEL_URL\"|g" capacitor.config.ts`, followed by `npx cap sync android`.

**Why this is a real footgun:**

The Capacitor CLI, when it encounters `capacitor.config.ts`, does NOT use `sed`-modified source directly. The Capacitor CLI imports `capacitor.config.ts` via `tsx` or a similar TS runtime under the hood, evaluating the TypeScript. The `sed` substitution on the `.ts` file would only work if the file is modified before the CLI reads it — which is the intent — BUT the existing `build-apk.sh` already demonstrates the problem: it calls `npx cap sync android` as step 2, and the current `capacitor.config.ts` hardcodes `http://brain.local:3000`.

More critically: **`cap sync` copies the evaluated config object to `android/app/src/main/assets/capacitor.config.json`** — this is the file the APK actually reads at runtime. This file was confirmed to currently contain `"url": "http://brain.local:3000"` (verified from `android/app/src/main/assets/capacitor.config.json` in the repo). If `sed` successfully patches `capacitor.config.ts` and `cap sync` then runs, the JSON is regenerated correctly.

**The actual risk is more subtle:** The `sed` substitution pattern `url: "http://brain.local:3000"` is fragile:
- If the TypeScript file has whitespace differences (tabs vs spaces), `sed` silently fails and the old URL remains.
- The `sed -i.bak` creates a backup `.ts.bak` file which, if left behind, will confuse TypeScript tooling.
- The `trap 'mv capacitor.config.ts.bak capacitor.config.ts' EXIT` trap fires on both success AND failure — but if `npx cap sync android` fails partway through, the `.json` file in `android/app/src/main/assets/` may already be partially overwritten with stale content from a previous run.
- **Most importantly:** The `sed` approach modifies a committed source file on every tunnel start. `git status` becomes dirty on every `dev:tunnel` run, polluting the working tree. This is operationally painful for a developer workflow.

**The deeper issue:** The research proposes Approach A for v0.5.0 and Approach B (`@capacitor/preferences` runtime URL) for v0.6.0+. But the named tunnel pivot changes the calculus: a named tunnel has a *stable URL*. The URL for `brain.arunp.in` never rotates. `capacitor.config.ts` can be permanently updated to `https://brain.arunp.in` at T-CF-X (one time), and `cap sync` is run once. No sed munging is needed at all. The research's entire §6 Approach A is predicated on quick-tunnel URL rotation, which is now irrelevant.

**Risk:** The plan author implements shell munging that is fragile, dirties the working tree, and is unnecessary for the named-tunnel case. Worse, if the research is used to write the plan before this is noticed, the plan will allocate task budget to a script that should simply be deleted.

**Fix:** For named tunnels, `capacitor.config.ts` gets a one-time update to `https://brain.arunp.in`. Full stop. Approach A (shell munging) should be deleted from the plan entirely. The build script `build-apk.sh` needs no URL injection logic.

**Which task owns the fix:** T-CF-X (capacitor config update) — a single one-time task, not a recurring workflow.

### B-4: Named tunnel persistence is undocumented — no guidance on what keeps the tunnel alive

**What the research claims:** The research discusses process lifecycle for quick tunnels (foreground process, Ctrl-C, etc.) but the named tunnel persistence story is completely absent. §9 says "Named tunnels have stable URLs (no re-pair on restart)" but doesn't describe how the tunnel daemon is kept running.

**Why this is a blocker:**

For a named tunnel, there are three persistence options:
1. **Foreground terminal:** User must keep a terminal open with `cloudflared tunnel run brain`. Closing the terminal kills the tunnel.
2. **`cloudflared service install` (launchd login agent):** Installs a macOS launch agent that starts at login. Lives in `~/Library/LaunchAgents/com.cloudflare.cloudflared.plist`. The tunnel starts when the user logs in and stops when they log out.
3. **`sudo cloudflared service install` (launchd boot daemon):** Installs a system launch daemon that starts at boot, before login. Lives in `/Library/LaunchDaemons/`. Uses `/etc/cloudflared/` for config.

The research doc implies the user just runs `cloudflared tunnel run` without addressing which of these is appropriate for the AI Brain use case. The research was written for a "developer runs `dev:tunnel`" mental model (quick tunnels), which is a foreground process. But named tunnels are positioned as "stable URL, always-on" — which implies a persistent daemon, not a terminal session.

**Sleep behavior for named tunnels:** The research inherits the quick-tunnel assumption that "cloudflared reconnects to the same tunnel after sleep/wake" [MEDIUM confidence, assumed]. For named tunnels this is architecturally cleaner (the tunnel UUID is stable; `cloudflared` reconnects using stored credentials), but the research provides zero empirical evidence for this on the actual Mac hardware. The 5–8 second reconnect window is specifically for quick tunnels; named tunnel reconnect time is unverified.

**Risk:** The user follows the plan, sets up a named tunnel, and has no idea whether it's supposed to run in a terminal window or as a system service. When the Mac restarts or the user closes their laptop, they don't know why the phone can't reach Brain anymore.

**Fix:** Plan v2.0 must declare explicitly: for the AI Brain named tunnel, use `cloudflared service install` (login agent, not boot daemon) so the tunnel starts automatically on login without requiring a permanent open terminal. The plan must document the launchd setup commands and verify this survives a Mac sleep/wake cycle.

**Which task owns the fix:** Dedicated tunnel daemon setup task in plan v2.0 (equivalent to the old T-5 mDNS setup task).

### B-5: Origin header validation for named tunnel requests is based on an unverified assumption

**What the research claims (§5):** "Cloudflare passes the Origin header through unchanged." The research cites "official CF docs — 'Cloudflare passes all HTTP request headers to your origin web server' with no modification of the Origin header listed."

**Why this needs scrutiny:**

Fetching the actual Cloudflare HTTP headers doc (`developers.cloudflare.com/fundamentals/reference/http-request-headers/`) reveals something the research doesn't mention: Cloudflare modifies `Accept-Encoding` (always sets to `"accept-encoding: br, gzip"`), `X-Forwarded-Proto`, and `Connection`. The page states "Cloudflare passes all HTTP request headers to your origin server" as a general rule, with specific exceptions listed. The Origin header is not in the exceptions list — which is consistent with it being passed through.

**However, the Origin header value itself matters for named tunnels:** When the APK WebView loads `https://brain.arunp.in`, the page origin becomes `https://brain.arunp.in`. All XHR/fetch requests from that page carry `Origin: https://brain.arunp.in`. This is NOT `*.trycloudflare.com`.

The current `validateOrigin()` in `src/lib/auth/bearer.ts` (verified from the live file) explicitly lists three allowed origins:
```typescript
const ALLOWED_ORIGINS = new Set([
  "http://localhost:3000",
  "http://127.0.0.1:3000",
  "http://brain.local:3000",
]);
```

None of these match `https://brain.arunp.in`. Every API request from the APK will be rejected with a 403 (`lan.bearer.reject-origin`) until this is fixed.

**The research recognizes this for quick tunnels** (Option B: pattern-match `*.trycloudflare.com`) but doesn't resolve it for named tunnels. For named tunnels, the fix is simpler and more precise: add `https://brain.arunp.in` as a static entry to `ALLOWED_ORIGINS`. This is better security posture than the `*.trycloudflare.com` wildcard the research recommended for quick tunnels.

**Risk:** Plan v2.0 may inherit the quick-tunnel origin fix (pattern-match wildcard) without noting that the named-tunnel fix is a static string. Or worse, both named-tunnel origin fix and `brain.local` removal may be missed entirely, leaving all APK requests in a permanent 403 loop.

**Fix:** Plan v2.0 must add `https://brain.arunp.in` to `ALLOWED_ORIGINS` and remove `http://brain.local:3000`. The `validateOrigin()` fix is a trivial two-line change, but it must be in the plan as an explicit task acceptance criterion.

**Which task owns the fix:** The `validateOrigin()` update task in plan v2.0 (analogous to P4 decision in the research).

### H-3: `androidScheme: 'http'` mismatch — consequence not fully traced for named tunnel

**Location:** §6 (MEDIUM confidence), risk register row 7.

The research correctly identifies that `androidScheme: 'http'` may conflict with an HTTPS `server.url`, rating it MEDIUM confidence and flagging it for empirical verification. But the consequence analysis is incomplete for the named tunnel case.

With `server.url = "https://brain.arunp.in"` (the post-pivot value) and `androidScheme: 'http'`, the Capacitor WebView's internal scheme assignment is inconsistent. In Capacitor 8, `androidScheme` controls the scheme used for local assets (served from the `webDir`). When `server.url` is set, Capacitor routes the WebView to the remote URL using whatever scheme that URL uses — but `androidScheme` can still affect how local resources and the offline fallback are served. The interaction is not definitively documented.

**What is confirmed from the live codebase:** `capacitor.config.ts` currently has `androidScheme: "http"` and `server.url: "http://brain.local:3000"` — consistent. After the tunnel pivot, `server.url` becomes `https://brain.arunp.in` while `androidScheme` stays `"http"` if not updated. The `.json` in `android/app/src/main/assets/` will be regenerated from the `.ts` by `cap sync`, so the mismatch lands in the APK only if both are not updated together.

**Fix:** The `androidScheme` must be changed to `"https"` (or removed to accept the default) when `server.url` is updated to `https://brain.arunp.in`. These are two config changes in a single commit — the plan must link them as a single atomic task acceptance criterion.

## §4 — Medium and Low-Severity Findings

### M-1: Tunnel credential storage (`cert.pem` + `<UUID>.json`) is undocumented

**What's missing:** The research says nothing about where named tunnel credentials live on disk or what happens if they are lost or compromised.

When the user runs `cloudflared tunnel login`, a `cert.pem` file is created in `~/.cloudflared/`. When `cloudflared tunnel create brain` runs, a `~/.cloudflared/<UUID>.json` credentials file is created. These are the only auth materials that allow the tunnel to run. If `~/.cloudflared/` is deleted or if the user gets a new Mac without migrating these files, the tunnel becomes inoperable and must be deleted and recreated.

The research's "state left behind: none" claim applies to quick tunnels only. For named tunnels, permanent state exists in `~/.cloudflared/`. The plan must include a step to back up `~/.cloudflared/<UUID>.json` alongside `debug.keystore.backup`.

**Severity:** MEDIUM. Loses tunnel on disk wipe without backup.

### M-2: `dev` script still binds to `127.0.0.1`, but named tunnel targets `localhost:3000`

The `package.json` `dev` script: `next dev -H 127.0.0.1`. The named tunnel config will point at `http://localhost:3000`. `localhost` resolves to `127.0.0.1` on most systems, but this is technically a loopback alias resolution — if `127.0.0.1` is the explicit bind address, both `127.0.0.1:3000` and `localhost:3000` work correctly. No actual bug here, but the plan should call out that the named tunnel `config.yml` should use `url: http://127.0.0.1:3000` (explicit) rather than `url: http://localhost:3000` (DNS lookup) to avoid any edge case where `localhost` resolves to `::1` (IPv6 loopback) on this Mac.

**Severity:** LOW-leaning-MEDIUM. IPv6 loopback vs IPv4 loopback is a real gotcha on macOS.

### M-3: The `disableChunkedEncoding` parameter and its SSE implications are unexplored

The origin config docs list `disableChunkedEncoding: false` as the default. For SSE, chunked encoding must remain enabled (SSE is delivered as a long chunked response). If any intermediary sets `disableChunkedEncoding: true`, SSE breaks silently (the response body is buffered until the connection closes, then delivered as one blob instead of a stream).

The research doc never mentions this parameter in the context of SSE verification. Before marking SSE as "working on named tunnel," the plan must verify that the default `disableChunkedEncoding: false` is in effect and that Cloudflare's edge is not buffering the SSE response.

**Severity:** MEDIUM. If this parameter defaults to false (as docs state) and no one touches it, SSE works. But the plan should call it out as a known SSE-breaking knob that must never be enabled.

## §5 — Gaps

### G-1: Tunnel credential rotation story is absent

The research has no answer to: what does the user do if `~/.cloudflared/<UUID>.json` is compromised or if the cert.pem expires?

Named tunnel credentials (the `<UUID>.json` file) are long-lived JWT-like tokens. They do not expire on a rolling basis. If compromised, the remediation is to delete the tunnel in the Cloudflare dashboard and recreate it — which changes the tunnel UUID and requires a new `cloudflared tunnel route dns` call (new CNAME), a new `config.yml`, and potentially a new APK build if the tunnel URL is somehow encoded in the APK (it isn't for named tunnels, but the plan should say so).

The cert.pem (account-level certificate) has a 10-year validity per Cloudflare docs. It is not rotated routinely. But if compromised, the user must re-run `cloudflared tunnel login` to get a new cert.pem.

The plan must include one paragraph in the README-equivalent task: "If you suspect tunnel credentials are compromised: (1) delete the tunnel in one.dash.cloudflare.com, (2) re-run setup from T-CF-2 onwards."

---

### G-2: Privacy posture for named tunnels is different from quick tunnels — but research treats them identically

The research notes (§5) that "Cloudflare's infrastructure technically can observe plaintext request and response bodies" because TLS terminates at the edge. This is correct for both quick and named tunnels.

However, named tunnels introduce an additional privacy dimension: the tunnel is tied to the user's Cloudflare account (`Arunever614@gmail.com`). Cloudflare can now correlate the tunnel traffic with a real account identity. For quick tunnels (no account), the traffic was still observable but not attributed. For named tunnels, Cloudflare logs are accessible in the Zero Trust dashboard and are tied to the account.

This is a minor concern for a personal notes app (the user's own data, trusted transit provider), but the research should have called it out explicitly. The plan's README note about "personal data flows through Cloudflare" should be updated to reflect that named-tunnel traffic is account-attributable.

---

### G-3: Observability beyond `/ready` is absent

The research §8 mentions `/ready` for local health checking and notes that quick tunnels have no Cloudflare dashboard. For named tunnels, there IS a dashboard. The user can view:
- Real-time tunnel status at `one.dash.cloudflare.com/[account-id]/zero-trust/networks/tunnels`
- Connection metadata (connector version, data center, connected since)
- Tunnel logs (via `cloudflared tail` command — not documented in the research)

The `cloudflared tail` command streams live request logs from the Cloudflare edge to the local terminal — this is a significant observability feature that the research entirely misses. It is equivalent to access logs but for tunnel traffic. This should be documented in the plan's troubleshooting section.

For named tunnels, this is simpler: since `brain.arunp.in` is stable, a GitHub Actions workflow can build the APK without any tunnel URL injection — the URL is baked into `capacitor.config.ts` permanently. The plan should include a note that CI/CD (if ever needed) is straightforward for named tunnels, unlike the quick-tunnel approach which required environment-variable injection.

### G-5: macOS sleep behavior for named tunnel daemon is unverified

The research (§3) states for quick tunnels: "the URL is stable during the session — cloudflared reconnects to the same tunnel (same subdomain) after a sleep/wake cycle as long as the process is still running." This is MEDIUM confidence, assumed.

For named tunnels running as a launchd login agent, the behavior is more complex:
- macOS suspends launch agents during sleep.
- On wake, launchd restarts any launch agents that exited during sleep.
- If `cloudflared` was running as a foreground process in a terminal, the QUIC connection drops during sleep and reconnects on wake (5–8 seconds per research).
- If `cloudflared` is a launchd agent, the agent may or may not be restarted automatically depending on the `KeepAlive` key in the plist.

The official Cloudflare macOS service docs confirm that `cloudflared service install` installs a launch agent, but do not describe the `KeepAlive` behavior in the plist. This is the difference between "the tunnel comes back automatically after sleep" and "the user must manually restart the tunnel after sleep."

**This must be empirically verified at T-CF-1 (setup smoke) before the plan is considered correct.**
