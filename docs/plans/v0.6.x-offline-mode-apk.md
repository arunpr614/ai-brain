## §1. Problem statement

The Android APK is a thin Capacitor WebView pointing at `https://brain.arunp.in`. When the user shares a URL, PDF, or note from another Android app → Brain, the capture flow does a synchronous POST to the tunnel and either succeeds immediately or fails.

Failure modes the user hits in real life:

1. **Phone has no internet** (airplane mode, subway, weak signal, data cap).
2. **Phone has internet but laptop is asleep** — Cloudflared can't reach the origin → tunnel returns 502.
3. **Phone has internet but laptop's `cloudflared` service stopped** — DNS resolves, connection refused at edge.
4. **Phone has internet, laptop awake, but `npm run dev` is not running** — Cloudflared probe returns 502 from origin.
5. **Server is up but returns a transient 5xx** (disk full, Ollama dead, enrichment queue wedged).

Today all five produce "capture failed," content is lost, user must remember to come back. That's a daily-use blocker.

## §2. Goal + non-goals

### In-scope (MVP)

- Android APK captures always succeed from the user's point of view (no "save failed" modal in the middle of a share).
- Offline / server-unreachable captures land in a local queue.
- Background retry attempts re-submit queued items opportunistically (on network regained, on app foreground, on user action, on interval — **all while the app is open**; see §4.1 honest scoping).
- User-visible inbox + nav-bar badge showing pending items + their state.
- Manual "Sync now" button to force retries.
- Clear error surface for items that have hit a terminal failure (auth bad, payload bad, version mismatch).

### Explicit non-goals

- **Background drain while the app is closed.** WorkManager / native scheduler tracked as v0.7.x roadmap (§4.10). This is a deliberate MVP scope cut after the WebView-freezing evidence; v1 plan implied it would work, v2/v3 honestly do not.
- Full offline reading of already-captured content (separate feature; out of scope).
- Offline capture for Chrome extension or web (offline notion doesn't apply).
- **Desktop-web offline capture** (laptop disconnected). The web `/capture` UI continues to fail visibly. APK-only because Capacitor share-target is the share path that needs offline coverage. (G-4)
- Local AI enrichment on device (no on-device model).
- End-to-end encryption of the outbox (Android app-private storage already sandboxed).
- Conflict resolution beyond "first-writer-wins dedup" (no three-way merge for same-URL-twice-offline).
- iOS (no iOS target).
- Multiple offline devices syncing together (one phone).

## §3. User stories

### 3.1 "I'm on the subway and want to save a Substack post" — REWRITTEN for honesty + Android share-target reality (B-6)

User opens Substack in subway, taps share → Brain. Phone has no internet.

**Android share-target behavior (corrected from v2):** when an external app shares to Brain, Android launches Brain's `MainActivity`, which loads the WebView. **The user is now inside Brain, not back in Substack.** The share-handler enqueues to the outbox, shows "Saved offline — will sync next time you open Brain" toast, and Brain stays in the foreground showing the saved state. To return to Substack the user taps back or swipes the recents.

Later that day, user opens Brain to browse their library (or shares another item). **On open, the retry trigger fires:** all queued items drain in the background while the user reads. A small badge "Synced 3 items" appears on the library.

**What v3 does NOT promise** (and v1 incorrectly implied): "automatic sync within 30s of signal returning while you're still walking around with Brain in your pocket." Logcat evidence (2026-05-13) shows the Brain WebView gets frozen by Android every ~30s when not foregrounded. JS retry timers don't fire reliably during freeze. **The MVP guarantee is "syncs reliably the next time you actively open the app."** WorkManager-backed background drain is an explicit v0.7.x feature (§4.10).

### 3.2 "My laptop is closed while I'm out"

User shares 3 articles over the morning from coffee shop. Laptop is asleep. Each share gets the "Saved offline" toast (Brain stays foreground each time; user swipes back to source app). Items queue locally. When user opens Brain later (lunch, at home) the retry triggers fire and items drain.

### 3.3 "Retries fail — I'm stuck"

Wi-Fi captive portal returns HTML on POST (either as a 200 with HTML body, a 302 redirect, or a 4xx with HTML — see §5.3). Retries cycle through transient-failure backoff. **For network-class failures, retries continue indefinitely** (no MAX_ATTEMPTS — see §5.3). The user sees a persistent "1 pending sync" badge while the conditions persist; once on real internet and Brain is opened, the sync drains.

For terminal failures (rotated bearer token → 401, malformed payload → 422, API version mismatch → 422 with `version_mismatch`), the item moves to `stuck` immediately and shows a clear, actionable error.

### 3.4 "I want to manually drain the queue right now"

User knows their laptop just woke up. They open Brain, see "Pending: 3" in the nav, tap → `/inbox` page → "Sync now" button → all three drain.

**Sync-now snapshot semantics (C-4):** Sync-now operates on a snapshot of `queued` items at trigger time. If the user shares a 4th item *during* the sync-now run, it joins the next retry cycle (next foreground / network-change / 30s tick), not the current one.

## §4. Architecture

### §4.1 Honest scoping — what Option A delivers and what it doesn't

The MVP is **Option A: pure WebView + IndexedDB outbox + JS retry orchestrator**. Decision rationale (refined from v1):

| What we get | What we don't |
|---|---|
| Zero new native code; matches "thin WebView" philosophy | Background drain when app is closed/frozen — Android freezes the WebView (logcat evidence) |
| Reliable enqueue at share time (share-target launches MainActivity → WebView active) | Timed retries while app is backgrounded for >5 min |
| Reliable retry on app foreground / explicit user action | "Subway story" automatic-sync-on-signal-return |
| Survives app force-kill (queue persists in IndexedDB) | iOS support (out of scope) |

**v0.7.x roadmap (§4.10)** adds Option C: native WorkManager + Capacitor plugin bridge that reads the same IndexedDB outbox and POSTs from native code while app is closed. Architecture chosen so the v3 outbox schema and orchestrator are reused unchanged when WorkManager lands — no schema migration on the Option A → C transition.

### §4.2 Data flow

```
User shares item → CapacitorShareTarget receives intent → MainActivity launches → WebView opens
  ↓
[INIT GATE] Outbox initialization (B-5):
  - storage.persist() request
  - IDB open + migration check
  - Replay any pre-init buffered share events (see §5.10)
  ↓ (only after init complete)
share-handler.tsx receives event with file URI / URL / text
  ↓
Build OutboxEntry (id, kind, payload | file_path, content_hash)
  ↓
Run dedup check (§5.2) — if dup, skip
  ↓
[STORAGE PRE-FLIGHT] navigator.storage.estimate() (C-3):
  - if usage / quota > 0.95 → reject with "Storage almost full" warning toast
  - if quota write throws despite pre-flight → surface non-blocking error
  ↓
Write entry to IndexedDB outbox (status='queued')   ← v3: skip 'pending' (C-6)
  ↓
[immediate] try direct POST via sync-worker
  ├─ success (2xx)  → mark synced, store server_id, show "Saved" toast
  ├─ network/5xx    → keep queued, show "Saved offline" toast
  └─ 401/422/4xx    → mark stuck, show error toast
  ↓
return control to user (immediate UI feedback regardless of network)
   — note: user is in Brain foreground (B-6); back/swipe to source app

Retry triggers (only when WebView is active and not frozen):
  - @capacitor/network → networkStatusChange to 'wifi' or 'cellular'
  - @capacitor/app    → appStateChange to active
  - 30s in-app interval (only ticks while WebView unfrozen)
  - User: tap "Sync now"
  ↓
For each queued entry where next_retry_at <= now:
  - try POST
  - on success → mark synced; reset attempts = 0; clear last_error (B-7)
  - on network/5xx → bump attempts, set next_retry_at = now + backoff(attempts)
  - on terminal 4xx → mark stuck
```

### §4.3 Outbox schema (IndexedDB) — discriminated union (A-2 + B-9)

```typescript
// IDB store: 'outbox', keyPath: 'id'
// Records are structured-clone-safe (NOT JSON-safe).
// Implementers: do NOT JSON.stringify before put(). Date round-trips natively;
// class instances and functions are blocked by structured clone. Blobs CAN be
// stored, but PDFs use filesystem path instead (see kind: 'pdf').

type OutboxEntryBase = {
  id: string;                     // client UUID
  status: 'queued' | 'synced' | 'stuck' | 'duplicate';   // (C-6: dropped 'pending')
  status_reason?: 'auth_bad' | 'payload_bad' | 'version_mismatch' | 'transient';
  attempts: number;               // retry counter; reset to 0 on synced (B-7)
  created_at: number;             // ms epoch
  last_attempt_at?: number;       // ms epoch; UI shows "Last tried 5 minutes ago" (B-7)
  last_error?: string;            // user-readable; reset to undefined on success
  next_retry_at?: number;         // ms epoch; ignored unless status='queued'
  server_id?: string;             // populated on success → ties to items.id
  content_hash: string;           // dedup key (see §5.2)
};

type OutboxUrlEntry = OutboxEntryBase & {
  kind: 'url';
  payload: { url: string; title?: string; text?: string };
};

type OutboxNoteEntry = OutboxEntryBase & {
  kind: 'note';
  payload: { title: string; body: string };
};

type OutboxPdfEntry = OutboxEntryBase & {
  kind: 'pdf';
  file_path: string;              // app-private filesystem path (Filesystem.Directory.Data)
  file_name: string;
  file_size: number;              // bytes — used for early-reject of >25 MB and dedup tier
  expected_sha256: string;        // computed via Web Worker at enqueue (B-2)
};

type OutboxEntry = OutboxUrlEntry | OutboxNoteEntry | OutboxPdfEntry;

// Indexes: by status, by created_at, by content_hash
```

**Why PDFs go to filesystem, not blob in IDB:** A-2 / A-3 critique. PDFs can be 5–10 MB; storing in IDB blobs hits the per-origin WebView quota fast and is more vulnerable to eviction than `Filesystem.writeFile` to app-private storage. The outbox row holds the path; the actual bytes live in app-private files.

### §4.4 Retention (B-4 — revised)

- `status='queued'`: retained until `synced` or `stuck` or user discard.
- `status='synced'`: **retained indefinitely** — user can review what synced in `/inbox` (no 24h auto-delete; previous v2 default contradicted the audit-trail purpose). PDF filesystem bytes ARE deleted on `synced` (the server now has them) — only the metadata row remains, which is small.
- `status='stuck'`: retain forever until user acts (Retry / Discard).
- `status='duplicate'`: delete after 1 hour (audit only; not user-facing).
- **Discard semantics (Q3):** hard delete of the outbox row + filesystem PDF (no `discarded` audit status).
- **Quota pressure escape valve:** if `usage / quota > 0.85` and the user has > 200 `synced` rows, the oldest synced rows are pruned silently. (`stuck` rows are never auto-pruned.)

## §5. Design decisions

### §5.1 Storage strategy: IndexedDB metadata + Filesystem blobs

Switch from "everything in IDB" (v1) to "metadata in IDB, blobs in filesystem":

| Data | Where | Why |
|---|---|---|
| Outbox row (id, kind, payload, status, etc.) | IndexedDB store `outbox` | structured query, sort by status |
| URL/note payloads | Inline in IDB row | small (<10 KB) |
| PDF bytes | `@capacitor/filesystem` writeFile to `Directory.Data` | survives quota pressure better; doesn't bloat IDB |
| Persistence request | `navigator.storage.persist()` at outbox-init | A-3: ask Chromium not to evict the outbox under memory pressure |

If `persist()` returns `false` (some Android WebView versions silently refuse), surface a one-time toast: "Your device may not preserve offline saves across long backgrounds. Sync as soon as you can."

### §5.2 Dedup (refined per critique B-3 + v3 B-2)

**Three-tier check order, applied in this exact sequence:**

1. **In-memory 2s window** (existing `isDuplicateShare()` at `src/lib/capture/dedup.ts:39`). Catches accidental double-taps. **Key:** `shareDedupKey(kind, hashPayload(payload))`. Unchanged from v0.5.0.
2. **Outbox 10-minute window.** Computed dedup key per kind:
   - `url` → `sha256(normalize_url(payload.url))` — strips utm_*, fragments, trailing slashes (computed inline; URL is short)
   - `note` → `sha256(payload.title + payload.body)` (computed inline)
   - `pdf` → **`sha256(file_bytes)` computed in a Web Worker** (B-2). Worker reads the filesystem path via `Filesystem.readFile` (base64), decodes, hashes, posts result back. UI thread unblocked. Hash is computed **before** enqueue completes — so dedup check can run before we commit the outbox row. Trade-off: ~600–1200 ms latency on share-receipt for 10 MB PDFs, but invisible to user (Brain WebView has loaded UI; spinner shown during hash).
3. **Server dedup.** Existing `findItemByUrl()` on `/api/capture/url:83` already does this; on `200 { duplicate: true, itemId }` the outbox row marks `synced` with the server id (no client-side error).

**Web Worker file:** `src/lib/outbox/sha256-worker.ts`. Uses `crypto.subtle.digest('SHA-256', ...)` on the decoded bytes. Capacitor WebViews support Web Workers (verified in OFFLINE-PRE; if not, fallback to `(file_name, file_size)` tuple-dedup with note that content-hash dedup is server-side only).

**On dedup hit (steps 1 or 2):** silently no-op. No outbox entry created. No user-facing toast.

### §5.3 Retry classification (refined per critique B-3 + v3 B-3 captive portal)

| HTTP / network condition | Class | Action |
|---|---|---|
| `fetch()` throws (DNS, socket, timeout, captive portal direct fail) | transient | retry indefinitely with backoff cap |
| 5xx (incl. 502, 503, 504) | transient | retry indefinitely |
| 429 with `Retry-After` | transient | retry after `Retry-After` seconds (or 60s default) |
| **2xx with non-JSON body / `res.json()` throws SyntaxError** (v3 B-3 captive portal proxy) | **transient** | retry indefinitely; this is captive portal HTML or transparent proxy interception |
| 401 / 403 (bearer auth) | terminal | `stuck`, reason `auth_bad`, "Re-pair your device in Settings" |
| 422 with `code: version_mismatch` | terminal | `stuck`, reason `version_mismatch`, "Update Brain to sync these items" |
| 422 / 400 (any other reason, with valid JSON body) | terminal | `stuck`, reason `payload_bad`, "Item couldn't be saved" |
| 200/201 with `duplicate: true` | success | `synced` with returned `itemId` |
| 200/201 fresh, JSON body parses | success | `synced` |

**Implementation note (B-3):** every response branch wraps `await res.json()` in try/catch. A SyntaxError on parse is routed to the transient bucket regardless of status code — this catches captive portals that return HTML with any status.

**Backoff for transient failures:** exponential with jitter, capped at 1 hour. Base 10s.

```
attempt 1  → +10s
attempt 2  → +20s
attempt 3  → +40s
attempt 4  → +80s
attempt 5  → +160s (~3 min)
attempt 6  → +320s (~5 min)
attempt 7  → +640s (~10 min)
attempt 8  → +1280s (~21 min)
attempt 9+ → +3600s (1 hour cap)
```

**No MAX_ATTEMPTS for transient failures.** A subway-trip-turned-flight outage of 4 hours is realistic; v1's 22-minute cap was wrong. The badge stays visible until success or user explicitly discards. **`attempts` resets to 0 on `synced`** (B-7); the UI in `/inbox` shows "Last tried 5 minutes ago" (computed from `last_attempt_at`), not the count.

When `@capacitor/network` reports network-state-change to online, every queued entry's `next_retry_at` resets to "now" — opportunistic retry on connectivity regain.

### §5.4 Network detection (B-2 v2 + v3 B-1 async fix)

Primary: **`@capacitor/network`** plugin. Fires `networkStatusChange` events through the JS bridge — more reliable than browser DOM `online`/`offline` events inside Capacitor WebViews.

Fallback: browser `window.online` event (defensive — if Capacitor plugin fails to register).

**Status query:** `await Network.getStatus()` returns `Promise<{ connected: connectionType }>`. (v3 B-1: corrected — Capacitor plugin methods always cross JS↔native bridge and return Promises.) Used for "should I bother trying" pre-flight checks before initiating a sync cycle.

### §5.5 API version compatibility (B-5)

Every outbox POST sends `X-Brain-Client-Api: 1` header. Server enforces:
- `/api/capture/{url,pdf,note}` reads header; if present and != current expected version, returns `422 { code: 'version_mismatch', message: 'Update Brain APK to sync these items' }`.
- Initial expected version: `1`. Bumped only when a breaking change requires re-encoding old queue entries.

This protects against the scenario where a user queues 30 items offline, doesn't open the app for a week, server gets upgraded with a breaking schema change in the meantime.

### §5.6 Notification UX (C-1)

- **Permission request:** fire on the *first successful enqueue* (Q2; not on app first-launch — too soon, user hasn't opted into the feature). Use `@capacitor/local-notifications.requestPermissions()`.
- **Granted path:** Android system notification fires on `0 → ≥1 stuck` transition. Tap → `/inbox`.
- **Declined path:** in-app nav-bar badge is the primary surface. Badge stays visible until count returns to 0.

### §5.7 `/inbox` page (C-4) + accessibility (v3 C-7)

`src/app/inbox/inbox-client.tsx` — **client component**, marked `"use client"`. Reads outbox via IDB, renders entries grouped by status, exposes per-item Retry / Discard + global Retry-all.

`src/app/inbox/page.tsx` is a thin server component that just renders `<InboxClient />` (no async data fetch on the server).

**(v3 §0.2 wording reconciliation):** the "no server dance" phrasing in §0 v2 row was loose — what's meant is "outbox content rendered entirely client-side; the page.tsx file is a thin server-component wrapper that does no data fetching." Same architecture, clearer wording.

**Accessibility contract (v3 C-7):**
- Outbox entries rendered as semantic `<ul>` / `<li>` with `role="list"`.
- Per-item action buttons are real `<button>` elements (not `<div onClick>`); keyboard-reachable; `aria-label` describes the action and target ("Retry sync for 'Substack — How to ship features'").
- Status badges have `aria-label` with full state name (`aria-label="Status: stuck — re-pair required"`); color is not the only signal (icon + text).
- Discard buttons get `aria-describedby` pointing to a "this can't be undone" hint (matches Q3 hard-delete semantics).

### §5.8 Sync-now rate-limit handling (C-3)

"Sync now" iterates a snapshot (taken at trigger time per §3.4 / C-4) of all `queued` entries. For each:
- If `next_retry_at <= now`, try the POST.
- If `next_retry_at > now` (still in 429 cooldown), skip and surface "1 item is rate-limited; try again in Xs" inline.

Items enqueued while sync-now runs join the next retry cycle, not the current snapshot.

### §5.9 IDB write-failure handling (v3 C-3)

Every IDB `put()` is wrapped:

```typescript
try {
  await preFlightStorageEstimate();   // throws QuotaWarning if usage > 95%
  await store.put(entry);
} catch (err) {
  if (err instanceof QuotaWarning) {
    showStorageFullToast();
    return;                           // do NOT enqueue
  }
  // rare: structured-clone error or actual quota exhaustion despite pre-flight
  reportClientError({ kind: 'idb_write', err: err.message });
  showGenericErrorToast();
  return;
}
```

If quota truly exhausts mid-write (rare — pre-flight should catch it), the share is dropped with a non-blocking error toast. This is preferable to a partial write that corrupts the outbox.

### §5.10 Outbox-init / share-event ordering (v3 B-5)

Race: Capacitor share-target dispatches `shareReceived` events as soon as the JS side calls `addListener`. If outbox init (`storage.persist()` + IDB open) is still in flight, an early share could arrive at a not-ready outbox.

**Solution:**

1. At app boot (in `src/app/layout.tsx` client-side bootstrap), kick off `initOutbox()` immediately. Returns a Promise.
2. The share-handler listener is registered **only after** `initOutbox()` resolves.
3. **Pre-init buffer:** if `CapacitorShareTarget` fires its initial event before the listener is attached (Capacitor caches it), the listener picks it up on registration. If the listener somehow misses it, a `pendingShareEvent` global (set by a minimal early listener that buffers and replays) ensures no event is lost.

**Test:** integration test that simulates a `shareReceived` event fired during the first 100 ms of app load → verifies the entry lands in the outbox.

## §6. Platform scope

- **In:** Android APK (Capacitor WebView), specifically `window.Capacitor?.isNativePlatform() === true`.

## §8. Performance + storage budget

### §8.1 Pre-execution measurement (OFFLINE-PRE / A-2 / v3 B-8 delivery fix)

**Decision derived from measurement:**
- if `quota >= 500 MB` → use generous limits (200 MB outbox cap)
- if `quota` 100–500 MB → moderate (50 MB cap)
- if `quota < 100 MB` → conservative; **drop PDF from MVP** and document `kind: 'pdf'` outbox entries as v0.7.x WorkManager territory (the worker writes directly to filesystem without WebView quota — see §4.10).

**SW availability decision:**
- if SW available → Workbox bg-sync is a real candidate for OFFLINE-1A spike
- if SW not available → spike narrows to `@capacitor-community/background-runner` + roll-own; Q5 dropped from user questions
