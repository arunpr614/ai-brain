## 3. Architecture decision: thin WebView vs bundled static

**Decision: thin WebView pointing at the Mac's live Next.js server. This is already the only viable option.**

The app is a Next.js 16 App Router application with server components, server actions, API routes, SSE streaming (for Ask/RAG), and a SQLite database. It has no `output: 'export'` in `next.config.ts` and cannot be statically exported without gutting the entire server-side architecture built in v0.1.0–v0.4.0.

The Capacitor APK is a thin shell: a WebView that loads `http://brain.local:3000`. All logic runs on the Mac. This is confirmed correct by:

1. R-CAP's §4 (`Static export vs App Router note`) already identified this.
2. The v0.0.1 spike used a static HTML page only because the spike tested the share-intent plugin in isolation — it did not test the full app.
3. The architecture matches the "local-first, Mac as server" premise of the entire project.

**`capacitor.config.ts` for production:**

```typescript
// capacitor.config.ts
import { CapacitorConfig } from '@capacitor/cli';

const config: CapacitorConfig = {
  appId: 'com.arunprakash.brain',
  appName: 'Brain',
  webDir: 'public',          // minimal fallback for offline splash; NOT the running app
  server: {
    url: 'http://brain.local:3000',   // mDNS hostname (primary)
    cleartext: true,                   // allow HTTP (required since Android API 28 blocks cleartext by default)
    androidScheme: 'http',
  },
  android: {
    buildOptions: {
      keystorePath: undefined,         // use debug keystore for v0.5.0
    },
  },
};

export default config;
```

**Cleartext approach: `server.cleartext = true` vs `network_security_config.xml`**

- `server.cleartext: true` in `capacitor.config.ts` sets `android:usesCleartextTraffic="true"` on the `<application>` tag — allows HTTP to ALL hosts. Simpler but broader than needed.
- `network_security_config.xml` with `<domain-config cleartextTrafficPermitted="true">` for `brain.local` and `192.168.*.*` only. More correct; restricts cleartext to known-local hosts.

**Use `network_security_config.xml` for production correctness.** The blanket `usesCleartextTraffic` attribute is a security anti-pattern even for internal apps. The `network_security_config.xml` approach also survives if we later add HTTPS for the Mac server on the same APK build.

**`webDir: 'public'`:** The `public/` folder already exists and contains static assets. It serves as the fallback bundle Capacitor loads when `server.url` is unreachable (offline splash). A minimal `public/offline.html` file is all that's needed for the offline screen (F-020).

## 4. LAN auth + CSRF + rate-limit wiring

### 4.1 Overview

The current `src/proxy.ts` checks for a `brain-session` cookie. For v0.5.0, it must also accept an `Authorization: Bearer <token>` header for requests from:
- The Android APK's WebView (which cannot set a `SameSite=Strict` cookie across the initial request)
- The Chrome extension popup (which is cross-origin: `chrome-extension://<id>`)

### 4.2 Next.js 16 proxy runtime clarification

**The comment in `src/proxy.ts` stating "edge runtime cannot use node:crypto" is now outdated.** As of Next.js v15.5.0 (stable), and continuing in Next.js 16, `proxy.ts` runs in the **Node.js runtime by default**. There is no `runtime` export option in proxy files — it always runs Node.js. This means `crypto.timingSafeEqual` and `crypto.createHmac` are available directly in `src/proxy.ts`.

The two-layer check described in the existing proxy comments (presence check in proxy, full HMAC verification in route handlers) is still valid as defense-in-depth, but the proxy can now also do HMAC verification for Bearer tokens without a separate route handler.

### 4.3 Bearer token strategy

**Token storage:** A single 32-byte hex token (`BRAIN_LAN_TOKEN`) stored in `.env` (already in `.gitignore`). Generated on first start if absent.

**Token embedding in APK:** The token is baked into `local.properties` as `brain.lanToken=<hex>` at APK build time, then read via a `BuildConfig` constant. This is the approach R-AUTH §8 recommends for v0.5.0.

**Token in extension:** Stored in `chrome.storage.local` (set via the extension's options/popup UI on first setup). Never baked into the extension source.

### 4.4 Proxy middleware sketch (not production code — planning reference)

```typescript
// src/proxy.ts — v0.5.0 changes (additions to existing check)

const BEARER_ROUTES = ['/api/capture'];   // only these routes accept bearer tokens

export function proxy(req: NextRequest) {
  const { pathname } = req.nextUrl;

  // 1. Public paths — unchanged
  if (pathname.startsWith('/api/auth') || PUBLIC_PATHS.has(pathname)) {
    return NextResponse.next();
  }

  // 2. Check session cookie (existing path — browser clients, APK WebView after first auth)
  const sessionToken = req.cookies.get(SESSION_COOKIE)?.value;
  if (sessionToken) {
    // HMAC presence check (proxy); full verification in route handler (unchanged)
    return NextResponse.next();
  }

  // 3. Check Bearer token (new path — APK first request, extension)
  if (pathname.startsWith('/api/') && BEARER_ROUTES.some(p => pathname.startsWith(p))) {
    const auth = req.headers.get('authorization');
    if (auth?.startsWith('Bearer ')) {
      const provided = auth.slice(7);
      const expected = process.env.BRAIN_LAN_TOKEN ?? '';
      if (provided.length === expected.length && provided === expected) {
        // timingSafeEqual not usable here (string comparison) — do HMAC check in route handler
        // The proxy check is just a fast-reject; route handler re-validates with timingSafeEqual
        return NextResponse.next();
      }
    }
  }

  // 4. Reject — unchanged redirect/401 logic
  if (pathname.startsWith('/api/')) {
    return NextResponse.json({ error: 'unauthenticated' }, { status: 401 });
  }
  const url = req.nextUrl.clone();
  url.pathname = '/unlock';
  url.searchParams.set('next', pathname);
  return NextResponse.redirect(url);
}
```

**Note on timing-safe comparison in proxy:** The proxy compares bearer tokens with a string equality check (fast reject). This is intentionally NOT timing-safe. The attack surface for a timing oracle on a local HTTP server is essentially zero. Full `timingSafeEqual` re-verification happens in the route handler (or in a shared `verifyBearerToken(req)` helper). This two-tier approach keeps the proxy lean.

**Note on BRAIN_LAN_TOKEN availability in proxy:** Next.js 16 proxy runs Node.js, and `process.env` is available. However, `BRAIN_LAN_TOKEN` must be set as a server-side env var (no `NEXT_PUBLIC_` prefix). Verify with `next.config.ts` env pass-through if needed.

### 4.5 CSRF posture

**APK WebView requests:**
- With `server.url = "http://brain.local:3000"`, the WebView page's origin is `http://brain.local:3000`.
- The Next.js server listens on `brain.local:3000`.
- These are **same-origin** from the browser's perspective.
- `SameSite=Strict` cookies ARE sent for same-origin requests — no CSRF issue for APK WebView navigation.
- For API calls from the APK that use Bearer token (not cookie), CSRF is moot by definition.

**Chrome extension requests:**
- Origin is `chrome-extension://<extension-id>`.
- This is cross-origin relative to `http://localhost:3000`.
- `SameSite=Strict` cookies are NOT sent — extension must use Bearer token for all requests.
- Origin-header validation in the route handler: accept requests where `Origin` is `null` (no Origin header from extension service workers), `http://localhost:3000`, `http://brain.local:3000`, or `chrome-extension://<known-id>`. Reject all others.

**Summary of CSRF posture for v0.5.0:**
- Browser (PIN-unlocked session) — `SameSite=Strict` cookie, same-origin → immune to CSRF.
- APK WebView (same-origin after `brain.local` resolves) — same-origin → immune to CSRF.
- Extension popup / service worker — Bearer token only; no cookie involved → CSRF irrelevant by design.
- `SameSite=Strict` on all cookies — unchanged from v0.3.1 F-043.

No double-submit cookie or separate CSRF token is needed for v0.5.0 given the bearer-token approach for cross-origin clients.

### 4.6 Rate limiter

**Where:** In `src/proxy.ts` (proxy layer, runs before route handlers, no DB access needed).

**Mechanism:** In-process `Map<string, number[]>` keyed by client IP, storing timestamps of recent requests within a sliding 60-second window.

**Constraint:** The proxy module is loaded once per server process. Hot-module reload resets the map (acceptable for dev). In prod (`npm run start:lan`), the map persists for the server's lifetime. This is acceptable for a single-user home app.

```typescript
// Rate limiter (inline in proxy.ts — planning reference, not production code)
const RATE_MAP = new Map<string, number[]>();   // ip -> [timestamp, ...]
const RATE_LIMIT = 10;       // requests
const RATE_WINDOW = 60_000;  // ms

function isRateLimited(ip: string): boolean {
  const now = Date.now();
  const times = (RATE_MAP.get(ip) ?? []).filter(t => now - t < RATE_WINDOW);
  if (times.length >= RATE_LIMIT) return true;
  RATE_MAP.set(ip, [...times, now]);
  return false;
}
```

Apply only to `BRAIN_LAN_TOKEN`-authenticated requests on `/api/capture/*`. Session-cookie-authenticated requests (browser) are exempt from this rate limiter (the user is already verified via PIN).

**Why not SQLite-backed?** A SQLite write per request on the rate-limit table introduces a DB connection in the proxy layer, which conflicts with the proxy's design as a fast, stateless interceptor. The in-process Map is sufficient for a single-user home tool.

## 5. mDNS + fallback plan

### 5.1 What ships as primary: `brain.local` via `bonjour-service`

**Registration** (in `src/instrumentation.ts`, alongside the backup scheduler):

```typescript
// src/instrumentation.ts (Node.js register hook — existing)
import { Bonjour } from 'bonjour-service';

export async function register() {
  // ... existing backup scheduler boot ...

  // mDNS advertisement (F-035)
  const bonjour = new Bonjour();
  bonjour.publish({
    name: 'Brain',
    type: 'http',
    port: 3000,
    host: 'brain.local',
  });
  // Do not call bonjour.destroy() — keep advertising for server lifetime
}
```

**How it works:** `bonjour-service` sends multicast DNS advertisements over UDP port 5353. Any device on the same LAN that queries `brain.local` gets the Mac's current LAN IP back. DHCP reassignment is transparent — the next query returns the new IP without any APK rebuild.

**macOS coexistence:** macOS has `/usr/sbin/mDNSResponder` running. `bonjour-service` uses pure Node.js UDP multicast (via `multicast-dns`) independently. Both can advertise services on the same LAN. macOS's `mDNSResponder` handles `.local` for Apple services (AirDrop, AirPrint, etc.); `bonjour-service` handles `brain.local` specifically. There is no conflict as long as the service name is unique (it is).

### 5.2 Android WebView mDNS resolution

**Findings:**
1. Chrome's DNS layer explicitly uses the **system resolver** for all `.local` hostnames (confirmed by Chromium DNS README: "For hostnames ending in '.local' the system resolver is always used").
2. Android's system resolver has supported mDNS since **Android 12 (API 31)**.
3. The target Pixel runs Android 14 (API 34) or Android 15 (API 35) — well above the API 31 threshold.
4. Chrome Android (and Android WebView) therefore delegates `brain.local` resolution to the system resolver, which uses the Android mDNS stack.

**Confidence level:** MEDIUM. This chain of logic is correct in theory. However, the v0.0.1 spike did NOT test mDNS on a real Pixel — it used `10.0.2.2` (AVD alias). The first real-device test may reveal quirks. A `PENDING: verify brain.local on Pixel` item must be in the v0.5.0 plan.

**Known WebView caveat:** Android WebView before API 34 (Chrome 108-equivalent) had a bug where `.local` mDNS resolution was blocked in some build configurations (related to Chromium's async DNS resolver treating mDNS differently from the synchronous system resolver). API 34+ WebViews use the system resolver path. Since the target is API 34 (`compileSdkVersion = 36`, `minSdkVersion = 24`), devices running Android 14 are safe.

### 5.3 Fallback plan (mandatory)

The `brain.local` mDNS advertises at the OS level, but we cannot guarantee the Android device resolves it on first test. Two fallback layers:

**Layer 1 — QR-code IP display (F-038, primary fallback):**
- Settings screen on the Mac's web UI shows a QR code containing `http://192.168.x.x:3000?token=<lanToken>`.
- The APK has a "Scan QR" setup screen (one-time, first launch) that reads the QR and stores the IP + token in `SharedPreferences`.
- When `brain.local` fails to resolve, the APK falls back to the stored IP.
- The IP is also shown as text for manual entry.

**Layer 2 — Configurable server URL in APK:**
- The APK settings screen has a text field to override the server URL (default: `http://brain.local:3000`; override: `http://192.168.1.17:3000`).
- This field is populated by QR scan or manual entry and stored in `SharedPreferences`.

**Decision tree at APK startup:**
```
Is brain.local resolvable? (timeout: 2s)
  YES → Use http://brain.local:3000
  NO  → Is a stored override IP in SharedPreferences?
          YES → Use stored IP:3000, show a toast "Using saved IP (brain.local unreachable)"
          NO  → Show "Setup" screen with QR scan + manual entry
```

**Exit criteria for F-035 (mDNS):** `brain.local` resolves on the Pixel AND appears on Mac library in ≤2 s. If mDNS fails on first device test, the QR-code IP fallback (Layer 1) is the go/no-go gate. The DHCP-reassignment-doesn't-require-APK-rebuild requirement is met by Layer 1 (QR scanned IP is stored) + Layer 2 (override field in settings).

## 6. Chrome MV3 extension path

### 6.1 Build tool: `@crxjs/vite-plugin@2.4.0`

**Decision:** Use `@crxjs/vite-plugin@2.4.0` over Plasmo.

**Rationale:**
- `@crxjs/vite-plugin@2.4.0` was released 2026-03-16 (6 weeks ago) — actively maintained.
- Plasmo `0.90.5` was last updated 2025-05-17 (12 months ago) — stagnant.
- crxjs is framework-agnostic and maps directly to a standard `manifest.json` — no Plasmo-specific abstractions to learn or fight.
- The extension is small (popup + background service worker + ~100 lines of logic). Full-framework Plasmo is over-engineered for this scope.
- crxjs HMR integration with Vite 8 provides a faster development loop.

**Extension lives in:** `extension/` at the project root (sibling to `src/`, not inside it). Separate `package.json`, separate Vite build. The Next.js build is unaffected.

### 6.2 Architecture

```
extension/
├── manifest.json          MV3 manifest
├── package.json           Vite + @crxjs/vite-plugin deps
├── vite.config.ts         crxjs plugin config
├── src/
│   ├── popup.html         Extension popup (80×300px)
│   ├── popup.ts           Reads active tab, sends to Brain
│   ├── background.ts      Service worker: context menu handler
│   └── settings.ts        Options page: configure Brain URL + token
└── public/
    └── icon.png           128×128 Brain icon
```

**No content script needed.** The popup reads the active tab URL via `chrome.tabs.query` — no DOM injection required.

### 6.3 Permission list (manifest.json)

```json
{
  "manifest_version": 3,
  "name": "Brain",
  "version": "0.5.0",
  "permissions": ["activeTab", "tabs", "contextMenus", "storage"],
  "host_permissions": [
    "http://localhost:3000/*",
    "http://brain.local:3000/*"
  ],
  "action": {
    "default_popup": "src/popup.html",
    "default_icon": "public/icon.png"
  },
  "background": {
    "service_worker": "src/background.ts",
    "type": "module"
  },
  "options_page": "src/settings.html"
}
```

**Permission justification:**
- `activeTab`: grants temporary access to the current tab on user click.
- `tabs`: required to read `tab.url` and `tab.title` (sensitive properties — not available with `activeTab` alone per Chrome docs).
- `contextMenus`: right-click "Save to Brain" on any page.
- `storage`: persist Brain server URL + token in `chrome.storage.local`.
- `host_permissions`: required for `fetch()` from service worker to `localhost:3000` or `brain.local:3000`.

### 6.4 Service worker localhost fetch (Chrome 147)

The Chrome 114 regression around service worker + localhost fetch is fully resolved in Chrome 147. Chrome 147 service workers can `fetch('http://localhost:3000/...')` without restrictions as long as the `host_permissions` in `manifest.json` includes the target host. This was confirmed by the Chrome DevRel team in the Chrome 120 release notes. No workaround needed.

**PENDING: verify this claim at plan time.** Run a minimal service-worker fetch to localhost and observe the Network panel in the Extension DevTools. If it fails, the fallback is to move the fetch to the popup page context (which definitely CAN fetch localhost) and only use the service worker for context-menu events.

### 6.5 Popup flow

1. User clicks Brain extension icon (or right-clicks → "Save to Brain").
2. Popup opens (or context menu handler fires).
3. `chrome.tabs.query({active: true, currentWindow: true})` → returns `{url, title}`.
4. Show pre-filled form: URL (read-only), title (editable), note (optional).
5. On submit: POST `{url, title, note}` to `http://brain.local:3000/api/capture/url` with `Authorization: Bearer <token>`.
6. Show success toast or error message in the popup.

**Token retrieval:** `chrome.storage.local.get(['brain_token', 'brain_url'])` is synchronous from the popup page context. If token is not set, show the settings page instead of the capture form.

### 6.6 Build pipeline

```bash

# npm scripts:

  "build": "vite build",

```

No separate `npm run build:extension` in the root `package.json` is needed for v0.5.0 (the planner can decide whether to add a root-level script).

## 7. Known risks

### RISK-1: Android WebView mDNS resolution not proven on real Pixel
**Details:** The v0.0.1 spike used AVD `10.0.2.2` alias. `brain.local` has never been tested on a Pixel. The chain (Chrome system resolver → Android mDNS stack) should work on API 34+, but has not been verified.
**Mitigation:** QR-code fallback (F-038 / Layer 1 above) is mandatory, not optional. Include a Day-1 Pixel test of `brain.local` as a blocking step before the F-035 exit criterion.
**Accept or escalate:** Escalate only if mDNS fails on real device AND the QR fallback also fails (unlikely).

### RISK-2: `@capgo/capacitor-share-target` event API differs from R-CAP docs
**Details:** R-CAP (`docs/research/android-share.md`) documents the capawesome plugin API (`appShareReceived`, `{type, text, url}`). The actual Capgo plugin API (confirmed in v0.0.1 spike logcat) is `shareReceived` and `{title, texts[], files[]}`. Any code copied from R-CAP's §4 will be wrong.
**Mitigation:** Use ONLY the spike's `www/index.html` code as the reference for plugin API, not R-CAP. Document the correct API explicitly in the v0.5.0 plan task for F-014 / CAP-6.
**Accept:** ACCEPTED — fully documented here; no escalation needed.

### RISK-3: `src/proxy.ts` comment about edge runtime is outdated
**Details:** The comment says "edge runtime cannot use node:crypto." Since Next.js 16 runs proxy in Node.js by default, `crypto` is available in `src/proxy.ts`. Copying the comment's logic into new bearer-token code may lead to unnecessary indirection (e.g., moving token verification to a route handler when proxy can do it).
**Mitigation:** Update the comment in F-016 task. Use `node:crypto` directly in proxy for HMAC comparison.
**Accept:** ACCEPTED — document in plan.

### RISK-4: `brain.local` conflicts with other `.local` names on network
**Details:** `bonjour-service` will publish `brain.local`. If another device on the LAN is already advertising `brain.local`, there will be a conflict.
**Mitigation:** The name `brain.local` is distinctive enough to avoid conflicts in a typical home network. In the unlikely event of a conflict, the user can override with a custom name in settings.
**Accept:** ACCEPTED — low probability, easy user workaround.

### RISK-5: Android Studio version below Capacitor 8's stated requirement
**Details:** Installed AS is 2023.1 (Hedgehog). Capacitor 8 docs say 2025.2.1 (Otter) is required. However, the v0.0.1 spike built a debug APK successfully using `./gradlew assembleDebug` with this exact AS installation.
**Mitigation:** Continue using CLI builds (`./gradlew`, `cap sync`). Do not rely on Android Studio IDE features during v0.5.0. If a future task REQUIRES AS IDE (e.g., device manager for new AVD), upgrade then.
**Accept:** ACCEPTED — CLI build path is proven; AS upgrade is low-friction if needed.

### RISK-6: Cold-start double-fire in `@capgo/capacitor-share-target`
**Details:** Confirmed in v0.0.1 spike: the plugin fires `shareReceived` twice on cold start. The second fire happens on app resume. Without dedup, the APK will submit the same URL twice.
**Mitigation:** F-041 (2-second dedup window) is already in the feature list. Implement using a `Set<string>` with a 2s TTL:
```typescript
const recentShares = new Map<string, number>(); // hash -> timestamp
function isDuplicate(hash: string): boolean {
  const t = recentShares.get(hash);
  if (t && Date.now() - t < 2000) return true;
  recentShares.set(hash, Date.now());
  return false;
}
```
**Accept:** ACCEPTED — implementation is trivial; already in scope as F-041.

### RISK-7: Chrome MV3 service worker fetch to localhost
**Details:** The Chrome 114 regression (service workers blocked from fetching localhost) was reported to be fixed. Chrome 147 is current. However, "PENDING: verify" applies here.
**Mitigation:** If service worker fetch fails, move the POST logic to popup context and use `chrome.runtime.sendMessage` from service worker → popup → fetch. This is a 30-minute fallback.
**Accept:** ACCEPTED — low risk at Chrome 147; fallback is trivial.

### RISK-8: `SameSite=Strict` cookie not sent on APK's first navigation
**Details:** When the APK WebView first navigates to `http://brain.local:3000`, there is no `brain-session` cookie (it's first launch). The user must PIN-unlock via the `/unlock` page. After unlocking, the session cookie is set and all subsequent navigation works. This is the intended flow but must be explicitly handled.
**Mitigation:** The APK `/capture` screen must handle the `401 → redirect to /unlock → return to /capture` flow. This is a UI task, not a blocking architectural issue.
**Accept:** ACCEPTED — normal auth flow.

### RISK-9: `network_security_config.xml` approach with dynamic LAN IPs
**Details:** The recommended `network_security_config.xml` uses domain-specific cleartext allowances (`brain.local`). If the user overrides to a raw IP (e.g., `192.168.1.17`), that IP must also be in the config. However, IPs vary by network.
**Mitigation:** Use a permissive domain-config for the `192.168.*.*` subnet range. Alternatively: use `server.cleartext = true` globally (simpler, slightly less correct but acceptable for a home-only tool) and add a comment. The planner decides.
**Recommendation:** Use `cleartext: true` for v0.5.0 simplicity; switch to `network_security_config.xml` in v0.10.0 hardening pass.
