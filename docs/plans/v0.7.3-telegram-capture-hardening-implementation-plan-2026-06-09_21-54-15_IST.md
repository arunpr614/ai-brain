# v0.7.3 Telegram Capture Hardening Implementation Plan

**Created:** 2026-06-09 21:54:15 IST  
**Status:** planned  
**Source audit:** `ReviewReport/TELEGRAM_CAPTURE_BOT_ADVERSARIAL_AUDIT_2026-06-09_21-46-36_IST.md`  
**Target branch:** `codex/v0.7.3-telegram-capture-hardening`  
**Primary goal:** Make Telegram capture safe enough to treat as a reliable always-on capture channel, with parity against Android/web where it matters and clear operational runbooks.

---

## 1. Executive Summary

The current Telegram bot implementation is a solid first version: it validates Telegram's webhook secret header, checks the configured owner user ID, and reuses the same capture helpers as Android/web. The audit found that the weak points are mostly around operational safety and public-webhook hardening rather than the core architecture.

This plan keeps the direct-dispatch design and the zero-dependency Telegram client. It adds:

- private-chat-only capture
- runtime payload validation
- durable Telegram `update_id` idempotency
- capture-channel provenance
- note/PDF duplicate protection
- webhook bad-secret throttling
- URL private-network protection
- Telegram client timeouts
- a real Telegram smoke script
- secret-hygiene cleanup guidance
- updated as-built documentation

---

## 2. Non-Goals

- Do not replace the Telegram client with Telegraf or another bot framework.
- Do not redesign the full capture pipeline.
- Do not add multi-user support.
- Do not make Telegram a general chat assistant.
- Do not create real production library items during automated release smoke unless explicitly enabled.
- Do not rewrite local git history automatically; risky local history/stash cleanup must be deliberate.

### T-2. Add Telegram Runtime Payload Validation

**Purpose:** Prevent malformed signed payloads from turning into runtime exceptions or broken error handling.

**Files**

- `src/lib/telegram/schema.ts` new
- `src/lib/telegram/types.ts`
- `src/lib/telegram/webhook-handler.ts`
- `src/app/api/telegram/webhook/route.test.ts`

**Design**

Add a narrow `zod` schema for only the Telegram fields Brain consumes:

- `update_id`: required integer
- `message`: optional object
- `message.message_id`: required integer if message exists
- `message.chat.id`: required integer
- `message.chat.type`: `private | group | supergroup | channel`
- `message.from.id`: optional integer
- `message.from.is_bot`: optional boolean
- `message.text`: optional string, max 100,000
- `message.caption`: optional string, max 10,000
- `message.entities`: bounded array
- `message.caption_entities`: bounded array
- `message.document.file_id`: required string if document exists
- `message.document.file_unique_id`: required string if document exists
- `message.document.file_size`: optional integer
- `message.document.mime_type`: optional string

**Behavior**

- Malformed JSON remains `200` silent, as today.
- Valid JSON that fails Telegram schema validation logs `telegram.webhook.invalid-payload` and returns `200`.
- Valid schema with no `message` returns `200`.

**Tests**

- Signed payload missing `chat` returns `200` and does not dispatch.
- Signed payload with invalid `chat.type` returns `200` and does not dispatch.
- Oversized `text` or too many entities returns `200` and does not dispatch.
- Valid owner message still dispatches.

**Acceptance**

- A forged signed payload missing required Telegram fields returns `200` and creates no item.
- No error path tries to access `msg.chat.id` unless schema validation has proved it exists.

### T-3. Enforce Private Chat Only

**Purpose:** Prevent the bot from saving group messages or leaking private Brain links into Telegram groups.

**Files**

- `src/lib/telegram/webhook-handler.ts`
- `src/app/api/telegram/webhook/route.test.ts`
- `docs/plans/v0.6.5-telegram-capture.md` as-built note

**Behavior**

- After schema validation and owner check, require `msg.chat.type === "private"`.
- For owner messages in group/supergroup/channel:
  - return `200`
  - log `telegram.webhook.non-private-chat`
  - do not call dispatch
  - do not send a group reply

**Tests**

- Owner in `group` chat creates no item and sends no acknowledgement.
- Owner in `supergroup` chat creates no item and sends no acknowledgement.
- Owner in `channel` chat creates no item and sends no acknowledgement.
- Owner in `private` chat still works.

**Acceptance**

- Owner messages outside private chat leak no item title or item URL.

### T-4. Add Durable Telegram Update Idempotency

**Purpose:** Prevent duplicate notes/PDFs from Telegram retries, webhook replays, or repeated delivery.

**Files**

- `src/db/migrations/011_telegram_updates.sql` new
- `src/db/telegram-updates.ts` new
- `src/db/telegram-updates.test.ts` new
- `src/lib/telegram/webhook-handler.ts`
- `src/lib/telegram/dispatch.ts`
- `src/lib/telegram/dispatch.test.ts`

**Schema**

Create `telegram_updates`:

```sql
CREATE TABLE telegram_updates (
  update_id   INTEGER PRIMARY KEY,
  message_id  INTEGER,
  chat_id     INTEGER,
  from_id     INTEGER,
  file_unique_id TEXT,
  item_id     TEXT REFERENCES items(id) ON DELETE SET NULL,
  status      TEXT NOT NULL CHECK (status IN ('received', 'ignored', 'captured', 'failed')),
  error       TEXT,
  created_at  INTEGER NOT NULL DEFAULT (unixepoch() * 1000),
  handled_at  INTEGER
);

CREATE INDEX idx_telegram_updates_status ON telegram_updates(status);
CREATE INDEX idx_telegram_updates_file_unique_id ON telegram_updates(file_unique_id)
  WHERE file_unique_id IS NOT NULL;
```

**Repository API**

- `claimTelegramUpdate(input): "claimed" | "duplicate"`
- `markTelegramUpdateIgnored(updateId, reason)`
- `markTelegramUpdateCaptured(updateId, itemId)`
- `markTelegramUpdateFailed(updateId, error)`
- `findTelegramDocumentByUniqueId(fileUniqueId)`

**Dispatch Contract Change**

Change `handleCaptureMessage()` to return a structured result:

```ts
type TelegramCaptureResult =
  | { status: "captured"; itemId: string; source: "url" | "youtube" | "note" | "pdf" }
  | { status: "duplicate"; itemId?: string; reason: string }
  | { status: "ignored"; reason: string }
  | { status: "failed"; reason: string; retryable: boolean };
```

The webhook handler owns update tracking. Dispatch owns capture logic.

**Tests**

- Same `update_id` posted twice dispatches once.
- Replayed `update_id` returns `200` without a second item.
- Captured update records `item_id`.
- Failed update records status and error.

**Acceptance**

- Reposting the same `update_id` creates at most one item.

### T-5. Add Capture Provenance Without Losing Content Type

**Purpose:** Make "captured via Telegram" visible without abusing `source_type`.

**Decision**

Use a new field named `capture_source`, not `source_type = telegram`.

`source_type` continues to mean content type:

- `url`
- `youtube`
- `pdf`
- `note`
- etc.

`capture_source` means ingestion channel:

- `web`
- `android`
- `extension`
- `telegram`
- `system`
- `unknown`

**Files**

- `src/db/migrations/012_capture_source.sql` new
- `src/db/client.ts`
- `src/db/items.ts`
- `src/app/api/capture/url/route.ts`
- `src/app/api/capture/note/route.ts`
- `src/app/api/capture/pdf/route.ts`
- `src/app/capture-actions.ts`
- `src/lib/telegram/dispatch.ts`
- relevant tests

**Schema**

```sql
ALTER TABLE items ADD COLUMN capture_source TEXT NOT NULL DEFAULT 'web'
  CHECK (capture_source IN ('web', 'android', 'extension', 'telegram', 'system', 'unknown'));

CREATE INDEX idx_items_capture_source ON items(capture_source);
```

If SQLite cannot add this exact `CHECK` constraint in-place in the current migration environment, use the same table-rebuild pattern already used by migrations 008/009.

**Insert Behavior**

- Existing web form actions: `capture_source = "web"`
- Bearer capture endpoint requests:
  - If caller sends a trusted header such as `X-Brain-Capture-Source`, allow `android` or `extension` only after bearer auth/proxy already passed.
  - If absent, default to `unknown` or `android` based on current clients. Prefer `unknown` to avoid false certainty.
- Telegram dispatch: `capture_source = "telegram"`
- System/backfill inserts: `system` if applicable, otherwise default.

**Tests**

- Telegram URL insert stores `source_type=url`, `capture_source=telegram`.
- Telegram YouTube insert stores `source_type=youtube`, `capture_source=telegram`.
- Telegram PDF insert stores `source_type=pdf`, `capture_source=telegram`.
- Telegram note insert stores `source_type=note`, `capture_source=telegram`.
- Existing capture tests still pass with defaults.

**Acceptance**

- Library/debug queries can distinguish content type from capture channel.
- Migration 009 docs are corrected so they no longer imply `source_type=telegram` is how bot provenance works.

### T-6. Add Telegram Note Dedup And Command Handling

**Purpose:** Avoid saving `/start`, `/help`, and accidental duplicate notes.

**Files**

- `src/lib/telegram/dispatch.ts`
- `src/lib/telegram/dispatch.test.ts`

**Command Behavior**

Before URL/note capture, detect bot commands from `entities` or leading text:

- `/start`: send concise help text, return ignored
- `/help`: send concise help text, return ignored
- other `/command`: send concise supported-capture message, return ignored

Do not save command messages as notes.

**Note Dedup Behavior**

Reuse the HTTP note route's title/body hash pattern:

- compute hash from normalized title/body
- call `isDuplicateShare(shareDedupKey("note", hash))`
- on duplicate, send a short "already received" acknowledgement and return duplicate/ignored

Durable `update_id` remains the primary replay guard; short-window dedup protects accidental double sends with distinct update IDs.

**Tests**

- `/start` sends help and creates no item.
- `/help` sends help and creates no item.
- `/unknown` creates no item.
- Duplicate text note within dedup window creates one item.

**Acceptance**

- Telegram commands do not pollute the library.
- Duplicate note sends within the short dedup window do not create duplicate notes.

### T-7. Harden Telegram PDF Capture

**Purpose:** Bring Telegram PDF handling closer to Android/web integrity and duplicate safety.

**Files**

- `src/lib/telegram/dispatch.ts`
- `src/lib/telegram/dispatch.test.ts`
- possibly `src/db/telegram-updates.ts`

**Behavior**

- Keep existing MIME and declared-size checks.
- After download, enforce `bytes.byteLength <= PDF_MAX_BYTES` even if Telegram did not provide `file_size`.
- Compute SHA256 after download.
- Log SHA256 on PDF failures.
- Use `file_unique_id` in `telegram_updates` to detect repeated Telegram document sends.
- If the same `file_unique_id` was previously captured, send an "already captured" acknowledgement with the existing item link when available.

**Tests**

- PDF with missing `file_size` but downloaded bytes over max is rejected.
- PDF failure log includes SHA256 when bytes were available.
- Same `file_unique_id` does not create duplicate item.

**Acceptance**

- Telegram PDF capture records or logs SHA256 and enforces size after download.
- Re-sent Telegram PDF document does not create duplicates when `file_unique_id` matches.

### T-8. Classify Telegram Failures

**Purpose:** Avoid losing messages silently when failures are transient.

**Files**

- `src/lib/telegram/dispatch.ts`
- `src/lib/telegram/webhook-handler.ts`
- `src/lib/telegram/dispatch.test.ts`
- `src/app/api/telegram/webhook/route.test.ts`

**Failure Classes**

User/content failures:

- unsupported document type
- URL unreadable
- PDF extraction failure
- too-short note
- unsupported media

Behavior: return `200`, notify owner if private chat, mark update failed/ignored.

Infrastructure/transient failures:

- DB insert failure before item creation
- Telegram file download network timeout
- Telegram API timeout before dispatch completes
- extractor unexpected exception

Behavior:

- If no item was created, mark update failed and return `503` only for clearly retryable failures.
- If item was created but acknowledgement failed, return `200` and log `telegram.ack.failed`.

**Tests**

- User-content failure returns `200`.
- Retryable infrastructure failure before item creation returns `503` and marks failed retryable.
- Ack failure after item creation does not duplicate item and returns `200`.

**Acceptance**

- Retryable failures are not treated the same as permanent user-content failures.

### T-9. Add Webhook Bad-Secret Rate Limiting

**Purpose:** Reduce public endpoint noise and log churn from unsigned/incorrect webhook requests.

**Files**

- `src/lib/telegram/webhook-rate-limit.ts` new
- `src/lib/telegram/webhook-handler.ts`
- `src/app/api/telegram/webhook/route.test.ts`

**Design**

Add a small in-memory limiter for bad-secret attempts:

- key by `cf-connecting-ip`, then `x-forwarded-for`, then `"unknown"`
- default: 20 bad-secret attempts per minute per key
- on limit: return `429` with `Retry-After: 60`
- do not rate-limit valid Telegram secret requests

**Tests**

- repeated wrong-secret calls eventually return `429`
- valid-secret calls are not blocked by previous bad-secret attempts from another key

**Acceptance**

- Public unauthenticated spam cannot create unlimited log churn.

### T-10. Add Shared Private-Network URL Guard

**Purpose:** Prevent all capture surfaces from fetching private/internal network URLs from the server.

**Files**

- `src/lib/capture/url-safety.ts` new
- `src/lib/capture/url.ts`
- `src/lib/capture/url-safety.test.ts` new
- `src/lib/capture/url.test.ts` or existing URL capture tests

**Guard Rules**

Reject:

- `localhost`
- `.localhost`
- loopback IPv4/IPv6
- RFC1918 private IPv4
- link-local IPv4/IPv6
- unique-local IPv6
- multicast/reserved ranges where practical
- cloud metadata IPs such as `169.254.169.254`

Implementation approach:

- parse hostname
- reject obvious hostnames immediately
- if hostname is an IP, classify directly
- if hostname is DNS, resolve with Node DNS and reject if any resolved address is private/reserved
- apply guard before `fetch()`

**Tests**

- rejects `http://localhost`
- rejects `http://127.0.0.1`
- rejects `http://10.0.0.1`
- rejects `http://192.168.1.1`
- rejects `http://169.254.169.254`
- rejects IPv6 loopback
- allows `https://example.com`
- mocked DNS resolution rejects domains resolving to private IP

**Acceptance**

- URL private-network protection applies to Telegram, Android, web, and extension because all share `extractArticleFromUrl()`.

### T-13. Implement Real Telegram Smoke Test

**Purpose:** Make the deploy-time `TELEGRAM_RELEASE=1` path actually work.

## 7. Risk Notes

- The `capture_source` migration touches the core `items` table. Keep migration tests focused and run the full suite.
- URL private-network blocking can reject edge-case legitimate intranet captures. That is acceptable for production default; document it.
- Returning `503` for retryable Telegram failures must be used carefully. If the handler creates an item and then returns `503`, Telegram can retry and duplicate unless `update_id` idempotency is already in place. Implement T-4 before T-8.
- Secret rotation is partly manual and should not be hidden inside app code.
