## 1. Problem statement

User browses the web; when a page they're reading is about something already in their Brain library, they have no way of knowing. The Chrome extension currently only *captures* pages — it can't tell the user "you've already saved 3 things about this topic."

Recall.it's augmented-browsing feature does this: highlights phrases on any page that match saved content, offers a side-panel to open/chat with the matches. Brain has no equivalent.

## 2. Goal

Give the user a passive, privacy-respecting signal — while they browse, their library reaches out and says "I have something about this" — without sending their browsing history to the Brain server.

## 3. Scope — what's in the MVP

Per user decision 2026-05-12 (plan umbrella §B): **title + tags + summary** local matching. No body-text matching; no server-side per-page matching.

### Shipping (AUG-1 through AUG-10)

| ID | Capability | Why it's in MVP |
|---|---|---|
| **AUG-1** | `GET /api/index/keywords` endpoint — returns `{ items: [{ id, title, tags, summary }] }` for all enriched items | Foundation; everything else depends on this |
| **AUG-2** | Extension content script that injects on all URLs | Required to see page text and inject highlights |
| **AUG-3** | `chrome.storage.local`-cached keyword index, refreshed every 5 minutes | Privacy floor; no per-page server calls |
| **AUG-4** | Case-insensitive substring match of page visible text against index | The actual highlighting engine |
| **AUG-5** | Shadow-DOM highlight badges over matched phrases | Visual indicator that doesn't clash with page CSS |
| **AUG-6** | Shadow-DOM side-panel showing matched items with titles + "Open in Brain" links | User's path to the saved content |
| **AUG-7** | Options-page toggle: "Augmented browsing ON/OFF" (default OFF) | Opt-in; user explicitly enables |
| **AUG-8** | Options-page per-site suppression list ("never highlight on these domains") | Essential escape hatch for sensitive sites |
| **AUG-9** | Global keyboard shortcut to toggle on/off for current tab | Ergonomic override |
| **AUG-10** | Plain-language error surfaces (index fetch failed, token expired, etc.) | Matches the P1a error-message style shipped in commit 59bba64 |

### Deferred (explicitly post-MVP)

From Recall's 13-row capability set in audit v2:
- Connection-counter widget (total matches across all open tabs)
- Hidden-entities registry (user "don't highlight this phrase ever" list)
- Server-side body-text matching (requires `POST /api/match` + rate-limiter work)
- AI chat from matched item (requires `/api/ask` bearer access, explicit non-goal per plan umbrella §A)
- Precision location mapping in body text (node.nodeValue offsets at scale)

## 4. Architecture

### 4.1 Data flow

```
Chrome page navigation
  ↓
Content script loads (injected via manifest content_scripts)
  ↓
Reads cached index from chrome.storage.local
  ├─ If stale (>5 min old): trigger refresh from /api/index/keywords
  └─ If missing: fetch immediately
  ↓
Walks visible DOM (TreeWalker over text nodes)
  ↓
For each text node: case-insensitive substring match against index
  ↓
For each match: wrap matched phrase in a shadow-DOM highlight element
  ↓
Click highlight → open shadow-DOM side-panel with matched items
  ↓
Click item link → chrome.tabs.create({ url: "https://brain.arunp.in/items/{id}" })
```

Zero per-page calls to Brain. The only Brain traffic is the 5-minute index refresh.

### 4.3 Manifest changes (exact)

```json
{
  "permissions": [
    "activeTab",
    "tabs",
    "contextMenus",
    "storage",
    "notifications",
    "scripting"
  ],
  "host_permissions": [
    "https://brain.arunp.in/*",
    "<all_urls>"
  ],
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "exclude_matches": ["https://brain.arunp.in/*"],
      "js": ["src/content.js"],
      "run_at": "document_idle"
    }
  ]
}
```

The `<all_urls>` permission is privacy-sensitive and will trigger a Chrome Web Store review warning. For this single-user personal extension it's acceptable — user is the only installer.

### 4.4 `/api/index/keywords` endpoint

- **Method:** `GET`
- **Auth:** cookie OR bearer (dual-check pattern from `src/app/api/capture/pdf/route.ts`)
- **Response shape:**
  ```json
  {
    "items": [
      { "id": "string", "title": "string", "tags": ["string"], "summary": "string or null" }
    ],
    "generated_at": 1715529600000
  }
  ```
- **Row count:** all items where `enrichment_state = 'done'` (cap at 5000 for payload sanity; if library grows beyond, add pagination later)
- **Caching:** client-cached in `chrome.storage.local` under key `brain_kw_index`; extension decides refresh cadence (5 min default)
- **No body text.** `summary` is the LLM-generated 1–3 sentence concise summary, usually <500 chars. Full response payload at 5k items: ~2 MB uncompressed. Serve gzipped by Next.js default.
- **Diagnostic logging:** use `console.warn`, never `logError` (see `src/lib/auth/no-destructive-gets.test.ts` gotcha)

## 5. Test plan

### Unit tests

- `extension/src/matcher.test.ts` — pure matching logic; no chrome APIs; test case-sensitivity, word boundaries, punctuation edges, Unicode.
- `src/app/api/index/keywords/route.test.ts` — cookie path, bearer path, unauth path; payload shape; excludes unenriched items.
- `src/lib/auth/bearer.test.ts` — `isBearerRoute("/api/index/keywords") === true`.

### Manual verification

1. `npm run build` (root) still clean.
2. `cd extension && npm run build` clean; new `dist/src/content.js` present.
3. Load unpacked in Chrome; reload Brain card; confirm permissions dialog lists the new `<all_urls>` scope (you approve once).
4. Options page: toggle Augmented Browsing ON; close options.
5. Navigate to a web page whose text contains a phrase matching a saved item's title or tag (use `/library` on Brain to pick a phrase). Expect: highlights on the matching phrases.
6. Click a highlight. Expect: shadow-DOM panel slides in with matched items; click-through opens the item in a new tab.
7. Open DevTools → Network. Browse 5 distinct pages. Confirm: only one `/api/index/keywords` request every 5 min; zero `/api/match` (route doesn't exist); no page text sent anywhere.
8. Open options; add `github.com` to suppression list; refresh a GitHub page; confirm: no highlights.
9. Rotate bearer token in Brain settings; confirm: extension shows the plain-language "token no longer works" message on next refresh, not silent failure.

## 7. Risks + mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `<all_urls>` permission scares user / flagged by Chrome Web Store | High | Medium | Personal-install, never submitted to store. Documented in options-page help text. |
| False-positive matches (common short words like "AI" triggering everywhere) | Medium | Medium | Word-boundary-aware regex; minimum phrase length of 3 chars; filter stopwords. |
| Match index gets huge at 5k+ items | Low (current library is small) | Low | Cap the index endpoint at 5k items; add pagination if needed. |
| Content script slows down page load on heavy sites | Medium | Medium | `run_at: document_idle`; debounced DOM walk; abort early if page >50k text nodes. |
| User forgets feature is on, sees highlights on sensitive sites | Medium | Low-Medium | Default OFF; suppression list; easy right-click-to-suppress. |

## 8. Non-goals (explicitly)

- Cross-lane impact: none. Lane C's cloud-migration work does not touch `extension/**` or any file this plan touches.
- Android / iOS / phone: structurally impossible — Chrome MV3 extensions don't exist on mobile.
- Chat-with-matched-item: excluded to preserve `/api/ask` security boundary.
- Body-text matching: deferred.
