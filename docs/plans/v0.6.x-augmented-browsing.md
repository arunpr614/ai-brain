## 1. Problem statement

User browses the web. When a page they're reading touches a topic already in their Brain library, they get no signal. The Chrome extension today only **captures** pages ‚Äî it never says "you've already saved 3 things about this."

Recall.it's augmented-browsing feature does this well; v2 audit catalogued 13 distinct capabilities. Brain has zero equivalent. MVP goal: deliver the core "did I already save something about this?" signal, safely and privately, without overshooting into Recall's 13-capability surface.

## 2. Goal + non-goals

### In-scope (MVP)

- Passive signal while browsing: highlights on matched phrases, click to see saved items.
- Zero browsing data sent to Brain server (local-only matching).
- Desktop-only by construction (Chrome MV3 extensions don't run on Android).
- Per-page on/off via popup button (simple, immediate escape hatch).
- Opt-in: default OFF until user enables.

### Explicit non-goals

- Body-text matching (summaries only in MVP).
- Per-site suppression **list** (one-click per-page disable only).
- SPA / dynamic-page support (Twitter, Gmail, Notion).
- Infinite-scroll re-scan.
- Keyboard shortcut (defer).
- Chat-from-match (preserves `/api/ask` security boundary from umbrella plan gotcha #1).
- Firefox, Arc, Safari ‚Äî Chrome + Chromium-Edge only.
- Analytics, telemetry ‚Äî explicitly not doing any tracking (beyond extension-local error logs).
- Performance optimization beyond the budget in ¬ß8.
- Onboarding tour ‚Äî ¬ß10 ships minimum discovery signal only.

## 3. Matching ‚Äî precise spec

Addresses critique ¬ß1.1. This is where 80% of execution risk lives; spelling it out.

### 3.1 Index shape (what the server sends)

```jsonc
{
  "version": 1,
  "generated_at": 1715529600000,
  "items": [
    {
      "id": "01HXYZ...",
      "title": "How to build AI products",
      "tags": ["pm", "ai", "onboarding"],
      "summary": "Three habits from the team that shipped..."  // may be null
    }
  ]
}
```

Items are included only when `enrichment_state = 'done'` AND `title` is present. `summary` may be `null` if enrichment hasn't run or the pipeline produced no summary.

### 3.2 Extracted match-targets (what the extension derives locally)

On receiving the index, the extension builds a match table in memory:

```
match_targets = []
for each item in index:
  add_target("title",   item.title,   item.id)         // full title phrase
  for each tag in item.tags:
    add_target("tag",   tag,          item.id)
  if item.summary is not null:
    for each keyphrase in extract_keyphrases(item.summary):
      add_target("summary", keyphrase, item.id)
```

`extract_keyphrases(summary)` runs client-side once per index refresh:

1. Lowercase the summary.
2. Tokenize on Unicode word boundaries (`\p{L}[\p{L}\p{N}'-]*`).
3. Drop stopwords using a fixed English list (~100 common words: `the, and, of, to, ...`). Bundled as a constant; user can't edit.
4. Build N-grams of length 2 and 3 from remaining tokens.
5. Keep N-grams whose total-char-length ‚â• 6 AND contain at least one word ‚â• 4 chars.

Rationale: single words are too noisy; 2/3-word phrases are specific enough to avoid false positives. Example: `"How to build AI products"` ‚Üí keyphrases `["build ai products", "ai products", "build ai"]`. Tag `"pm"` is kept as-is (2 chars, already short by design).

### 3.3 Match rules on a page

For each target, match it against the page text ONLY if:

| Target kind | Min total chars | Word-boundary required? | Case-sensitive? |
|---|---|---|---|
| title | ‚â• 8 | yes | no |
| tag | ‚â• 3 | yes | no |
| summary keyphrase | ‚â• 6 (already enforced in 3.2) | yes | no |

"Word boundary required" means the regex match uses `\b` on both sides (or equivalent Unicode-aware boundary). This is what prevents `"pm"` from matching inside "3pm" or "compile."

Titles under 8 chars are dropped from match-targets entirely (e.g. an item titled `"Ship"` would match too noisily; better to lose that signal than poison the feature).

### 3.4 Scoring when multiple targets match on one page

Each matched target contributes a score to its item:

- tag match: +1
- title match: +5
- summary keyphrase match: +2

Item's total score = sum over all its matched targets (counted once per page, not per occurrence). The side-panel orders items by score descending. In MVP only top 10 items are shown.

### 3.5 Hard limits (perf budget ‚Äî see ¬ß8 for full budget)

- Max 5000 match-targets per index (if more, drop lowest-score tags first).
- Max 2000 unique matches per page scan (stop early).
- Max 500 ms of matching CPU per page before aborting with graceful "scan incomplete" state.

## 4. DOM scan ‚Äî static-only MVP

Addresses critique ¬ß1.2.

### 4.1 Scope of what gets scanned

- **In:** `document.body` text nodes reachable by `document.createTreeWalker(document.body, NodeFilter.SHOW_TEXT)` at `document_idle`.
- **Out (skipped):** text nodes whose ancestor chain contains any of: `<script>`, `<style>`, `<noscript>`, `<textarea>`, `<input>`, `<select>`, `contenteditable="true"` elements, elements with `aria-hidden="true"`, elements whose computed `display === 'none'` or `visibility === 'hidden'`.
- **Out (MVP):** content added after initial `document_idle`. No `MutationObserver`. This means Twitter feed posts loaded after scroll, Gmail message bodies, Notion docs ‚Äî none will highlight. User will see highlights on traditional article sites (blog posts, news, Substack, Medium archives, Wikipedia, HN submissions); SPAs will appear not to work.

### 4.2 Known limitation (communicated to user via onboarding)

The popup's Augmented-Browsing status area will say: `"Works on most article pages. Dynamic pages like Gmail or Twitter won't highlight ‚Äî content loads after the scan."` So the user has a mental model that matches behavior.

### 4.3 Debounce + one-shot

The content script runs once per page load, at `document_idle`. It does NOT re-run on scroll or DOM change in MVP. Re-run is deferred; see ¬ß13 open questions.

## 5. Highlighting ‚Äî overlay, not DOM mutation

Addresses critique ¬ß1.3.

### 5.1 Approach

**Do not wrap matched text in `<span>` elements.** Instead:

1. For each match, build a `Range` from `(text_node, start_offset, end_offset)`.
2. Call `range.getClientRects()` to get the pixel bounding boxes of the matched characters.
3. Render absolutely-positioned highlight `<div>` elements in a fixed-position shadow-DOM overlay container, positioned to align with those rects.
4. Reposition overlays on `window.resize`, `scroll`, `orientationchange` via a throttled (rAF-coalesced) handler.

Benefits (per critique ¬ß1.3):
- No page text mutation ‚Üí no reflow, no CSS-selector breakage, no contentEditable interference.
- Host page DOM untouched; content scripts with DOM modifiers coexist peacefully.

Drawbacks acknowledged:
- Scroll + resize listeners have ongoing CPU cost. Mitigation: `requestAnimationFrame`-coalesced, early-exit if no matches on this page.
- If the host page reflows (e.g. image loads late, expands content), overlay positions drift until next scroll/resize tick. Acceptable for MVP.

### 5.2 Hover target

Each highlight `<div>` has `pointer-events: auto` and a subtle background tint. Click opens the side-panel (¬ß6) pinned to the item(s) matched by that highlight.

## 6. Panel UX

Addresses critique ¬ß1.4.

### 6.1 Panel location + z-index

- Fixed-position shadow-DOM element, bottom-right corner, 360px wide, max 70vh tall.
- z-index: `2147483647` (max signed 32-bit int ‚Äî the conventional "top" value used by Chrome extensions). Ties with some cookie-consent banners; that's the best any overlay can do without browser help.
- If the host page covers the bottom-right (common: Intercom, Drift chat widgets), user can click a drag handle in the panel header to move it. Position persisted to `chrome.storage.local` per-origin.

### 6.2 Theme

- Panel respects `prefers-color-scheme` via `@media` inside the shadow-DOM stylesheet.
- Host page's theme doesn't matter ‚Äî panel is fully isolated.

### 6.4 Per-tab disable (replaces v1's per-site list)

v1 planned a persistent per-site allowlist/denylist. v2 replaces with a simpler mechanism:

- One button in the panel: "Turn off Brain on this page."
- Click ‚Üí content script removes all overlays + refuses to re-inject for this tab until reload or navigation.
- No settings UI, no list to manage. If user wants AB permanently disabled for a site, they disable it at the extension level via popup toggle (¬ß10).

Rationale: full per-site rules is a 1-day UX design problem on its own. Per-tab disable ships in 1 hour and handles 90% of the need.

## 7. Index endpoint ‚Äî `/api/index/keywords`

Addresses critique ¬ß1.5, ¬ß1.6, ¬ß1.7.

### 7.1 Contract

- **Method:** `GET`
- **Auth:** session cookie OR bearer (dual-check). Reference route: read `src/app/api/capture/pdf/route.ts` at execution time to confirm the pattern; ¬ß12 gates the implementation on that verification.
- **Query params:**
  - `?since=<int>` ‚Äî optional Unix ms timestamp; if provided, response includes only items where `enriched_at > since`. Response's `generated_at` becomes the new baseline client sends on next request.
  - `?limit=<int>` ‚Äî optional; default and max 5000.
- **Response headers:**
  - `ETag: "<content-hash>"`
  - `Cache-Control: private, max-age=0` (client always revalidates via `If-None-Match`)
- **Response body:** ¬ß3.1 shape.
- **Status codes:**
  - `200` with body (success).
  - `304` Not Modified (client's `If-None-Match` matched server ETag).
  - `401` auth missing/invalid.
  - `500` server error (logged via `console.warn` only ‚Äî **not** `logError`; see ¬ß7.3).

### 7.2 Payload budget

At 5000 items √ó ~0.55 KB average = ~2.75 MB uncompressed JSON ‚Üí ~700 KB gzipped. Well under Chrome Storage's 10 MB per-extension quota. On first install, cold fetch latency over Cloudflare tunnel expected 1‚Äì3 seconds; shown in the popup as a one-time "Building index..." state (¬ß10).

### 7.3 Why `console.warn`, never `logError`

There is a test at `src/lib/auth/no-destructive-gets.test.ts` that statically scans every GET route handler file for string patterns suggesting state mutation. `logError` writes to `data/errors.jsonl` ‚Üí counts as a mutation ‚Üí the test fails. If the endpoint hits an error case worth logging, use `console.warn` (stdout only, non-persistent). Test ships as part of this feature's CI gate.

### 7.4 `BEARER_ROUTES` update

Add `/api/index/keywords` to the list at `src/lib/auth/bearer.ts:67`. Add positive-membership assertion in `src/lib/auth/bearer.test.ts`. Do NOT add `/api/ask` or `/api/search` in this feature (unrelated security boundary per umbrella plan gotcha #1).

### 7.5 Refresh trigger strategy (replaces v1 "5-minute poll")

Addresses critique ¬ß2.2. In priority order:

1. **On popup open** (user clicked toolbar icon) ‚Äî always refresh. Chrome service workers wake up here reliably.
2. **On capture complete** ‚Äî existing `captureUrl()` in `capture.ts` gets a post-success hook; on success, it messages the service worker which schedules an index refresh in ~2 seconds (giving enrichment a head-start).
3. **On extension startup** ‚Äî one-time refresh if index is older than 1 hour.

No timed polling. No `setInterval`. No Chrome Alarms. The index is lazily current; at worst it's 1 hour stale. If the user captured something 30 minutes ago and it's not highlighted yet, opening the popup refreshes on the spot.

### 9.2 Root-repo tests (unchanged harness)

### 9.3 Manual verification script

For each shippable commit, the human-in-the-loop runs:

1. Build extension: `cd extension && npm run build`.
2. Load unpacked in Chrome; reload Brain card.
3. Check popup displays and toggle works (¬ß10).
4. Enable AB via toggle.
5. Navigate to a known-matching article (a blog post on a topic you've saved). Expect: highlights appear within 2s; panel opens on click.
6. Navigate to a page with no matches. Expect: no highlights, no panel.
7. Navigate to Twitter or Gmail. Expect: either no highlights or only the pre-render content highlighted; popup warns about SPAs.
8. Click "Turn off Brain on this page" in panel. Expect: highlights removed; reload brings them back.
9. Capture a new URL via the existing popup. Within ~5 seconds of capture, open popup again ‚Üí confirm index refreshed (popup shows new item count).
10. Privacy verification: DevTools ‚Üí Network. Browse 5 pages. Confirm: zero requests to Brain server during browsing. Only requests are (a) popup-triggered refresh, (b) capture POSTs, (c) initial install index fetch.
11. Performance: all budgets in ¬ß8 hold on a typical article page.

Any failure ‚Üí fix before commit, or defer the commit.

Addresses critique ¬ß6 "plan that names its own unknowns."

1. **SPA support ‚Äî what's the follow-on?** After MVP ships, the biggest real-world gap is Twitter/Gmail/Notion. Options: `MutationObserver` with DOM-diff-driven incremental scan; dedicated site adapters for popular SPAs; browser-level messaging from a web-app side (unrelated to extension). Needs its own plan.
2. **`<all_urls>` vs `optional_host_permissions` granting flow.** Unclear what the exact Chrome UX looks like when the welcome tab's "Turn on" button triggers a permission request. Will verify at commit 3; may need a fallback in commit 5 (e.g. "permission denied ‚Äî feature stays off").
3. **Panel drag handle across shadow-DOM boundary.** Drag-to-move uses pointer events; need to confirm Chrome's event capture behavior inside a closed shadow root. Will test in commit 4; fall back to a fixed position if problematic.
4. **Index scaling past 5000 items.** Current cap; user's library will exceed this eventually. Pagination + region-of-interest refresh (fetch only items matched by recent page scans) is a follow-on feature.
5. **Tokenizer + stopword list localization.** Bundled English only. A German/Hindi/Japanese page will match poorly. Unicode word-boundary regex handles the tokenization; stopwords do not. Non-English users get degraded signal; noting as known.
6. **ETag computation cost at server.** Computing an ETag requires hashing the payload. For 5000 items on each GET this is ~5ms of hash work. Acceptable in isolation; if popup + capture together cause bursts, consider caching ETag by `max(enriched_at) over items`. Implement simple-hash first; optimize only if ¬ß8 budget is breached.

These are known-unknowns I'll resolve during execution. They are explicitly NOT blocking; committing to an MVP that defers them honestly is better than padding this plan with guessed answers.
