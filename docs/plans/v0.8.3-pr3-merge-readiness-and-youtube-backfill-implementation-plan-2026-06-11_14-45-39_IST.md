## 7. Phase E - Next Feature: Batch Backfill Job

Start this only after PR #3 is either merged or intentionally accepted as the base for the next branch.

### Goal

Heal older YouTube captures that were saved without transcripts by queueing transcript recovery in batches.

### Product Behavior

The user should not have to resend old YouTube links manually. The app should periodically find old transcript-weak YouTube items and queue recovery work.

### Functional Requirements

1. Identify eligible YouTube items:
   - source is YouTube
   - URL is present
   - transcript is missing, weak, or metadata-only
   - item is not already terminally impossible, unless forced
2. Queue jobs idempotently:
   - repeated backfill runs must not create duplicate active jobs
   - old failed jobs should obey retry policy
   - terminal private/deleted/unavailable states should not burn retries
3. Support bounded batches:
   - `limit`
   - `olderThan`
   - optional `dryRun`
   - optional `force`
4. Leave an audit trail:
   - job source should be distinguishable from Telegram/API/manual actions
   - attempts should show backfill source
5. Provide safe operator feedback:
   - number scanned
   - number eligible
   - number enqueued
   - number skipped because already queued
   - number skipped because terminal

### Proposed Implementation Shape

Add a small backfill service around the existing transcript job system:

- `src/lib/capture/youtube-transcript/backfill.ts`
  - finds eligible items
  - applies skip rules
  - enqueues recovery jobs
  - returns structured summary
- `src/lib/capture/youtube-transcript/backfill.test.ts`
  - idempotency tests
  - eligibility tests
  - terminal skip tests
  - dry-run tests
- Optional admin/operator script:
  - `npm run backfill:youtube-transcripts -- --limit 50 --dry-run`
- Optional cron entry after dry-run confidence:
  - daily small batch
  - conservative limits
  - visible logs

Use the existing recovery job table rather than inventing a second queue.

### Backfill Eligibility Rules

An item should be eligible if:

- it has a YouTube URL,
- transcript/body content is empty or below the quality threshold,
- current capture quality indicates metadata-only or transcript-missing,
- there is no active queued/running transcript job for the same item,
- the most recent failure is retryable or old enough to retry.

An item should be skipped if:

- it already has a good transcript,
- it has a terminal failure such as private/deleted/unavailable,
- it already has an active job,
- it has exceeded retry policy,
- it has no usable YouTube video ID.

### Backfill Validation

Automated checks:

```bash
npm run typecheck
npm test -- youtube-transcript
npm test -- transcript
npm run lint
```

Manual checks:

1. Run dry-run against a copy of the database.
2. Confirm candidate counts look sane.
3. Run a tiny live batch, for example limit 5.
4. Confirm jobs are queued.
5. Start worker and confirm jobs settle.
6. Confirm Review page improves or resolves affected items.
7. Confirm repeated backfill does not duplicate jobs.

### Backfill Rollout Plan

1. Dry-run locally on copied SQLite database.
2. Run limit 5 in local/staging.
3. Run limit 25 against production-like data after backup.
4. Enable daily job only after manual runs are clean.
5. Keep initial daily limit low.
6. Increase gradually after observing provider reliability.

### Backfill Rollback Plan

1. Disable cron/scheduled invocation.
2. Stop transcript worker if it is causing provider pressure.
3. Leave existing completed transcript attempts intact.
4. If a bad batch creates noisy queued jobs, mark those jobs cancelled/skipped with an auditable reason.
5. Restore DB backup only if data corruption occurs, not for normal failed attempts.

### Exit Criteria

- Old YouTube metadata-only items can be found and queued in bounded batches.
- Backfill is idempotent.
- Operator can run dry-run safely.
- Daily scheduling is conservative and observable.
