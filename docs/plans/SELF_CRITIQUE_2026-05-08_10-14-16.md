## 1. Architecture-level findings

### A-1 · **P0 · OPEN** — There is no CSRF protection on the server actions, and the project is planned to be reachable over the LAN

`src/app/taxonomy-actions.ts` and the bulk actions planned in the v0.3.1 doc all run as server actions invoked via HTML `<form action={...}>`. Next.js's server-action encryption guards against action-ID enumeration but does **not** on its own stop a remote origin from POSTing a form to `http://<my-mac>.local:3000/...`. The v0.5.0 APK plan (per `ROADMAP_TRACKER.md`) explicitly expands scope to "+CSRF, +token rotation" — meaning today it is **not** there. The Handover §M9 also flags this as a watch-out.

This is fine for `localhost` but the project wants LAN use (mDNS) at v0.5.0. The v0.3.1 plan adds **bulk delete** as a server action. A destructive action with no CSRF guard is the single highest-risk addition in the polish batch.

- **Action:** Either (a) explicitly narrow v0.3.1 server to `127.0.0.1` binding so LAN exposure is impossible until v0.5.0, or (b) add an Origin-header check to the bulk delete action now. (a) is cheaper; do (a) and add a comment in `src/instrumentation.ts`.

### A-2 · **P1 · OPEN** — Single-process enrichment worker is tied to the web server's lifecycle

`src/instrumentation.ts` starts the enrichment worker via `register()` on every Node runtime boot. In Next 16 dev, that includes HMR restarts. Two risks:
- During fast-refresh, a module reload could `startEnrichmentWorker()` a second time; the `running` flag in `enrichment-worker.ts` guards against re-entry in the **same module instance**, but HMR gives you a **new** module instance each time. Two concurrent workers would both pass the `running=false` check, each poll, each call `claimNext()` — the DB transaction is atomic so they won't double-run a job, but they **will** both back off on Ollama unreachable and both log. Low-harm but ugly and invalidates the "single worker" design note at line 5 of the file.
- When `next build` runs, the worker boots inside the build process if `register()` runs. Need to verify.

- **Action:** Add a process-level guard (`globalThis.__enrichmentWorkerStarted`) instead of a module-level flag, *and* assert `process.env.NODE_ENV !== "test"` to keep tests from booting it. Also document in the worker file when HMR double-boots.

### A-3 · **P1 · OPEN** — `sweepStaleClaims` runs once per worker boot, not periodically

`enrichment-worker.ts:46` calls `sweepStaleClaims()` exactly once before entering the poll loop. If the server never restarts but the worker hits a non-throwing hang (e.g., a `fetch()` to Ollama that neither resolves nor rejects because the socket wedges), the stale claim is never swept. `STALE_CLAIM_MS` = 90 s but there's no code path that re-checks.

- **Action:** Move the sweep inside the loop, gated on `Date.now() - lastSweepAt > STALE_CLAIM_MS`. Three lines.

### A-4 · **P2 · OPEN** — `MAX_ATTEMPTS = 3` + infinite sweep means a permanently-broken item can retry forever across reboots

`handleFailure` flips state to `error` after `MAX_ATTEMPTS`, but `sweepStaleClaims` only looks at `state = 'running'` — so `error` rows don't retry. Good. **However**, if the job gets stuck in `pending` (e.g., the worker crashes mid-`claimNext` transaction), `attempts` resets semantics are fine but there's no visible signal to the user that a specific item is failing *repeatedly before* hitting the cap. `EnrichingPill` polls the item's `enrichment_state`, not attempt count.

- **Action:** Surface `attempts` in `/api/items/[id]/enrichment-status` so the UI can show "retrying (2/3)".

### A-5 · **P1 · NOTED → escalated to P0 concern** — Edge runtime proxy does zero cryptographic verification

`src/proxy.ts` only checks cookie **presence**. The handover's rationale is correct: Edge can't use `node:crypto`. But the cookie value is signed with an HMAC key that lives in the `settings` table; if a user ever copies their cookie onto another device that's fine, but **a malicious browser extension that reads the cookie** can replay it, and since the cookie has no expiry / rotation policy documented, a stolen cookie is valid indefinitely.

- **Action:** (i) Add an `expires_at` claim to the signed cookie and verify it in `verifySessionToken`. (ii) Document cookie rotation policy in `03_Secrets_and_Configuration.md`. (iii) Consider setting `SameSite=Strict` (currently unspecified in my read).

### A-6 · **P1 · NOTED** — SQLite as a single file means any write-lock contention between worker + request handlers blocks both

`better-sqlite3` is synchronous and uses WAL by default (need to verify migration `001` sets `PRAGMA journal_mode=WAL`). If it doesn't, `BEGIN IMMEDIATE` transactions in the worker's `tx()` and a request handler's write will serialize and any long enrichment write will block UI writes. On a personal tool this is probably invisible, but bulk operations in v0.3.1 (`F-207`) wrap many `deleteItem` / `attachTagToItem` calls in a transaction — worth confirming WAL is on.

- **Action:** Check `001_initial_schema.sql` for `PRAGMA journal_mode=WAL` and `PRAGMA synchronous=NORMAL`. If missing, add to `src/db/client.ts` on connection open (pragmas must be set per-connection, not persisted in-file for all SQLite versions).

### A-7 · **P1 · OPEN** — The backup scheduler has no restore path documented, only `VACUUM INTO`

`src/lib/backup.ts` (115 lines) does `VACUUM INTO data/backups/brain-<ts>.sqlite`. There is no documented **restore** procedure in the handover or planning docs. For a local-first personal tool the entire reliability story leans on this.

- **Action:** Add a `scripts/restore-from-backup.sh` one-liner + a paragraph in `07_Deployment_and_Operations.md` explaining: stop server → copy chosen backup over `data/brain.sqlite` → start server. Even one sentence documented prevents a 3am panic.

### A-8 · **P2 · NOTED** — FTS5 "LIKE fallback" duplicates code paths

`searchItems` has an FTS5 branch and a LIKE fallback. Fine for robustness, but the fallback is dead code unless migrations run out of order or the DB is opened read-only. If it has **never been hit in practice**, remove it — the LIKE scan on a 10k-item library is an unpleasant silent failure mode anyway.

- **Action:** Deferred concern; revisit at v0.4.0 when FTS5 behaviour is stressed by hybrid search.

### A-9 · **P1 · OPEN** — `sqlite-vec@^0.1.6` is pinned at a pre-1.0 version; breaking changes between minor releases are routine

The handover lists `sqlite-vec@^0.1.6`. The caret range on a 0.x dependency is effectively "any minor version ≥ 0.1.6 and < 0.2.0" — and the project hasn't used it yet (v0.3.0 doesn't exercise the extension). By the time R-VEC starts, `^0.1.6` could resolve to 0.1.15 with different SQL syntax. The R-VEC plan should **pin exactly**, run the bench, and only relax the pin after v0.4.0 ships.

- **Action:** The R-VEC plan mentions "pin the version in package.json during the spike" — good — but doesn't state *exact pin*. Change `^0.1.6` to `0.1.6` in `package.json` at the start of R-VEC work. Already listed in the spike's §7 as a risk; upgrade to an action item.

### A-10 · **P2 · OPEN** — No monitoring or alerting exists; the only observability is `console.log`

The enrichment worker logs to stdout. There is no aggregated error sink, no weekly error report, no sign that anything is wrong unless the user opens the item and notices an `error` pill. For a personal tool this is okay at v0.3.0 but becomes a real gap once v0.4.0 RAG adds a second async pipeline.

- **Action:** Write findings into a lightweight `data/errors.jsonl` rotation file in `handleFailure`. Later phases can surface it in `/settings`.

### A-11 · **P1 · OPEN** — `instrumentation.ts` returns before doing anything under non-Node runtimes but doesn't warn

`process.env.NEXT_RUNTIME !== "nodejs"` — correct guard — but if someone moves a route to Edge by mistake, the worker won't start and failures are silent. Add a `console.log` for the non-nodejs branch so the boot log is unambiguous.

- **Action:** Cosmetic but cheap. One line.

### A-12 · **P2 · OPEN** — The architecture's "single-user" assumption is not enforced anywhere

The handover and this codebase repeatedly assume "single-user personal tool". Nothing in the schema, auth, or server enforces it — if a second PIN were created, the `settings.auth.pin_hash` write would just overwrite. That's probably intentional ("re-setup" behaviour) but it's not documented as a property, and a future bug could reintroduce multi-user state.

- **Action:** Add an assertion in `setup-actions` that refuses to overwrite an existing `pin_hash` unless the user explicitly chose "reset install". Today the check is by `PUBLIC_PATHS` matching `/setup`, not by state.

## 2. Planning-doc findings (BACKLOG + v0.3.1 + R-VEC)

### P-1 · **P1 · ACTIONABLE** — v0.3.1 plan's T-4 (title de-hyphenation) heuristic has a latent false-positive

The proposed rule is "more hyphens than spaces". Counter-example from my own test cases:
- `State-of-the-Art 2026` has **3 hyphens and 1 space** → rule fires → title becomes `State Of the Art 2026`. Wrong; the original was correct.

The plan acknowledges this ambiguity in the test table but doesn't commit to a fix. The heuristic needs refinement **before** T-4 starts, not during.

- **Action:** Tighten the rule to: fires only when **every word is a single hyphen-separated token with no spaces at all** OR when hyphens > 2× spaces. Rewrite T-4's heuristic section before implementing. Add `State-of-the-Art 2026` as a **must-not-rewrite** test case.

### P-2 · **P1 · ACTIONABLE** — v0.3.1 plan commits to `node:test` as a hard choice with no precedent in the repo

T-4 proposes introducing the first-ever test file in the project. The plan says "Node's built-in `node:test` for parity with the existing zero-dep posture. If that feels heavier than the project wants, inline an assertion block in a `scripts/...ts` one-shot". That's two competing decisions punted into the task. Pick one **now**.

- **Action:** Recommend `node:test` (standard, zero-dep, survives CI) and add a two-line `npm test` script to `package.json` that globs `src/**/*.test.ts`. Set the precedent deliberately; don't discover it mid-task.

### P-3 · **P1 · NOTED** — v0.3.1 plan has no back-pressure on concurrent bulk ops

`bulkDeleteItemsAction` wraps N `deleteItem` calls in a transaction. But nothing stops the user from clicking Delete twice rapidly; the second action will wait on the first's write lock, which on a 500-item batch could be hundreds of milliseconds. UX-wise the user sees nothing.

- **Action:** In T-5c, after firing a batch action, disable the bulk bar buttons until the server response returns (`useTransition` with `isPending`). Add this to the task explicitly; don't rely on the implementer to guess.

### P-4 · **P1 · ACTIONABLE** — v0.3.1 plan has **no** e2e smoke script — every success criterion says "manual smoke"

Every SC-N under §2 says "Manual smoke". For a non-technical owner in an AI-assisted workflow, manual smoke across 6 criteria = fatigue → criteria get skipped → regressions land. Especially painful since v0.3.1 introduces the first multi-item destructive path (bulk delete).

- **Action:** Add a **T-5d.5** step: a `scripts/smoke-v0.3.1.mjs` that (a) creates two items via API, (b) runs bulk-tag, (c) asserts the tags land, (d) bulk-delete, (e) asserts gone. Not a full test suite — a single-script smoke. ~40 lines.

### P-5 · **P2 · OPEN** — v0.3.1 plan references artifacts with slightly inconsistent versioning

The plan's header says "Previous HEAD `dad9ca6`" but the actual HEAD at commit time is `5e4804f` (this commit itself). Minor, but the provenance is misleading.

- **Action:** Update the plan to read "Baseline `5d1c390` (v0.3.0 release); Planning committed at `5e4804f`". Trivial.

### P-6 · **P1 · ACTIONABLE** — v0.3.1 plan's F-207 batch actions do not revalidate `/collections/[id]` or `/settings/tags`

`bulkAttachCollectionAction` revalidates `"/"` only. If the user is currently on `/collections/[id]` viewing that collection, they won't see the added items until a hard reload. Same for the tag page.

- **Action:** Add `revalidatePath("/collections/[id]", "layout")` and `revalidatePath("/settings/tags")` to the three actions. Update §T-5b in the plan.

### P-7 · **P2 · NOTED** — v0.3.1 plan treats `500-item cap` as the hard limit but does nothing about it at the UI

The zod schema rejects batches > 500 items. The library currently renders `listItems({ limit: 100 })` — so 500 is way beyond what's selectable today. The cap is good defense but the UX should never allow reaching it. Not a bug, but the UI should match.

- **Action:** Either lift the list limit with pagination (later work) or document that the 500 cap is defense-in-depth only.

### P-8 · **P1 · ACTIONABLE** — R-VEC plan measures latency but not memory

The R-VEC plan has three pass/fail thresholds (p50, p95, build time). **Missing:** peak RSS during the 50k build. On an M1 Pro with 32 GB and the app also running Next dev (~1.5 GB) + Ollama (7B model ≈ 5 GB resident), a 50k × 768-dim × float32 build that spikes to 10+ GB RSS could swap and invalidate the whole benchmark.

- **Action:** Add an RSS metric to S-2 and S-3 in R-VEC-spike.md. Fail the GREEN verdict if peak RSS > 12 GB at 50k, or > 4 GB at 10k (the target corpus).

### P-9 · **P1 · ACTIONABLE** — R-VEC plan uses random vectors but doesn't note the embedding distribution mismatch clearly enough

Section §7 handwaves "Acceptable for latency-only measurement (sqlite-vec doesn't learn from data distribution)". This is mostly true but `sqlite-vec` internally can choose between brute-force cosine and an approximate index depending on `vec0` version. If the actual version at spike time ships an ANN index, synthetic-random vs real-embedding distribution **will** shift the p95.

- **Action:** Confirm `sqlite-vec`'s current index structure during S-1. If it's ANN, run the p95 harness on both synthetic and a 1000-vector real-Ollama-embedding slice. If it's brute-force only, the synthetic caveat stands.

### P-10 · **P2 · OPEN** — R-VEC plan's "decision gate" is binary-ish but the fallbacks aren't costed

The plan says "If RED, escalate to FAISS sidecar". A Python sidecar introduces Python as a runtime dependency — a significant increase in install complexity for a non-technical user. The plan should weight this against YELLOW ("cap corpus size") before recommending either.

- **Action:** Add a §6 sub-table that costs each alternative (FAISS sidecar: needs Python + IPC; LanceDB: pure Rust, single binary, worth preferring; Chroma: heavier). LanceDB should probably be the primary fallback over FAISS-via-Python.

### P-11 · **P2 · NOTED** — BACKLOG.md's "never delete rows" rule conflicts with its own closure-via-strikethrough pattern

BACKLOG.md §6 item 4 says "Never delete rows — history matters". Item 2 says closed items "move into §5 at next phase rollover". Those two rules combined mean §5 grows unboundedly over the project's lifetime. Fine for 10 phases; becomes a scroll-tax for 100.

- **Action:** Cap §5 at "last 30 days of closed items"; older closures move to a `docs/history/backlog-archive-YYYY-QN.md`. Not urgent. Note for the v0.4.0 or later refactor.

### P-12 · **P1 · ACTIONABLE** — v0.3.1 plan's release step `git push --tags` runs without confirming what `main` contains

T-6 says `git tag v0.3.1 && git push --tags (user confirms before push)`. But nothing else in the task says what the diff should look like first. An inexperienced committer could tag `main` while it has accidental WIP or the `node_modules` change the handover mentions as part of the working tree.

- **Action:** Prepend to T-6: "Run `git status` — working tree must be clean. Run `git log v0.3.0..HEAD --oneline` — all entries must be v0.3.1 phase commits, no unrelated drift."

## 3. Meta-findings (about the planning posture itself)

### M-1 · **P1 · OPEN** — The plan assumes all work happens in a single session with no context-reset

v0.3.1 plan has six tasks (T-1..T-6). Each commit is atomic, but the plan doesn't specify what state the `RUNNING_LOG.md` should be in between tasks — so if the next agent resumes mid-phase after a context reset, there's no breadcrumb that says "T-3 done, T-4 in progress, start here". The handover pattern this project already uses works; v0.3.1 should reuse it.

- **Action:** After each T-n commit, append a **one-line** `RUNNING_LOG.md` entry. Not the full end-of-phase narrative — just "T-3 F-302 shipped at commit <sha>".

### M-2 · **P1 · OPEN** — The plan files sit under `docs/plans/` but are not linked from `BUILD_PLAN.md`

`BUILD_PLAN.md` is cited across the handover as the phase-plan source of truth at v0.3.0. The new `docs/plans/v0.3.1-polish.md` breaks that convention by living elsewhere. Two sources of truth = drift.

- **Action:** Either (a) move `docs/plans/v0.3.1-polish.md` content into `BUILD_PLAN.md` under a v0.3.1 heading, or (b) add a pointer from `BUILD_PLAN.md` top to `docs/plans/`. Minor; pick one before T-1.

### M-3 · **P2 · OPEN** — "Full AI-assisted" means the reviewer of these plans is the author

This document itself is a partial mitigation, but the structural risk remains: the same agent writes the architecture, the plan, and the self-critique. Without an independent reviewer (the user, a second AI, or a cross-check run), systematic blind spots persist across docs.

- **Action:** Run `gsd-review` or equivalent cross-AI review on v0.3.1 plan before execution if the user has it available. If not, have the user read §1 and §2 of this document as a spot-check.

### M-4 · **P2 · OPEN** — No rollback rehearsal

The v0.3.1 plan §8 describes rollback but never rehearses it. Given the plan introduces the first destructive batch operation, a 2-minute `git revert` dry-run on a throwaway branch before shipping would be cheap insurance.

- **Action:** Add a single bullet to T-6: "Before tagging: `git revert --no-commit HEAD && git reset --hard HEAD` to confirm revert produces a working app. Don't skip."
