## 1. Executive Summary

The first v0.8.3 implementation slice created the transcript recovery queue, Review surface wiring, duplicate-link enqueue behavior, and provider retry improvements. The follow-up adversarial review found no P0 blocker, but it found one merge/deploy-blocking P1:

> The transcript recovery worker can silently stop after an uncaught per-job or DB exception while the UI still tells the user recovery is queued or retrying.

This hardening plan makes the recovery loop trustworthy before any provider expansion, deployment, or release packaging. The plan deliberately does not add `yt-dlp`, browser scraping, ASR, or a new provider. The goal is to make the current recovery loop reliable, test-protected, auditable, and honest.

Recommended execution order:

1. Fix worker crash safety.
2. Add duplicate YouTube resend tests.
3. Record manual transcript/notes fixes as durable attempts.
4. Implement Review focus links.
5. Classify terminal YouTube failures correctly.
6. Add small provider fallback/ranking test coverage.
7. Run full validation and prepare commit/PR notes.

---

## 2. Release Objective

YouTube captures that fail transcript extraction must enter a background recovery workflow that:

- keeps running even when one job fails unexpectedly;
- preserves durable attempt history for provider and manual outcomes;
- lets duplicate Telegram/API resends re-queue recovery instead of dead-ending;
- takes the user to the exact Review item when recovery is queued;
- avoids wasting retries on terminal YouTube states;
- remains testable, observable, and safe to deploy as a first slice.

### Step 1: Fix Worker Crash Safety

**Priority:** P1, release blocker
**Primary file:** `src/lib/queue/transcript-worker.ts`
**Likely tests:** new `src/lib/queue/transcript-worker.test.ts` or focused additions if a worker test file already exists.

#### Problem

`startTranscriptRecoveryWorker()` calls `void loop()` without a catch. `loop()` awaits `runOne(job)` directly. `runOne()` only catches upgrade failures in the success path, so exceptions from DB helpers or unexpected provider/attempt transitions can reject the loop promise. If that happens, `state.running` may stay true and the worker may not restart in the same process.

#### Implementation design

Make the worker resilient at two levels.

1. **Top-level loop lifecycle**

   Add a wrapper such as:

   ```ts
   function startLoop(): void {
     void loop().catch((err) => {
       const message = err instanceof Error ? err.message : String(err);
       console.error(`[transcript] worker crashed: ${message}`);
     });
   }
   ```

   Inside `loop()`, use `try/finally` so `state.running = false` happens for every exit path:

   ```ts
   async function loop(): Promise<void> {
     const state = workerState();
     try {
       while (!state.stopRequested) {
         // existing loop
       }
     } finally {
       state.running = false;
       console.log("[transcript] worker stopped");
     }
   }
   ```

   Keep `stopRequested` semantics intact.

2. **Per-job safety boundary**

   Replace direct `await runOne(job)` with a safe wrapper:

   ```ts
   await runOneSafely(job);
   ```

   `runOneSafely(job)` should:

   - call `runOne(job)`;
   - catch any thrown error;
   - try to record a `transcript_attempts` row with:
     - `provider = "transcript_worker"`, or the known provider if available;
     - `state = "retryable_error"`;
     - `retryable = true`;
     - `errorCode = "worker_exception"`;
     - `errorMessage = message`;
   - call `markTranscriptJobRetryable(job.id, attemptIdOrNull, nextTranscriptRetryAt(job.attempts), ...)`;
   - if recording the attempt itself fails, still try to mark the job retryable with `attemptId = null`;
   - if even marking retryable fails, log loudly and rely on stale claim sweep as the final backstop.

3. **Sweep and claim resilience**

   Keep sweep/claim outside the per-job wrapper, but ensure an exception in sweep/claim cannot leave `state.running` permanently true. Top-level `finally` covers this.

4. **Optional helper extraction**

   Export small pure/testable helpers only if needed:

   - `safeTranscriptErrorMessage(err)`
   - `nextTranscriptRetryAt()`
   - possibly `handleTranscriptJobException(job, err)`

   Avoid exporting the infinite loop just for tests.

#### Test plan

Add worker tests that do not require running the infinite background loop if possible.

Recommended minimal test shape:

1. Create a metadata-only YouTube item and claim a transcript job.
2. Simulate a worker exception in the per-job handling path.
3. Assert the job is no longer stuck in `running`.
4. Assert it is `retryable_error` or `manual_needed` depending on max attempts.
5. Assert `last_error_code = "worker_exception"`.
6. Assert a transcript attempt exists when attempt recording did not throw.
7. Add a second case where attempt recording fails and the job still moves out of `running` with `last_error_code = "worker_exception"`.

If dependency injection would be cleaner, add a small test-only or internal function that accepts dependency overrides:

```ts
export async function runTranscriptJobSafelyForTests(
  job: TranscriptJobRow,
  deps?: Partial<TranscriptWorkerDeps>,
): Promise<void>
```

Keep the production API small and avoid large architecture changes.

#### Acceptance criteria

- One bad job cannot stop the worker loop.
- A failed job does not remain `running` unless every DB update path failed.
- `state.running` is reset if the loop exits unexpectedly.
- Tests cover the failure mode that caused the adversarial P1.
- Existing transcript worker behavior still passes current DB/helper tests.

### Step 3: Make Manual Transcript Fixes Auditable

**Priority:** P2
**Files:**

```text
src/db/transcript-jobs.ts
src/db/item-upgrades.ts
src/db/transcript-jobs.test.ts
src/app/api/capture/url/route.test.ts
src/lib/telegram/dispatch.test.ts
```

#### Problem

Manual pasted transcript/notes upgrades currently mark the transcript job `done` but do not create a `transcript_attempts` success row. The job state becomes correct, but the attempt history is incomplete.

#### Implementation design

Add a helper in `src/db/transcript-jobs.ts`:

```ts
export function recordManualTranscriptResolutionForItem(input: {
  itemId: string;
  provider?: string;
  transcriptChars?: number | null;
}): TranscriptJobRow | null
```

Behavior:

1. Load existing transcript job for the item.
2. If no job exists, return null.
3. If job is already `done`, do not duplicate success attempts. Return the existing job.
4. Determine the next attempt number safely:
   - use `Math.max(job.attempts, maxRecordedAttemptNumber) + 1`;
   - or, if preserving `attempts` as already-consumed only, ensure the inserted attempt number does not collide.
5. Insert a `transcript_attempts` row:
   - `provider = "manual_user_text"` by default;
   - `state = "success"`;
   - `retryable = false`;
   - `errorCode = null`;
   - `transcriptChars = input.transcriptChars`;
6. Update `transcript_jobs`:
   - `state = "done"`;
   - `claimed_at = NULL`;
   - `completed_at = now`;
   - `next_run_at = NULL`;
   - `last_attempt_id = attemptId`;
   - `last_provider = "manual_user_text"`;
   - clear last error fields;
   - update `attempts` to at least the inserted attempt number.

Wire `upgradeItemCaptureContent()` to call this helper when:

- the item has an active transcript job;
- the upgrade is YouTube/Shorts;
- the content method indicates user-provided/manual text, especially `youtube_user_provided_text`.

For provider success, keep existing worker success path as the source of the success attempt. Avoid double-recording provider success via `upgradeItemCaptureContent()`.

Recommended condition:

```ts
const isManualYoutubeUpgrade =
  isYoutube &&
  (input.content.extraction_method === "youtube_user_provided_text" ||
   input.content.capture_quality === "user_provided_full_text");
```

Then call `recordManualTranscriptResolutionForItem({ itemId, provider: "manual_user_text", transcriptChars: input.content.body.length })`.

#### Tests

1. DB helper test:
   - create item and job;
   - call manual resolution helper;
   - assert job done;
   - assert one success attempt with `provider = "manual_user_text"`.
2. API paste test:
   - existing metadata-only YouTube item has transcript job;
   - POST same URL with long pasted transcript;
   - assert item upgraded;
   - assert job done;
   - assert success attempt.
3. Telegram paste test:
   - existing metadata-only YouTube item has job or mocked dependency records manual helper path;
   - send URL plus pasted text;
   - assert upgrade and manual attempt path.

#### Acceptance criteria

- Manual recovery is visible in attempt history.
- No duplicate manual success rows on repeated idempotent calls.
- Provider recovery still records exactly one provider success attempt.

### Step 5: Classify Terminal YouTube Failures Correctly

**Priority:** P2
**Files:**

```text
src/lib/capture/youtube-transcript/recovery.ts
src/lib/capture/youtube.ts
src/lib/capture/youtube.test.ts
new or existing recovery tests
```

#### Problem

`recoverYoutubeTranscriptForItem()` catches all thrown errors and marks them retryable. Private, deleted, invalid, or unavailable videos should not burn retry attempts.

#### Implementation design

Import `YoutubeCaptureError` from `src/lib/capture/youtube.ts` and add explicit classification:

```ts
if (err instanceof YoutubeCaptureError) {
  return classifyYoutubeCaptureError(err);
}
```

Suggested mapping:

| Error code | Recovery state | Retryable | Error code to store | Message |
|---|---:|---:|---|---|
| `video_unavailable` | `manual_needed` | false | `video_unavailable` | existing error message |
| `invalid_url` | `manual_needed` | false | `invalid_youtube_url` | existing error message |
| `live_stream` | `retryable_error` for early attempts | true | `live_stream_captions_pending` | live captions unavailable |
| `fetch_failed` | `retryable_error` | true | `innertube_fetch_failed` | existing error message |
| `no_captions` | `manual_needed` | false | `captions_unavailable` | existing error message |

Notes:

- Current `extractYoutubeVideo()` often returns metadata-only for no transcript instead of throwing `no_captions`; keep both paths safe.
- If `YoutubeCaptureError` does not carry status code, do not invent one.
- Preserve original error message in attempt history, but do not leak sensitive URLs or tokens. YouTube URLs here are normal source URLs, not secrets.

#### Tests

Add recovery tests that mock or indirectly trigger:

1. `YoutubeCaptureError("video_unavailable")` -> `manual_needed`, `retryable = false`.
2. `YoutubeCaptureError("invalid_url")` -> `manual_needed`, `retryable = false`.
3. `YoutubeCaptureError("fetch_failed")` -> `retryable_error`, `retryable = true`.
4. `YoutubeCaptureError("live_stream")` -> `retryable_error`, `retryable = true`, unless the team chooses terminal behavior after max attempts in DB layer.

If direct mocking is awkward, extract `classifyYoutubeCaptureError()` as a pure exported helper for tests.

#### Acceptance criteria

- Terminal YouTube states do not burn all retry attempts.
- Retryable YouTube states still requeue properly.
- Attempt history stores actionable error codes.
