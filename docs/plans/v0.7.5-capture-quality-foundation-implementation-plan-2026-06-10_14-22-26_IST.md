## 2. Executive Decision

Build the capture-quality foundation now. Do not start with LinkedIn scraping, a browser extension, Gmail integration, or a third-party extraction provider.

The immediate release should ship:

1. A durable capture quality data model.
2. A platform router for YouTube, YouTube Shorts, Substack, LinkedIn, and generic URLs.
3. Richer YouTube body construction using current fields, oEmbed, and optional YouTube Data API metadata.
4. A Substack adapter that is Readability-first with RSS/metadata enrichment.
5. A LinkedIn server adapter that saves metadata-only previews, plus an explicit user-paste path for full text.
6. Library/detail UI labels so weak captures are visible and repairable.
7. A repeatable capture-quality smoke harness based on the executed spike.
8. Capped raw artifact storage so captures can be reprocessed later.

The important correction from the spikes: **Substack should not be RSS-first yet.** The spike showed Readability was stronger than RSS across the sampled public posts, so RSS should enrich or recover, not lead.

## 3. Current Codebase Snapshot

Relevant current files:

- `src/lib/capture/url.ts` extracts generic articles with fetch + Readability.
- `src/lib/capture/youtube.ts` extracts YouTube transcripts through InnerTube/timed-text and includes the v0.7.4 oEmbed anti-bot metadata fallback.
- `src/lib/capture/types.ts` exposes a small `CapturedContent` interface.
- `src/app/api/capture/url/route.ts` dispatches only between YouTube and generic URL extraction.
- `src/app/api/capture/note/route.ts` captures plain notes and already has a simple title/body contract.
- `src/lib/telegram/dispatch.ts` routes Telegram messages to PDF, command, URL, or note capture. It currently chooses URL capture before note capture.
- `src/db/items.ts` inserts captured items into the `items` table.
- `src/db/client.ts` defines the `ItemRow` type. `items.source_type` currently stays broad: `url`, `pdf`, `note`, `youtube`, etc.
- Latest migration number is `012_capture_source.sql`; the next migration should be `013_capture_quality.sql`.
- Current UI relies mostly on `source_type` and `extraction_warning`.

Important constraint:

- Keep `source_type` as the broad existing storage category. Add platform and quality fields instead of trying to force `linkedin` or `substack` into `source_type`.

## 4. Non-Goals

Do not build these as part of v0.7.5:

- Server-side LinkedIn full-text scraping.
- Fake-account, cookie-backed, or headless LinkedIn automation.
- YouTube audio/video download and server-side transcription.
- Bypass logic for YouTube anti-bot challenges.
- Gmail API ingestion for Substack.
- Chrome extension or browser DOM capture.
- Firecrawl, Browserless, or Jina as the default first extraction path.
- Unlimited raw HTML storage in SQLite.
- Production deploy, commit, or push without a separate explicit user instruction.

These are allowed as later work:

- Optional YouTube Data API enrichment when `YOUTUBE_DATA_API_KEY` exists.
- Optional Jina public-page fallback after the local adapters ship.
- Manual Substack email body capture.
- Chrome extension/browser capture for LinkedIn and logged-in pages.

```sql
CREATE INDEX IF NOT EXISTS idx_capture_artifacts_item ON capture_artifacts(item_id);
CREATE INDEX IF NOT EXISTS idx_capture_artifacts_kind ON capture_artifacts(kind);
```

### 6.2 Storage Design

Add:

```text
src/db/capture-artifacts.ts
src/lib/capture/artifacts.ts
```

Store artifact files under:

```text
data/artifacts/captures/<item_id>/
```

Extractor result should be able to carry:

```ts
interface CaptureArtifactCandidate {
  kind: string;
  contentType: string;
  body: string | Uint8Array;
  suggestedFilename: string;
}
```

The route/dispatch insert flow should:

1. Extract content and artifact candidates.
2. Insert the item.
3. Write capped/sanitized artifacts under the item directory.
4. Insert `capture_artifacts` rows.
5. If artifact write fails, log the artifact error but do not fail the item capture.

### 6.3 Caps And Sanitization

Initial caps:

| Artifact kind | Cap |
|---|---:|
| YouTube oEmbed JSON | 64 KB |
| YouTube Data API JSON | 256 KB |
| YouTube timed-text XML | 2 MB |
| HTML page snapshot | 512 KB |
| JSON-LD/OpenGraph metadata JSON | 128 KB |
| RSS entry/feed excerpt | 512 KB |
| User-pasted text | 256 KB |

Rules:

- Do not store cookies, bearer tokens, Telegram secrets, browser local storage, or raw request headers.
- Strip `<script>`, `<style>`, tracking fragments, and obvious analytics payloads from HTML artifacts.
- Store large artifacts as files, not SQLite blobs.
- Record `sha256` and `size_bytes` for every artifact.

Acceptance:

- Deleting an item deletes artifact rows through `ON DELETE CASCADE`.
- A follow-up cleanup task can remove orphan files if a filesystem write partially fails.
- Local artifact storage does not write into `.next/standalone/data`.

## 7. Platform Router

Add:

```text
src/lib/capture/platform.ts
src/lib/capture/platform.test.ts
```

Suggested API:

```ts
export type CapturePlatform =
  | "youtube"
  | "youtube_short"
  | "substack"
  | "linkedin"
  | "generic_article";

export interface PlatformDetection {
  platform: CapturePlatform;
  canonicalUrl: string;
  videoId?: string;
  sourceType: "url" | "youtube";
}

export function detectCapturePlatform(rawUrl: string): PlatformDetection;
```

Detection rules:

- `youtube.com/watch`, `youtu.be`, `m.youtube.com/watch`, `youtube.com/embed` -> `youtube`
- `youtube.com/shorts/<id>` -> `youtube_short`
- `*.substack.com/p/*`, `substack.com/@*/p/*`, and custom domains with Substack metadata/feed discovery -> `substack`
- `linkedin.com/posts/*`, `linkedin.com/feed/update/*`, `linkedin.com/pulse/*`, `linkedin.com/in/*/recent-activity/*`, `linkedin.com/company/*/posts/*` -> `linkedin`
- everything else -> `generic_article`

Canonicalization:

- Keep the existing YouTube canonical URL behavior for duplicate detection.
- Strip common tracking params from Substack and LinkedIn URLs: `utm_*`, `trk`, `trackingId`, `li_fat_id`.
- Preserve the URL path and canonical content identifier.

Update dispatchers:

- `src/app/api/capture/url/route.ts`
- `src/lib/telegram/dispatch.ts`

Acceptance:

- URL API and Telegram both use the same platform router.
- YouTube Shorts are stored with `source_platform = "youtube_short"` while keeping `source_type = "youtube"`.
- Substack and LinkedIn are stored with `source_type = "url"` plus specific `source_platform`.
- Duplicate detection uses canonical URLs.

### 8.2 Behavior

Keep the v0.7.4 anti-bot oEmbed fallback.

Add a structured body builder:

```text
Title: ...
Channel: ...
Published: ...
Duration: ...
URL: ...
Capture quality: metadata_plus_transcript

Description:
...

Chapters:
00:00 Intro
...

Transcript:
[0:00] ...
```

Quality rules:

- Transcript plus metadata header -> `capture_quality = "metadata_plus_transcript"`
- Transcript without rich metadata -> `capture_quality = "transcript"`
- No captions but public metadata exists -> `capture_quality = "metadata_only"`
- Anti-bot plus oEmbed fallback -> `capture_quality = "metadata_only"`

Metadata sources:

1. InnerTube `videoDetails` for title, author, duration when available.
2. YouTube oEmbed for title/author fallback.
3. Optional YouTube Data API when `YOUTUBE_DATA_API_KEY` exists.

Data API behavior:

- Add `YOUTUBE_DATA_API_KEY=` to `.env.example`.
- If absent, skip Data API without warning to the user.
- If present, call `videos.list` with fields covering `snippet`, `contentDetails`, `status`, `statistics`, and thumbnails.
- Treat quota/auth/network errors as enrichment misses, not capture failures.
- Save the raw Data API JSON as a capped artifact.

Shorts behavior:

- Use the same video ID and transcript path.
- Set `source_platform = "youtube_short"`.
- If transcript is short or unavailable, save metadata-rich body and be honest in UI.

### 8.3 Tests

Add or update tests for:

- Normal YouTube transcript -> `metadata_plus_transcript`.
- YouTube Short transcript -> `source_platform = "youtube_short"`.
- No caption tracks -> metadata-only saved item.
- Anti-bot InnerTube response plus oEmbed success -> metadata-only saved item.
- Data API absent -> capture still succeeds.
- Data API quota/auth error -> capture still succeeds through fallback.
- Data API success -> `published_at`, `thumbnail_url`, `description`, duration, and body header are populated.
- Structured body parser does not duplicate title/channel when fields are missing.

Acceptance:

- A public YouTube link is saved even if transcript extraction is blocked by YouTube anti-bot sign-in.
- Public YouTube videos with transcripts become more searchable because title/channel/description are in the body and metadata fields.
- YouTube Shorts are saved reliably and labeled honestly.

### 9.2 Capture Ladder

Use the spike-corrected order:

1. Detect/canonicalize Substack URL.
2. Fetch page HTML.
3. Extract JSON-LD Article/BlogPosting metadata.
4. Extract Open Graph metadata.
5. Run Readability and keep this as the primary body candidate.
6. Discover RSS feed:
   - page `<link rel="alternate" type="application/rss+xml">`
   - `https://<host>/feed`
7. Match RSS entry by canonical URL, normalized URL, slug, or title similarity.
8. Use RSS for author/date/description/body only when it is clearly better than Readability.
9. Detect paywall/preview signals from the extracted body and metadata.

Quality rules:

- Clean full body -> `capture_quality = "full_text"`
- Short/truncated preview with paywall indicators -> `capture_quality = "paywall_preview"`
- Metadata-only fallback -> `capture_quality = "metadata_only"`

Metadata fields:

- `published_at` from JSON-LD/RSS when present.
- `thumbnail_url` from JSON-LD/Open Graph.
- `description` from JSON-LD/Open Graph/RSS.
- `author` from JSON-LD/RSS/Readability.

### 9.3 Tests

Use saved fixtures where possible to avoid live-page drift:

- Public Substack post where Readability works.
- Public Substack post with JSON-LD metadata.
- RSS feed match enriches author/date/description.
- RSS mismatch does not replace a good Readability body.
- Paywall/preview page is marked `paywall_preview`.
- Custom-domain feed discovery works through `<link rel="alternate">`.

Acceptance:

- Public Substack posts usually save full text with author/date/description.
- Paid/subscriber-only captures are not presented as full text when only preview content was available.
- Substack captures do not regress generic article capture.

Behavior:

2. Fetch public page metadata only.
3. Extract:
   - Open Graph title
   - Open Graph description
   - canonical URL
   - image
4. Save a metadata body.
5. Set:
   - `source_type = "url"`
   - `source_platform = "linkedin"`
   - `capture_quality = "metadata_only"`
   - `extraction_method = "linkedin_opengraph"`

Do not attempt logged-in DOM extraction on the server.

Telegram acknowledgement:

```text
Saved LinkedIn link as metadata only. For full text, paste the post text with the link.
```

### 10.3 User-Paste Full-Text Path

Support a message/API request that contains both a LinkedIn URL and pasted post text.

Recommended route behavior:

- Reuse the existing optional `note` field in `/api/capture/url` if practical, because `CaptureUrlBody` already accepts it but does not use it today.
- For LinkedIn only, if `note` is present and has enough non-URL text, save the note as the body with the LinkedIn URL as provenance.
- For Telegram, if the message includes a LinkedIn URL plus meaningful text outside the URL, capture the remaining text as user-provided full text instead of discarding it.

Stored fields:

- `source_type = "url"`
- `source_platform = "linkedin"`
- `capture_quality = "user_provided_full_text"`
- `extraction_method = "user_paste"`
- `source_url = canonical LinkedIn URL`
- `body = pasted post text`

Acceptance:

- Sending only a LinkedIn URL creates a useful metadata-only item.
- Sending LinkedIn URL plus pasted text creates a high-quality full-text item.
- The UI explains the difference clearly.
- No LinkedIn cookies, tokens, or scraped authenticated DOM are stored.

## 18. Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Live web pages drift and smoke scores change | Keep fixture-based unit tests for core parsing; treat live smoke as release signal, not first-day CI blocker |
| YouTube Data API key absent | Make Data API optional; keep InnerTube/oEmbed behavior |
| YouTube quota/auth error | Gracefully skip Data API enrichment and save through existing ladder |
| Substack RSS mismatch | Use RSS as enrichment/fallback only; never replace a stronger Readability body without evidence |
| LinkedIn metadata is thin | Tell user it is metadata-only and provide paste path |
| Raw artifact storage grows | Enforce caps, filesystem storage, and later cleanup job |
| Migration typing misses old rows | Conservative nullable fields and backfill rules |
| UI over-warns expected weak captures | Separate quality labels from warnings |
| Tests write to default DB | Keep test setup imports before DB modules and verify local default DB remains clean |
