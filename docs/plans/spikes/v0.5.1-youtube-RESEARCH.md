**Researched:** 2026-05-11
**Scope:** YouTube transcript extraction for AI Brain v0.5.1 — server-side extractor, extension trigger, APK share path, data model, and test strategy.

## 1. How to Fetch a YouTube Transcript Programmatically (2026 State)

### Official YouTube endpoint shape

YouTube does not publish an official third-party transcript API. The two endpoints used in practice:

1. **InnerTube `/youtubei/v1/player`** — POST, JSON body. Returns the full player response including `captions.playerCaptionsTracklistRenderer.captionTracks[]`, each containing a `baseUrl` that points at the actual timed-text XML.
2. **Timed-text XML** — fetched via GET from the `baseUrl` returned above. The URL shape is `https://www.youtube.com/api/timedtext?v=<id>&lang=en&fmt=srv3&...` (query params vary; they are embedded in the `baseUrl` YouTube returns, not constructed manually).

Both endpoints are unauthenticated for public videos. The InnerTube endpoint does not set `Access-Control-Allow-Origin` for arbitrary origins, so direct browser/extension fetches hit CORS (addressed in §2).

### Community library comparison

| Library | npm version (2026-05-11) | Last publish | Approach | Zero-dep? | Node-only? |
|---------|--------------------------|-------------|----------|-----------|------------|
| `youtube-transcript` | 1.3.1 | 2026-04-25 [VERIFIED: npm registry] | InnerTube POST (Android client context) + HTML-scrape fallback; XML parse | Yes — only `fetch` | No — pure Web APIs |
| `youtubei.js` | 17.0.1 | 2026-03-16 [VERIFIED: npm registry] | Full InnerTube client, platform shims for Node/browser/CF-Worker/React Native | No — 15 MB unpacked, 2047 files [VERIFIED: npm registry] | No, but complex |

`youtube-transcript` is the right choice for this project:
- 44 KB unpacked vs 15 MB for `youtubei.js` [VERIFIED: npm registry]
- Zero production dependencies
- Uses only `fetch` — runs in Node.js Next.js server, extension service worker, and any environment with `fetch`
- Actively maintained: last publish 2026-04-25, v1.3.0→v1.3.1 released in 2026

`youtubei.js` is more capable (live chat, downloads, full info) but costs 15 MB, has platform-specific shims, and its transcript path (`info.getTranscript()`) was returning HTTP 400 in December 2025–January 2026 and remains intermittently broken as of v16–v17 per open issues #1102 and #1118. [CITED: https://github.com/LuanRT/YouTube.js/issues/1102, https://github.com/LuanRT/YouTube.js/issues/1118]

### Known breakage (last 6 months)

Two issues in `youtube-transcript`:

- **Issue #49 (Jan 2026):** `fetchTranscript` throws "Cannot read properties of undefined (reading 'playerCaptionsTracklistRenderer')" when that field is missing — videos without captions, age-gated, or region-locked. Fix: wrap in try/catch and return a typed "no_captions" error, not a JS crash. [CITED: https://github.com/Kakulukian/youtube-transcript/issues/49]

- **Issue #43 (ongoing):** Requests from **production server IPs** (Render, DigitalOcean, Vercel) are blocked by YouTube with "transcript is disabled / blocked by YouTube." Local dev works fine. [CITED: https://github.com/Kakulukian/youtube-transcript/issues/43]

**Implication for AI Brain:** The server runs on the user's Mac behind a Cloudflare Named Tunnel. A Mac on residential/office IP is very unlikely to be in YouTube's datacenter block list. This is a known risk for cloud-hosted deployments; it is not a problem for the local-first architecture of this project.

The library's Android client context (`clientName: "ANDROID", clientVersion: "20.10.38"`) is the current workaround that keeps it working on most IPs. [CITED: https://raw.githubusercontent.com/Kakulukian/youtube-transcript/master/src/index.ts]

### What `fetchTranscript` returns per segment

```typescript
// Source: youtube-transcript v1.3.1 src/index.ts
interface TranscriptResponse {
  text: string;      // plain-text spoken words, HTML entities decoded
  duration: number;  // milliseconds
  offset: number;    // start time in milliseconds
  lang?: string;
}

const segments = await fetchTranscript("dQw4w9WgXcQ");
// segments[0] => { text: "We're no strangers to love", duration: 2800, offset: 0 }
```

### Fields NOT in the transcript library

`fetchTranscript` only returns timed text. Video title, channel, duration, and description come from the **InnerTube player response** JSON that the library already fetches internally. To expose them you either:
- Call `fetchTranscript` (cheap re-parse won't re-fetch) and also call YouTube's oEmbed endpoint (`https://www.youtube.com/oembed?url=...&format=json`) for title and channel — no auth required, no CORS on server side. [ASSUMED — oEmbed shape; verify before implementation]
- Or use a second InnerTube POST yourself for `basic_info`.

**Recommendation:** Use oEmbed for title + channel (one extra GET, zero extra dep), set `author = channel_name`, and store `duration_seconds` in a dedicated column (see §4).

### Failure modes

| Condition | Library behavior | Correct handling |
|-----------|-----------------|------------------|
| No captions (CC disabled, auto-caps not generated) | Throws crash on `playerCaptionsTracklistRenderer` (issue #49) | Catch, return `YoutubeCaptureError("no_captions", …)` |
| Age-gated / login required | InnerTube returns player with no caption tracks | Same — "no_captions" error |
| Private / unlisted / deleted | InnerTube returns error JSON; `captionTracks` absent | Catch, return `YoutubeCaptureError("unavailable", …)` |
| Live stream (active) | Caption tracks absent or have `isLive` flag | Detect and return `"live_stream"` error |
| Very long video (>4 h, ~14 400 segments at 1 s each) | Transcript XML can be 1–3 MB; all in memory | Cap at 7 200 segments (~2 h) or truncate with `extraction_warning` |
| YouTube rate-limit / IP block | `fetchTranscript` throws network/HTTP error | Surface as `"fetch_failed"` like the URL extractor |

**Shorts (<60 s):** Auto-captions are sometimes generated but often absent. Treat same as no-captions — return `extraction_warning: "shorts_no_transcript"` and store `body = "[No transcript available for this Short]"` so the item is still captured with title/URL.

## 2. Chrome/Edge Extension Approach

### CORS in service workers with `host_permissions`

When a URL is listed in `host_permissions`, the Chrome extension service worker bypasses the browser's CORS enforcement for outgoing `fetch()` calls to that origin. The CORS preflight is skipped entirely. [CITED: https://developer.chrome.com/docs/extensions/develop/concepts/network-requests — "A script executing in an extension service worker or foreground tab can talk to remote servers outside of its origin, as long as the extension requests host permissions."]

This means the extension service worker **can** fetch `https://www.youtube.com/youtubei/v1/player` directly if `https://www.youtube.com/*` is in `host_permissions`.

### Client-side vs server-side extraction decision

| Approach | Pros | Cons |
|----------|------|------|
| **Client-side (extension service worker)** | No extra host permission on Brain server; works offline from server; faster UX | Extension must add `https://www.youtube.com/*` to `host_permissions` (visible in Chrome Web Store permission prompt); `youtube-transcript` must bundle into the extension (adds ~44 KB to extension bundle); duplicates logic |
| **Server-side (extension POSTs URL → Brain server extracts)** | Single implementation; extension stays thin; no change to extension host_permissions; consistent with how URL capture already works | One round-trip over CF tunnel before user sees "Saved"; server on Mac must be running |

**Recommended approach: server-side extraction.**

The extension currently POSTs every URL to `/api/capture/url`. YouTube URLs would go through the same path — the server detects the YouTube URL and calls the new `extractYoutubeVideo()` function instead of `extractArticleFromUrl()`. The extension code changes by exactly zero lines. This is consistent with the project's "server-side is the truth" model.

Extension changes needed: none in the core flow. A UX polish (show a YouTube-specific notification — "Transcript saved") is optional.

### `youtube-transcript` in a service worker

The library uses only `fetch` and `RegExp` — no Node-specific imports. It is safe to `import` in a service worker context if bundled via Vite (which the extension already uses). Size impact: ~44 KB unpacked → ~10 KB gzipped. Well within extension bundle budget. [VERIFIED: npm registry unpackedSize]

However, given the server-side recommendation above, bundling into the extension is unnecessary for v0.5.1.

### No-captions failure mode in the extension

If the Brain server returns `422 { error: "no_captions" }`, the extension shows: "No transcript available for this video. Title saved without transcript." — consistent with existing `describe(result)` error handling in `background.ts`.

## 3. Capacitor/APK (Android) Approach

### URL shape from Android YouTube app share

The YouTube Android app produces a `youtu.be/<id>` short URL as `texts[0]` in the `shareReceived` payload (e.g., `https://youtu.be/dQw4w9WgXcQ`). [ASSUMED — confirmed by widespread developer reports; Android YouTube app always uses youtu.be short links for the share sheet]

The `share-handler.tsx` already routes `texts[0]` that passes `looksLikeUrl()` to `POST /api/capture/url`. A `youtu.be/` URL is a valid HTTPS URL and will pass that check. So the APK flow **requires zero code changes** as long as the server correctly handles all YouTube URL variants.

### Normalize to canonical URL on server

All YouTube URL variants must normalize to `https://www.youtube.com/watch?v=<id>` before the `findItemByUrl` dedup check. This prevents `youtu.be/X` and `youtube.com/watch?v=X` from being stored as two separate items.

```typescript
// src/lib/capture/youtube.ts — normalizeYoutubeUrl()
const YOUTUBE_PATTERNS = [
  /^https?:\/\/(?:www\.)?youtube\.com\/watch.*[?&]v=([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/youtu\.be\/([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:www\.)?youtube\.com\/shorts\/([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:m\.)?youtube\.com\/watch.*[?&]v=([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:www\.)?youtube\.com\/embed\/([a-zA-Z0-9_-]{11})/,
];

export function extractVideoId(url: string): string | null {
  for (const re of YOUTUBE_PATTERNS) {
    const m = url.match(re);
    if (m) return m[1];
  }
  return null;
}

export function canonicalYoutubeUrl(videoId: string): string {
  return `https://www.youtube.com/watch?v=${videoId}`;
}
```

### APK vs extension: same server path?

Yes. Both flow through `POST /api/capture/url` on the Brain server. The server detects the YouTube URL, calls `extractYoutubeVideo()`, and returns `{id, duplicate}` — identical contract to URL capture. No APK-specific code beyond what already exists.

## 4. Data Model Choices

### `source_type`

The `items` table CHECK constraint already includes `'youtube'` as a valid value (migration `001_initial_schema.sql`, line 18). [VERIFIED: codebase read] Use `source_type: "youtube"` — it enables filtering, display customization, and type-specific UI later without touching the CHECK constraint.

### `body` content

Store the transcript as **plain text in `body`**, formatted as:

```
[00:00] We're no strangers to love
[00:03] You know the rules and so do I
...
```

Rationale:
- The existing chunker, FTS5 index, and embedding pipeline operate on `body` with zero changes.
- Timestamp prefixes `[MM:SS]` are human-readable and searchable ("find where they discussed X").
- Plain text avoids a new `transcript_segments` table and a new JSON column in v0.5.1.

A separate `transcript_segments` table (with millisecond-precision timestamps for deep-link UX) is a valid v0.5.2+ enhancement. Defer it.

### Chunking strategy for transcripts

The existing token-based chunker (400–800 tokens, paragraph-boundary split) works well for transcripts because:
- Transcript lines separated by `\n` become one block per timestamp.
- The paragraph-flush at blank lines doesn't apply (transcripts have no blank lines by default), so long transcripts will naturally form 400-token windows via the sentence-split fallback.

**Add one tweak:** Before chunking, group transcript lines into 30-second windows and insert a double newline between windows. This gives the chunker natural paragraph boundaries aligned to speech rhythm, and improves retrieval precision.

```typescript
// Group segments into ~30s paragraphs before chunkBody()
function formatTranscriptBody(segments: TranscriptResponse[]): string {
  const WINDOW_MS = 30_000;
  let windowStart = 0;
  const paragraphs: string[] = [];
  let lines: string[] = [];

  for (const seg of segments) {
    if (seg.offset - windowStart > WINDOW_MS && lines.length > 0) {
      paragraphs.push(lines.join("\n"));
      lines = [];
      windowStart = seg.offset;
    }
    const ts = msToTimestamp(seg.offset);
    lines.push(`[${ts}] ${seg.text.trim()}`);
  }
  if (lines.length) paragraphs.push(lines.join("\n"));
  return paragraphs.join("\n\n");
}

function msToTimestamp(ms: number): string {
  const s = Math.floor(ms / 1000);
  const m = Math.floor(s / 60);
  const h = Math.floor(m / 60);
  if (h > 0) return `${h}:${String(m % 60).padStart(2, "0")}:${String(s % 60).padStart(2, "0")}`;
  return `${m}:${String(s % 60).padStart(2, "0")}`;
}
```

### Citation UX with time anchors

The existing citation system emits `[CITE:<chunk_id>]` and renders a clickable reference to the item page. For YouTube, the rendered citation should deep-link to `https://www.youtube.com/watch?v=<id>&t=<seconds>`.

Since `body` already contains `[MM:SS]` prefixes in every chunk, the citation renderer can parse the first timestamp in the retrieved chunk text to compute `t`. No schema change needed for v0.5.1.

A dedicated `start_offset_ms` column on `chunks` (for time-anchored citations without string parsing) is a v0.5.2+ enhancement.

### New column: `duration_seconds`

Add `duration_seconds INTEGER` to `items` via migration `007_youtube_duration.sql`. Null for non-video items. Used for display ("1h 23m") and for enforcing the 2-hour cap on ingestion.

The schema already has `total_pages` (null for non-PDF) and `total_chars` — `duration_seconds` follows the same nullable pattern.

## 5. URL Detection and Routing in the Capture Pipeline

### Where to detect

Detect YouTube URLs **after** `findItemByUrl(url)` dedup check, **before** calling `extractArticleFromUrl()`. The detection goes in `route.ts`:

```typescript
// src/app/api/capture/url/route.ts (v0.5.1 change)
import { extractVideoId } from "@/lib/capture/youtube";

// ... after dedup check, before extractArticleFromUrl:
const videoId = extractVideoId(url);
if (videoId) {
  // route to YouTube extractor
  article = await extractYoutubeVideo(videoId, url);
} else {
  article = await extractArticleFromUrl(url);
}
```

**Why after dedup:** Dedup uses `source_url` as stored. After normalization, the canonical URL is `https://www.youtube.com/watch?v=<id>`, so dedup must run on the canonical form. Extract video ID first for normalization, then run dedup against the canonical URL.

Revised order:
1. Extract video ID (if YouTube URL)
2. Normalize URL to canonical form
3. `findItemByUrl(canonicalUrl)` dedup
4. Route to YouTube or Readability extractor

### YouTube URL patterns

```typescript
// Patterns that count as "YouTube" (all normalize to watch?v=)
const YOUTUBE_PATTERNS = [
  /^https?:\/\/(?:www\.)?youtube\.com\/watch.*[?&]v=([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/youtu\.be\/([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:www\.)?youtube\.com\/shorts\/([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:m\.)?youtube\.com\/watch.*[?&]v=([a-zA-Z0-9_-]{11})/,
  /^https?:\/\/(?:www\.)?youtube\.com\/embed\/([a-zA-Z0-9_-]{11})/,
];
```

Embed URLs (`/embed/<id>`) are included because the extension context-menu fires on embedded YouTube players.

### Shorts

Auto-captions exist for some Shorts but are absent for most music-only or speech-free Shorts. Handle identically to no-captions: save the item with `body = "[No transcript available for this Short]"` and `extraction_warning: "shorts_no_transcript"`. The title + thumbnail URL are still useful for the library.

## 6. Failure-Mode Catalogue

| Failure | Detection | API response | User-facing message |
|---------|-----------|-------------|---------------------|
| No captions (CC disabled / no auto-caps) | `playerCaptionsTracklistRenderer` absent or empty tracks | `422 { error: "no_captions" }` | "No transcript available. Item saved with title only." |
| Age-gated / sign-in required | InnerTube returns no caption tracks | `422 { error: "no_captions" }` | Same as above |
| Private / unlisted / deleted | InnerTube returns error; no `videoDetails` | `422 { error: "video_unavailable" }` | "Video is private, deleted, or unavailable." |
| Live stream (active, no VOD yet) | `captionTracks` absent or `isLive: true` in player response | `422 { error: "live_stream" }` | "Live stream — transcript not available yet. Try again after the stream ends." |
| Transcript > 7 200 segments (~2 h) | `segments.length > 7200` check | `201` with `extraction_warning: "transcript_truncated_2h"` | Saves first 2 h; warning shown in item detail |
| YouTube rate-limit / IP block | `fetchTranscript` throws HTTP 4xx/5xx | `422 { error: "fetch_failed", message }` | "YouTube blocked the request. Try again later." |
| Network failure mid-fetch | `fetch` throws / AbortError | `422 { error: "fetch_failed", message }` | "Network error fetching transcript." |
| oEmbed title fetch fails | oEmbed GET returns non-200 | Degrade gracefully: use video ID as title, continue | `extraction_warning: "title_fetch_failed"` |

**No new `UrlCaptureError` codes needed.** Add a parallel `YoutubeCaptureError` class with codes: `"no_captions"`, `"video_unavailable"`, `"live_stream"`, `"fetch_failed"`. The route handler catches it and maps to 422, same pattern as `UrlCaptureError`.

## 7. Evaluation and Test Strategy

### The fundamental challenge

`youtube-transcript` calls a live YouTube endpoint. Mocking at the HTTP layer (`node:test` `fetch` mock or `vi.stubGlobal`) is the only way to make tests deterministic.

### Recommended test architecture

```
src/lib/capture/youtube.test.ts
├── Unit: extractVideoId() — no network
├── Unit: canonicalYoutubeUrl() — no network
├── Unit: formatTranscriptBody() — no network, fixed input array
├── Unit: extractYoutubeVideo() — mock fetchTranscript + oEmbed fetch
│     ├── happy path: returns ExtractedArticle shape
│     ├── no_captions: throws YoutubeCaptureError("no_captions")
│     ├── video_unavailable: throws YoutubeCaptureError("video_unavailable")
│     └── truncation: segments > 7200 → extraction_warning set
└── Integration smoke (tagged @smoke, excluded from CI):
      scripts/smoke-youtube.ts — real network, 3 fixture video IDs
```

### Fixture video IDs for smoke test

Use 3 real videos with known properties:

| Video ID | Property tested |
|----------|----------------|
| `dQw4w9WgXcQ` | Normal video with English auto-captions |
| `jNQXAC9IVRw` | First YouTube video ever (short, minimal captions) |
| One Shorts ID | Shorts — expected `no_captions` or short transcript |

Store expected title strings in the smoke script for a basic sanity check.

### Fixture cache

Capture one real InnerTube response + timed-text XML response and commit to `src/lib/capture/__fixtures__/youtube-player-response.json` and `youtube-timedtext.xml`. The unit test's `fetch` mock returns these fixtures. This eliminates network dependency in CI and makes the "what changed when YouTube breaks us" diff obvious.

### Rollback plan

If YouTube breaks `youtube-transcript` in the future:

1. `extraction_warning: "transcript_unavailable"` is already returned on no-captions — the item is saved without a body. No data loss.
2. The `extractYoutubeVideo()` function is isolated in `src/lib/capture/youtube.ts`. Swap the library or switch to manual `fetch` of InnerTube without touching route handlers.
3. The smoke test (`npm run smoke:youtube`) will immediately surface breakage before users report it.

## 8. Scope Recommendation

### Minimum shippable v0.5.1

These pieces together make YouTube capture genuinely useful:

1. **`src/lib/capture/youtube.ts`** — `extractVideoId()`, `canonicalYoutubeUrl()`, `extractYoutubeVideo()`, `YoutubeCaptureError`. Single new file. Returns `ExtractedArticle` shape so no route changes downstream.
2. **`src/app/api/capture/url/route.ts`** — add YouTube URL detection + routing branch (~8 lines).
3. **`007_youtube_duration.sql`** — add `duration_seconds INTEGER` column.
4. **`package.json`** — add `youtube-transcript@^1.3.1`. One new dep, justified: 44 KB, zero transitive deps, actively maintained.
5. **`src/lib/capture/youtube.test.ts`** — unit tests with fixture mocks.
6. **`scripts/smoke-youtube.ts`** — real-network smoke script, not in CI.

No extension changes. No APK changes. No new UI components. The item detail page already renders `body` as text — timestamped transcript lines display immediately.

**The user experience:** Share `https://youtu.be/dQw4w9WgXcQ` from Android or right-click in Chrome → "Save to Brain" → item appears with title, channel as author, and full timestamped transcript. FTS and semantic search work on day one.

### Explicitly defer to v0.5.2+

- **`transcript_segments` table** — millisecond-precision time anchors, chapter markers, per-segment retrieval. Worth it when citation deep-links are a real user ask.
- **`start_offset_ms` on `chunks`** — needed for "jump to 4:32 in YouTube" citation links from the Ask UI.
- **Chapter marker extraction** — InnerTube player response contains `chapters` when the video description has timestamps. Useful but not critical for first launch.
- **oEmbed thumbnail URL** — store as `thumbnail_url` on items for richer library display.
- **Shorts detection + skip** — could silently not capture Shorts at all, but saving with a warning is more honest.
- **User-configurable transcript language** — `TranscriptConfig.lang` supports this; expose in Settings later.
- **Rate-limit retry with exponential backoff** — only needed if the Mac IP ever gets blocked.
- **Description-based body fallback** — when no captions, use video description as `body` instead of the empty placeholder. Useful but adds another InnerTube call.

### Key decisions for the user (planner must confirm)

1. **oEmbed for title/channel vs. a second InnerTube call.** oEmbed is simpler (one GET, no body parsing) but only returns title + author + thumbnail URL — not duration. Duration requires parsing `videoDetails.lengthSeconds` from the InnerTube player response that `youtube-transcript` already fetches internally. The recommended approach is to restructure `extractYoutubeVideo()` to do the InnerTube POST itself (one call, gets captions + duration + title), rather than calling `youtube-transcript` as a black box and adding an oEmbed call. **Does the user prefer one self-contained InnerTube POST (slightly more code, all fields), or `youtube-transcript` + oEmbed (simpler code, no duration)?**

2. **No-captions behavior for Shorts.** Current proposal: save the item with an empty-ish body and `extraction_warning`. Alternative: silently return a duplicate-like 200 and don't save at all. Which is preferred?

3. **2-hour transcript cap.** A 2-hour cap means a 3-hour lecture saves only the first half. Is that acceptable, or should the cap be higher (e.g., 4 h, 14 400 segments) at the cost of a potentially 3 MB body field?
