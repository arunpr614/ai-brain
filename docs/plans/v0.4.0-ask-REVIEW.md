## Patches (non-blocking)

### P-1 — T-5 should define `nomic-embed-text` install check, not assume it

Current T-5 exit criterion is "embed-and-retrieve round-trip returns top-1." But if the user hasn't run `ollama pull nomic-embed-text` yet, the pipeline will hit a 404. §8 mentions this risk and says T-5 "detects `model not found`, prints `ollama pull ...`, retries once" — but that logic isn't in any task description; it's buried in a risk row.

**Fix:** add to T-5 exit criteria: "first call to `/api/embed` with missing model returns a structured `EMBED_MODEL_NOT_INSTALLED` error containing the exact `ollama pull` command; a unit test exercises this path." Cost: ~20 min.

### P-2 — SC-7 p95 < 2 s is aggressive given Ollama cold-start

`bench-ask.mjs` (T-18) measures "first token" latency, but Ollama generates a ~5 s cold start on first call after idle (model load into RAM). On a warmed model, p95 < 2 s is achievable on M1 Pro for a 7B Q4 model at top-k=8 × ~500 tokens. On a cold model, the first request blows past 2 s every time.

**Fix options:**
- (a) Lower the bar: change SC-7 to "p95 < 2 s on *warm* model; first-ever request may take up to 8 s."
- (b) Add a keep-alive: `/api/ask` pre-pings Ollama if > 5 min idle.
- (c) Accept the risk in-plan — note in §7 that SC-7 is warm-model only.

**Recommendation:** (a) + (c). Measure warm-only in the bench, document the cold-start behaviour as a known characteristic.

### P-3 — Embedding pipeline failure recovery is under-specified

T-5 says "3× retry with backoff" but doesn't specify: what happens after retry 3 fails? The enrichment-worker precedent (F-045, F-050) is to mark the job failed, log to `errors.jsonl`, and leave retry to `sweepStaleClaims()`. The embedding-jobs sibling table (T-3) needs the same shape or we'll end up with orphaned unembedded items that nothing re-queues.

**Fix:** T-3 schema for `embedding_jobs` must include `attempts INTEGER DEFAULT 0`, `claimed_at INTEGER`, `last_error TEXT`, mirroring `enrichment_jobs`. T-5 should exit-criterion require: "after 3 retries, job is marked failed and `errors.jsonl` gets an entry; sweeping re-queues after 10 min." Cost: ~10 min spec tightening, already present-pattern in code.

### P-4 — `[CITE:chunk_id]` drop-orphan logic is a silent failure

Current T-9 says "filter against actual retrieved chunk IDs, drop orphans." If the LLM cites well but gets *one* chunk_id wrong, the user sees the rest of the answer with a missing citation — no indication anything went wrong. Over time, if the LLM orphan-rate creeps up (prompt drift, model update, long context), we'll have degraded citations and no signal.

**Fix:** log orphan citations to `errors.jsonl` with a metric counter. Plan §5.7 says "append latency metrics to errors.jsonl when ask > 10 s" — add orphan-citation events to the same sink. Acceptance: a test that feeds a known-bad LLM response and asserts the orphan is logged.

## Items explicitly considered and left alone

- **No benchmark for 1024-dim.** User-approved in v1.1, explicit rationale in §12 Closed. Not second-guessing.
- **Sibling `embedding_jobs` table vs extending `enrichment_jobs`.** Plan picks sibling with stated rationale (§5.2). Both are defensible; sibling is slightly more code but easier to reason about. Keep.
- **Auto-title threads deferred to v0.4.x.** Correct scope call.
- **SSE vs streaming server action.** Deferring to T-8 skeleton measurement is the right move; both are cheap to try.
- **Related-items via mean-of-chunk-centroids.** Simple first pass, note to revisit. Keep.

## Items that felt tempting to add but shouldn't block v0.4.0

- **Hybrid BM25+vector re-ranking.** Already out of scope. Don't re-open.
- **Per-item cost caps.** Single-user local LLM — no cost to cap. Don't re-open.
- **Cross-encoder re-ranking.** Would be genuine quality improvement but needs a separate model and adds latency. Measure naive top-k first, add if recall@k turns out noisy.
