## 2. Why This Comes First

The project is in a partially-shipped state:

- Web build has passed locally.
- Production is alive.
- v0.7.2 provider guardrails are implemented locally.
- The new Android APK exists locally.
- The new changes are not yet fully released and verified end-to-end.

Starting a fresh roadmap feature now would stack new work on top of an unclosed release. That makes it harder to know whether future issues came from v0.7.2, Android packaging, production deployment, or the next feature lane.

The correct next move is release discipline:

- First: make the current work real in production.
- Second: verify the Android build that contains it.
- Third: record the outcome.
- Fourth: start the next product lane from a clean, truthful baseline.

### Gate C - APK Verified

Can mark v0.7.2 shipped only if:

- APK installs and launches.
- Settings provider status works.
- Item detail semantic indexing state works.
- No critical Android-only regression is found.

Decision:

- Pass: update roadmap and mark v0.7.2 shipped.
- Fail: document blocker, build a corrected APK with a bumped version/code, and retest.

## 8. Rollback Plan

Rollback is required if production is broken after deploy.

Rollback steps:

1. Identify last known good production deployment.
2. Revert production to the last known good deployment.
3. Confirm production health.
4. Record rollback timestamp and reason in running log.
5. Create a targeted hotfix plan for the failed release.

Do not continue to Android verification until production is stable again.

## 9. Risk Register

| Risk | Impact | Mitigation |
|---|---|---|
| Unrelated v0.7.3 changes accidentally included | Release becomes hard to verify | Use clean release branch and inspect diff before deploy |
| Gemini billing still fails despite updated billing | Settings may show degraded status | Verify with a safe provider-status check before marking resolved |
| Provider status leaks raw provider errors | Security/privacy issue | Inspect payload and UI copy before release |
| APK points to stale production behavior | Android appears unchanged | Deploy production first, then test APK |
| APK version not bumped for a rebuilt APK | User cannot distinguish builds | If a new APK is created, bump version/code and filename |
| Android device not connected | APK test may be delayed | Provide APK path for manual install; record what was and was not verified |
| Semantic indexing state has no good test item | Verification incomplete | Use or create one indexed and one unindexed/pending item |
