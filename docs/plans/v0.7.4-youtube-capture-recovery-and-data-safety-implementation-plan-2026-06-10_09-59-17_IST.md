# v0.7.4 YouTube Capture Recovery And Data Safety Implementation Plan

**Created:** 2026-06-10 09:59:17 IST  
**Status:** implemented locally, not committed or deployed  
**Source report:** `ReviewReport/AI_BRAIN_ROOT_CAUSE_REPORT_2026-06-10_09-46-14_IST.md`  
**Target branch:** `codex/v0.7.3-telegram-capture-hardening` or a follow-up branch cut from it after review  
**Primary goal:** Make Telegram and API YouTube captures save reliably when production YouTube transcript extraction is blocked by an anti-bot sign-in challenge, and prevent test/runtime data from being shipped into production again.

---

## 1. Executive Summary

The root-cause investigation found two separate issues.

1. Telegram YouTube capture failed because production Hetzner requests to YouTube InnerTube returned `LOGIN_REQUIRED` with `Sign in to confirm you're not a bot`. The videos were public and extract correctly from the local Mac. The app collapsed the production challenge into the misleading message `Video is private, deleted, or unavailable`, then threw before inserting a Library item.
2. Dummy Library records were created by unit-test fixtures that wrote into the default local database. A standalone build then copied that local `data/brain.sqlite` into `.next/standalone/data/`, and the deploy script synced that directory into `/opt/brain/`, overwriting production runtime data.

Cleanup has already removed the dummy rows from production and local default DBs. Two recurrence fixes are already present locally but still need normal review, commit, and deployment:

- `src/lib/items/status.test.ts` now imports test DB setup before DB modules.
- `scripts/deploy.sh` now excludes `.next/standalone/data/` from production sync.

The remaining product fix is to add a YouTube anti-bot fallback. If InnerTube is blocked but YouTube oEmbed can still identify the public video, Brain should save a metadata-only YouTube item instead of failing the capture. Telegram should tell the user the link was saved, while being honest that transcript extraction was blocked by YouTube.

## 2. Non-Goals

- Do not add a browser automation or cookie-backed YouTube scraper in this release.
- Do not attempt to bypass YouTube's anti-bot challenge.
- Do not restore older production Library data from encrypted B2 backups as part of this implementation; backup decryption was blocked during the investigation.
- Do not redesign the full capture/enrichment pipeline.
- Do not deploy unrelated dirty work without explicitly reviewing and staging it.
- Do not create automated production Library items during smoke tests unless a test flag explicitly allows cleanup afterward.

- Sending an affected public YouTube link to Telegram creates a Library item.
- The item has:
  - `source_type = "youtube"`
  - canonical YouTube URL
  - title from oEmbed
  - author/channel from oEmbed

### T-2. Model YouTube Playability Status Explicitly

**Purpose:** Stop treating every missing `videoDetails` response as private/deleted/unavailable.

**Files**

- `src/lib/capture/youtube.ts`
- `src/lib/capture/youtube.test.ts`

**Implementation**

- Extend the internal `InnerTubePlayer` type with:
  - `playabilityStatus.status`
  - `playabilityStatus.reason`
  - optionally `playabilityStatus.messages`
- Add a small helper:

```ts
function getPlayabilityReason(player: InnerTubePlayer): string | null
```

- Add a challenge detector:

```ts
function isYoutubeAntiBotChallenge(player: InnerTubePlayer): boolean
```

It should return true when:

- `playabilityStatus.status === "LOGIN_REQUIRED"`
- and the reason/messages contain a phrase like `not a bot` or `sign in to confirm`

- When `videoDetails` is missing:
  - if anti-bot challenge is detected, route to the oEmbed metadata fallback in T-3
  - otherwise throw `YoutubeCaptureError("video_unavailable", message)` with a more accurate message that includes the playability reason when present

**Acceptance**

- Unit tests can distinguish anti-bot challenge from genuine unavailable/private responses.
- The old generic unavailable message is not used for the anti-bot case.

### T-3. Add YouTube oEmbed Metadata-Only Fallback

**Purpose:** Save public YouTube links even when production transcript extraction is blocked.

**Files**

- `src/lib/capture/youtube.ts`
- `src/lib/capture/youtube.test.ts`

**Implementation**

- Add oEmbed URL builder:

```ts
function youtubeOEmbedUrl(canonicalUrl: string): string
```

Expected endpoint:

```text
https://www.youtube.com/oembed?format=json&url=<canonical-url>
```

- Add response type:

```ts
interface YoutubeOEmbedResponse {
  title?: string;
  author_name?: string;
  author_url?: string;
  provider_name?: string;
  thumbnail_url?: string;
}
```

- Add fetch helper using the existing timeout pattern:

```ts
async function fetchYoutubeOEmbed(videoId: string): Promise<YoutubeOEmbedResponse | null>
```

- Keep timeout short enough for Telegram UX, preferably using the existing `FETCH_TIMEOUT_MS` unless a lower oEmbed timeout is introduced.
- Treat non-200 oEmbed as no fallback.
- Treat malformed oEmbed JSON as no fallback, not a crash.
- Add metadata-only result helper:

```ts
function antiBotMetadataOnlyResult(
  videoId: string,
  oembed: YoutubeOEmbedResponse,
): YoutubeExtracted
```

Returned shape:

```ts
{
  title: oembed.title || videoId,
  author: oembed.author_name || null,
  duration_seconds: null,
  source_url: canonicalYoutubeUrl(videoId),
  body: "[Transcript unavailable: YouTube blocked Brain's server transcript request with an anti-bot sign-in check.]",
  extraction_warning: "youtube_antibot_metadata_only",
}
```

**Decision**

- Only use this fallback when InnerTube explicitly shows an anti-bot sign-in challenge and oEmbed returns a valid public-video response.
- If oEmbed also fails, throw `YoutubeCaptureError("video_unavailable", "YouTube blocked transcript extraction with a sign-in challenge, and metadata fallback failed.")`.

**Acceptance**

- `extractYoutubeVideo()` returns a valid `YoutubeExtracted` object for mocked InnerTube anti-bot + oEmbed success.
- `extractYoutubeVideo()` still throws for mocked anti-bot + oEmbed 404.
- No transcript body is invented.

## 7. Rollback Plan

### If Deploy Breaks The Service

- Revert to the previous deployed application version using the existing production deployment process.
- Keep `/opt/brain/data/` intact.
- Do not restore a local DB over production.
- Restart `brain.service`.
- Verify provider status and item count.

### If YouTube Fallback Saves Bad Items

- Disable the fallback by reverting the `youtube.ts` fallback commit.
- Delete only the affected metadata-only items by `extraction_warning = 'youtube_antibot_metadata_only'` after taking a backup.
- Keep the deploy data exclusion and test isolation fixes.

### If Production Item Count Changes Unexpectedly

- Stop the service.
- Copy `/opt/brain/data/brain.sqlite` to a timestamped backup before touching it.
- Compare against the pre-deploy count and backup.
- Restore only from a production backup, never from local `.next/standalone/data/`.

## 8. Risks And Mitigations

### Risk: oEmbed Also Becomes Blocked

Mitigation:

- Fallback should fail honestly with a specific error saying YouTube blocked transcript extraction and metadata fallback failed.
- Production smoke should surface this before relying on Telegram manually.

### Risk: Metadata-Only Items Have Low Ask Quality

Mitigation:

- Body must clearly state transcript is unavailable.
- Warning code should be visible in Library/detail.
- Longer-term transcript-provider or user-authorized browser/cookie path can be considered separately.

### Risk: Unrelated v0.7.3 Dirty Changes Ride Along

Mitigation:

- Stage files explicitly by workstream.
- Review `git diff --cached` before commit.
- Avoid broad staging commands.

### Risk: Test Data Reappears Locally

Mitigation:

- Keep DB setup imports first in all DB-reaching tests.
- Add a quick post-test DB-count check for status tests.
- Consider a future guard in `src/db/client.ts` that warns when tests open the default DB without `BRAIN_DB_PATH`.
