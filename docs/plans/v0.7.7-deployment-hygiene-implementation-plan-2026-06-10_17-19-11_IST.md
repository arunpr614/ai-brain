# v0.7.7 Deployment Hygiene Implementation Plan

**Created:** 2026-06-10 17:19:11 IST  
**Author:** Codex  
**Problem:** the v0.7.6 production deploy succeeded, but `scripts/deploy.sh` stopped at the authenticated health check because the local `.env` value for `BRAIN_API_TOKEN` was stale. Production itself was healthy when checked with the token from `/etc/brain/.env`.  
**Primary goal:** make production deploys finish cleanly without copying the production bearer token back to the laptop or printing it in logs.  
**Recommended path:** change `scripts/deploy.sh` so its authenticated health check uses the server's production token safely, in-place on the server. Keep local-token health checks as an explicit optional mode, not the default.

---

## 1. Executive Summary

The deploy script currently treats the laptop's `BRAIN_API_TOKEN` as the source of truth for the production health check. That was reasonable while local and production tokens were expected to match, but it is now brittle:

1. Production owns the real runtime token in `/etc/brain/.env`.
2. Local `.env` can drift after a production token rotation, device pairing flow, manual edit, or old workspace copy.
3. A stale local token makes a successful deployment look failed.
4. The safest fix is not to sync secrets back to the laptop; it is to perform the authenticated health check from the server, using the server token without exposing it.

This is a deployment hygiene release. It should not change app behavior, capture behavior, user data, Telegram behavior, or production configuration beyond the deploy script's verification path.

---

## 2. Current Behavior

### Deploy Flow Today

`scripts/deploy.sh` does this:

1. Loads local `.env`.
2. Requires local `BRAIN_API_TOKEN`.
3. Verifies the remote `/etc/brain/.env` contains `BRAIN_API_TOKEN`.
4. Builds and syncs the app.
5. Restarts the production service.
6. Calls `https://brain.arunp.in/api/health` from the laptop using the local token.

### Failure Mode Observed

The deployment succeeded, but step 6 returned `401` because the local token was stale. A manual server-side health check with production's real token returned `200`.

### Root Cause

The deploy health check is tied to a local copy of a production secret. That local copy is not guaranteed to remain current.

---

## 3. Desired Behavior

After this change:

- A production deploy should finish cleanly when production is healthy.
- The deploy script should not require the local laptop to hold the production `BRAIN_API_TOKEN` for the default deploy path.
- The production token should stay on the server and should never be echoed into local logs.
- If production health fails, the script should still fail loudly with a useful status code.
- Operators should still have an explicit opt-in mode to test with a local token when they intentionally need to validate device/client pairing.

---

## 4. Non-Goals

- Do not rotate `BRAIN_API_TOKEN` as part of this fix.
- Do not print, copy, or persist the production token locally.
- Do not change the semantics of `/api/health`.
- Do not weaken auth on `/api/health`.
- Do not add an unauthenticated production health endpoint in this release.
- Do not deploy any capture-quality changes as part of this work.
- Do not touch production data.

---

## 5. Design Decision

### Recommended Decision

Make the default deploy health check run over SSH on the production server:

1. SSH to `BRAIN_SSH_HOST`.
2. Source `/etc/brain/.env` under `sudo`.
3. Use `curl` from the server to call `${BRAIN_BASE_URL}/api/health`.
4. Send the header `Authorization: Bearer ${BRAIN_API_TOKEN}` only inside the remote shell.
5. Return only the HTTP status to the local deploy script.

This keeps the token where it belongs and verifies the public production URL after restart.

### Why Not Just Update Local `.env`

Updating local `.env` is a useful short-term workaround, but it is not the durable fix:

- It spreads a production secret to another machine/file.
- It can drift again after the next rotation.
- It makes deploy correctness depend on local secret hygiene instead of production runtime truth.
- It may hide the difference between "production is healthy" and "this laptop is paired correctly."

The local token should be treated as a client credential, not as the deploy system's source of truth.

## 6. Implementation Plan

## Phase 1 - Refactor Deploy Health Check

### Files

- `scripts/deploy.sh`

### Tasks

1. Rename the current `health_check` to `local_authenticated_health_check`.
2. Add a new default function named `remote_authenticated_health_check`.
3. Make `health_check` choose the token source using an environment variable:
   - default: `BRAIN_DEPLOY_HEALTH_TOKEN_SOURCE=remote`
   - optional: `BRAIN_DEPLOY_HEALTH_TOKEN_SOURCE=local`
4. Remove the default local-token requirement from `local_env_preflight`.
5. Keep remote-token preflight:
   - verify `/etc/brain/.env` exists
   - verify it contains `BRAIN_API_TOKEN=`
6. Fail fast if `BRAIN_DEPLOY_HEALTH_TOKEN_SOURCE=local` and local `BRAIN_API_TOKEN` is missing.
7. Keep the visible deploy step label as "Authenticated health check", but include the token source in the log line:
   - `Authenticated health check (remote token)`
   - `Authenticated health check (local token)`

### Proposed Shell Behavior

The remote check should conceptually do this, while carefully quoting values:

```bash
ssh "${SSH_HOST}" "sudo bash -lc '
  set -euo pipefail
  set -a
  source /etc/brain/.env
  set +a
  status=\$(curl --silent --show-error --output /dev/null --write-out \"%{http_code}\" \
    --header \"Authorization: Bearer \${BRAIN_API_TOKEN}\" \
    \"${BASE_URL}/api/health\")
  [[ \"\${status}\" == \"200\" ]] || {
    echo \"[deploy] health check returned \${status}, expected 200\" >&2
    exit 1
  }
'"
```

The actual implementation should avoid leaking the token through `set -x`, logs, or interpolated local output. The token must be expanded only by the remote shell after `/etc/brain/.env` is sourced.

### Acceptance Criteria

- Running deploy without a valid local `BRAIN_API_TOKEN` can still pass the authenticated health check when production is healthy.
- Running deploy with a broken production token still fails at health check.
- The token is not printed in terminal output.
- Existing `BRAIN_BASE_URL`, `BRAIN_SSH_HOST`, and `BRAIN_REMOTE_DIR` overrides still work.

## Phase 2 - Add a Safe Local Token Mode

### Files

- `scripts/deploy.sh`
- Optional documentation file if a deploy runbook exists.

### Tasks

1. Keep local-token health check support for intentional client-pairing validation.
2. Document the explicit opt-in:

```bash
BRAIN_DEPLOY_HEALTH_TOKEN_SOURCE=local bash scripts/deploy.sh
```

3. In local mode, preserve the current behavior:
   - load `.env`
   - require `BRAIN_API_TOKEN`
   - call `${BASE_URL}/api/health` from the laptop
4. If local mode returns `401`, make the message clearer:
   - "health check returned 401 with local token; production may be healthy but this local token is stale"

### Acceptance Criteria

- Default deploy validates production health using production token.
- Local mode remains available and intentionally strict.
- A local stale-token failure is correctly described as a client-token mismatch, not necessarily a failed deployment.

## Phase 3 - Add Lightweight Script Validation

### Files

- `scripts/deploy.sh`
- Optional new test/helper:
  - `scripts/test-deploy-health-check.mjs`, or
  - `scripts/test-deploy-health-check.sh`

### Preferred Testing Approach

Keep testing lightweight. This script is operational shell, so the highest-value checks are:

1. Syntax check:

```bash
bash -n scripts/deploy.sh
```

2. Static safety inspection:
   - no `echo "$BRAIN_API_TOKEN"`
   - no command that writes the remote token into local `.env`
   - no `set -x`

3. Mocked function test, if easy:
   - run `deploy.sh` functions in a controlled environment with fake `ssh` and `curl`
   - assert `remote` is default
   - assert `local` requires local token
   - assert a remote `401` exits non-zero

### Acceptance Criteria

- `bash -n scripts/deploy.sh` passes.
- The script defaults to remote-token mode.
- Test or manual verification proves no token is printed.

## Phase 4 - Production Verification

### Pre-Deploy Checklist

1. Confirm no unrelated changes are staged.
2. Confirm the deploy script diff only touches deployment hygiene.
3. Confirm production currently has `/etc/brain/.env` with `BRAIN_API_TOKEN=`.
4. Confirm a fresh backup exists if a full deploy is being run.

### Validation Commands

Run the normal validation gates:

```bash
npm run typecheck
npm run lint
npm test
bash -n scripts/deploy.sh
```

Then run the deploy:

```bash
BRAIN_AI_PROVIDER_WARN_ONLY=1 bash scripts/deploy.sh
```

### Expected Output

- The script reaches the authenticated health check.
- The health check reports success using the remote token.
- The script continues to the remote AI provider check.
- Telegram webhook reachability returns `401` without secret, as expected.
- The script finishes with:

```text
[deploy] Release candidate deployed and health-checked.
```

### Post-Deploy Checks

1. Production service is active.
2. Authenticated health check returns `200`.
3. Telegram webhook unauthenticated check returns `401`.
4. No token value appears in local terminal output.
5. No local `.env` token change was made by the deploy.

## 7. Rollback Plan

This change only affects `scripts/deploy.sh`.

If the new remote-token health check fails unexpectedly:

1. Do not rotate any tokens.
2. Check production directly from the server using `/etc/brain/.env`.
3. If production is healthy, run one deploy with explicit local mode only if needed:

```bash
BRAIN_DEPLOY_HEALTH_TOKEN_SOURCE=local bash scripts/deploy.sh
```

4. Revert only the deploy-script change if remote quoting or SSH behavior is broken.

Production app behavior should not require rollback because the implementation does not touch runtime application code.

## 8. Risks And Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Bad shell quoting expands the token locally | Secret leak risk | Review command quoting carefully; avoid local interpolation of `${BRAIN_API_TOKEN}`; verify logs before commit |
| Remote curl validates from server, not laptop | It may miss a laptop-specific network issue | That is acceptable for deploy health; laptop/client pairing remains covered by optional local mode |
| `/etc/brain/.env` is unreadable under sudo | Deploy fails before false success | Keep `remote_env_preflight`; produce clear error |
| Public URL hairpin from server behaves differently | Could fail despite service health | If this happens, add a second check: remote localhost service check plus laptop unauthenticated reachability |
| Operator expects local token deploy behavior | Confusion | Add explicit log and env var documentation |

## 9. Definition Of Done

This work is complete when:

- `scripts/deploy.sh` defaults to server-token authenticated health checks.
- Local token mode remains available only by explicit opt-in.
- The deploy script no longer fails only because the laptop has a stale token.
- Validation passes:
  - `npm run typecheck`
  - `npm run lint`
  - `npm test`
  - `bash -n scripts/deploy.sh`
- One production deploy finishes cleanly end-to-end.
- No production token is printed or copied locally.
- The final commit is scoped to deployment hygiene.

## 11. Recommendation

Execute the script change rather than manually syncing the local token. Manual token syncing is a bandage; using the production token in-place is the right operational shape. It keeps production secrets on production, makes deploys repeatable, and avoids another false failed deploy after the next token rotation.
