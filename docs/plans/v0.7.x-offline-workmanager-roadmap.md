**Version:** roadmap v1.0
**Date:** 2026-05-13
**Owner:** Arun (Lane L when work begins)
**Source:** `docs/plans/v0.6.x-offline-mode-apk.md` v3 §4.1 + §4.10
**Status:** ROADMAP ONLY — not scheduled. Promoted from "vague v0.7.x mention" to "concrete design sketch" per plan v3 A-1 fix.

---

## §1. Why this exists

The v0.6.x offline mode (OFFLINE-1..10, shipped on `lane-l/feature-work`) commits to **Option A**: a pure WebView + IndexedDB outbox + JS retry orchestrator. This delivers:

- Reliable enqueue at share time
- Reliable retry on app foreground / network change / 30s tick
- Survives app force-kill (queue persists in IDB)

It does **not** deliver the "subway story" — automatic background sync within 30s of signal returning while the phone is in the user's pocket and Brain is closed. Empirical evidence (`adb logcat`, 2026-05-13) shows the Brain WebView gets frozen by Android every ~30s when not foregrounded; JS timers don't fire reliably during freeze.

v0.7.x WorkManager (Option C) closes that gap with a native Kotlin worker that reads the same IDB outbox and POSTs from native code while the app is closed.

This document exists so cross-references in v0.6.x plan §2/§3.1/§12/§4.10 land on a real design, not vague-future words.

---

## §2. What stays from v0.6.x (zero-migration goal)

The v0.6.x design was deliberately schema-stable so the v0.7.x bridge can drop in without re-encoding existing queue entries. What v0.7.x reuses unchanged:

| v0.6.x asset | Reused in v0.7.x | Notes |
|---|---|---|
| Outbox IDB schema (`brain-outbox` DB, `outbox` store, three indexes) | Yes | Native worker reads the same store via the WebView's Chromium IDB on disk |
| Discriminated-union OutboxEntry types | Yes | `kind: 'url' \| 'note' \| 'pdf'` with the same payload shapes |
| `content_hash` dedup keys (worker SHA256 + URL normalization) | Yes | Already computed at enqueue; native worker just reads them |
| Backoff math (`baseDelayMs`, exponential + jitter, 1-hour cap) | Yes — re-implemented in Kotlin | Same constants; preserves cross-tier consistency |
| Classifier table (5xx → transient, 422+code → version_mismatch, etc.) | Yes — re-implemented in Kotlin | Same disposition shape |
| `X-Brain-Client-Api` header + 422 contract | Yes | Native worker sends the same header |
| `/inbox` UI + nav-bar badge | Yes | Reads the same outbox the worker writes to |
| Stuck-state notifications (plan §5.6) | Replaced | WorkManager has its own foreground-service notification model; the existing in-WebView notification path becomes redundant |

## §3. What changes / is added

### §3.1 Native Kotlin worker

**Class:** `BrainOutboxWorker extends androidx.work.CoroutineWorker`
**Module:** `android/app/src/main/java/com/arunprakash/brain/outbox/BrainOutboxWorker.kt`

```kotlin
class BrainOutboxWorker(
  ctx: Context,
  params: WorkerParameters,
) : CoroutineWorker(ctx, params) {

  override suspend fun doWork(): Result {
    // 1. Open the WebView's IndexedDB read-write — see §3.4 storage bridge
    // 2. Read all status='queued' rows whose next_retry_at <= now
    // 3. For each: build the request per kind, OkHttp POST, classify,
    //    update the row's status + next_retry_at + attempts back to IDB
    // 4. For PDFs: read bytes from app-private filesystem via the same
    //    path the WebView wrote them to (Directory.Data prefix)
    // 5. On 2xx for PDFs: delete the filesystem blob (parity with v0.6.x)
    // 6. Result.success() if no rows remain queued; Result.retry() to
    //    reschedule earlier than the periodic interval if rows remain
  }
}
```

### §3.2 Constraints + scheduling

```kotlin
val periodicWork = PeriodicWorkRequestBuilder<BrainOutboxWorker>(
  15, TimeUnit.MINUTES, // Android-enforced minimum
).setConstraints(
  Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED)
    .setRequiresBatteryNotLow(true)
    .build(),
).build()

WorkManager.getInstance(ctx).enqueueUniquePeriodicWork(
  "brain-outbox-drain",
  ExistingPeriodicWorkPolicy.KEEP,
  periodicWork,
)
```

Plus `OneTimeWorkRequest` triggers on:
- `BOOT_COMPLETED` broadcast (drain anything queued before phone reboot)
- `CONNECTIVITY_CHANGE` broadcast (drain immediately when network returns)
- `MyFirebaseMessagingService` push wake (optional — out of v0.7.x scope)

### §3.3 Manifest + permissions

```xml
<!-- AndroidManifest.xml additions -->
<uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED" />
<uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" />
<uses-permission android:name="android.permission.WAKE_LOCK" />
<!-- Android 14+: foreground service for long-running work -->
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_DATA_SYNC" />
```

### §3.4 Storage bridge — the design tension

Two architectures are viable; pick at v0.7.x kickoff:

**Option C-1 — Direct IDB read from native.** Chromium WebView stores IndexedDB at a known path (`/data/data/com.arunprakash.brain/app_webview/Default/IndexedDB/`). The native worker reads the LevelDB files directly. **Pro:** zero round-trip, single source of truth. **Con:** brittle vs Chromium upgrades; LevelDB format isn't a documented contract; concurrent access vs the WebView is undefined.

**Option C-2 — Capacitor plugin bridge.** A custom Capacitor plugin exposes the outbox via a Kotlin service the worker can call. **Pro:** the WebView is the source of truth; bridge talks via documented IPC. **Con:** the worker must wake a JS context to read the queue; defeats some of the "while app is closed" benefit; the WebView would need a no-UI startup path.

**Option C-3 — Mirror to a SQLite store.** Write queue mutations to a parallel SQLite database that both the WebView (via a Capacitor SQLite plugin) and the native worker can read. **Pro:** clean separation; SQLite contract is stable. **Con:** doubles every write at enqueue time; keeping IDB and SQLite in sync is itself a problem.

**Provisional pick:** Option C-3 if the v0.6.x outbox grows beyond a few hundred rows in steady state; C-1 if it stays small (<100 rows) and we're willing to accept Chromium-version coupling. C-2 is rejected because it forfeits the closed-app advantage.

**Decision deferred to v0.7.x kickoff** when we have real OFFLINE-PRE measurements + 1–2 weeks of v0.6.x outbox population data.

## §4. PDF handling post-pivot

If OFFLINE-PRE measurement (commit `a5aeb38`, run on Pixel 7 Pro post-cutover) reveals WebView quota < 100 MB and v3 §0.1 Q1 forces PDF out of the v0.6.x MVP, v0.7.x can write PDFs to filesystem from native code without the WebView quota constraint at all. The native worker:

1. On share, foreground intent still hands bytes to the WebView (Capacitor share-target stays unchanged)
2. WebView writes to Directory.Data via the same `pdf-storage.ts` API
3. Native worker reads from the same Directory.Data path
4. POST + delete-on-2xx as today

The WebView's IDB quota only constrains the metadata row (~1 KB), which is irrelevant. So PDF support can return in v0.7.x even on a small-quota device.

## §5. Notifications

WorkManager's foreground-service variant requires a sticky notification while work is running. This collides with the v0.6.x "0 → ≥1 stuck transition" notification UX.

**Plan:** at v0.7.x cutover, replace the WebView-side `LocalNotifications` plugin usage with:
- Native foreground-service notification ("Brain syncing…") shown during active worker runs, dismissed at completion
- Native heads-up notification on any stuck-row transition, channel separate from the foreground service so the user can mute one without the other

The user-facing surface stays the same: tap → /inbox.

## §6. Testing strategy

WorkManager testing is harder than the v0.6.x pure-JS suite. The plan:

- **Unit tests:** `BrainOutboxWorkerTest` using `androidx.work:work-testing`'s `TestWorkerBuilder`. Stub IDB read and OkHttp client. Cover the same disposition table as v0.6.x `classify.test.ts` to confirm cross-tier consistency.
- **Instrumentation tests:** Espresso + a real device; queue 5 rows, force-kill the app, set the device to airplane mode then back, assert the queue drains within one periodic cycle.
- **Backbox parity test:** run the same set of OutboxEntry inputs through both v0.6.x JS and v0.7.x Kotlin; assert byte-for-byte equality of the outbound HTTP body. Catches regressions in the Kotlin re-implementation.

## §7. v0.7.x scope ordering

If/when v0.7.x kicks off:

1. **Spike: storage bridge** — pick C-1 / C-2 / C-3 with a 2-day spike at the start, before any productionization.
2. **Re-implement classifier + backoff in Kotlin** — pure-logic; high test coverage; mirrors v0.6.x byte-for-byte.
3. **BrainOutboxWorker scaffolding** — schedule + constraints + WorkInfo observer.
4. **PDF path** (if it was scope-cut from v0.6.x).
5. **Foreground-service notification UX** — replace the v0.6.x LocalNotifications path.
6. **Espresso instrumentation tests** — the canary scenarios from plan v3 §9.1.
7. **Shadow ship** — both v0.6.x and v0.7.x active simultaneously for 1 week; compare drain success rates; flip the v0.6.x JS retry interval to 5 min (instead of 30s) to reduce duplication, leaving only the BOOT/CONNECTIVITY one-shots in JS.
8. **v0.7.x cutover** — disable the v0.6.x JS retry orchestrator entirely (it stays in code as a fallback for any device where WorkManager is suspended by aggressive battery optimization).

## §8. Open questions for v0.7.x kickoff

1. Storage bridge choice (§3.4) — C-1 vs C-3 depends on outbox size data we don't have yet
2. Push-wake (`MyFirebaseMessagingService`) — out of scope for v0.7.x base, but worth one paragraph in v0.7.x v1 plan: would allow "instant sync on user-driven trigger" without polling
3. Battery-optimization handling — Samsung / Xiaomi / OnePlus aggressively kill background work; should we surface a one-time "exclude Brain from battery optimization" prompt?
4. Concurrent access — if WebView and worker both want to mutate the same IDB row, who wins? The current Option A has no contention because the worker doesn't exist; Option C-1 needs an explicit lock
5. Doze-mode behavior — `setRequiresBatteryNotLow(true)` already biases against Doze; should we also `setRequiresDeviceIdle(false)` to opt INTO Doze runs?

---

## §9. What this roadmap does NOT promise

- A timeline. v0.7.x is unscheduled; this doc exists so the v0.6.x plan's cross-references are credible.
- Backwards compatibility for the v0.6.x JS retry path indefinitely. After v0.7.x ships, the JS path becomes the OEM-aggressive-battery-killer fallback only.
- iOS support. Brain has no iOS target.
- Background drain on app uninstall + reinstall. The outbox IDB is wiped on uninstall.
