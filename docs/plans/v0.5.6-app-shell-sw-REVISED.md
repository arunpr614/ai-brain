## §0. Executive summary (read first, even if skimming)

**Where we are:**

- v0.5.6 SHELL-1..6 shipped: SW source + registration + APK build + tests + docs.
- v0.5.6 SHELL-7 shipped: aggressive `clients.claim()` + `controllerchange`-triggered reload.
- Device test 2026-05-14 (Pixel 7 Pro, post-data-wipe, fresh install): **still ERR_INTERNET_DISCONNECTED on offline cold-launch**.
- DevTools snapshot from earlier: `brain-shell-v1` cache populated (SW *did* run install) but Service Workers panel for `https://brain.arunp.in` empty (no active SW for the origin).

**What this means:**

The SW is installing and caching successfully, but on the registering page's lifecycle the SW is not becoming the controller in time for the *next cold-launch*. Two more SHELL-7-style code changes (claim+reload) didn't fix it. Continuing to guess at the root cause with code edits is the wrong move.

**Decision:**

1. **Stop guessing.** Use the CDP automation work (already partly done — `chrome-remote-interface` is in `devDependencies`) to programmatically inspect the WebView state. Confirm exactly: did the SW register? what is its `status` and `runningStatus`? what URL is the page on? does the WebView origin match `https://brain.arunp.in`?
2. **Act on the self-critique.** §6 of the critique listed 15 remediation items; items #1, #2, #4, #5, #6 are blockers for trusting the inspector. Fix those before running.
3. **Run the inspector script.** Capture real evidence. Decide the next code change based on data, not hypothesis.
4. **Then iterate.** The fix may be: registration timing, scope path, Capacitor WebView SW restriction (real possibility), or something else. We can't tell until we see the live state.

This is a **diagnose-first** plan. No more SHELL-N code commits until we have CDP evidence.

**Goal:** apply self-critique items #1, #2, #7, #8, #9 (script bugs + edge cases) before running so we don't waste a run on an avoidable hang.

**File:** NEW `scripts/inspect-webview.mjs` (~180 lines after hardening)

**Changes from §F draft:**

1. Add `adb` preflight: try `adb version`, exit with friendly error if not in PATH.
2. Wrap the entire body in a 30s deadline. If exceeded, dump partial state and exit non-zero.
3. Handle multi-PID `pidof` output (split + take first).
4. Filter `targets.find` by `t.url.startsWith(APP_ORIGIN)` not just `t.type === 'page'`.
5. Fix the runningStatus assertion to allow all 4 valid values, not just 2.
6. Wrap `client.close()` in try/catch so it doesn't mask the real error.
7. Log the page URL prominently — if it's `/unlock` or `/setup-apk`, the SW will not have intercepted page navigations yet (PIN unlock state warning).

**Tests:** none. The script is the test.

**Commit:** `chore(scripts): hardened CDP inspector for WebView debug (DIAG-1)`

### §3.2 DIAG-2 — Run the inspector and capture real evidence

**Goal:** answer these questions empirically, not by guessing:

| Question | How DIAG-2 answers it |
|---|---|
| Is the WebView reachable via CDP at all? | `CDP.List({port: 9222})` returns non-empty array |
| What URL is the WebView on right now? | `pageTarget.url` |
| Is there a service worker target? | `targets.filter(t => t.type === 'service_worker')` |
| What is the SW `status` and `runningStatus`? | `ServiceWorker.workerVersionUpdated` event |
| What caches exist for `https://brain.arunp.in`? | `CacheStorage.requestCacheNames` |
| Are the expected URLs in `brain-shell-v1`? | `CacheStorage.requestEntries` |

**Procedure:**

1. User opens Brain on Pixel (online, fresh from reinstall).
2. Wait for library to render + automatic reload (the SHELL-7 `controllerchange` reload, if it fires).
3. Run: `node scripts/inspect-webview.mjs > /tmp/inspect-output.txt 2>&1`
4. Capture `/tmp/inspect-output.txt`.
5. Save it to `docs/research/inspect-webview-output-2026-05-14.md` for traceability.

**Decision matrix from output:**

| Scenario | What inspector shows | Diagnosis | Next fix |
|---|---|---|---|
| A | SW status `activated`, runningStatus `running`, all 3 caches populated, `brain-shell-v1` has 6 entries | The SW *is* active. The cold-launch failure is something else (cache name mismatch? scope?). Inspect HTTP request flow. | Investigate fetch interception |
| B | SW status `redundant` or no SW versions reported | Registration failed silently; cache exists from a prior SW that got superseded | Registration timing / scope fix |
| C | Only `brain-shell-v1` cache exists; no `brain-static-v1` or `brain-pages-v1` | SW activated once, did install precache, then never intercepted any other fetches | SW fetch handler not firing → scope or origin issue |
| D | CDP can't enumerate any service_worker target, but page works | The Capacitor WebView is intercepting the SW registration but not exposing it to CDP | This may be a Capacitor WebView limitation (real possibility) |
| E | Some other state | Document and decide |

**Commit:** `docs(research): captured WebView inspector output (DIAG-2)` (just commits the captured output to the research doc)

### §3.3 DIAG-3 — Targeted fix based on DIAG-2 evidence

**Conditional plan; depends on DIAG-2 outcome.**

**If A (SW active but offline still fails):**
- Cache hit path is broken. Test: from `chrome://inspect` Console, run `caches.match("/")` — does it return a Response? If no, the cache key has a mismatch (e.g., trailing slash, query string). Fix the SW.
- Likely fix: change SW fetch handler to normalize URLs before cache.match.

**If B (SW redundant or absent):**
- Registration is failing. Inspect with `await navigator.serviceWorker.getRegistrations()` from Chrome DevTools Console.
- Likely fix: move registration earlier (inline `<script>` in `<head>` before any fetch fires), or fix scope path.

**If C (only shell-v1 cached):**
- SW activated but `fetch` handler never fires for new requests after install.
- Likely fix: SW scope is set wrong, or the `androidScheme: "https"` + `server.url` combination breaks SW interception in Capacitor.
- Worst-case discovery: Capacitor WebView fundamentally doesn't support SW for `server.url` apps. This would force Path B (`next export`) or a redesign.

**If D (Capacitor doesn't expose SW to CDP):**
- Architectural — Path C (SW for app shell) is not viable in Capacitor's `server.url` mode.
- Pivot to Path B (next export) for v0.5.7 OR to a Capacitor-native offline cache (e.g., precache via Capacitor Filesystem on first install + serve via custom WebView interceptor).

**Commit:** depends on path. Each fix gets its own atomic SHELL-N commit.

## §4. Critical decision: how long do we iterate before pivoting?

The original plan §5 listed "Capacitor WebView doesn't respect SW registrations" as the highest-severity risk. That risk has now partially activated. The threshold for pivoting:

| Iteration | What we know | Decision |
|---|---|---|
| Now (post-SHELL-7) | SW caches but doesn't intercept on next cold-launch | Run DIAG-2 |
| After DIAG-2 | We know the SW state empirically | DIAG-3 fix attempt |
| After 1 more code change | Still failing | Strongly consider Path B or hybrid |
| After 2 more code changes | Still failing | Pivot — Path C is not viable in Capacitor's `server.url` mode |

**Hard stop:** if 3 total code-change attempts in v0.5.6 don't ship a working offline cold-launch, declare Path C infeasible and revert v0.5.6 to docs-only (the SW source stays in `public/` but doesn't ship as a registration); start v0.5.7 with Path B (next export). The 12 OFFLINE-* commits stay merged because they're correct *online* — they only fail offline due to the missing app shell.
