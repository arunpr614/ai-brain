## 1. Findings the Stage 4 review missed

### C-1 — The body-header fix (H-3) silently changes enrichment for existing items too

**Severity:** HIGH

The plan's §5.2.1 prepends `title\n\nchannel\n\nduration\n\nsource\n\n---\n\n<transcript>` to YouTube item bodies so the enrichment's 12,000-char cap still sees the metadata. That's fine for new YouTube items. But the change is implemented inside `extractYoutubeVideo`, and the body is written to the `items.body` column. **The chunker and vector embedder then chunk the body including the header.** Which means:

- The first chunk of every YouTube item will contain the header. The chunker might treat the header as its own small chunk (because the `---` separator and blank lines trigger paragraph boundaries), producing a ~5-line chunk of pure metadata.
- That metadata chunk gets its own embedding. Semantic retrieval over "Channel: Lenny's Podcast" or "Source: https://..." will now match against questions about those literal phrases.
- When Ask cites a chunk, the citation may be the header chunk — a useless citation for the user ("see the source URL for this answer").

**The plan doesn't address this.** It asserts §5.5 is "zero changes" to the retrieval pipeline, but the content flowing through it is now structurally different.

**Recommended fix:** In `extractYoutubeVideo`, write the full `body` with header, but pass the **transcript-only** portion to a future enrichment-specific field — OR accept the limitation and add a test asserting the first chunk of a YouTube item contains the header (at least then we know what behavior we're shipping).

Simpler alternative: **reverse the H-3 fix.** Append the header **at the end** of the body, not the start. Enrichment's `slice(0, 12000)` starts at 0, so an end-of-body header is invisible to enrichment — which was H-3's whole point. Then the chunker processes the transcript first; header becomes a trailing chunk that naturally has low relevance to most queries. This keeps the fix while avoiding the polluted-first-chunk problem.

Actually — re-reading H-3, the simplest fix is to pass enrichment a different input than retrieval. The enrichment pipeline already does this for `summary`: it sends `title + summary + body` to the embedder (`enrichment-worker.ts:67-71`). The body-column stays the transcript; enrichment gets the synthesized text via the **`title` column**, which is already passed as `Original title` in the enrichment user prompt. **Set `title` to something like `"<video title> [<channel>, <duration>]"` and leave `body` as pure transcript.** Enrichment sees the context via title; retrieval stays clean.

This is the correct fix. The plan's v1.1 chose the wrong implementation of H-3.

**Effort to correct plan:** 15 min (rewrite §5.2.1).

### C-2 — `youtube-transcript` + self-driven InnerTube POST is a hidden contradiction

**Severity:** HIGH

Decision D-v0.5.1-1 (§13) locks in "self-contained InnerTube POST, using `youtube-transcript` only for its XML parser." The Stage 4 review's Dependencies section flagged the risk that `youtube-transcript` might not export its XML parser independently. My plan v1.1 absorbed this by adding an "export-surface audit" to T-YT-3.

But that audit has a binary outcome, and the plan doesn't branch:
- **If the parser IS independently exported:** fine, proceed as planned.
- **If the parser is NOT exported** (realistic — small libraries often inline their parsers): the plan says "either inline the XML parsing or call `fetchTranscript()`". But `fetchTranscript()` does its OWN InnerTube POST. So if I call `fetchTranscript()`, **I double-call InnerTube** — once to get title+duration, once to get captions.

That double-call problem isn't small:
- Two HTTP requests per YouTube capture — doubles the extraction latency
- Two chances to hit YouTube's rate limit
- Two client-context fingerprints — if YouTube starts correlating, we're more likely to get blocked

**The real question I dodged:** if `youtube-transcript` doesn't export its parser, is D-v0.5.1-1 still the right decision? Probably not — in that case, `youtube-transcript` + oEmbed (2 calls) is no worse than inline-InnerTube + `fetchTranscript` (2 calls), and oEmbed is simpler code. The plan should branch the decision.

**Recommended fix:** Make T-YT-3 a real gate, not just an audit. If the parser is not exported, D-v0.5.1-1 is rescinded and the plan falls back to `youtube-transcript@~1.3.1` direct call + oEmbed — exactly what the research spike originally suggested. Update the plan to say "T-YT-4 has two possible shapes; T-YT-3 selects which."

**Alternative (simpler):** Just copy the ~30 lines of XML parsing into `youtube.ts` and remove the dep entirely. Zero new deps. Zero external-library breakage exposure. The XML parser for `srv3`/`srv1` formats is genuinely small (~20-30 lines). This would be the cleanest path of all.

**Effort to correct plan:** 30 min to rewrite T-YT-3 + T-YT-4 with the branch, or 15 min to pivot to "inline the parser, drop the dep entirely."

### C-3 — Plan claims "zero extension/APK changes" but the user will see a YouTube-specific UX gap immediately

**Severity:** MEDIUM

SC-9 + the whole "zero extension/APK changes" narrative is technically true but misleading. After v0.5.1 ships:

- User shares a YouTube URL from Android → backend takes 3-8 seconds (InnerTube POST + XML fetch + embedding). The Capacitor share intent's "Saved to Brain" toast already fires immediately for URL capture; for YouTube there's a longer delay before the item is retrievable. If the user goes to Library and doesn't see the item within 2 seconds, they'll think it failed.
- The extension's "Save to Brain" popup shows "Saving..." → "Saved." The YouTube save will sit on "Saving..." for 5-10 seconds longer than URL captures. No user visual signal that something slower is happening; they'll think the extension hung.

The existing `ExtractedArticle` path typically returns in < 2 seconds (fetch + JSDOM + Readability). YouTube with an InnerTube POST + timedtext fetch + XML parse will be measurably slower. The plan's §7 NFRs say "p95 < 8 s" for YouTube latency — that's a real 4× regression vs URL capture from the user's perspective.

**What the plan misses:** there's no ACK-then-process pattern. The route returns 201 only after extraction + insert. A YouTube-aware client change would be valuable: extension popup shows "Fetching transcript…" when it detects a YouTube URL, different from "Saving…" for articles. That's a minor client tweak but SC-9 forbids it.

**Recommended fix:** Either (a) relax SC-9 to allow a 1-file extension copy change ("Fetching transcript..." state for YouTube URLs — detected by regex in the popup), or (b) acknowledge the latency UX gap as a v0.5.2 item and add a one-line warning to the extension README that YouTube saves take 5-10 seconds.

**Effort to correct plan:** 5 min (option b) or 30 min (option a during execution).

### C-4 — T-YT-9 fixture capture has no documented how-to; the plan assumes I'll figure it out

**Severity:** MEDIUM

The plan lists T-YT-9 as "Commit fixture files captured from a real video" with exit "Files present; `npm test` uses them." But **how** do I capture those fixtures? The InnerTube POST is a POST with a specific body shape. The timedtext is the response of an opaque `baseUrl`. If I don't document the capture procedure in T-YT-9, then when the fixtures rot (6-12 months from now), the next agent reading the plan won't know how to regenerate them.

This is not a bug that blocks execution, but it bakes in a maintenance hazard.

**Recommended fix:** Add to T-YT-9: "Capture procedure: run `node scripts/capture-youtube-fixture.mjs dQw4w9WgXcQ > __fixtures__/youtube-player-response.json` (script to be created in T-YT-9 itself)." Alternatively, document the curl commands in a comment at the top of the fixture JSON.

**Effort to correct plan:** 10 min.

### C-5 — Migration 007 creates a column the LLM enrichment doesn't use; the data model is already accreting

**Severity:** LOW

`duration_seconds` is a new nullable column on `items` used only for display in the YouTube item UI (which this plan doesn't actually build) and for the H-3 header (which goes into `body`, not its own column). **So the column is technically unused by this release.** The actual value flows via the body header to enrichment and via the body text to retrieval.

**Why this is a real concern and not pedantry:** the `items` table already has `total_pages` (PDF-only), `total_chars` (all types), `extraction_warning` (text-valued), `author` (often null). Every release that adds "one little nullable column" tilts the schema toward wide-sparse. In six releases we'll have 12 source-type-specific columns on `items` and no clean way to query them. This is the first of what will become many.

A cleaner model: `items_meta` as a JSON column. Pay the complexity cost once, never add another migration for a type-specific field.

**But this is scope creep for v0.5.1.** My critique: the plan should acknowledge this trajectory, not fix it now. Right now `duration_seconds` on `items` is fine — just document that the "add a column per source type" pattern needs a rethink before v0.6.0 starts adding more.

**Recommended fix:** Add to the "Out of scope / deferred" section: "Schema: wide-sparse column accretion on `items` — will need an `items_meta` JSON refactor before v0.6.0 if any new source_type needs a metadata field."

**Effort to correct plan:** 5 min.

## 2. Places where v1.1 absorbed a review finding incompletely

### A-1 — H-1 resolution is inconsistent between §5.3 and §6.4

`§5.3` route pseudocode correctly uses `shareDedupKey("url", url)`. But T-YT-6 in the task table still says the dedup test "saving `youtu.be/X` then `youtube.com/watch?v=X` returns `duplicate: true`" — **but the dedup check uses the canonical URL, so you'd never HIT the dedup branch on the canonical URL if it wasn't the first save**. The actual flow is `isDuplicateShare(shareDedupKey("url", canonical))` → if present in the 2-second window, return 200 duplicate. On a second save of the same URL, the `findItemByUrl(canonical)` branch returns duplicate (historical), not the in-memory window. The plan's SC-5 verification says "saving any variant a second time" — within or outside the dedup window isn't specified.

Impact: the new integration test added in T-YT-6 might fail intermittently if the 2-second window expires between the two saves. The test should force one case explicitly (write two URLs, ensure gap > 2s, assert historical-dedup path).

**Fix:** Clarify SC-5 + T-YT-6 test spec: "second save ≥ 3 seconds after first → `duplicate: true` via `findItemByUrl` (historical), NOT via `isDuplicateShare` window." One sentence, important.

### A-2 — M-1's `CapturedContent` shared type is mentioned in §5.3 but never specified

§5.3 says "`CapturedContent` is a shared base type (to be added at T-YT-4 in `src/lib/capture/types.ts` or inlined)." But `ExtractedArticle` (existing) has `excerpt` and `html_length` fields that `YoutubeExtracted` doesn't. For `ExtractedArticle` to satisfy `CapturedContent`, either:
- `CapturedContent` must NOT include `excerpt`/`html_length` (then `ExtractedArticle` is a wider type)
- Or `YoutubeExtracted` must include them (`excerpt: null, html_length: 0` fallback)

The plan doesn't pick. The review's M-1 raised this; my v1.1 punted with "at T-YT-4 in `src/lib/capture/types.ts` or inlined."

**Fix:** Define `CapturedContent` explicitly in §5.2:
```ts
export interface CapturedContent {
  title: string;
  body: string;
  author: string | null;
  source_url: string;
  extraction_warning: string | null;
  duration_seconds?: number | null;
}
```
Note: `ExtractedArticle` IS already structurally a valid `CapturedContent` (all 5 required fields). `YoutubeExtracted` becomes its own type declaration of `CapturedContent & { duration_seconds: number | null }`. Done — no field gymnastics needed.

## 3. Assumptions I didn't flag in the plan but should have

### U-1 — The plan assumes enrichment works on YouTube items at all

The enrichment pipeline runs on every new item (queue → LLM → category/summary/tags). The plan assumes YouTube items flow through this unchanged. But:

- The enrichment prompt classifies items into `CATEGORIES` (Newsletter, Blog Post, Podcast Episode, Tutorial, Case Study, Reference, Announcement, Data Report, Social Post, Forum Discussion, Video Page, Landing Page, General, Other). **There's no "YouTube Video" category.** The LLM will categorize as "Video Page" or "General" — probably "Video Page" for most.
- The enrichment asks for "5 verbatim quotes pulled verbatim from the article." Transcript lines with timestamps like `[0:00] Yeah so I was thinking` aren't great "quotes." The LLM will either include the timestamps in the quote (odd) or strip them (loses the anchor).
- The title-rewrite step will see the YouTube title which is usually already clean; it may "rewrite" the title in ways that lose the channel name.

**None of these break; they just produce subtly worse enrichment for YouTube items.** The plan should either note this as acceptable (enrichment is best-effort; user can edit) or add a small YouTube-specific enrichment prompt variant as a follow-up.

**Recommended fix:** Add to "Out of scope" section: "YouTube-specific enrichment prompt (video-friendly quotes, `Video Page` vs `YouTube` category distinction) — v0.5.2 if enrichment output proves unsatisfactory."

### U-2 — The plan assumes no session/bot-detection escalation from YouTube

The research spike's A3 assumption ("residential IP is safe") is well-reasoned but static. YouTube increasingly uses behavioral signals — burst of transcript fetches from one IP, or correlated Android-client-context-spoofing requests, can trigger progressive blocks.

If I capture 50 YouTube videos in one session (Lenny binge), the tunnel server fires 50 InnerTube POSTs from my Mac IP in a few minutes. That's a detectable pattern.

**Plan impact:** SC-6's "capture one long-form video, wait for embedding backfill" downplays the risk. The realistic mode of failure is "user captures 20 videos on day one, fine; day two they get mysterious `fetch_failed` errors because of escalating rate limits."

**Recommended fix:** Add a rate-limit concern to §8 Risks: "Bulk-capture escalation — YouTube escalates anti-bot signals after N requests/hour/IP. Mitigation: (1) no retries on `fetch_failed` in v0.5.1 (single shot), (2) document a 'capture rate' soft limit of ~20 videos/hour in the extension README, (3) expose the bulk-rate-limit retry behavior as a v0.5.2 item if it hits users in practice."

## 4. Weaknesses in the task breakdown

### TB-1 — T-YT-10 release gate is thin

Every prior release in this repo (v0.3.1, v0.4.0, v0.5.0) had a dedicated test commit for the smoke script. T-YT-10 bundles "version bump + tracker updates + RUNNING_LOG + tag + push" into one task. That's a lot of different concerns for one commit.

At minimum, the smoke commit (T-YT-8) and the release commit (T-YT-10) should be separate; the plan has them separate, but T-YT-10's exit criterion of "all smokes + tests green" is post-hoc — the plan should call out running `npm run smoke && npm test` as an explicit gate **before** version bump.

**Recommended fix:** Prepend to T-YT-10: "Precondition — run full gate (`npm run typecheck && npm run lint && npm run build && npm test && npm run smoke`). On red, stop; do not tag."

### TB-2 — The execution order diagram is wrong

§14 shows T-YT-9 → T-YT-4 → T-YT-5. But T-YT-4 needs fixtures to type-check against a real InnerTube response shape. In practice, T-YT-9 must complete **before** T-YT-4 can even begin (you can't write the parser for a shape you haven't seen). The diagram has this right. But it shows T-YT-5 after T-YT-4, which means the tests that require fixtures are separate from the extractor that requires fixtures. T-YT-4 + T-YT-5 could be one commit — writing the extractor alongside its tests is how the TDD-ish pattern normally works in this codebase.

**Recommended fix:** Merge T-YT-4 + T-YT-5 into one task (or at least one commit pair).

### TB-3 — No task for removing the `[No transcript available]` rendering edge case from Library

Plan says "item detail page already renders `body` as plain text. Timestamped lines render naturally." True — but the `[No transcript available for this video]` placeholder for no-captions Shorts is the only thing in the body. The item detail page will render this as a single-line gray card. No warning pill. The user can't distinguish "I captured this Short and it genuinely has no transcript" from "the save failed partially."

**Recommended fix:** Add a tiny Library display affordance: when `extraction_warning = "no_transcript"`, render a small "no transcript available" badge on the item card. One-file change in `src/components/library-item-card.tsx` or equivalent.

Actually: this violates SC-9 (if we consider the shared library UI "extension-adjacent"). It doesn't — SC-9 is about `extension/` and `android/` trees, not `src/components/`. Fine to add.

## 5. What the plan got right

Not everything is broken. Worth naming the strong points so the above critiques are properly weighted:

- **Scope is genuinely small.** One new file, one migration, one dep (or zero if we inline the parser per C-2 alternative). This is appropriate for a tactical release.
- **The server-side-only approach is correct.** No extension/APK rebuilds required, both clients get it for free — the reasoning in §1 is sound.
- **The 2-hour cap at 7,200 segments is defensible.** Covers 99% of real content; honest warning on truncation.
- **Rollback story is real.** Soft (git revert), hard (backup restore), library swap (isolated file) — all three are credible paths.
- **Test strategy is sane.** Fixture-mocked unit + opt-in real smoke + existing regression suite. This matches the project's pattern.
- **The dedup-before-extractor ordering is right.** Canonical URL → `findItemByUrl` → extractor branch. This is the cheapest valid dedup.
- **The plan absorbed the Stage 4 review findings.** H-1 answered cleanly, H-2 threaded through types correctly, H-3 addressed (but chose wrong implementation — see C-1).

But doing 55 minutes of plan rewrite NOW is cheaper than doing 2-3 hours of rework mid-execution.

**Recommend:** absorb items 1-3 into plan v1.2, then start T-YT-2.
