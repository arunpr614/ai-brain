## §0. The bug this plan fixes

**Discovered:** 2026-05-14 manual matrix run. Pixel 7 Pro, APK 0.5.5, airplane mode → Share URL → Brain.

**Symptom:** Brain shows the Android system error page:

```
Web page not available
The web page at https://brain.arunp.in/ could not be loaded because:
net::ERR_INTERNET_DISCONNECTED
```

The 12 OFFLINE-* commits ship correct code, but the code never runs because the WebView cannot fetch the bundle that contains it.

**Root cause:** `capacitor.config.ts` sets `server.url: "https://brain.arunp.in"` — the APK is a thin WebView. Cold-start with no network = no bundle = no `share-handler.tsx`, no outbox, no IndexedDB code path.

**Why this wasn't caught:**

- Plan v3 §3 (architecture) describes the "share enqueue happy path" as if `share-handler.tsx` is already loaded.
- Plan v3 has **zero references** to "service worker", "app shell", "cold start offline", or "WebView bundle caching" (verified: `grep service.worker docs/plans/v0.6.x-offline-mode-apk.md` → 0 hits).
- OFFLINE-PRE measured `serviceWorker: available` and `controller: false` — we recorded both correctly but never connected them to "we don't have one registered, so cold-start-offline = dead app."
- All 425 unit tests run in Node with the bundle pre-imported. The Bucket A scenarios in the manual matrix start with a Brain that's already loaded (from the prerequisite "PIN unlock + library loads" step).

This is Plan v3 critique-3 item #1, manifested at execution time. M5 §5.2 hypothesized "every plan should go through 2 critique cycles before execution" and predicted that matrix testing might surface a critique-3. It did, with the largest possible severity: **the entire offline-mode feature is unreachable on the device.**

## §2. Goal

Make the OFFLINE-* code reachable on the Pixel when the WebView cold-starts with no internet. Specifically:

| State | Today | After v0.5.6 |
|---|---|---|
| Cold-start online (first ever) | Loads, hydrates, share works | Same — also installs SW transparently |
| Cold-start online (second+ time) | Loads from server | Same — SW updates cache; serves on next offline run |
| Cold-start **offline**, never online before | ERR_INTERNET_DISCONNECTED | Same — cannot conjure a bundle from nothing; show `offline.html` "Open Brain online once to enable offline mode" |
| Cold-start **offline**, was online before | **ERR_INTERNET_DISCONNECTED** ❌ | **App-shell loads from SW cache; share-handler runs; outbox enqueues** ✅ |
| Warm app + lose network mid-share | Outbox enqueues (correct today) | Same |

Non-goals:

- No offline reading of captured items (library view stays online-required per v3 §12).
- No offline /ask (always server-required).
- No background sync (still v0.7.x WorkManager territory).
- No PWA install (this is APK-side only; the SW also helps the regular web origin but that's incidental).

## §3. Architecture choice — picked: app-shell SW (Path C from session discussion)

| Path | Effort | Picked? | Why / why not |
|---|---|---|---|
| **A. Service Worker app-shell cache** | Medium (~1 day) | NO standalone — folded into C | The SW alone caches but doesn't change Capacitor wiring |
| **B. `next export` static export** | Large (~1 week) | NO | Brain uses server components heavily (`/items/[id]`, `/inbox`, `/api/*`). Dynamic routes require server. Conversion is invasive. |
| **C. SW for app shell + keep `server.url`** | Medium (~1 day) | **YES** | Smallest delta. Existing `webDir: "public"` + `offline.html` Capacitor fallback already exists as a hint. We register a SW from inside the app on first load; SW caches the shell + `_next/static/*`; on offline cold-start, SW returns cached shell; React hydrates from cached chunks; share-handler runs against the cached app. |

### §3.1 Why not @serwist/next or @ducanh2912/next-pwa?

`@serwist/next` (the modern fork of next-pwa) is the obvious "Next.js + SW" library. **Rejected — but only as a build-time integration.** Reasons:

1. **Schema ownership over the cached resources.** Same logic as v3 §3 rejecting Workbox for outbox: we want explicit control over which routes are cached, what stale-while-revalidate strategy applies, and how the app behaves on cache miss. Library magic that "just works" tends to surprise on offline edges.
2. **Build-config blast radius.** Adding `withSerwist` to `next.config.ts` couples Next.js builds to a library that injects a SW + manifest at build time. We don't want a build-time coupling for a 100-line SW.
3. **Capacitor `webDir: "public"` already gives us a static fallback path.** We can put the SW source at `public/sw.js` and register it from a tiny client-side initializer. Zero build-time integration.

We **borrow** Workbox runtime APIs as a model (cache-first for static, network-first for HTML, fallback to cached shell) but **roll the SW by hand** in ~150 lines.

### §3.2 The cache layers

```
┌─────────────────────────────────────────────────────────────┐
│ APK WebView                                                 │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  Service Worker (sw.js, served from same origin)    │  │
│   │                                                     │  │
│   │  Cache 1: shell-v1                                  │  │
│   │    /              (Next.js root HTML)               │  │
│   │    /share-target  (share intent landing route)      │  │
│   │    /inbox         (where queued items live)         │  │
│   │    /items/:id     ❌ NOT cached (dynamic)           │  │
│   │                                                     │  │
│   │  Cache 2: static-v1                                 │  │
│   │    /_next/static/**  (chunks, css)                  │  │
│   │    /favicon.ico                                     │  │
│   │    /file.svg, /globe.svg, etc.                      │  │
│   │                                                     │  │
│   │  NEVER cached:                                      │  │
│   │    /api/**          (always network)                │  │
│   │    /_next/image/**  (heavy; on-demand)              │  │
│   │    POST anything    (always network or fail)        │  │
│   └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### §3.3 Caching strategy per route family

| Route pattern | Strategy | On offline cache miss |
|---|---|---|
| `/`, `/inbox`, `/share-target` | **stale-while-revalidate** | Show `offline.html` |
| `/_next/static/**` | **cache-first** | network (will fail → SW returns generic 504) |
| `/_next/image/**` | network-only | natural broken image |
| `/api/**` GET | network-only | natural error |
| `/api/**` POST | network-only | natural error → outbox transport classifies it |

**Library offline reading discovery (2026-05-14):** `/` and `/items/[id]` are Next.js **server components** — they call `listItems()` / `getItem()` directly against SQLite at request time. There are no `/api/items` GET endpoints. So "library offline reading" is achieved by caching the **rendered HTML pages** (`/`, `/items/[id]`). Trade-off: the cached library is a snapshot from the last online visit; new captures from other devices won't appear until next online refresh. Acceptable for v0.5.6 — strictly better than the current ERR_INTERNET_DISCONNECTED.

**HTML page caching strategy:**

| Route | Strategy | Cache name |
|---|---|---|
| `/` (library) | stale-while-revalidate | `pages-v1` |
| `/items/:id` (detail) | stale-while-revalidate; cache populated as user visits | `pages-v1` |
| `/inbox` | stale-while-revalidate | `pages-v1` |
| `/share-target` | precached | `shell-v1` |
| `/ask`, `/ask/*` | network-only (server-required) | not cached |
| `/capture` | precached shell | `shell-v1` |
| `/setup-apk`, `/unlock` | network-only | not cached |
| Anything else (HTML) | network-first → cache-fallback → `offline.html` | shows offline.html (user already knows what that means) |

**Rationale:**

- **stale-while-revalidate** for `/`: the user gets the last-cached HTML instantly, and the SW updates cache in the background. Trade-off: first paint after a UI deploy may be 1 cycle stale. For Lane L iteration this is acceptable.
- **cache-first** for `/_next/static/**`: chunk filenames are content-hashed by Next.js, so `cache-first` is safe — new build = new filename = new cache entry. Old chunks linger in cache until eviction.
- **network-only for `/api/**`**: the outbox already has the right semantics (queue on transient, mark stuck on permanent). The SW must not interfere — it would add a layer of confusion on top of the outbox classifier.

### §3.4 SW lifecycle

1. **First online cold-start (clean install or fresh data):** Brain loads from server. A tiny initializer in the root layout calls `navigator.serviceWorker.register("/sw.js")`. SW installs (event `install`) → opens both caches → calls `cache.addAll(SHELL_URLS)` → activates.
2. **Subsequent online starts:** Brain loads from network; SW intercepts and serves from cache + revalidates in background. Cache stays warm.
3. **Offline cold-start (was online before):** WebView issues GET `/`. SW intercepts. Cache hit → returns cached HTML. HTML references `_next/static/chunks/main-<hash>.js` which the SW also has cached. React hydrates. `share-handler.tsx` (which lives in those chunks) runs. Share intent is captured. Outbox enqueues. Toast appears.
4. **Offline cold-start (NEVER online before):** WebView issues GET `/`. SW intercepts. Cache empty → SW falls through to network → fails → SW returns the bundled `/offline.html`. User sees "Open Brain online once first" message.
5. **Update:** when a new SW is registered (different `sw.js` hash), the new SW installs in parallel, activates on next cold-start, deletes old cache versions in `activate` event.

### §4.2 SHELL-2 — Register SW from the app

**Files:**

- NEW `src/lib/client/register-sw.ts` (~25 lines)
- EDIT `src/app/layout.tsx` — add `<RegisterSW />` client component near `<ShareHandler />`. Must be client-side only.

**Why a separate client component, not inline `<script>`:** Next.js 16 server components cannot register SWs; the registration must run after hydration. A `"use client"` component with `useEffect(() => navigator.serviceWorker.register("/sw.js"), [])` is the canonical pattern.

**Guard against non-APK contexts:** the SW only registers in `"https"` origins (skips Capacitor's old `androidScheme: "http"` if anyone reverts) AND in environments where `navigator.serviceWorker` is defined. SSR safe (no-ops on the server).

**Tests:** unit test for `register-sw.ts` mocking `navigator.serviceWorker` — verify it doesn't throw when SW is unavailable, registers correctly when it is. ~5 tests.

**Commit:** `feat(sw): register service worker from root layout (SHELL-2)`

## §5. Risks + mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| **Capacitor WebView doesn't respect SW registrations** | HIGH if true; would invalidate the entire approach | OFFLINE-PRE on the APK (see §6 below) probes `serviceWorker.controller` after registration. If still null after 2s wait + reload, this approach fails and we go to Path B (static export). The current Chrome browser probe already shows `serviceWorker: available`; APK should match since it's the same Chromium engine. |
| **SW caches stale shell across UI deploys** | MEDIUM | stale-while-revalidate strategy: user sees last cache, SW fetches new shell in background; next reload uses fresh. Acceptable for Lane L iteration. Mitigation if a critical bug ships: bump cache version (`brain-shell-v2`) → activate event purges old. |
| **`/api/**` requests fail loudly without outbox** | LOW | outbox already classifies network-class errors as transient; behavior unchanged from v0.5.5. |
| **First-ever offline launch shows offline.html, user thinks it's broken** | LOW | offline.html v0.5.6 update adds clear "open Brain online first" copy. |
| **SW + Capacitor share-target plugin race** | MEDIUM | `share-handler.tsx` already has init lifecycle gating + `pendingShareEvent` buffer (per plan v3 §4 line 456). SW activation happens during page load, BEFORE shareReceived fires (intent is delivered post-launch). Test in §4.3 verifies. |
| **Cache filling on a slow network** | LOW | shell precache is ~5 small URLs. Static cache fills lazily as user navigates. First-load over slow LTE: ~200KB total shell precache. |
| **Multiple SW versions confusing during dev** | MEDIUM | dev-only `?nosw=1` query parameter that `register-sw.ts` checks; if present, calls `navigator.serviceWorker.getRegistrations()` + unregister. Documented in commit message. |

## §8. What this plan does NOT do

- Not a full PWA (no manifest install prompt; this is APK-side primarily).
- Not offline /api/items reading. The library will render the shell but show empty state when API calls fail. **A future plan (v0.7.x or later) can add IndexedDB-cached library items**, but it's out of scope here.
- Not background sync. Still v0.7.x WorkManager.
- Not a fix for "device never online before" — that case still shows offline.html with helpful copy. By design.
