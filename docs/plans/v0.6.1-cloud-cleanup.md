# v0.6.1 — Cloud-Cleanup Implementation Plan

## 0. One-sentence goal

Stop the v0.6.0 cloud server from lying about itself in user-facing copy, fix the
two-and-a-half real security-hygiene gaps the cutover left behind, and rename the
legacy LAN/Mac identifiers — without touching auth architecture or breaking any
paired devices.

---

## 1. Scope and non-scope

### In scope (this phase)

| Tier | Items | Why included |
|------|-------|--------------|
| **DO-THIS-WEEK** | T-1 setup-page privacy claim | Only finding delivered at first-run trust moment; pure string change |
| **T1** (one PR) | T-2 `Secure` cookie · T-3 security headers (XFO/nosniff/HSTS) · T-4 `cf-connecting-ip` log | Three small, related, server-side changes; testable together; ~2–3h |
| **T2** (one PR) | T-5 stale version/mode strings · T-6 unlock recovery copy · T-7 extension "your Mac" copy · T-8 offline.html copy · T-9 reachability strings · T-10 setup-apk verify-error | Pure copy cleanup; user-visible; no protocol changes |
| **T2-rename** | T-11 `BRAIN_LAN_TOKEN` → `BRAIN_API_TOKEN` (dual-read) · T-12 `/settings/lan-info` → `/settings/device-pairing` · T-13 delete `getLanIpv4()` · T-14 `OLLAMA_DOWN_BACKOFF_MS` rename | Renames with documented sequencing |
| **T2-tooling** | T-15 SwiftBar plugin trim to single probe · T-16 `rotate-token.sh` default URL · T-17 `restore-from-backup.sh` warning header | Mac-side scripts that broke at cutover |
| **T2-honesty** | T-18 layout `<meta description>` · T-19 settings backup-path label · T-20 sidebar version | Truthfulness in always-visible chrome |

### Explicitly out of scope (phase backlog or future)

- CSP nonce wiring (S3b) — M-L effort with measurable UI breakage risk; do as its own phase after at least one weekend of soaking the simpler headers.
- Per-device tokens (audit items A + C) — auth architecture change; needs its own design doc, DB migration, and APK pairing-flow rework. Schedule only if a second device or second user is on the roadmap.
- Rate-limit raise to 120/min (S4) — only if user actually hits 429s in real use. Currently theoretical.
- B2 off-site backup (D-18 carry-over) — a real durability gap, but **not in this phase**. It's its own phase: v0.6.2 — Backup hardening. Tracked separately.
- `robots.txt` — 2-line hygiene; bundle into the next coincidental commit, not worth a task.
- `/debug/quota` route gating — defer until someone has time for a debug-flag refactor.
- Chrome extension URL configurability (E5) — only if a second deployment exists.

## 5. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Header rollout breaks an embedded view (e.g., extension popup that frames Brain) | Audit complete: extension does NOT iframe Brain content — it makes API calls only. X-Frame-Options DENY is safe. |
| Cookie `Secure` flag breaks local dev | `process.env.NODE_ENV` guard ensures dev still works over `http://localhost`. |
| Extension copy lands before T-12 (route rename) | Sequenced explicitly — T-12 must merge first. |
| `BRAIN_LAN_TOKEN` rename breaks paired devices | Devices store the token VALUE, not the variable NAME. Phase 11a dual-reads both, so devices are unaffected during the transition. |
| HSTS lock-in (browsers cache) | The `max-age` is 2 years. If we ever need to revoke, browsers will still enforce until cache expiry. Acceptable risk — we control the domain. |
| User reads new privacy copy and decides not to use the app | That's the correct outcome if the disclosure changes their decision. The current state (lying to them) is worse. |

1. The setup page no longer lies (T-1).
