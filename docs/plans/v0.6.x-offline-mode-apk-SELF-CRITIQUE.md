# Self-Critique — `docs/plans/v0.6.x-offline-mode-apk.md` v1.0

**Date:** 2026-05-12
**Critic:** AI agent (same-agent adversarial pass, not independent review)
**Target:** `docs/plans/v0.6.x-offline-mode-apk.md` v1.0 (written this session, uncommitted)
**Companion plans (for comparison):** AB v2 (shipped `1538705`), Graph v2 (drafted same session)

> **Summary verdict:** 14 structural gaps. **3 are blockers** (PDF offline path is broken under the v0.5.3 CapacitorHttp-disabled regime; no library-count baseline for storage budget; IndexedDB blob storage may not survive the WebView's OS eviction policy). **6 are HIGH.** **5 are MEDIUM.** Plan v1 reads confident but has the same structural weakness that AB v1 and Graph v1 had — confident architecture on thin evidence, hand-waved edge cases, missing Open-Questions discipline was present but insufficient. A v2 rewrite is needed before the first code commit.

## §A. Blockers (3)

### A-1 — PDF offline path is incompatible with v0.5.3's CapacitorHttp-disabled fix

**Evidence:**
- `capacitor.config.ts:48` — `CapacitorHttp.enabled: false` (shipped this session in commit `9712dd5` to fix the APK PIN unlock loop).
- `src/components/share-handler.tsx:239–245` (comment + code) — **explicitly depends on CapacitorHttp's patched `fetch()` to resolve Android `content://` URIs** into Blobs without loading as base64. Under the plain WebView `fetch()`, `content://` URIs are **not resolvable** — browser-native `fetch` rejects the scheme.
- Plan v1 §5.6 says: "at share-receipt, read the file via `fetch(uri)` → `blob()` → store in `file_blob` column."

**Why it's a blocker:** The current share-handler PDF path was written when CapacitorHttp was enabled. Today (v0.5.3), CapacitorHttp is disabled. **There is no working PDF share path on the current APK — the existing code would fail on the `fetch(uri)` call today**, and the plan inherits that broken assumption. Either:
- (a) the existing code is already broken for PDFs in v0.5.3 and nobody noticed yet — in which case offline-PDF is a moot feature until the underlying bug is fixed, or
- (b) there's a fallback I haven't seen — in which case the plan must document which code path it's building on.

**Impact if un-fixed:** OFFLINE-4 commit would enqueue a null blob for every PDF share, creating silent data loss where the user thinks they saved something but the queue contains zero bytes.

**Fix for v2:**
- Before writing code, run an empirical test: share a PDF from the Files app on a v0.5.3 APK device. Does `fetch(content://...)` work? If no, this plan's PDF story needs a different mechanism (e.g., a Capacitor plugin for file resolution, or base64-over-bridge).
- Explicitly document the verified-or-broken state in §5.6.
- If PDF is broken under v0.5.3, drop PDF offline from MVP and ship OFFLINE for URL + note only.

### A-2 — Storage budget unmoored from actual usage

**Evidence:**
- Plan v1 §8 sets "IndexedDB storage cap 50 MB, warn at 40 MB, reject PDFs > 20 MB." These numbers are plucked — no measurement.
- Android WebViews enforce per-origin IndexedDB quotas that vary by device, free-space heuristic, and WebView version. On low-storage devices, the effective quota can be ≤ 10 MB before eviction.
- The same session's `docs/research/graph-view-tooling.md` explicitly flagged "library size unknown" as a research gap for Graph; the offline plan didn't do the equivalent.

**Why it's a blocker:** the user's typical Substack post PDF is 1–3 MB (per `project_lenny_login_paywall_tier.md` memory — login tier varies output). "20 PDFs in a subway trip" is a realistic scenario. That's 40–60 MB. The plan's cap of 50 MB hits exactly at this boundary. A real subway scenario could silently start rejecting shares, producing exactly the "saved offline" user-trust break the feature is supposed to prevent.

**Fix for v2:**
- Measure current Android WebView IndexedDB quota on the user's actual device (adb shell).
- Publish a **measured** cap, not a round-number guess.
- Define explicit "quota pressure" fallback: when approaching cap, either reject new captures *with a clear user message* OR evict `status='synced'` older than 1 hour (not 24).
- Consider: store PDF blobs on the Android filesystem (app-private directory) instead of inside IndexedDB, with IndexedDB holding just the file path. Would require a Capacitor Filesystem plugin call — trade-off against the "zero new native code" goal.

### A-3 — IndexedDB persistence under WebView OS pressure is undocumented

**Evidence:**
- Android WebViews can, under OS memory pressure, evict IndexedDB data for backgrounded apps. Chrome's guarantees on "persistent storage" (`navigator.storage.persist()`) are browser-specific and not all WebViews honor them.
- Plan v1 assumes persistence but doesn't call `navigator.storage.persist()` anywhere.
- User story 3.2 "My laptop is closed while I'm out" and 3.3 "Retries fail — I'm stuck" both depend on the queue surviving **hours** of app-closed state.

**Why it's a blocker:** if the WebView evicts IndexedDB data the feature's core promise fails silently — the item isn't "saved offline," it's "saved until Android needs the memory back, then deleted without notice."

**Fix for v2:**
- Add `await navigator.storage.persist()` on outbox-init. If it returns `false`, fall back to user-visible warning ("Your device won't guarantee offline saves will persist — sync as soon as you can").
- Document: what is the actual Android WebView behavior on our specific target API level (Android 13, 14)?
- Consider a shadow copy of outbox metadata in `@capacitor/preferences` (which uses SharedPreferences and is more survivable) — even if blobs live in IndexedDB, at least metadata and recent text-only captures are more robust.

## §B. High-severity gaps (6)

### B-1 — Option A (pure WebView) eliminates the subway use case

**Evidence:** Plan v1 §4.1 commits to "Option A — Pure WebView" rationalized by "user's stated flow is share → close → later open." But the §3.1 user story explicitly says: "arrives at station, signal comes back, item syncs within ~30 seconds" — **assuming the app is still open in memory**. In reality:

- User shares from Substack. Brain opens briefly via share-target. User returns to Substack.
- Android may close Brain's WebView process 2 minutes later to reclaim memory.
- Train arrives at station. Signal returns. Brain is not running. **No retry fires.**
- User opens Brain an hour later. Sync finally happens.

**The "subway" story sold the feature, but Option A doesn't deliver it.** What Option A delivers is "syncs next time you actively open Brain." That's closer to user story 3.2 than 3.1.

**Fix for v2:**
- Rewrite §3.1 to match Option A's actual behavior OR commit to Option C (hybrid) for MVP.
- If sticking with Option A, rewrite §3.1 user expectation: "syncs next time you open Brain."
- Consider a **deferred** v0.7.x commit to add WorkManager — plan this as the explicit follow-on, not a vague "might add later."

### B-2 — No validation of `online` event reliability in Capacitor WebView

**Evidence:** Plan v1 §4.2 retries trigger on "browser `online` event." The `window.online` event is a **browser DOM event** and its reliability inside a Capacitor WebView is inconsistent — historically, Android WebView has fired `online` events spuriously (when only DNS changed, not internet) and missed them entirely (on captive portal transitions).

**Why it matters:** if `online` fires late or not at all, the 30s interval is the real backstop. User experience becomes: "the app thinks it's syncing but it's waiting up to 30s after signal returns." That's not broken, but it's slower than the plan advertises.

**Fix for v2:**
- Document the actual Capacitor WebView `online` event reliability on the target API level.
- Consider using `@capacitor/network` (first-party Capacitor plugin, already battle-tested) instead of the browser event. Its `networkStatusChange` event is more reliable.
- Adjust user-facing copy to match reality: "syncs within ~30 seconds of signal returning" (not "instantly").

### B-3 — Dedup content-hash windowing is incoherent

**Evidence:** Plan v1 §5.2 says "If another outbox entry has the same hash and `status ∈ {pending, queued, synced}` within the last 10 minutes → skip the new enqueue, mark the result as 'duplicate.'"

**Problems:**
1. Dedup against `status='synced'` only covers the 24-hour retention window (§4.3 deletes synced after 24h) — but the 10-minute dedup window is smaller. Fine in isolation; confusing in combination.
2. "Mark the result as 'duplicate'" — mark *what* result? There's no enqueued entry yet to mark. This reads like the dedup decision creates a no-op outbox entry for audit, but that's not said.
3. The existing `isDuplicateShare()` helper (`src/lib/capture/dedup.ts:39`) has a **2-second** window. The plan's 10-minute window is a different mechanism; the plan says "keep both." But what if the 2s window catches it and the 10m window would have too? Order of checks undefined.
4. The content hash scheme (`sha256(kind + sorted_payload_fields)`) includes every field — but the user might share the same URL with different highlighted text from the source app. That changes the payload, bypasses hash, produces duplicates.

**Fix for v2:**
- Specify the dedup **key**, not the "payload hash": for URLs, it's `(kind, normalized_url)`; for notes, the content hash; for PDFs, the file hash.
- Specify the **check order** explicitly: (1) 2s in-memory dedup, (2) 10m outbox-level dedup, (3) server-side dedup.
- Clarify what "mark as duplicate" means in UI terms: is there a toast? a silent no-op? an entry in `/inbox`?

### B-4 — Retry budget (~22 minutes) is too short for realistic outages

**Evidence:** Plan v1 §5.3 sets MAX_ATTEMPTS=8 → max 22 min before `stuck`. Realistic user outage patterns:
- **Laptop asleep overnight** — 8+ hours.
- **User takes flight with laptop** — 4+ hours.
- **Long weekend away** — 48+ hours.

After 22 minutes of active retries (which only fire while app is open), items go `stuck` and require manual "Retry all." That's a lot of clicking for a user who comes home from a weekend trip with 30 items that hit stuck state.

**Fix for v2:**
- Revisit: either extend MAX_ATTEMPTS to 12+ (4+ hours in backoff max) or skip `stuck` state for certain failure classes (network failures should retry indefinitely; only *client-side* bugs should ever hit `stuck`).
- Table per §5.4 already classifies failures correctly — extend the decision: "network failures never stuck; auth failures stuck immediately; 4xx-non-auth stuck immediately."

### B-5 — No plan for what happens when server schema/API drifts during a long offline window

**Evidence:** Scenario: user queues 20 items offline. A week later, laptop comes online. During that week, `/api/capture/url` was changed (server-side) to require a new field. All 20 items fail on sync with 422 → immediately `stuck`.

Plan v1 §5.4 correctly classifies 422 as `stuck` with "malformed payload (client bug)." But it doesn't address that **the client is doing the right thing — the server changed underneath.**

**Fix for v2:**
- Consider a lightweight API version header (`X-Brain-Client-Api: 1`) — if server replies 422 with a version-mismatch code, classify as client-needs-upgrade and show a distinct error ("Update the Brain app to sync these items").
- This is minor, but the plan skips the cross-version scenario entirely.

### B-6 — Captive-portal detection via `content-type: text/html` is too narrow

**Evidence:** Plan v1 §10 risk table says "Captive portal returns HTML on POST → detect `content-type: text/html` → retry." Real captive portals:
- Cloudflare's Warp blocks with a 200 OK and **JSON** (it's a legitimate API response).
- Some captive portals return **HTTP 302 redirects** without changing content-type.
- Some return **TLS handshake rejection** before content-type is negotiable.
- Some return legitimate JSON containing a login link.

Detecting captive portal is a known-hard problem; "check content-type" solves a small fraction of cases.

**Fix for v2:**
- Drop the claim that captive portal is handled robustly. Instead: "any non-2xx, non-401/403/422 response → retry up to N times then stuck."
- Accept that captive portals are classified as "transient network failure" and retried — not detected-as-portal.

## §C. Medium-severity gaps (5)

### C-1 — `@capacitor/local-notifications` permission UX handwaved

**Evidence:** §7 Commit 7 says "Android 13+ `POST_NOTIFICATIONS` permission." But:
- Plan doesn't specify **when** to request the permission (on first share? on first stuck? never request, just silently fail?).
- If the user declines, the plan says "feature degrades gracefully — still works, just no OS notifications." But the user expectation set in user story 3.3 is "persistent notification." Reducing that silently is a UX break.

**Fix for v2:** Permission request specified to fire on first successful enqueue; if user declines, prominent in-app badge becomes the primary surface AND the `/inbox` link is pinned to the share-target toast.

### C-2 — No RESUME-after-kill test in the manual matrix

**Evidence:** Plan v1 §9.3 includes "Queued items + force-kill app + reopen → items still there; drain on reopen." **This is the most important test** for an offline feature. It's buried in row 8 of a 9-row table with no special emphasis.

**Fix for v2:** Elevate to a top-level "critical scenario" in §9.3. Also add: "force-kill during a PDF enqueue (before blob finishes writing) → on reopen, partial entry is either correctly resumed or safely discarded (not leaked)."

### C-3 — `retry now` button behavior under rate limiting not specified

**Evidence:** §5.4 says "Response 429 → retry after Retry-After header or 60s." But the user can press "Sync now" to bypass the backoff. If they hammer it, the client will spam the server, get more 429s, and the server's rate limiter (`DEFAULT_RATE_LIMIT = 30/min/token`) will extend the 429 window.

**Fix for v2:** "Sync now" should respect the next-retry-at for items already in 429-cooldown. Or: "Sync now" forces a single-pass retry, not an override-all. Document.

### C-4 — Plan doesn't specify whether `/inbox` is a new bearer-auth API route or purely client-side

**Evidence:** The outbox is client-side data (IndexedDB). `/inbox` page should not need a server round-trip. But Next.js + RSC model means `/inbox/page.tsx` runs on the server by default. Plan v1 §7 Commit 6 says "server component shell; reads initial state via a client call (outbox is client-side data)" — that works but is subtle and will confuse future maintainers.

**Fix for v2:** Make `/inbox` a pure client component with `"use client"` — no server component shell — since all data is client-local.

### C-5 — No prior-art evaluation

**Evidence:** Graph plan v2 spent an entire research doc on "what does Logseq / Obsidian / Foam use?" Offline plan v1 has zero prior-art check. Plenty of Capacitor apps have built offline queues (e.g., Ionic's official examples, `@capacitor-community/workbox`, PWA service-worker background sync). Plan should evaluate whether an existing pattern / library covers this.

**Candidates worth a 30-minute look:**
- **Workbox background-sync** — handles exactly this use case for PWAs; Capacitor-compatible.
- **`@capacitor-community/background-runner`** — native background-task runner.
- **`redux-persist` + offline queue middleware** — battle-tested in React Native land.

**Fix for v2:** 30-minute prior-art spike. Either adopt an existing tool or document *why* we're rolling our own.

## §E. Patterns across all three plans this session (AB v1, Graph v1, Offline v1)

All three v1 plans had the same structural pattern:

| Pattern | AB v1 | Graph v1 | Offline v1 |
|---|---|---|---|
| Confident architecture without empirical validation | ✓ (span-wrap, 5-min poll) | ✓ (d3-force, no bench) | ✓ (Option A, no WorkManager eval) |
| Hand-waved edge cases in prose | ✓ (match strategy) | ✓ (hit-testing) | ✓ (captive portal, kill-resume) |
| Missing Open Questions section | ✓ | ✓ | *present but incomplete* |
| No prior-art check | ✓ | ✓ | ✓ |
| No performance budget | ✓ | ✓ | *present but unmoored from measurement* |

**Systemic takeaway:** one-pass plan writing produces structurally deficient plans. The "draft → critique → revise" loop is not optional — it's the delivery mechanism. Future plans should be written with the critique lens applied from the first draft, not retroactively.

**Memory to consider:** a feedback entry — "plan docs in this project must include §0 change-log, §N Open Questions, explicit prior-art table, and measured (not guessed) performance budgets. Write critique-first, not critique-after."

## §G. Open questions for the user

1. **Blocker A-1 triage:** should we empirically test PDF share on v0.5.3 *before* writing v2, or rewrite v2 around URL/note-only offline and treat PDF as a v0.7.x follow-on?
2. **Blocker A-2 triage:** acceptable to require a Capacitor Filesystem plugin for PDF storage (breaks the "zero new native code" norm), or keep blobs in IndexedDB with a smaller cap?
3. **Option A vs C (B-1):** subway story is real user intent. Is "app must be opened to sync" an acceptable MVP trade-off, or should we bite the WorkManager bullet now?
4. **Infinite vs bounded retries (B-4):** does the user prefer "it keeps trying forever, you never have to manually click retry" (infinite for network errors) or "predictable worst-case, then a notification"?
5. **Prior-art spike (C-5):** is a 30-min eval of Workbox / `@capacitor-community/background-runner` worth the delay, or ship the hand-rolled version?
