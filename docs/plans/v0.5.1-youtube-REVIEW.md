# v0.5.1 YouTube Plan — Cross-AI Review (Stage 4)

**Review date:** 2026-05-11
**Reviewer:** Plan agent (Claude Code Plan sub-agent)
**Verdict:** PROCEED-WITH-CHANGES

## Executive summary

The plan is structurally sound and the research spike is high quality. The architecture decision to do a self-contained InnerTube POST rather than delegating to `youtube-transcript` as a black box is the right call. Three things need to be fixed before T-YT-2 begins: a type-system compile error hidden in the route handler (`shareDedupKey` rejects `"youtube"` as a source literal), a missing column in `InsertCapturedInput` / `insertCaptured` that will cause a TypeScript error on `duration_seconds`, and the enrichment pipeline's 12,000-character body truncation, which will silently swallow the majority of any long transcript before it reaches the LLM. Everything else is either fine or a low-priority polish item.

---

## Severity findings

### HIGH (must absorb before execution)

**H-1 — `shareDedupKey` type signature rejects `"youtube"`; compile error in route handler**

The existing `shareDedupKey` function in `src/lib/capture/dedup.ts` declares its first argument as `source: "url" | "note" | "pdf"`. The plan's route-wiring pseudocode in §5.3 calls `shareDedupKey("url", url)` on the canonical URL, which incidentally avoids this — but if the implementer reads §5.3 and follows a `source: videoId ? 'youtube' : 'url'` pattern that would be natural to write, TypeScript refuses to compile. The plan does not explicitly state the intent.

Recommended fix: Add a note to T-YT-6 and §5.3 explicitly stating that `shareDedupKey`'s source literal must remain `"url"` for YouTube canonical URLs (since the key identifies payload shape, not source type). Effort: 5 minutes to add a sentence.

**H-2 — `insertCaptured` and `ItemRow` have no `duration_seconds` field; the T-YT-2 task scope is understated**

The plan's T-YT-2 correctly lists `src/db/items.ts` as a file to change. But `InsertCapturedInput` currently has no `duration_seconds` field, and the `INSERT INTO items` statement in `insertCaptured` (items.ts lines 33–50) does not include that column. The plan's §5.3 pseudocode uses a runtime duck-typed `"duration_seconds" in article` check rather than threading the field through `InsertCapturedInput`, which is fragile. Also, if `YoutubeExtracted` is not assignable to `ExtractedArticle`, the union type `article: ExtractedArticle | YoutubeExtracted` in route.ts will cause TypeScript to flag every field access on `article` that isn't shared between the two types.

Recommended fix: Add `duration_seconds?: number | null` to `InsertCapturedInput`, update the `INSERT` statement, and make the route pass `duration_seconds: videoId ? youtubeResult.duration_seconds : null` straightforwardly. Effort: 30 minutes, absorbed into T-YT-2.

**H-3 — Enrichment pipeline hard-truncates `body` at 12,000 characters; a 2-hour transcript will be ~99% discarded before LLM enrichment**

The enrichment prompt in `src/lib/enrich/prompts.ts` caps `body` at `MAX_BODY_CHARS = 12_000` before passing to the LLM. A 2-hour video at ~60 chars/line × 7,200 lines = ~432,000 characters. The enrichment pipeline will summarize only the first ~200 lines — roughly the first 3 minutes. Summaries will be misleading, categories likely wrong, LLM-rewritten titles may miss the subject.

This is not a regression introduced by v0.5.1 — it is a pre-existing limitation made acutely visible by this feature. The plan does not mention it.

Recommended fix: Prepend a synthetic header `title\n\nchannel_name\n\nduration: MM:SS\n\n<transcript>` to the transcript body before storage so that the truncated enrichment slice sees the most important metadata. Alternatively, add a short description of this limitation and defer the fix to v0.5.2. Either is acceptable; the plan must acknowledge it. Effort: 15 minutes to decide and document; 30 minutes to implement the prepend.

### MEDIUM (should absorb; defer to v0.5.x only with explicit reason)

**M-1 — `YoutubeExtracted` is not a subtype of `ExtractedArticle`; the route handler union type is fragile**

`ExtractedArticle` includes `excerpt: string | null` and `html_length: number` that `YoutubeExtracted` does not. TypeScript will only allow access to the intersection of both types in the union. The practical compile error is contained (neither field is used by `insertCaptured`), but the structural mismatch is a maintenance hazard.

Recommended fix: Extract a shared `CapturedContent` base interface that both satisfy, and make `insertCaptured` accept that. Alternatively, extend `YoutubeExtracted` with `excerpt: null` and `html_length: 0`. Effort: 1 hour.

**M-2 — `live_stream` error missing from T-YT-5 test list**

The plan's `YoutubeCaptureError` type includes `"live_stream"`, and the research maps it to 422. But T-YT-5 lists only 8 cases without `live_stream`. A live-stream URL will hit an unexercised code path in production.

Recommended fix: Add a 9th unit test case mocking `isLive: true`. Update exit criterion to "9 cases." Effort: 30 minutes.

**M-3 — `smoke-v0.5.1.mjs` DB-boot assertion doesn't match the static-only pattern of existing smokes**

Existing smokes are purely static. The plan's assertion "(1) migration 007 applied on fresh DB" requires a DB boot. If run against `data/brain.sqlite`, it mutates developer state.

Recommended fix: Make the migration assertion static (parse the SQL file, assert it contains the ADD COLUMN statement), or use `BRAIN_DB_PATH=:memory:`. Effort: 30 minutes.

### LOW (nice-to-haves; file as backlog)

**L-1 — No negative test for `extractVideoId` against near-miss URLs**

Channel URLs (`youtube.com/channel/UCxxx`), playlist URLs, and `youtu.be/` with no ID should return `null`. Add negative-case tests so the Readability-fallback path is deliberate, not accidental.

**L-2 — `formatTranscriptBody` with zero segments produces empty string; `body` is NOT NULL**

Guard: if the formatted body is empty after transcript fetch, treat as `no_captions`.

**L-3 — Telemetry log type mismatch**

The plan's `capture.youtube.fetched` goes to `data/errors.jsonl` via `logError`, but that function is for errors. Skip the success log (the item row is the record) or add `logEvent` to the sink.

**L-4 — `^1.3.1` semver range allows minor-version drift on install**

For a library wrapping an undocumented API, `~1.3.1` (patch-only) is more conservative. Low priority given the lockfile pins exactly.

## Risk-register check

All mitigations are real except where noted. The "migration 007 IF NOT EXISTS" wording in §4 is slightly wrong (SQLite doesn't support that clause on ADD COLUMN; the migration runner's applied-ID tracking is what makes it safe) — the plan's own parenthetical acknowledges this. No action needed, but wording could mislead.

The "very long transcript blows memory" risk is partially mitigated: 7,200 segments × 200 chars = 1.44 MB body. SQLite handles it; Node.js heap handles it. Real pressure is during formatting (all segments in one array), not storage. Sufficient for v0.5.1.

## Dependencies sanity check

**`youtube-transcript@^1.3.1`**: Version correct per research. `^` semver concern noted at L-4.

**Zero transitive deps**: Research-verified; T-YT-3 confirms on lockfile.

**IP-block concern**: A3 assumption is well-reasoned. The Cloudflare tunnel doesn't proxy outgoing requests; InnerTube fetch originates from the Mac's own IP. Mitigation assessment is correct.

**Hidden coupling risk — `youtube-transcript` XML parser export surface**: The plan's D-v0.5.1-1 decision uses `youtube-transcript` for XML parsing only while driving the InnerTube POST ourselves. This assumes the library exports the XML parser separately. If it only exports `fetchTranscript()` (high-level), the implementation will either need to call `fetchTranscript()` (duplicate InnerTube call) or inline the ~30 lines of XML parsing.

Verification step missing from T-YT-3: after `npm install`, read `node_modules/youtube-transcript/src/index.ts` and confirm the export surface before writing T-YT-4.

## Alternative paths not considered

**Should `fetchTranscript()` be called directly?** Decision D-v0.5.1-1 locks in self-contained InnerTube POST for good reasons (title, author, duration from one call). This creates the coupling concern above; verify export surface before T-YT-4.

**Should `duration_seconds` come from InnerTube instead of a migration?** No — persistent storage is needed for display. Migration pattern matches `total_pages` precedent.

**Should this be a single PR?** Task breakdown adds genuine value given new dep, migration, new file, type changes, fixtures, two smoke scripts. Not over-engineered.
