### A-2 — Schema `payload: object` does not support PDFs

**Evidence:**
- §4.3 schema (lines 156–172): `payload: object   // the request-body verbatim (JSON-safe)`
- §5.2 dedup (line 207): PDF content hash = `sha256(file_bytes)`
- Today's PDF capture path: `/api/capture/pdf` accepts `multipart/form-data` (`FormData` with `pdf` blob field + `x-expected-sha256` header) — see `src/components/share-handler.tsx:262`

**Why blocker:** PDFs are uploaded as `FormData`, not JSON. The `payload: object` field is "JSON-safe" but FormData with a Blob is not JSON-serializable. The schema as written **cannot represent a PDF outbox entry**. Either:
- Schema needs `payload: object | { kind: 'pdf', file_path: string, expected_sha256: string }` — the file lives on filesystem (per §5.1) and the row carries metadata only, OR
- A separate `pdf_metadata` field, OR
- The plan is internally inconsistent.

**Fix for v3:** rewrite the schema to make explicit how PDF entries differ from URL/note entries. Example:

```typescript
type OutboxEntry =
  | { kind: 'url'; payload: { url: string; title?: string }; ... }
  | { kind: 'note'; payload: { title: string; body: string }; ... }
  | { kind: 'pdf'; file_path: string; file_name: string; expected_sha256: string; ... };
```

Discriminated union is the right shape; the v2 collapsed structure obscures the per-kind difference.

## §B. High-severity gaps (9)

### B-1 — `Network.getStatus()` claimed synchronous; actually async

**Evidence:**
- §5.4 (line 249): *"Status query: `Network.getStatus()` returns `{ connected, connectionType }` synchronously for 'should I bother trying' pre-flight checks."*
- Capacitor docs / npm `@capacitor/network`: `getStatus()` returns `Promise<ConnectionStatus>`. **All Capacitor plugin methods are async** — they cross the JS↔native bridge.

**Why high:** the implementer would write `if (Network.getStatus().connected)` and get an always-truthy Promise. Code wouldn't behave as the plan describes. Wasted iteration cycle.

**Fix for v3:** *"Status query: `await Network.getStatus()` returns `{ connected, connectionType }` — await before pre-flight check."*

### B-2 — SHA256 of 10 MB PDF blocks the UI thread

**Evidence:**
- §5.2 (line 207): PDF dedup key = `sha256(file_bytes)` via "Filesystem readFile + crypto"
- `Filesystem.readFile()` returns base64; converting to bytes + computing SHA256 in JS for a 10 MB file takes ~600–1200 ms on M1-class hardware, longer on a phone CPU
- Share flow is synchronous — user is staring at the share-target activity until enqueue completes

**Why high:** a >1s freeze on share-receipt feels broken. Two problems: (a) the SHA256 is computed before the dedup check, so even a duplicate (which we'll discard) costs the full hash. (b) The UI has no progress indicator during the hash.

**Fix for v3:**
- Skip content-hash dedup for PDFs; dedup PDFs on `(file_name, file_size)` tuple instead — cheaper, almost as good in practice (different PDFs with same name+size are rare).
- OR: defer the SHA256 to background (after enqueue, before sync attempt) — accept that the dedup window for PDFs is "by name+size at enqueue, by hash at sync."
- OR: hash on a Web Worker so the UI thread isn't blocked. Capacitor WebView supports `Worker`; cost is ~30 lines of code.

### B-3 — Captive portal still mis-classified after the v1 fix attempt

**Evidence:**
- §5.3 (line 216): *"`fetch()` throws (DNS, socket, timeout, captive portal HTML) → transient → retry indefinitely"*
- Reality of captive portals:
  - Sometimes returns **HTTP 302 → login URL** (fetch follows redirect, ends at HTML page → 200 with HTML body, **not a throw**)
  - Sometimes returns **200 OK with HTML body** to a POST request (cleartext interception)
  - Sometimes returns **403 with HTML body**
- Server contract: `/api/capture/url` returns `application/json`. If response is HTML (any status code), `await res.json()` throws `SyntaxError: Unexpected token '<'`.

**Why high:** v1 critique B-6 said "drop captive portal detection." v2 dropped the detection. But the new classifier in §5.3 only handles `fetch() throws` and HTTP status codes. **It does not handle "got 200 OK but the body is HTML."** That case will throw inside the response-parsing code path and be silently untyped.

**Fix for v3:** Add explicit row to §5.3 table:
| Response 2xx with `content-type` not `application/json` (or JSON parse fails) | transient | retry indefinitely; this is captive portal / transparent proxy |

Plus: in `classify.ts`, wrap `res.json()` in try/catch and route the catch to the transient bucket.

### B-4 — Retention deletes audit trail after 24h offline

**Evidence:**
- §4.4 (line 179): *"`status='synced'`: delete after 24 hours."*
- §3.4 / §4.2: user opens Brain → "Synced N items" badge → user closes browser
- Scenario: user shares 100 items over a busy week, syncs them all; goes on vacation 25 hours later; comes back to find `/inbox` is empty (everything deleted). Loss of "what did I save and when" trail.

**Why high:** this isn't a bug, it's a design choice — but the plan claims `/inbox` is "review what synced recently" (line 179 again) while immediately deleting the entries that constitute the review. The retention contradicts the user-facing function.

**Fix for v3:** options:
- (a) Retain `synced` entries until storage pressure (50% quota → start aging), not on a timer.
- (b) Retain forever; rely on user discard. Deleting auto is a small win for a meaningful loss.
- (c) Move to a `synced_at + 24h` retention with explicit user-toggle "Keep history" → makes the trade-off visible.

Plan v3 should pick one and document the user-facing implication.

### B-5 — Race condition: outbox must initialize before share-handler

**Evidence:**
- Capacitor share-target's mechanism: when an external app shares, Android launches Brain's `MainActivity` which loads the WebView, which loads `https://brain.arunp.in/`. The share event is delivered as soon as the JS side calls `CapacitorShareTarget.addListener('shareReceived', …)`.
- `src/components/share-handler.tsx:101` registers the listener inside a `useEffect`, which runs after the component mounts, which runs after the page hydrates.
- Outbox initialization (`navigator.storage.persist()` + IDB `open()`) is async.
- v2 plan does not specify the relative ordering of "outbox ready" and "shareReceived event delivered."

**Why high:** if the share event fires before outbox is initialized, the handler either (a) drops the share, or (b) tries to enqueue with a not-ready outbox and gets an error. Both are silent data loss. A user shares from Substack expecting "Saved offline" and gets nothing.

**Fix for v3:** explicitly:
- Outbox-init must complete before `shareReceived` listener is attached.
- If the listener is attached and an event is buffered before init completes, it must be queued in memory and replayed after init.
- Add an integration test: simulate a `shareReceived` event firing during the first 100 ms of app load.

### B-6 — User-flow narrative in §3.1 is incorrect about Android share-target behavior

**Evidence:**
- §3.1 (line 85): *"Brain opens via share-target → enqueues to outbox → shows 'Saved offline' toast → control returns to Substack."*
- Android reality: when a share-target receives an intent, it becomes the foreground app. The user is **inside Brain after the share**, not back in Substack. Returning to Substack requires the user to tap back or swipe.

**Why high:** the user story narrates an experience the user won't have. This affects the toast design too — if Brain is foreground after share, a toast inside Brain works. If we want "share and stay in source app," we'd need to call `Activity.finish()` or similar after enqueue, which is a different design.

**Fix for v3:** rewrite §3.1 to match actual behavior: *"Brain opens via share-target → enqueues to outbox → shows 'Saved offline' confirmation → user taps back to return to Substack OR closes Brain to swipe back."* Decide whether the post-share UX is "Brain stays in foreground showing the success state" or "Brain auto-finishes after toast" — both are valid; pick one and write it down.

### B-7 — `attempts` field grows unbounded for indefinite retries

**Evidence:**
- §5.3 (line 239): *"No MAX_ATTEMPTS for transient failures."*
- §4.3 schema: `attempts: number`
- Scenario: user puts phone in drawer, network is flaky for 3 days. With 1-hour cap on backoff, that's ~72 retry attempts. `attempts: 72` is fine; `attempts: 720` (a month) is still fine numerically. But the field never resets — even after eventual success, the next item shipped on the same outbox queue starts fresh, but if the same item somehow re-enters retry, the count carries.

**Why high (medium-high):** not a numerical bug but a UX one — `/inbox` page presumably shows "attempts: 72" in the row, which surfaces a useless counter at large values. Also: if a stuck-then-retry-succeed happens, attempts isn't reset.

**Fix for v3:**
- On `synced`, reset `attempts = 0` and clear `last_error` (already does the latter per schema comment).
- In `/inbox` UI, show "Last tried 5 minutes ago" not "attempts: 72."

### B-8 — Quota-measurement HTML delivery mechanism undefined

**Evidence:**
- §8.1 (line 311): *"write `scripts/measure-webview-quota.html` — a tiny standalone HTML the user loads inside Brain WebView once."*
- Reality: the Brain WebView only loads `https://brain.arunp.in/...`. There's no mechanism to load arbitrary file:// HTML inside it. The user can't paste a URL into the WebView (no address bar in a Capacitor app).

**Why high:** the OFFLINE-PRE measurement gate cannot be executed as described. The whole "measure quota first" architecture rests on this script being runnable.

**Fix for v3:** options:
- Add the measurement as a route on the Next.js server: `/debug/quota` that runs `navigator.storage.estimate()` and shows the result. User navigates to `https://brain.arunp.in/debug/quota` from inside the Brain WebView. Simple, requires no special delivery.
- OR: bake the measurement into the existing `/settings` page as a one-click button.
- OR: add it to the boot path of the OFFLINE-1 outbox-init code so it logs once on first run via the existing `/api/errors/client` channel.

### B-9 — `payload: object` is JSON-safe but stored in IndexedDB which supports structured clone (more than JSON)

**Evidence:** §4.3 schema comment: `payload: object   // the request-body verbatim (JSON-safe)`. IDB stores by structured-clone, which is a superset of JSON (supports Date, Map, Set, Blob, etc.) but blocks functions and class instances.

**Why high:** the comment is misleading — implementers reading "JSON-safe" might `JSON.stringify` before storing (unnecessary serialization cost). Or might assume Date objects round-trip through JSON (they don't — Date → string).

**Fix for v3:** *"`payload: object` — structured-clone-safe; do NOT JSON.stringify before storing. Date instances round-trip; class instances don't."*

## §C. Medium-severity gaps (8)

### C-1 — §5.7 contradicts §0 change-log claim about `/inbox` architecture

**Evidence:**
- §0 (line 39): *"v2 decision: **Pure client component** (`"use client"`) — outbox is client-local data, no server dance"*
- §5.7 (line 269): *"`src/app/inbox/page.tsx` is a thin server component that just renders `<InboxClient />`"*

These are reconciled if you read §5.7 carefully: page.tsx is a server stub that renders an all-client child. But the §0 claim "no server dance" contradicts having a server component at all.

**Fix for v3:** in §0, drop "no server dance" — say *"`/inbox` content rendered entirely client-side; page.tsx is a thin wrapper."*

### C-2 — Desktop-web offline behavior unaddressed

**Evidence:** §6 (line 284): *"In non-APK contexts the feature is inert; share-handler does direct POSTs as today."*

**Reality:** desktop browser users *can* be offline (laptop on plane, browser tab open). A direct POST will fail. The plan provides no path; the user just sees the failure.

**Why medium:** not the primary use case (desktop is generally always-online via the local server), but the plan claims comprehensive offline coverage and explicitly excludes one obvious case.

**Fix for v3:** explicitly note: *"Desktop web offline (laptop disconnected) is out of scope; the feature is APK-only because Capacitor share-target is the share path that needs offline coverage. Desktop captures via `/capture` web UI continue to fail visibly."*

### C-3 — IDB write failure (quota exhausted) not handled in §4.2

**Evidence:** §4.2 data flow shows: write outbox → try POST → handle response. There is no branch for "outbox write fails." If quota is full, IDB throws on `put()`.

**Why medium:** a real edge case at scale — user with full quota tries to enqueue and the write fails. Plan §10 says "reject new enqueue at 100%" but doesn't specify how to detect that or what the user sees.

**Fix for v3:** add explicit branch in §4.2 — pre-flight `navigator.storage.estimate()` before each enqueue; if `usage / quota > 0.95`, show a "Storage almost full" warning and reject the enqueue. If the write itself throws despite pre-flight, surface a non-blocking error.

### C-4 — Sync-now ordering with newly enqueued item not specified

**Evidence:** §3.4: user opens Brain with 3 queued items, taps Sync now → all drain. Implicit assumption: the 3 items drain in a known order.

**Reality:** if the user shares a 4th item *while* sync-now is in progress, when does it sync? In the same pass? In the next pass? Plan doesn't say.

**Fix for v3:** specify: *"Sync-now operates on a snapshot at trigger time. Items enqueued during a sync-now run join the next retry cycle, not the current one."*

### C-5 — Commit 5 is shippable in a broken state (no auto-retry)

**Evidence:** §7 ordering — commit 5 wires share-handler into outbox; commit 6 adds retry triggers. Between commits 5 and 6, the outbox accumulates entries that will never auto-drain (only manual sync-now works).

**Why medium:** Lane L convention is "every commit ships green and works." Commit 5 violates that.

**Fix for v3:** either (a) merge commit 5 + 6 into a single commit, or (b) document explicitly that commit 5 ships behind a feature flag (`OUTBOX_ENABLED=false`) until commit 6 lands.

### C-6 — §10 risk table off-by-one: "queued vs stuck" should be "five states"

**Evidence:** §10 (line 388): *"User confused by 'queued' vs 'stuck' state language → user-test all four states with you."*

**Reality:** §4.3 schema lists FIVE statuses: `pending | queued | synced | stuck | duplicate`.

**Fix for v3:** "five states" — and possibly drop `pending` from the user-facing surface entirely (it's a momentary state during initial POST). User-facing states are 4: queued / synced / stuck / duplicate. Document.

### C-7 — `/inbox` accessibility unaddressed

**Evidence:** Graph plan v2 §0 explicitly added accessibility (text-list fallback, keyboard nav). Offline plan v2 §5.7 just says "renders grouped by status" — no mention of keyboard reachability, screen-reader labels, semantic HTML structure.

**Why medium:** consistency with Graph plan v2 quality bar. `/inbox` is an interactive page with destructive actions (Discard) — accessibility matters more here than Graph.

**Fix for v3:** add to §5.7: *"Page uses semantic `<ul>` of entries with `<button>` elements for actions; ARIA labels on status badges; keyboard-reachable Retry/Discard."*

### C-8 — Manual matrix missing today's PDF-from-Files-app regression

**Evidence:** §9.4 includes "Share PDF, phone offline" but not the specific scenario that broke today: "Share PDF from Files app, phone online" — which exposed the `fetch(/data/...)` 404 bug. Plan v2 doesn't add this as a non-regression test.

**Why medium:** the bug we just fixed has no test; future refactor could re-break it without anyone noticing.

**Fix for v3:** add row: *"Share PDF from Files app (bare-path URI), phone online → outbox enqueues + syncs; file content matches via SHA256 round-trip on the server."*

## §E. Pattern across v1 → v2 → (v3 needed)

The v2 rewrite addressed every v1 critique item but introduced new gaps in three categories:

| Category | v1 gaps | v2 gaps | Pattern |
|---|---|---|---|
| **Cross-references** | none flagged | A-1 (broken §4.10), A-3 (tag contradiction) | adding new sections without verifying cross-refs |
| **Schema rigor** | none flagged | A-2 (PDF doesn't fit), B-9 (JSON-safe vs structured-clone) | "let me sketch the type" without checking real-world inputs |
| **Async/runtime details** | hand-waved | B-1 (sync vs async API), B-2 (UI-blocking SHA256), B-5 (race), B-8 (delivery mechanism) | architecture written in prose; the **gritty timing details** are where bugs hide |

**Systemic takeaway:** the v2 rewrite improved *strategic* correctness (right architecture, right scope) but didn't apply the same rigor to *tactical* correctness (right API, right ordering, right edge cases). Future plan revisions should include a "tactical pass" — read the plan as if executing it line-by-line and flag every place where a real implementer would say "wait, but how exactly…"
