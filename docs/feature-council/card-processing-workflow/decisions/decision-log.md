# Card Processing Workflow — decision log

**Status:** **Explored — not implemented**
**Decision authority:** Feature-council recommendation for stakeholder review; none of these records authorizes implementation.
**Evidence baseline:** repository `1cb5d36f37611e60442b4f2c4433b45455273500`, wiki `88a3520038703108a0533501c7a384c6def7b74e`, 2026-07-11 review package.

| ID | Decision | Alternatives rejected/deferred | Rationale | Validation / reopen trigger |
|---|---|---|---|---|
| CPW-001 | Recommend Direction B: dedicated **Processing** section with Inbox-first landing; Board and List are secondary views. | A board-first; C Library-integrated. | Fastest next decision, strongest mobile/accessibility fit, lowest project-management drift. | Reopen if observed users primarily need whole-workload spatial planning. |
| CPW-002 | Use **Processing** as section name and **Inbox** as its landing job. | Workflow, Queue, Backlog, Board, Tasks. | Processing describes the behavior; Inbox describes only one status. | Run five-second comprehension test against Inbox and Queue before implementation authorization. |
| CPW-003 | Desktop Processing is a Library peer. Mobile starts under More with a Library Inbox summary and “Saved to Inbox” capture feedback. | Immediately displace a primary mobile nav item; hide entry under More without summary. | Preserves current mobile IA while giving the queue two explicit discovery paths. | If ≥20% of test users fail to find Processing unaided, promote it to primary mobile navigation. |
| CPW-004 | Workflow status, User tags, AI topics, capture/enrichment quality, SRS Review, and archive are independent concepts. | Reuse tags/categories; reuse SRS `cards`. | Current `items` and taxonomy evidence prove different lifecycles. | No reopen without repository evidence that replaces these concepts. |
| CPW-005 | Every genuinely new item is enrolled in Inbox in the same transaction as capture. Duplicate/repair/re-enrichment paths preserve existing workflow state. | Client-only default; later asynchronous assignment; resetting duplicates. | Prevents invisible gaps and lifecycle loss across ingestion channels. | Must pass every-ingestion-path and raw-insert integrity matrix before rollout. |
| CPW-006 | Existing items receive a valid dormant baseline but are excluded from Processing and metrics until explicit enrollment. Choices: selected items, recent 30 days capped at 25 after count preview, or all. | Silent all-history backfill; permanently unclassified null status; uncapped “recent.” | Avoids surprise debt while preserving schema validity and explicit consent. | Reopen cap after a usability test with real owner backlog sizes. |
| CPW-007 | Statuses are Inbox, To Do, In Progress, Done. Any active status may move to any other; same-state action is a no-op. | Strict forward-only flow; terminal Done. | Lightweight personal processing requires correction and re-entry without PM ceremony. | Retain unless dogfood shows harmful churn. |
| CPW-008 | No manual rank in first release. Inbox orders by current Inbox-entry time; other active statuses by latest status change; Done by latest completion; Archive by archive time. | Persisted drag rank; priority field. | Deterministic order is cheaper, safer under pagination/concurrency, and sufficient for discovery. | Reopen only with repeated evidence that users cannot process without within-column prioritization. |
| CPW-009 | Pointer drag is desktop enhancement only and changes status, not rank. Native source-specific Move controls are the primary universal path. | Drag-only; ARIA grid keyboard model; mobile drag. | Preserves accessibility and mobile parity. | Production drag remains off until cancellation, AT, reduced-motion, and virtualization focus gates pass. |
| CPW-010 | Archive is a Done-only `workflow_archived_at` lifecycle attribute, never automatic. Restore returns to Done; Reprocess is an explicit atomic compound command that unarchives and enters Inbox. | Terminal Archive status; deletion; auto-archive on Done. | The workflow prefix prevents confusion with global content archive; restoration stays unambiguous. | Archive downstream matrix requires stakeholder approval. |
| CPW-011 | Workflow archive hides only from active Processing. Library, search, Ask, Related, detail, quality/Review eligibility, duplicate detection, export, and independent workers continue with an Archived badge/metadata where surfaced. | Global content hiding; “where practical” ambiguity. | “Archive from Processing” must not resemble deletion or change knowledge availability. | Any downstream exception is a no-go until explicitly approved and documented. |
| CPW-012 | Canonical item detail remains the existing full route with Processing return context. Quick preview is read-only and removable if it does not improve time/confidence. Notes edit only on canonical detail. | Editable modal/drawer as canonical detail; notes in quick preview. | Avoids duplicate detail/note contracts and allows real history/return/draft testing. | Remove preview if moderated comparison shows no measurable value. |
| CPW-013 | First-release mutations are single-item only. Batch move/archive is deferred. | Partial per-item batches; atomic whole-batch transaction. | V1 reviews found batch safety untestable and unnecessary to validate the core single-item trust model. | Reopen after single-item dogfood plus a separately prototyped partial-outcome/receipt contract. |
| CPW-014 | Saved data can be read and filtered offline; workflow mutations require a connection and are not queued. | Offline optimistic queue; last-write-wins. | Honest recovery is safer than an unproven mutation journal. | Reopen only with a separately designed durable sync/outbox architecture. |
| CPW-015 | Mutations use expected version + mutation ID; 409 returns the current snapshot. Optimistic UI remains pending until confirmed, rolls back source-locally on failure, and reconciles unknown outcomes. | Last-write-wins; page-wide rollback. | Protects multi-tab/device trust in SQLite single-owner deployment. | Must pass lost-response and concurrent-tab E2E before rollout. |
| CPW-016 | Undo is one level, visible and server-eligible for 10 seconds after confirmation, linked to the original event, and Undo-origin events cannot themselves be undone. Success/replay returns `undoEligibleUntil`; an expired request returns a bounded `undo_expired` outcome plus current truth. Redo is an ordinary new move. | Arbitrary Undo chains; state-only client reversal. | Keeps the client, server, accessibility announcement, and metric rules deterministic. | Reopen only with a proven parity/graph-reduction model or an accessibility finding that requires a longer window. |
| CPW-017 | Authoritative current Inbox age is `workflow_inbox_entered_at`, updated transactionally on capture, enrollment, move/reprocess to Inbox, and Undo restoration; ignored outside active Inbox. | `captured_at`; expensive event-only derivation. | Current Inbox dwell is not capture age, especially for legacy enrollment and reprocessing. | Query plan/p95 gate at 10k/50k is mandatory. |
| CPW-018 | Persistent headline metrics are Inbox now + oldest current entry, Processed this week versus Added this week, and Completed this week. “Added” counts only genuinely new initialized captures; legacy enrollment, duplicate/repair, reprocess/return, and ordinary move-to-Inbox are excluded. “Processed” counts one effective exit per Inbox-entry episode; first-ever Triaged remains diagnostic. | Processed today as persistent headline; calling all Inbox entries “Added”; any transition; first-ever only. | Separates capture pressure from recurring work without counting churn outside Inbox. | Today may appear only as short-lived action feedback; validate pressure/debt perception. |
| CPW-019 | Metrics describe currently retained items; hard delete removes their event history and may recompute past totals downward. | Anonymous immutable rollups. | Aligns privacy-first deletion with honest data semantics. | UI/help copy and cache invalidation tests required before implementation. |
| CPW-020 | Schema addition and guard installation run in the normal boot migration; legacy baseline/enrollment-event population is a separate resumable, idempotent, observable job. | One synchronous event-per-item boot transaction. | Limits startup/WAL/disk risk and supports interruption/resume. | Measured backup/free-space/wall-time rehearsal is a rollout no-go gate. |
| CPW-021 | Application capture transaction supplies valid workflow fields/events; DB trigger is a guard for raw/future inserts. If guard integrity is uncertain, Processing fails closed and a repair queue/readiness check blocks enablement—captures never silently become dormant. | Trigger-only initialization; disabling trigger while UI remains active. | Eliminates silent Inbox-default violations during degraded rollback. | Startup integrity query must remain zero for new invalid rows. |
| CPW-022 | Filter algebra is OR within User tags, OR within AI topics, AND across facets. View-scoped matching counts and unfiltered Inbox total are labeled separately. | One blended tag facet; ambiguous all-scope “matching.” | Matches current taxonomy model and fixes the v1 trust defect. | Multi-value UI remains implementation-stage validation; prototype proves only one value per facet plus cross-facet AND. |
| CPW-023 | Stakeholder approved Direction B for continued design exploration. Board and List share one compact **Group & sort** control with comprehensive grouping and sorting choices. Non-status grouping changes layout only; Move remains the only status-changing control. | Separate Board/List organization models; using drag as sorting; letting grouped columns imply status mutations. | One predictable organization model is easier to learn and preserves the workflow/status contract. | Reopen option breadth or defaults after task-based usability testing with realistic libraries. |
| CPW-024 | Direction B is intentionally designed and reviewable in both Light and Dark appearance modes, with the current selection persisted in the prototype URL. | Dark-only exploration; automatic inversion without component-level review. | Both appearances are part of the requested design surface and need equivalent hierarchy, contrast, and responsive behavior. | Production implementation still requires design-token mapping, preference persistence, forced-colors, and manual contrast/reflow validation. |
| CPW-025 | The Board/List organization trigger and popover use the product's compact utility-control density rather than primary-task sizing: 36px desktop trigger, 322px popover, 50px rows, 12–13px content type, and small-radius treatment; mobile preserves a 44px trigger target. | Oversized 42px trigger, 390px popover, 68px rows, 16px labels, 24px icons, and 18px radius. | Stakeholder feedback showed that the larger treatment broke alignment with adjacent filters and the existing compact menu pattern. | Reopen only if measured usability testing shows the compact values or row targets are hard to use. |

## Explicit future opportunities

- Batch operations with durable per-item receipts and honest partial outcomes.
- Manual priority/rank after deterministic-order dogfood.
- Quick-preview removal or expansion based on measured value.
- Primary mobile navigation promotion if the More + Library-summary model fails discovery.
- Offline mutation journal only as a separate sync project.

## No authorization

This ledger records an explored recommendation. No production application, schema, migration, API, feature flag, rollout, or merge is authorized.
