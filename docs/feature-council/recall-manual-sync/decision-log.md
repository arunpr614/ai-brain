# Recall manual sync decision log

Status: Active implementation decision record
Baseline: `origin/main` at `1cb5d36` on 2026-07-11
Authority: supplied PRD, supplied rendered designs, current code and design system, current wiki, then older records

| ID | Decision | Rationale and consequence | Status |
| --- | --- | --- | --- |
| RMS-D001 | “Last successful sync” means the server completion instant of the most recent fully validated guarded apply, automatic or manual. A no-new-items apply qualifies; dry runs, blocked runs, failures, and partial failures do not. | The checkpoint is a coverage boundary and a terminal `done` row precedes final wrapper validation. A new post-validator success marker is required. | Approved |
| RMS-D002 | Store canonical instants as epoch milliseconds/UTC and display every absolute Recall time in `Asia/Kolkata` with a visible `IST` suffix. | The supplied PRD explicitly overrides the goal’s default device-local principle. Today/Yesterday are computed on the IST calendar. | Approved |
| RMS-D003 | Insert Recall immediately after AI services and before conditional My notes and Data & Privacy. | This honors the explicit placement without removing or redesigning the newer My notes section. | Approved |
| RMS-D004 | The browser creates only a durable intent record. A trusted worker executes the complete existing wrapper: dry run, validation, backup, proof-backed apply, final validation, shared checkpoint, and core lock. | Running Recall from the route or reimplementing the importer would violate the safety and privilege boundary. | Approved |
| RMS-D005 | Use SQLite for a single-active manual request with a bounded idempotency key, immediate enqueue/claim transactions, a five-minute terminal cooldown, and 30-minute unclaimed expiry. | SQLite is already authoritative and sufficient for the single-owner deployment; no new queue dependency is justified. | Approved |
| RMS-D006 | New and deduplicated active manual requests both return `202` with the same safe schema and an explicit `deduplicated` boolean. | This preserves the supplied accepted-request contract and avoids an unnecessary client branch. | Revised after adversarial review |
| RMS-D007 | Authentication precedes feature evaluation. Unauthenticated GET/POST returns `401`; authenticated flag-off GET returns a safe disabled status and flag-off POST returns a safe unavailable response. | This matches proxy and route defense-in-depth behavior without leaking feature state. | Approved |
| RMS-D008 | Do not coalesce a manual request into an automatic result in v1. If the full-wrapper lock is occupied, leave the manual request queued and retry delivery of that same request. | Proving window coverage and linkage is more complex than queue-after and a false coalesced success would breach trust. | Approved |
| RMS-D009 | Add a trusted-runtime `flock` shared by daily and manual Recall units while retaining `tryAcquireRecallSyncLock`. Manual attempts wait briefly then requeue; automatic attempts wait for the tested maximum or use an explicit retry so a manual overlap cannot skip the daily run. | The current core lock is released between stages; adversarial review rejected unconditional non-blocking behavior because it could regress daily sync. | Revised after adversarial review |
| RMS-D010 | Persist a running execution record and aggregate progress. Convert a failed apply to product `partial_failure` only when persisted imported/upgraded counts are non-zero. | The current per-card apply can write before a later throw while the outer catch reports zero. Zero-write reassurance requires proof. | Approved |
| RMS-D011 | Show a trusted persisted systemd next-elapse snapshot when available; otherwise show “Schedule unavailable.” Never fabricate an exact time from the nominal timer because it has randomized delay. | A manual sync must not change the timer, and the web process must not gain service-control privileges. | Approved |
| RMS-D012 | Use path activation for fast pickup and a 60-second trusted safety timer for lost-wake recovery. The marker is empty/non-secret; SQLite is authority. | This meets fast-start and durability goals without granting the web process `systemctl` or command execution. | Approved |
| RMS-D013 | Use Radix Dialog for modal focus management, a dedicated transition-only polite live region, at least 44px mobile controls, current semantic tokens, and explicit reduced-motion behavior. | Rendered prototype inspection found escaping focus, missing restoration, undersized targets, weak muted contrast, and a parallel palette. | Approved |
| RMS-D014 | Add loading, never-synced, automatic-running, long-running, offline/status-unknown, session-expired, rate-limited, request-expired, and unavailable variants to the v2 UX. | These are required by the goal/PRD but missing from the supplied eight-state board. | Approved |
| RMS-D015 | Retain previous last-success and next-schedule metadata in queued, running, blocked, failed, and partial states. | Transient state must not erase the user’s durable freshness context. | Approved |
| RMS-D016 | No third-party analytics and no cancellation event in v1. Use request rows and source-safe structured lifecycle logs only. | Cancel intentionally creates no request, and the app has no product analytics subsystem. | Approved |
| RMS-D017 | The manual UI flag defaults off. This task may deliver code, units, docs, wiki, and a PR, but may not enable, merge, or deploy the feature. | Production enablement requires separate authorization and host credential-boundary verification. | Approved |
| RMS-D018 | Introduce a distinct `brain-recall` service identity and shared restricted data group. Deliver a root-owned Recall credential only to Recall units; the web identity must be unable to read the credential or open the wrapper lock. | Checked-in units share one user/environment. Adversarial review rejected leaving identity separation optional; systemd credentials may harden the split but do not replace it. | Revised after adversarial review; host proof gate |
| RMS-D019 | Create a focused revised HTML state prototype rather than redesigning Settings. | Material state/accessibility gaps require clarification, but the supplied card composition remains the visual baseline. | Approved |
| RMS-D020 | `recall_sync_executions` is the whole-wrapper authority. Runner owns run rows/progress; lifecycle completion/failure atomically updates execution, linked manual request, and last-success. Worker may claim, spawn, requeue, and reconcile but may not invent a terminal outcome. | Prevents split-brain terminal state after crash/final validation. | Approved after adversarial review |
| RMS-D021 | A PID-bound trusted heartbeat updates the active execution every 15 seconds; stale recovery requires at least three missed beats plus lock/process/run evidence. Exact service timeout is derived from tested wrapper bounds, not guessed. | Stage-only updates cannot distinguish a healthy long run from a crash. | Approved after adversarial review |
| RMS-D022 | Confirmation does not dismiss on overlay click. Add a local **Requesting sync…** state, qualify offline data as last known with a last-checked time, compose cooldown into the terminal result, reduce gutters at 320px, and never label a past schedule as “Next.” | Resolves v1 UX race, stale-data, retry, dismissal, and reflow findings. | Approved after adversarial review |
| RMS-D023 | Review-ready completion requires local/static/fixture proof with units and feature disabled. Production enablement separately requires host identity/credential/permission/unit proof and explicit authorization. | Prevents an out-of-scope production action from being smuggled into PR completion or the security gate from being weakened. | Approved after adversarial review |
| RMS-D024 | Only a newly created lifecycle occurrence may enter core work. Existing running or terminal occurrences return without re-entry; stale reconciliation requires a stale heartbeat, a free held outer lock, and linked-run evidence. | Stable occurrence keys alone do not prevent crash/restart duplication. | Approved after QA/adversarial remediation |
| RMS-D025 | Exact authenticated request-ID and idempotency-key acknowledgements are independent of the reduced latest activity and expose only bounded safe fields. | A hidden/offline tab must resolve accepted request A even after a newer request B supersedes the main status. | Approved after QA/adversarial remediation |
| RMS-D026 | Client absence resolution is server-provided and derived from the shared SQLite busy timeout/retry constants plus margin; the current bound is 20 seconds. | Prevents the browser from releasing ambiguity before a maximally contended enqueue can finish. | Approved after targeted residual closure |
| RMS-D027 | Deployment preparation holds one private remote lock continuously across all application and Recall runtime switching, permissions, migrations, units, tmpfiles, and daemon reload, and verifies daily timer state before release. | A preflight-only lock probe has a time-of-check/time-of-use race with the enabled daily timer. | Approved after QA/adversarial remediation |
| RMS-D028 | Local review readiness may use built worker/lifecycle, real `flock`, multi-process SQLite, crash injection, and fake-systemd path/fallback fixtures. It must label actual Linux identity/credential/unit/timer proof as a separate authorized enablement gate. | Provides meaningful process evidence without mutating production or overstating host proof. | Approved after final review |

## Final delivery boundary

All implementation numeric bounds and lifecycle contracts are derived or locked by tests. The remaining work is documentation/Wiki publication and review-ready PR delivery. Merge, deployment, feature enablement, production timer mutation, credential access, and real Recall data access remain outside authorization.
