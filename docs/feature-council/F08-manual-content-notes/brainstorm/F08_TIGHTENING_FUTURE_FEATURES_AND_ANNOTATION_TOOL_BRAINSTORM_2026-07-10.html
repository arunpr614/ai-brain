Detailed brainstorm · 2026-07-10

From a note editor to a personal thinking layer

Opportunities to tighten the production manual-notes experience, extend AI Brain with explainable knowledge workflows, and evolve the annotation tool into verified product intent.

Baseline: production-verified F08 manual notes Audience: Product, design, engineering, QA Status: Brainstorm, not implementation commitment

Executive summary

The hard foundations are already present: canonical Markdown, autosave and recovery, conflicts, versions, exact search, opt-in semantic indexing, provider-specific consent, source-aware citations, Related influence, and verified cleanup. The next release should focus on trust, explainability, and stronger connection to the act of reading.

Core product thesis The note should become the place where evidence, interpretation, questions, decisions, and future review meet—without blurring user writing, source material, and AI output.
  1. Tighten trust. Clarify local safety, server save, AI permission, and semantic-index state.
  2. Connect notes to reading. Add source-linked highlights and annotations.
  3. Make retrieval controllable. Add My notes only, Original only, and mixed Ask policies.
  4. Build explicit connections. Add wikilinks, backlinks, and typed relationships.
  5. Compound into learning. Add review, synthesis, and explainable graph workflows.

Working principles

Portable by default

Canonical Markdown remains the source of truth. Complexity must not compromise export, recovery, or long-term ownership.

Authorship stays visible

User writing, quotations, AI digests, and AI suggestions remain labeled and separately controllable.

Privacy is a state

The UI continuously communicates whether content is local, saved, eligible, indexed, or being removed.

Edges explain themselves

Every connection must expose the passages, explicit link, or semantic reason that created it.

Recovery is reversible

Restore, merge, AI acceptance, and bulk actions should create new state without silently destroying old state.

Mobile is first-class

Optimize mobile for quick capture, highlighting, review, and follow-up—not merely a compressed desktop editor.

1. Tighten the current implementation

Prioritize confidence and comprehension before expanding the editor into a larger workspace.

Unified save, sync, permission, and index status

LayerUseful statesUser meaning
Local draftDrafting, recovered, waitingThe browser has the newest safe copy.
Canonical saveSaving, saved, failed, conflictThe server’s canonical Markdown state.
AI eligibilityPrivate, permission needed, includedWhether remote or semantic use is permitted.
Semantic indexWaiting, indexing, indexed, removing, failedWhether Ask and Related can use this generation.

Keep the default presentation compact. Put epoch, generation, provider, and recovery detail in an optional status popover.

High-value polish

Version history as a working tool

  • Inline or side-by-side Markdown diff.
  • Preview without restoring.
  • Copy a paragraph from an older version.
  • Compare any two versions.
  • Restore as a new reversible version.
  • User-authored checkpoint labels.
Editor polish

Better Markdown, not a block editor

  • Slash commands and discoverable shortcuts.
  • Find/replace and heading outline.
  • Tasks, tables, quotes, links, and code blocks.
  • Smart lists and normalized paste.
  • Focus mode and word count.
  • Sticky mobile toolbar above the keyboard.

Search and Ask explainability

Search

  • Original, AI digest, and My notes filters.
  • My notes only preset.
  • Exact phrase versus related-concept reasons.
  • Deep link to the matching heading or paragraph.
  • Show when exact search is current but indexing is pending.

Ask

  • All, Original only, My notes only, and mixed policies.
  • Prefer My notes without hiding original evidence.
  • Heading-aware citation labels.
  • Retrieval summary by source kind.
  • Rerun with a different source policy.

Privacy and consent

  • Show provider, effective model, destination class, and purpose together.
  • Place permission disclosure beside the per-note AI switch.
  • Show “Removing from AI…” until purge converges.
  • Add a content-free privacy dashboard for counts, jobs, purges, and approvals.
  • Add a guarded “Revoke and remove all manual-note AI data” operation.
  • Investigate encrypted note-only export and later encryption at rest.

Operational confidence and metrics

Add chaos coverage for disconnects, IndexedDB failure, multi-tab delete/restore races, large-note mobile performance, provider-model consent changes, and full backup restoration. Useful private metrics include recovery usage, conflict resolution, search-to-note opens, note-context Ask usage, opt-in/revocation, and version restores. Raw note text and queries should never become general analytics.

2. Additional features

Build features that deepen the read–think–connect–review loop.

Best next major feature: source-linked highlights and annotations Select text in an article, transcript, PDF, or captured body and attach a comment, question, takeaway, counterargument, evidence label, task, or card link. Preserve a stable source locator, quote context, source version, user commentary, and an orphaned state.
Connections

Wikilinks and backlinks

Use [[ autocomplete to create explicit links. Support typed relationships such as supports, contradicts, expands, and example of.

Connections

Explainable graph

Expose why every edge exists and let users inspect passages, approve suggestions, rename relationships, or remove edges.

Synthesis

Cross-card workspace

Compare selected cards, find agreement and contradiction, build an outline, and create a cited synthesis card.

Learning

Review engine

Resurface notes, unresolved questions, checklists, flashcards, and changes in the user’s thinking.

Structure

Note templates

Insert editable Markdown for research, meetings, decisions, retrospectives, learning summaries, and claim/evidence analysis.

Action

Task aggregation

Collect open Markdown tasks with stable links back to their source lines while preserving clean export.

AI assistance

Proposals, not rewrites

Suggest structure, questions, claims, action items, or clarity edits with Accept, Reject, and Insert controls.

Capture

Voice and append

Dictate notes, append selected browser text, share into an existing card, or OCR a screenshot.

Portability

Markdown ecosystem

Support Markdown-folder import/export, stable URLs, and optional Obsidian-style wikilinks and frontmatter.

Smart collections powered by notes

  • Notes with open questions or incomplete tasks.
  • Recently changed thinking.
  • AI-enabled versus private-only notes.
  • Notes mentioning a project, person, or concept.
  • Cards with manual notes but no recent review.

Every smart collection should expose an understandable, editable rule.

3. Annotation tool updates

Evolve the visual annotation tool from a pin on a screenshot into a semantically anchored, context-rich, verifiable change request.

Foundation

Semantic anchors

  • DOM or accessibility node.
  • Component identity when discoverable.
  • Selected text range or field/label pair.
  • Route, viewport, theme, and state.
  • Uncertain remapping when a target disappears.
Context

Automatic evidence capture

  • Screenshot, crop, element text, role, and name.
  • Computed styles and scroll position.
  • Open dialog, menu, loading, or validation state.
  • Preceding interaction steps.
  • Candidate component and source file.
Intent

Type and scope

  • Polish, copy, behavior, accessibility, responsive, data, question, feature.
  • This element versus every component instance.
  • Current breakpoint versus all supported widths.
  • Current state versus every equivalent state.
Definition of done

Acceptance criteria

  • Exact copy or alignment.
  • Breakpoint and non-regression boundaries.
  • Focus behavior and accessibility.
  • Timing or state-transition expectations.
  • Asset and crop preservation.

Preview and verification workflow

  1. User anchors an annotation and specifies intent.
  2. Agent summarizes the interpretation and affected scope.
  3. Tool identifies likely components and files.
  4. User reviews a proposed visual or behavioral preview.
  5. Agent applies the accepted change.
  6. Tool captures a same-state before/after comparison.
  7. Acceptance checks run; the annotation becomes Verified or Reopened.

Durable work-item lifecycle

Support Open, Clarification needed, Ready, In progress, Applied, Verified, Rejected, and Reopened. Add comment threads, batching, duplicate/conflict detection, commit and PR linkage, original/final screenshots, and export to GitHub or Jira.

Interaction, responsive, and time-based feedback

  • Annotate hover, focus, drag, dialogs, menus, and loading transitions.
  • Capture keyboard navigation and mobile software-keyboard behavior.
  • Attach feedback to a short recording and timestamp.
  • Preserve the action sequence needed to reproduce the state.
  • Preview the same semantic target across representative viewports.
Long-term differentiator: annotation-to-test generation Responsive feedback can become a viewport test; accessibility feedback can become a focus or role assertion; behavior feedback can become an interaction test; and visual feedback can become a stable before/after checkpoint.

Privacy and sharing

  • Detect and redact secrets, private names, account data, and sensitive content.
  • Let users crop or mask evidence before sharing.
  • Distinguish private annotations from team-visible annotations.
  • Show what screenshot, DOM text, and metadata will be sent to an agent.
  • Support a local-only mode for sensitive applications.

5. Suggested experiments

Trust-bar comprehension

Test whether users can identify if text is safe locally, saved, eligible for AI, indexed, or failed. Target 90% correct interpretation without help.

Source annotation flow

Test highlight → annotate → view in My notes → revisit source across article, transcript, and PDF. Target creation in under 15 seconds.

Ask source policy

Test whether users can predict which evidence is used by All, My notes only, Original only, and Prefer My notes modes.

Annotation acceptance criteria

Compare clarification turns, rework, completion time, and cross-breakpoint regressions with and without structured criteria.

6. Ideas to avoid or defer

Complexity that does not yet earn its cost
  • A full Notion-style block editor.
  • Multiple independent notes per card before the single-note model is exhausted.
  • Automatic AI rewriting of canonical notes.
  • A graph-first redesign without edge explanations.
  • Automatic remote processing of every note.
  • Real-time collaboration before the personal workflow is excellent.
  • Pixel-only annotation anchors.
  • Broad annotation-driven changes without preview.

7. Decision prompts

  1. Is the next release about trust and polish, reading-linked annotation, or knowledge connections?
  2. Are source annotations structured records, canonical Markdown, or a hybrid reference model?
  3. Should My notes only Ask be global, item-level, or both?
  4. Which explicit relationship types must ship before a graph?
  5. Are note tasks personal reminders or integrations with another task system?
  6. What minimum context must an annotation capture before an agent can act without clarification?
  7. Which acceptance checks can be generated reliably today?
  8. Which low-risk annotations may apply directly, and which always require preview?
Read → highlight → interpret → connect → ask → revisit → refine
Observe → anchor → specify intent → preview → apply → verify → preserve as a test

These loops reinforce the same product value: turning informal human intent into durable, trustworthy, and explainable knowledge.
