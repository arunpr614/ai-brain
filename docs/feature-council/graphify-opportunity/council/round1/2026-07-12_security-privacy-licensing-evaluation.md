## 2. Perspective-specific principles and conflicts

I apply the following principles:

1. **Owner boundary before feature richness.** A private single-owner product boundary is not evidence of future tenant or role isolation. Concentrating relationships increases the impact of the documented weak browser-unlock boundary.
2. **Consent follows every derivative.** Note-derived influence must disappear immediately from UI, caches, indexes, snapshots, explanations, and measurements when consent or eligibility changes.
3. **Associations are sensitive data.** A path, shared theme, inferred affinity, label count, title cluster, or provenance trail can reveal private facts even when raw content is absent.
4. **Minimize egress and telemetry.** No new provider call is justified by these candidates without an explicit purpose, consent, data-minimization rule, and failure behavior. Diagnostics and product measurement must exclude titles, URLs, bodies, note text, paths, and sensitive labels.
5. **Owner records remain authoritative.** Derived state must be versioned, stale-aware, fully rebuildable, and deletable without preserving graph-only facts.
6. **Exact dependency review, not license adjacency.** Native/no-new-dependency designs have the best current posture. Any selected package requires an exact lock, hashes, SBOM, license obligations, reachability review, blocking vulnerability scan, maintenance owner, and exit plan.
7. **Rollback includes cleanup.** Hiding a route is insufficient if snapshots, queues, telemetry, cached relationships, or provider artifacts remain.

The main conflict in this perspective is that the safest option can defer possible user benefit. I therefore do not treat incomplete evidence alone as proof that no problem exists; B-00 must have a real timebox and exit rule.

## 6. Preferred option

**Preferred: B-00 Bounded Discovery / Defer.** It is the only option that can reduce the dominant uncertainty without increasing production exposure. Its safeguards—consented or fictional/redacted inputs, outcome-class rather than content recording, no production instrumentation, a four-week maximum, and a required exit decision—fit least privilege and data minimization. This preference is not a claim that users have no problem; it is a decision to avoid concentrating private relationships or creating mutation/derivative state before value is demonstrated.

The B-00 protocol should explicitly name the research-record owner, access list, retention/deletion date, publication review, consent withdrawal path, and prohibition on screenshots/raw library exports. If those are not approved, even B-00 should remain paused.

## 7. Strongest minority / contrary case

The strongest contrary case is **a narrowly constrained C-01 minimal-cue test**: source-kind and current/stale state are adjacent to existing Related/citation behavior, may need no new dependency, provider call, persistent relationship, or route, and can be removed behind a feature flag. Compared with C-02 it avoids source mutations; compared with C-03 it avoids concentrated paths and whole-library projection.

That case still does not justify advancement now. The packet does not show that users misinterpret current results, that the cues materially beat opening sources/citations, that every displayed origin is faithful, or that note-derived influence disappears across all surfaces after consent withdrawal. A full explanation panel is especially unsafe without actual contribution reconstruction.

## 8. Major assumptions and risks

1. **Authentication concentration risk:** the browser PIN boundary has no discovered throttle, lockout, or durable attempt record. A relationship summary can expose more sensitive information per successful unlock than current item-at-a-time surfaces.
2. **Owner/tenant ambiguity:** current single-owner behavior and shared bearer patterns do not prove future tenant/role isolation. Every candidate route/action must fail closed on owner mismatch.
3. **Consent and deletion lag:** manual-note `purged` event assertion is missing, non-note semantic producers are absent, event rows cascade with the item, and no consumer/durable deletion tombstone exists. Derived state can outlive eligibility unless an independent complete watermark/reconciliation design is proven.
4. **Sensitive associations:** labels, counts, shared themes, pair explanations, paths, hubs, and no-path behavior may reveal or imply people, health, finances, work projects, or affinities even without content text.
5. **False authority:** similarity, model-generated topics, and path structure are not facts or causes. Post-hoc explanations and undirected/hub shortcuts can mislead.
6. **Telemetry leakage:** outcome metrics can accidentally capture titles, URLs, note text, labels, relationship paths, free text, or high-cardinality identifiers. Event schemas must be allowlisted and tested.
7. **Provider egress:** no candidate should add a model-written explanation or ambient provider selection. If any provider call is later proposed, exact fields, purpose, retention, provider, region, consent, and failure path must be approved.
8. **Dependency/license drift:** Graphify core is MIT at the pinned commit, but its default runtime is broad, `[all]` reaches an AGPL-3.0-only optional Pascal grammar, some optional paths have other obligations, and upstream security scans are non-blocking. MIT compatibility does not make direct integration acceptable.
9. **Rollback incompleteness:** hiding UI does not delete snapshots, queues, caches, telemetry, provider artifacts, or versioned relationships. Exit must enumerate and remove every derivative while preserving owner data.
10. **Research opportunity cost:** B-00 may delay a useful improvement or produce too few natural episodes. One timebox and a forced exit decision prevent indefinite deferral.

These risks align with the frozen risk register's open sensitive-memory, inferred-relationship, dependency, auth, viewer, and artifact-path concerns ([risk register](../../RISK_REGISTER.md)).
