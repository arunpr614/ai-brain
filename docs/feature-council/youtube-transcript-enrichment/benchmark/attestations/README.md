# A1 evidence-bound attestations

These records document the coordinator's **provisional private-research clearance** for the isolated benchmark described by protocol v2. They are technical evidence classifications, not legal opinions, owner consents, production approvals, or permission to publish full transcripts. `review_required: true` is invariant: production use, broader retention/publication, or third-party transfer still requires an authorized stakeholder/legal-policy determination. It does not block the narrower local, no-provider, source-preservation run explicitly authorized by this research goal. Source-page and third-party caveats remain controlling. Full text stays in the private workspace, model upload and embedding are prohibited, publication is limited to hashes/derived metrics, and the records expire on 2026-10-14.

Every file uses [attestation schema 1.2](../A1_ATTESTATION.schema.json). The publication-safe records are [YT-01](YT-01.json), [YT-02](YT-02.json), [YT-03](YT-03.json), [YT-04](YT-04.json), [YT-05](YT-05.json), [YT-06](YT-06.json), [YT-07](YT-07.json), [YT-08](YT-08.json), and [YT-09](YT-09.json). The tested class is narrower than a general creator upload: it covers an official source-published sidecar locally associated with the source page's exact YouTube link. Its `input_contract` binds the file format, language, duration, cue count, last-cue coverage, explicit completeness state, and expected supported-class outcome so run-time flags cannot silently redefine the sealed cell. `complete` reflects a cited source assertion; `partial` preserves a known coverage gap; `unknown` is allowed only for a predeclared rejection and cannot enter an eligible A1 run. A future owner/user-upload contract would additionally need principal/authority, grant scope, acceptance, revocation, and provider-transfer fields and is not validated here.

The `youtube_side_verified: false` field is intentional: official source-row association is sufficient for this source-supplied A1 preservation study, but no YouTube caption track, API response, signed-in browser state, or media hash was used. It is not evidence that these sidecars are YouTube caption tracks or byte-equivalent to a YouTube-hosted artifact.
