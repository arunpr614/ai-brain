Weak or metadata-only captures retain provenance and appear in Needs Upgrade or attention Review. Per-item repair can refresh recoverable content; YouTube repair accepts user-provided VTT/SRT/TXT/Markdown under explicit policy and records transcript source/segments before resetting derived state.

The official-caption adapter needs an authorized path and is shown as not wired. Owned-media validation/adapters exist, but the route deliberately returns `503 provider_disabled`. Neither is an active product feature.

Current primitives do not prove the complete FCP-001 single Repair Center or full parity across web, Android, extension, Telegram, and Recall.

Primary files: quality/result/artifact modules, `src/app/needs-upgrade/`, `src/app/review/`, `src/app/items/[id]/repair/`, transcript APIs/workers/repositories and tests.

## User problem, entry points, and journey

The owner needs to know when “saved” is not “usable,” understand why, and safely improve a weak source without creating a duplicate or losing provenance. Entry points are capture result links, `/needs-upgrade`, `/review`, item trust/quality panels and `/items/[id]/repair`. Journey: inspect reason/evidence → choose supported repair → validate policy/input → replace or reprocess atomically → observe enrichment/index state.

## State and failure matrix

| State | Behavior |
|---|---|
| Empty queue | Explains that no current source needs attention |
| Loading/claiming | Server page load or durable transcript job claim without presenting success early |
| Repair success | Existing item identity retained; provenance/version changes; derived state requeued |
| Retryable failure | Job records attempt/error/cooldown and remains recoverable |
| Manual required | Explicit unsupported/gated provider or user transcript action |
| Inactive | Official-caption and owned-media options remain labeled unavailable; owned-media route returns 503 |

## Architecture, data, APIs, AI, and security

Quality/result policy in `src/lib/capture/` feeds attention rules in `src/lib/review/attention.ts` and repair policy in `src/lib/repair/item-repair.ts`. Transcript routes call parsers/policy and write `transcript_jobs`, attempts, policy decisions, sources and segments before resetting enrichment/index state. The transcript worker applies cooldown/retry/manual transitions. User-provided text is not generated by AI; later enrichment/embedding follows normal provider rules. Rights/retention assertions and authenticated same-item mutation are required.

API entrypoints include capture transcript, item repair/enrich actions and the inactive owned-media transcript route. Related current features are Capture, Review/Needs Upgrade, Enrichment and Search/Ask; related ideas are the planned unified Repair Center and Reading Studio.
